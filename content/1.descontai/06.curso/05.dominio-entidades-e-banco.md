Toda API precisa **guardar dados em algum lugar**. Os cupons, os usuários, as empresas — nada
disso pode viver só na memória, senão sumiria quando a aplicação reiniciasse. Esse "lugar" é o
**banco de dados**. Nesta aula você vai entender o que é um banco relacional, como uma classe
Java vira uma tabela (JPA/ORM) e como o Spring te dá comandos de banco quase de graça (Repository).

## O que é um banco de dados relacional

O Descontai usa o **PostgreSQL**, um banco de dados **relacional**. A ideia central é simples:
os dados ficam organizados em **tabelas**, e uma tabela é basicamente uma **planilha do Excel**.

- Cada **tabela** guarda um tipo de coisa (uma tabela de usuários, uma de empresas, uma de cidades).
- Cada **linha** (ou *registro*) é um item: um usuário específico, uma empresa específica.
- Cada **coluna** é um campo daquele item: o nome, o email, a data de criação.

Olha como seria a tabela `cidades` do Descontai:

| id | nome | created_at |
|----|------|------------|
| 1 | Recife | 2026-01-10 08:00 |
| 2 | Olinda | 2026-01-10 08:01 |

::callout{type="info" title="Chave primária: o \"RG\" da linha"}
Repare na coluna **`id`**. Ela é a **chave primária** (*primary key*): um número **único** que
identifica cada linha sem ambiguidade. Nunca existem duas cidades com `id = 1`. É como o RG de
uma pessoa — pode existir mais de uma "Maria Silva", mas o RG é único. Sempre que você quiser
apontar para uma linha específica, você usa a chave primária.
::

### Chave estrangeira: a "cola" entre tabelas

E se um usuário mora em uma cidade? A gente **não** repete o nome da cidade dentro da tabela de
usuários (isso duplicaria dados e daria divergência). Em vez disso, a tabela `usuario` guarda só
o **id** da cidade, numa coluna chamada `cidade_id`. Isso é uma **chave estrangeira**
(*foreign key*): uma coluna que aponta para a chave primária de **outra** tabela.

| id | nome | email | cidade_id |
|----|------|-------|-----------|
| 1 | João | joao@x.com | 1 |
| 2 | Ana | ana@x.com | 2 |

O `cidade_id = 1` do João é a "cola" que liga a linha dele à cidade "Recife" da tabela `cidades`.
É assim que tabelas se **relacionam**: uma aponta para a outra por id. No Descontai, um `Produto`
(cupom) aponta para uma `Empresa` (`empresa_id`) e para uma `Cidade` — porque todo cupom **pertence
a** uma empresa.

## ORM e JPA: transformando tabela em objeto Java

Aqui está o problema: no banco, os dados são **linhas e colunas**. No código Java, a gente pensa
em **objetos** (um objeto `Usuario` com `.getEmail()`). Ficar traduzindo um no outro na mão, escrevendo
SQL para cada operação, seria repetitivo e cheio de bugs.

A solução é o **ORM** (*Object-Relational Mapping*): uma ferramenta que faz essa tradução automática.
No mundo Java, o padrão de ORM se chama **JPA** (*Jakarta Persistence API*), e a implementação que o
Spring Boot usa por baixo é o **Hibernate**.

::callout{type="info" title="ORM em uma frase"}
**ORM** é um tradutor entre dois mundos: o mundo das **tabelas** (banco) e o mundo dos **objetos**
(Java). Você mexe com objetos Java normais; o ORM gera o SQL (`INSERT`, `SELECT`, `UPDATE`) por baixo
dos panos. Uma **classe** vira uma **tabela**, um **objeto** (instância) vira uma **linha**, e um
**atributo** vira uma **coluna**.
::

## As anotações que fazem a mágica

Como o JPA sabe que a classe `Cidade` corresponde à tabela `cidades`? Através de **anotações** —
aqueles `@AlgumaCoisa` em cima da classe e dos campos. Vamos ver na entidade mais simples do
Descontai:

```java
@Entity
@Table(name = "cidades")
public class Cidade extends BaseEntity {

    @Column(nullable = false, unique = true)
    private String nome;

    // getters e setters...
}
```

Cada anotação tem um papel:

::fields
::field{name="@Entity"}
Marca a classe como uma **entidade** JPA — ou seja, "esta classe vira uma tabela no banco". Sem ela,
o JPA ignora a classe.
::
::field{name="@Table(name = ...)"}
Diz **qual tabela** essa classe representa. Aqui, a classe `Cidade` mapeia a tabela `cidades`.
Se os nomes fossem iguais, essa anotação seria opcional.
::
::field{name="@Column"}
Configura uma **coluna**. `nullable = false` significa que o campo é obrigatório (não pode ser vazio
no banco); `unique = true` impede duas cidades com o mesmo nome. Sem `@Column`, o JPA ainda mapeia o
campo, só usando os padrões.
::
::

### Chave primária: @Id e @GeneratedValue

Toda entidade precisa de uma chave primária. No Descontai isso está no `BaseEntity` (já já explicamos):

```java
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
```

- **`@Id`** marca o campo que é a chave primária.
- **`@GeneratedValue(strategy = IDENTITY)`** diz "não sou eu que invento o id — deixa o **PostgreSQL**
  gerar sozinho, de forma incremental" (1, 2, 3...). Você cria um usuário sem `id`; ao salvar, o banco
  atribui o número e o JPA devolve preenchido.

### Relacionamentos: @ManyToOne e @JoinColumn

Lembra da chave estrangeira `cidade_id`? No Java, ela vira um **objeto de verdade**, não um número solto.
Olha o `Usuario`:

```java
@ManyToOne
@JoinColumn(name = "cidade_id")
private Cidade cidade;
```

- **`@ManyToOne`**: "**muitos** usuários podem apontar para **uma** cidade". (Muitos-para-um.)
- **`@JoinColumn(name = "cidade_id")`**: diz **qual coluna** do banco guarda essa chave estrangeira.

O ganho é enorme: em vez de você ter só o número `cidade_id = 1` e ter que ir buscar o nome à parte,
você escreve `usuario.getCidade().getNome()` e o JPA busca a cidade certa automaticamente. O `Produto`
usa o mesmo padrão para amarrar cada cupom à sua empresa:

```java
@ManyToOne(optional = false)
@JoinColumn(name = "empresa_id")
private Empresa empresa;
```

O `optional = false` reforça uma **regra de negócio**: não existe produto sem empresa. Todo cupom tem
dono.

::callout{type="tip" title="Por que \"ManyToOne\" e não \"OneToMany\"?"}
A pergunta é: de que lado está a chave estrangeira? A coluna `empresa_id` mora na tabela `produtos` —
então é o **Produto** que aponta para a **Empresa**. Do ponto de vista do produto, são **muitos**
produtos para **uma** empresa: `@ManyToOne`. Pense sempre "muitos DE MIM para um DELE".
::

## O BaseEntity: campos que toda tabela tem

Repare que `Cidade`, `Usuario`, `Empresa` e `Produto` todos fazem `extends BaseEntity`. Por quê?
Porque **quase toda tabela** precisa dos mesmos três campos de infraestrutura: um `id`, a data de
criação e a data da última atualização. Repetir isso em cada classe seria copiar-e-colar sem fim.

A solução é uma **superclasse mapeada**:

```java
@MappedSuperclass
public abstract class BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    @PrePersist
    public void onCreate() {
        this.createdAt = LocalDateTime.now();
    }

    @PreUpdate
    public void onUpdate() {
        this.updatedAt = LocalDateTime.now();
    }
}
```

Três coisas importantes aqui:

- **`@MappedSuperclass`**: diz ao JPA "esta classe **não é** uma tabela própria; em vez disso, **injete
  os campos dela** diretamente na tabela de cada filha". Assim, a tabela `usuario` ganha as colunas
  `id`, `created_at` e `updated_at` sem precisar de nenhum `JOIN` extra.
- **`@PrePersist`** e **`@PreUpdate`** são **callbacks do ciclo de vida**: o `onCreate()` roda
  automaticamente logo antes do primeiro `INSERT` (preenchendo `createdAt`), e o `onUpdate()` roda antes
  de cada `UPDATE` (atualizando `updatedAt`). Você nunca seta essas datas na mão — o JPA cuida.
- **`updatable = false`** no `created_at` garante que a data de criação **nunca** seja sobrescrita
  depois. É imutável por design.

::callout{type="info" title="A diferença entre @MappedSuperclass e @Entity"}
Uma classe `@Entity` **é** uma tabela. Uma classe `@MappedSuperclass` **não é** uma tabela — ela só
empresta seus campos para as filhas. Por isso `BaseEntity` é `abstract`: você nunca cria um `BaseEntity`
sozinho, só `Usuario`, `Empresa`, etc. É uma forma de reaproveitar código sem custo de SQL.
::

## O padrão Repository: comandos de banco de graça

Já temos as entidades. Agora, como a gente **salva** e **busca** no banco? Aqui entra a joia do Spring
Data JPA: o **Repository**. Você só cria uma **interface** e o Spring gera a implementação sozinho.

```java
public interface UsuarioRepository extends JpaRepository<Usuario, Long> {

    Optional<Usuario> findByEmail(String email);

    boolean existsByCodigoIndicacao(String codigoIndicacao);
}
```

Repare: **não tem nenhum código dentro**. Só declarações. E mesmo assim ele já faz muita coisa.

### 1. Métodos CRUD prontos, herdados de JpaRepository

Ao estender `JpaRepository<Usuario, Long>` (onde `Usuario` é a entidade e `Long` é o tipo da chave
primária), você **ganha de graça** os comandos mais comuns:

| Método herdado | O que faz | SQL equivalente |
|----------------|-----------|-----------------|
| `save(usuario)` | Cria ou atualiza | `INSERT` ou `UPDATE` |
| `findById(1L)` | Busca pela chave primária | `SELECT ... WHERE id = 1` |
| `findAll()` | Lista todos | `SELECT * FROM usuario` |
| `delete(usuario)` | Remove | `DELETE FROM usuario WHERE id = ?` |
| `count()` | Conta quantos há | `SELECT count(*)` |

Você não escreve nenhum desses — eles já vêm.

### 2. Query methods: o nome do método vira SQL

E o `findByEmail(String email)`? Esse é um **query method**. O Spring **lê o nome do método** e monta
o SQL automaticamente. `findBy` + `Email` = "busque um usuário `WHERE email = ?`". A convenção é tão
poderosa que dá para compor filtros só nomeando:

```java
// "conte usuários com validade >= a data informada"
long countByValidadeGreaterThanEqual(LocalDate data);

// "liste usuários que têm indicador = true"
List<Usuario> findByIndicadorTrue();

// "busque pelo hash do token de reset"
Optional<Usuario> findByResetTokenHash(String resetTokenHash);
```

O Spring entende palavras-chave como `findBy`, `existsBy`, `countBy`, `GreaterThanEqual`, `True`, `And`,
`Or` — e traduz tudo para SQL. Para o dia a dia, isso elimina uma montanha de código repetitivo.

::callout{type="tip" title="E quando o nome não dá conta?"}
Para consultas mais complexas (junções, agregações), você ainda pode escrever a query na mão com a
anotação `@Query`, como o `UsuarioRepository` faz no `buscarParaAdmin(...)` para filtrar usuários com
paginação. Mas para o comum — buscar por email, por id, listar — os query methods resolvem sem esforço.
::

## Optional: o jeito seguro de dizer "pode não existir"

Repare que `findByEmail` não retorna `Usuario`, e sim `Optional<Usuario>`. Isso é de propósito. Quando
você busca por um email, **pode ser que ninguém tenha aquele email**. O que retornar? Antigamente
retornava-se `null`, e aí o programa quebrava com o temido `NullPointerException` se você esquecesse de
checar.

O **`Optional`** é uma "caixinha" que ou **tem** o valor dentro, ou está **vazia** — e ele te **obriga**
a lidar com o caso vazio:

```java
Optional<Usuario> talvezUsuario = usuarioRepository.findByEmail("joao@x.com");

if (talvezUsuario.isPresent()) {
    Usuario u = talvezUsuario.get();   // achou
} else {
    // não existe usuário com esse email — trate aqui
}

// ou, de forma mais elegante:
Usuario u = usuarioRepository.findByEmail("joao@x.com")
        .orElseThrow(() -> new RuntimeException("Usuário não encontrado"));
```

::callout{type="tip" title="Por que isso importa"}
`Optional` transforma um risco silencioso (esquecer de checar `null`) em algo **explícito no tipo**.
Ao ver `Optional<Usuario>`, você já sabe, só de olhar a assinatura, que "pode não vir nada" — e o
compilador te empurra a tratar. É segurança de graça.
::

## Soft delete: apagar sem apagar de verdade

Um último detalhe importante nas entidades `Empresa` e `Produto`. Se um usuário resgatou um cupom e
depois a empresa é excluída, o histórico apontaria para o nada — quebrando relatórios e integridade.
Por isso o Descontai usa **soft delete** (exclusão lógica): em vez de remover a linha, marca uma coluna
`deleted_at` com a data da "exclusão".

```java
@Entity
@Table(name = "empresas")
@SQLDelete(sql = "UPDATE empresas SET deleted_at = NOW() WHERE id = ?")
@Where(clause = "deleted_at IS NULL")
public class Empresa extends BaseEntity {
    // ...
}
```

- **`@SQLDelete`** troca o `DELETE` real por um `UPDATE deleted_at = NOW()`. Ou seja, chamar
  `empresaRepository.delete(empresa)` **não apaga** — só carimba a data.
- **`@Where(clause = "deleted_at IS NULL")`** faz o Hibernate adicionar, em **toda** consulta dessa
  entidade, um filtro que **esconde** os registros excluídos. Para o resto da aplicação, é como se a
  empresa tivesse sumido — mas a linha continua lá, preservando o histórico.

::callout{type="info" title="Por que não apagar de verdade?"}
Porque em sistemas reais os dados quase nunca podem sumir sem deixar rastro: há histórico de uso,
auditoria e a possibilidade de recuperar algo excluído por engano. O soft delete dá o melhor dos dois
mundos — o registro "some" para o usuário, mas continua íntegro no banco.
::

## Resumindo

- Um **banco relacional** (PostgreSQL) guarda dados em **tabelas** (planilhas), com **linhas**
  (registros) e **colunas** (campos).
- A **chave primária** (`id`) identifica cada linha de forma única; a **chave estrangeira** (`cidade_id`)
  é a "cola" que liga uma linha a outra tabela.
- O **ORM/JPA** traduz **classe → tabela**, **objeto → linha**, **atributo → coluna**, usando anotações:
  `@Entity`, `@Table`, `@Column`, `@Id` + `@GeneratedValue`, e `@ManyToOne` + `@JoinColumn` para
  relacionamentos.
- O **`BaseEntity`** (`@MappedSuperclass`) centraliza `id`, `createdAt` e `updatedAt` para todas as
  entidades, preenchidos automaticamente por `@PrePersist`/`@PreUpdate`.
- O padrão **Repository** te dá `save`/`findById`/`findAll` de graça ao estender `JpaRepository`, e
  transforma o **nome** de métodos como `findByEmail` em SQL (query methods).
- **`Optional`** deixa explícito que um resultado "pode não existir", eliminando `NullPointerException`.
- O **soft delete** (`@SQLDelete` + `@Where`) marca `deleted_at` em vez de apagar, preservando histórico.
