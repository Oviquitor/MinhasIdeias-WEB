Até agora você viu o mapa: o que é uma API, as ferramentas, a estrutura do projeto e as camadas.
Nesta aula a gente **entra numa única rota e a percorre inteira** — do momento em que o app manda
o pedido até o momento em que a API devolve a resposta. Vamos usar o **login real** do Descontai
(`POST /api/v1/auth/login`) e passar por cada peça, na ordem em que o pedido as encontra. No fim,
uma requisição HTTP vai deixar de ser mágica e virar uma sequência de passos que você consegue
apontar com o dedo.

## O caminho completo, de relance

Antes de detalhar cada peça, olha o trajeto que um pedido de login faz. Ele **desce** pelas
camadas, toca o banco, e a resposta **sobe** de volta:

```
  Cliente (app Flutter)
        │  POST /api/v1/auth/login  { "email": "...", "senha": "..." }
        ▼
┌──────────────────────────────────────────────┐
│ 1. AuthController  (camada api)                │  ← recebe o HTTP, valida a borda
│      @PostMapping("/login")                    │
└───────────────────┬──────────────────────────┘
                    │  login.execute(req)
                    ▼
┌──────────────────────────────────────────────┐
│ 2. LoginUseCase  (camada application)          │  ← orquestra a regra de negócio
│      @Transactional                            │
└───────────────────┬──────────────────────────┘
                    │  users.findByEmail(...)
                    ▼
┌──────────────────────────────────────────────┐
│ 3. UsuarioRepository  (camada infra) → Banco   │  ← busca os dados
└───────────────────┬──────────────────────────┘
                    │  monta TokenResponse
                    ▼
        Resposta HTTP 200 OK + JSON
        (sobe de volta pelo mesmo caminho)
```

Cada caixa é uma **camada** com uma responsabilidade única (você viu isso na Aula 4). O pedido
desce, o banco responde, e o resultado sobe transformado em JSON. Vamos peça por peça.

## Passo 1 — O Controller: a porta de entrada

O **Controller** é a primeira peça do código Java que um pedido HTTP encontra. É ele quem "atende
a porta". No Descontai, os endpoints de login ficam no `AuthController`:

```java
@RestController
@RequestMapping("/api/v1/auth")
public class AuthController {

    private final LoginUseCase login;
    // ... outros use cases injetados via construtor

    @PostMapping("/login")
    public ResponseEntity<?> login(@RequestBody @Valid LoginRequest req, HttpServletRequest httpReq) {
        String ip = httpReq.getRemoteAddr();
        if (!rateLimiter.tryConsume(ip)) {
            return ResponseEntity.status(HttpStatus.TOO_MANY_REQUESTS)
                    .header("Retry-After", "600")
                    .body(new ApiError(429, "Too Many Requests",
                            "Muitas tentativas de login. Tente novamente em 10 minutos.",
                            httpReq.getRequestURI()));
        }
        return ResponseEntity.ok(this.login.execute(req));
    }
}
```

Cada anotação (aquelas palavras com `@` na frente) tem um papel. São **instruções para o Spring**,
não código que "roda" no sentido comum — o Spring as lê e sabe o que fazer:

::fields
::field{name="@RestController" type="na classe"}
Diz ao Spring: "esta classe atende requisições HTTP e o que os métodos retornam já é o **corpo da
resposta** (vira JSON automaticamente)". Sem isso, o Spring nem olharia para essa classe como um
ponto de entrada web.
::
::field{name="@RequestMapping(\"/api/v1/auth\")" type="na classe"}
Define o **prefixo** de todas as rotas desta classe. Tudo aqui dentro começa com `/api/v1/auth`.
::
::field{name="@PostMapping(\"/login\")" type="no método"}
Liga o **verbo POST** + o caminho `/login` a este método. Juntando com o prefixo da classe, forma
a rota final: `POST /api/v1/auth/login`. É assim que o Spring sabe qual método chamar para cada URL.
::
::field{name="@RequestBody" type="no parâmetro"}
"Pegue o **corpo (body) JSON** do pedido e transforme num objeto `LoginRequest` para mim." Isso é a
**desserialização** (explicada logo abaixo).
::
::field{name="@Valid" type="no parâmetro"}
"Antes de me entregar esse objeto, **valide** os campos dele." É o gatilho do Bean Validation
(Passo 2).
::
::

::callout{type="info" title="Como o Spring \"liga\" a URL ao método (roteamento)"}
Quando a aplicação sobe (você viu na Aula 3), o Spring varre todos os `@RestController` e monta uma
**tabela de rotas**: uma lista que diz "`POST /api/v1/auth/login` → método `login()` do
`AuthController`". Quando um pedido chega, o Spring olha o **verbo + a URL**, consulta essa tabela e
chama o método certo. Isso se chama **roteamento**. Você nunca escreve um `if url == "/login"` na
mão — as anotações descrevem o mapa, e o Spring faz a busca.
::

Repare numa coisa importante: **o Controller quase não tem lógica**. Ele extrai o IP, checa o rate
limiter (a proteção contra força bruta da Aula 3) e **delega** o trabalho de verdade com
`this.login.execute(req)`. Ele é um recepcionista, não o especialista. Quem resolve o login é o
**UseCase** (Passo 3).

## Passo 2 — O DTO de entrada e a validação na borda

Aquele `LoginRequest` que chega no método é um **DTO** (Data Transfer Object): um objeto simples,
cujo único trabalho é **carregar dados** de um lado para o outro. Olha como ele é enxuto:

```java
public class LoginRequest {
    @Email @NotBlank
    private String email;
    @NotBlank
    private String senha;

    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
    public String getSenha() { return senha; }
    public void setSenha(String senha) { this.senha = senha; }
}
```

Os campos `email` e `senha` batem exatamente com o JSON que o app manda. E cada campo tem
anotações de **Bean Validation** — regras que o campo precisa obedecer:

| Anotação | O que exige | Se falhar… |
|----------|-------------|------------|
| `@NotBlank` | O texto não pode ser nulo nem vazio/só espaços | pedido rejeitado |
| `@Email` | O texto tem que ter formato de e-mail (`algo@algo.com`) | pedido rejeitado |
| `@Size(min=, max=)` | O texto tem que ter um tamanho dentro da faixa | pedido rejeitado |

::callout{type="info" title="De onde vem o @Size?"}
No `LoginRequest` a gente só usa `@Email` e `@NotBlank` — para logar, basta o campo estar
preenchido. Mas o `@Size` aparece em outros DTOs quando um limite de tamanho importa (ex.: uma
senha nova precisa ter no mínimo X caracteres). A ideia é sempre a mesma: **declarar a regra
ao lado do campo**, e o Spring a aplica sozinho quando vê o `@Valid`.
::

### Por que validar logo na borda?

Lembra do `@Valid` no Controller? É ele que dispara essas regras. Se o app mandar um e-mail
malformado ou a senha vazia, o Spring **rejeita o pedido antes de chamar o UseCase** e devolve
`400 Bad Request` sozinho. A regra de negócio nem chega a rodar.

Isso se chama **validar na borda** (validar na entrada). A vantagem: o resto do código pode
**confiar** que os dados já vieram com o formato mínimo esperado. É como o segurança na porta da
balada conferir a carteirinha — lá dentro ninguém precisa checar de novo se você é maior de idade.

::callout{type="tip" title="DTO x Entidade — a distinção que separa o júnior do pleno"}
Existe uma classe `Usuario` (a **Entidade**, veremos na Aula 5) que representa a linha do usuário no
banco — com hash de senha, role, saldo, datas, tudo. **Nunca** recebemos nem devolvemos a Entidade
direto na rota. Em vez disso, usamos **DTOs** como o `LoginRequest` (entrada) e o `TokenResponse`
(saída). Por quê?

- **Segurança**: se você devolvesse a Entidade, poderia vazar campos internos (hash de senha, flags).
  O DTO expõe **só o que o cliente precisa ver**.
- **Estabilidade**: a estrutura do banco pode mudar sem quebrar o contrato com o app — o DTO é uma
  "casca" independente.
- **Clareza**: o DTO de entrada diz exatamente o que a rota aceita; o de saída, exatamente o que ela
  devolve.

Guarde a frase: **Entidade é do banco; DTO é da conversa com o cliente.**
::

### Como o JSON vira objeto Java (desserialização)

O app manda isto no corpo do pedido:

```json
{
  "email": "ana@email.com",
  "senha": "minhaSenha123"
}
```

Isso é **texto**. O Java não entende texto de JSON diretamente — ele trabalha com objetos. Então o
Spring (usando uma biblioteca chamada **Jackson**, que veio junto no `starter-web`) faz a
**desserialização**: lê o JSON, cria um `new LoginRequest()`, chama `setEmail(...)` e `setSenha(...)`
com os valores encontrados, e entrega o objeto pronto no seu método. (É por isso que o DTO tem
*setters*.) Na volta, o caminho inverso — objeto Java virando texto JSON — se chama **serialização**.

## Passo 3 — O padrão UseCase: um caso de uso, uma responsabilidade

Quando o Controller chama `this.login.execute(req)`, quem responde é o `LoginUseCase`. Antes de ver
o código dele, entenda o **contrato** que ele assina — uma interface genérica que todo caso de uso do
Descontai implementa:

```java
public interface UseCase<I, O> {
    O execute(I input);
}
```

Leia assim: "um `UseCase` recebe uma entrada do tipo `I` e devolve uma saída do tipo `O`, através de
um único método `execute`". O `<I, O>` são **tipos genéricos** — placeholders que cada caso de uso
preenche. No login, `I` é `LoginRequest` e `O` é `TokenResponse`:

```java
@Component
@Transactional
public class LoginUseCase implements UseCase<LoginRequest, TokenResponse> {

    @Override
    @Transactional
    public TokenResponse execute(LoginRequest input) {
        // Busca pelo e-mail — erro proposital vago para não revelar se o e-mail existe
        Usuario u = users.findByEmail(input.getEmail())
                .orElseThrow(() -> new BusinessException("Credenciais invalidas"));

        // Compara a senha enviada (plaintext) com o hash BCrypt armazenado
        if (!encoder.matches(input.getSenha(), u.getPasswordHash())) {
            throw new BusinessException("Credenciais invalidas");
        }

        String accessToken  = jwt.generateToken(u.getEmail());
        String refreshToken = refreshTokenService.gerarRefreshToken(u);

        return TokenResponse.completo(accessToken, refreshToken, u.getId(), u.getNome(), /* ... */);
    }
}
```

::callout{type="info" title="Por que separar cada caso de uso numa classe própria?"}
Cada operação de negócio importante — logar, registrar, renovar token, redefinir senha — vira **uma
classe UseCase**. Em vez de um "service gigante" com trinta métodos, você tem várias classes
pequenas com um `execute` cada. Isso traz três ganhos:

- **Uma responsabilidade por classe**: abriu o `LoginUseCase`, você sabe que ali só tem login.
- **Fácil de testar**: dá para testar o login isoladamente, injetando *mocks* das dependências (Aula 11).
- **Fácil de achar**: precisa entender o cadastro? Procure `RegisterUserUseCase`. O nome é o mapa.

Repare que o `LoginUseCase` recebe suas dependências (`UsuarioRepository`, `PasswordEncoder`, etc.)
**pelo construtor** — a injeção de dependência da Aula 3 em ação. Ele não dá `new` em nada; declara o
que precisa e o Spring entrega.
::

## Passo 4 — @Transactional: tudo ou nada

Repare na anotação `@Transactional` sobre o método `execute`. Ela ativa uma **transação** de banco de
dados. Uma transação é um **bloco "tudo ou nada"**: ou todas as operações de banco dentro dele dão
certo e são gravadas juntas (*commit*), ou, se alguma falhar, **todas são desfeitas** (*rollback*),
como se nada tivesse acontecido.

::callout{type="tip" title="A analogia da transferência bancária"}
Transferir R$ 100 do Ana para o Beto são **duas** operações: tirar 100 do Ana e somar 100 no Beto.
Imagine que a energia caísse **entre as duas** — o dinheiro sumiria do Ana e nunca chegaria no Beto!
A transação impede isso: ou as duas acontecem, ou nenhuma. Nunca "pela metade".
::

No login, o `execute` gera um refresh token e **salva o hash dele no banco**. O `@Transactional`
garante que, se esse *save* falhar, a transação é desfeita e **nenhum token é devolvido ao cliente** —
evitando um estado inconsistente onde o app recebe um token que o banco não conhece. É o "tudo ou
nada" protegendo a integridade dos dados.

## Passo 5 — O Repository: falando com o banco

Dentro do UseCase, a linha `users.findByEmail(input.getEmail())` é onde a API finalmente **toca o
banco de dados**. O `users` é um `UsuarioRepository`:

```java
public interface UsuarioRepository extends JpaRepository<Usuario, Long> {

    Optional<Usuario> findByEmail(String email);
    // ... outros métodos
}
```

Duas coisas chamam a atenção — e as duas são "mágica" boa do Spring Data (Aula 5):

- **É só uma interface, sem código.** Você declara o método `findByEmail`, mas não escreve o SQL.
  O Spring **implementa sozinho** em tempo de execução, lendo o *nome* do método: `findByEmail` vira
  "busque um usuário onde o campo `email` seja igual ao parâmetro". A convenção de nome **é** a query.
- **O retorno é `Optional<Usuario>`.** Um `Optional` é uma "caixinha" que pode estar cheia (achou o
  usuário) ou **vazia** (não achou nenhum e-mail igual). Isso força você a lidar com o caso "não
  existe" — no login, o `.orElseThrow(...)` faz exatamente isso: se a caixa vier vazia, lança o erro
  de credenciais inválidas.

## Passo 6 — Montando a resposta: DTO de saída + ResponseEntity

Com o usuário em mãos e a senha conferida, o UseCase monta o **DTO de saída**, o `TokenResponse`, com
só o que o cliente precisa: os tokens e os dados públicos do perfil.

```java
return TokenResponse.completo(accessToken, refreshToken, u.getId(), u.getNome(), /* ... */);
```

Esse `TokenResponse` sobe de volta até o Controller, que o embrulha num **`ResponseEntity`**:

```java
return ResponseEntity.ok(this.login.execute(req));
```

O `ResponseEntity` é o objeto que representa a **resposta HTTP completa** — o `.ok(...)` define o
status **200 OK** e coloca o DTO no corpo. Repare que outros métodos do mesmo controller usam status
diferentes de propósito: o `register` usa `ResponseEntity.status(HttpStatus.CREATED)` (**201**, "criei
algo"), e o `logout` usa `ResponseEntity.noContent()` (**204**, "deu certo, sem corpo"). Escolher o
status certo faz parte de desenhar uma boa rota (Aula 1).

Na saída, o Spring/Jackson **serializa** o `TokenResponse` de volta para JSON, e o app recebe algo assim:

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "token": "eyJhbGciOiJIUzI1NiJ9...",
  "refreshToken": "d3f1c2a0-...",
  "id": 42,
  "nome": "Ana",
  "cidade": "Recife",
  "role": "USER"
}
```

E o ciclo fecha: **o app perguntou, a API respondeu.** Exatamente o par request/response da Aula 1 —
só que agora você conhece cada peça que rodou por dentro.

## Resumindo

- Uma rota é uma **cadeia de camadas**: Controller → UseCase → Repository → (banco) → Response.
- O **Controller** (`@RestController`, `@RequestMapping`, `@PostMapping`) atende o HTTP e **delega**;
  o `@RequestBody @Valid` desserializa o JSON e dispara a validação.
- O **DTO de entrada** (`LoginRequest`) carrega os dados e traz as regras do **Bean Validation**
  (`@NotBlank`, `@Email`, `@Size`) — validar **na borda** deixa o resto do código confiar nos dados.
- **DTO não é Entidade**: nunca expomos a Entidade do banco direto — por segurança, estabilidade e clareza.
- O padrão **`UseCase<I, O>`** dá a cada operação de negócio uma classe própria com um `execute` —
  testável, isolada e fácil de achar.
- **`@Transactional`** torna o bloco "tudo ou nada": se algo falha, tudo é desfeito.
- O **Repository** (`findByEmail`) fala com o banco só declarando um método; o retorno `Optional`
  força tratar o "não achei".
- A resposta é um **DTO de saída** (`TokenResponse`) embrulhado num **`ResponseEntity`** com o status
  certo, serializado de volta para JSON.

::callout{type="tip" title="Fixando"}
Pegue outra rota qualquer do projeto e tente traçar a mesma cadeia no papel: qual o Controller? qual o
DTO de entrada e suas validações? qual o UseCase? o que ele busca no Repository? qual o DTO de saída e
o status? Se você conseguir desenhar esse caminho sozinho, entendeu a anatomia de uma rota.
::
