Toda API séria precisa responder duas perguntas o tempo todo: **quem é você?** e **o que
você pode fazer aqui?**. Se a API errar na primeira, um estranho entra como se fosse você.
Se errar na segunda, um usuário comum apaga dados de admin. Esta aula é sobre como o
Descontai responde a essas duas perguntas — do jeito que o mercado faz.

## Autenticação x Autorização: a dupla que todo mundo confunde

São coisas **diferentes** e acontecem **nessa ordem**:

- **Autenticação** = *provar quem você é*. Você chega na portaria e mostra o documento. A
  portaria confere e conclui: "ok, é o João mesmo".
- **Autorização** = *o que você pode fazer* depois de identificado. O João entrou, mas só
  pode subir até o 3º andar. O apartamento da cobertura é só do síndico.

::callout{type="info" title="Uma frase para nunca mais esquecer"}
**Autenticação** vem antes: é o login, o "quem é você". **Autorização** vem depois: é a
permissão, o "o que você pode". Lembra da Aula 1? `401 Unauthorized` = falhou a
**autenticação** (não provou quem é). `403 Forbidden` = falhou a **autorização** (provou
quem é, mas não tem permissão). Os dois erros existem justamente porque são etapas separadas.
::

Nesta aula vamos ver **as duas**: primeiro como o Descontai prova quem você é (senha, JWT,
o filtro-porteiro) e depois como decide o que você pode fazer (rotas públicas, roles,
`@PreAuthorize`).

## Nunca, jamais, guarde senha em texto puro

Regra número um da segurança de verdade: **o banco de dados nunca deve saber a sua senha**.

Parece contraintuitivo — como o sistema confere o login se não guarda a senha? A resposta é
o **hashing**. Em vez de guardar `minhasenha123`, guardamos o resultado de "triturar" essa
senha por uma função matemática **de via única**: dá para transformar a senha no hash, mas
é **impossível** transformar o hash de volta na senha.

No Descontai usamos **BCrypt**. Olha como ele é configurado no `SecurityConfig`:

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

No cadastro, aplicamos o BCrypt sobre a senha e guardamos **só o hash**. Repare que o
`CustomUserDetails` deixa isso explícito — ele nunca devolve a senha crua:

```java
@Override
public String getPassword() {
    // "passwordHash" já é o resultado do BCrypt — nunca a senha em texto plano
    return user.getPasswordHash();
}
```

No login, o Spring pega a senha que você digitou, aplica o **mesmo** BCrypt e compara os
dois hashes. Se batem, a senha está certa — **sem nunca ter guardado a senha original**.

::callout{type="warning" title="Por que BCrypt e não MD5/SHA-1 — segurança de verdade"}
Você poderia pensar: "qualquer hash serve, não?". **Não.** BCrypt tem dois superpoderes que
um SHA-1 simples não tem:

- **Salt automático**: ele joga um valor aleatório junto com a senha antes de triturar.
  Assim, duas pessoas com a senha `123456` geram hashes **diferentes**. Isso mata as
  *rainbow tables* (tabelas gigantes de senhas comuns já pré-calculadas).
- **Lento de propósito**: BCrypt é intencionalmente devagar (fator de custo ajustável). Para
  o login isso é imperceptível, mas para um atacante que quer testar bilhões de senhas por
  segundo, essa lentidão é uma muralha.

**Nunca** use MD5 ou SHA-1 para senha, **nunca** invente seu próprio esquema, e obviamente
**nunca** guarde a senha em texto puro. Se o banco vazar (e bancos vazam), a diferença entre
BCrypt e texto puro é a diferença entre "os hashes são inúteis para o atacante" e "todas as
senhas dos seus usuários estão na internet".
::

Analogia: BCrypt é **triturar a senha num liquidificador sem botão de "desfazer"**. Você
consegue triturar de novo e comparar as vitaminas, mas nunca reconstruir a fruta.

## O JWT: uma pulseira de festival assinada

Provamos quem você é no login. Mas o HTTP é **sem memória** — a próxima requisição chega
"do zero", sem saber que você já logou. Como a API te reconhece a cada pedido sem pedir a
senha toda hora?

Com um **token**. No login, a API te entrega um crachá digital chamado **JWT** (JSON Web
Token). Em toda requisição seguinte, você mostra esse crachá no header
`Authorization: Bearer {token}`.

Pense numa **pulseira de festival**: ao entrar você mostra o ingresso (login) e ganha uma
pulseira **assinada/lacrada**. Depois é só mostrar a pulseira — ninguém confere seu RG de
novo. E se alguém tentar falsificar a pulseira, o lacre não bate e o segurança percebe na hora.

### As 3 partes do JWT

Um JWT é um texto com **três partes separadas por ponto**: `header.payload.signature`.

| Parte | O que tem | Analogia |
|-------|-----------|----------|
| **header** | O algoritmo usado (aqui, `HS256`) | O tipo da pulseira |
| **payload** | Os dados (no Descontai: o `sub` = email, mais `iat` e `exp`) | O que está escrito na pulseira |
| **signature** | A **assinatura** que prova que o token não foi adulterado | O lacre que não dá para falsificar |

::callout{type="warning" title="O payload é LEGÍVEL — não guarde segredo lá dentro"}
As duas primeiras partes são só **Base64**, não criptografia. Qualquer um consegue ler o
payload do seu token (cole num site como o jwt.io e veja). O que protege o token **não é
esconder** o conteúdo — é a **assinatura**. Por isso: **nunca** coloque senha, dado sensível
ou segredo no payload. O Descontai coloca só o email e as datas, nada mais.
::

### Como a assinatura protege (e por que é "stateless")

A assinatura é feita com **HMAC-SHA256**: a API mistura o `header` + `payload` com uma
**chave secreta** que só ela conhece (o `jwt.secret`, que — lembra da Aula 3? — vem de
variável de ambiente, nunca do código). O resultado é a terceira parte.

Se um atacante mudar **um único caractere** do payload (ex.: trocar seu email pelo de outra
pessoa), a assinatura não bate mais, porque ele **não tem a chave secreta** para gerar a
assinatura nova. O token é rejeitado.

E aqui está a mágica do **stateless**: para validar o token, a API **não precisa guardar
sessão nenhuma**. Ela só recalcula a assinatura com a chave secreta e confere se bate. A
prova de que o token é legítimo é **matemática**, não uma consulta a uma tabela de sessões.

::callout{type="tip" title="Por que \"stateless\" faz a API escalar fácil"}
Se o servidor guardasse sessão em memória, cada instância da API só reconheceria os usuários
que logaram *nela*. Com 5 instâncias atrás de um load balancer, viraria bagunça (ou você
precisaria de um banco de sessões compartilhado). Com JWT **stateless**, qualquer instância
valida qualquer token sozinha — é só ter a mesma chave secreta. Adicionar mais servidores
para aguentar mais usuários fica trivial.
::

Vamos ver isso no código. O `JwtTokenProvider` tem os dois métodos que importam.
**Gerar** o token no login:

```java
public String generateToken(String subject) {
    Date now = new Date();
    Date exp = new Date(now.getTime() + expirationMs);

    return Jwts.builder()
            .setSubject(subject)          // "sub": quem é o dono (o email)
            .setIssuedAt(now)             // "iat": quando foi emitido
            .setExpiration(exp)           // "exp": quando expira
            .signWith(key, SignatureAlgorithm.HS256) // assina com HMAC-SHA256
            .compact();                   // vira header.payload.signature
}
```

E **validar** o token a cada requisição:

```java
public boolean isValid(String token) {
    if (!enabled) return false;
    try {
        // O parse verifica assinatura HMAC-SHA256 E expiração de uma vez.
        // Se qualquer coisa estiver errada, ele lança exceção.
        Jwts.parserBuilder().setSigningKey(key).build().parseClaimsJws(token);
        return true;
    } catch (Exception e) {
        // Token adulterado, expirado ou malformado → simplesmente inválido.
        return false;
    }
}
```

Repare numa decisão de design importante: o token **não guarda a role** do usuário. Só o
email. Assim, se a role mudar no banco, um token antigo não continua carregando a permissão
velha — a role é sempre buscada fresquinha (veremos no filtro logo abaixo).

## O que é um "filtro" numa cadeia de filtros

Antes de qualquer requisição chegar no seu controller, ela passa por uma **fila de filtros**
— a *filter chain* do Spring Security. Cada filtro é um "inspetor" que olha a requisição,
faz seu trabalho e decide: **deixa passar** para o próximo, ou **barra aqui**.

Pense na entrada de um aeroporto: raio-x, checagem de passaporte, checagem do cartão de
embarque… uma **fila** de conferências. Se você passa por todas, chega no portão (o
controller). Se travar em uma, nem chega perto do avião.

O filtro de segurança do Descontai é o `JwtAuthenticationFilter` — o **porteiro** que lê e
valida o crachá JWT.

## O JwtAuthenticationFilter: o porteiro da API

Este filtro roda **uma vez por requisição** (ele estende `OncePerRequestFilter`). O trabalho
dele, em ordem:

::steps
### Lê o crachá
Pega o header `Authorization` e verifica se começa com `Bearer `. Se não tiver, deixa a
requisição seguir **sem autenticar** (uma rota pública vai funcionar; uma protegida será
barrada mais à frente).

### Valida a assinatura e a expiração
Chama o `tokenProvider.isValid(token)`. Se o token foi adulterado ou expirou, responde
**401 na hora** e a requisição morre aqui.

### Descobre quem é o usuário
Extrai o email do payload (`getSubject`) e busca o usuário no banco — assim pega a role
**atualizada** e confirma que a conta ainda existe.

### Carimba o "crachá validado" no contexto
Cria a autenticação com as **roles** do usuário e a guarda no `SecurityContextHolder`. A
partir daqui, qualquer parte do código sabe quem está logado.
::

O trecho central:

```java
String auth = request.getHeader(HttpHeaders.AUTHORIZATION);

if (auth != null && auth.startsWith("Bearer ")) {
    String token = auth.substring(7); // remove o "Bearer "

    if (!tokenProvider.isValid(token)) {
        response.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Token invalido ou expirado");
        return; // barra aqui — nem chega no controller
    }

    String email = tokenProvider.getSubject(token);
    var user = users.findByEmail(email).orElse(null); // busca a role fresca

    if (user != null) {
        var principal = new CustomUserDetails(user);
        var authToken = new UsernamePasswordAuthenticationToken(
                principal, null, principal.getAuthorities()); // credencial null: o JWT já provou
        SecurityContextHolder.getContext().setAuthentication(authToken);
    }
}

filterChain.doFilter(request, response); // passa para o próximo da fila
```

Aquela linha `SecurityContextHolder.getContext().setAuthentication(...)` é o coração de tudo:
é o momento em que a API passa a **reconhecer** você durante aquela requisição.

## O SecurityConfig: as regras da casa

Se o filtro é o porteiro, o `SecurityConfig` é o **regulamento do prédio**. Ele define, num
lugar só, o que é público, o que é protegido e como a API se defende. Vamos por partes.

### Stateless + CSRF desligado

```java
.csrf(csrf -> csrf.disable())
.sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
```

`STATELESS` diz ao Spring: **nunca crie sessão HTTP**. Faz sentido — nossa "sessão" é o
próprio JWT. E o CSRF (uma proteção contra ataques que abusam de **cookies de sessão**) pode
ser desligado com segurança justamente porque **não usamos cookies de sessão**. Desligar CSRF
sem entender isso seria um erro grave; aqui é uma decisão consciente.

### Rotas públicas x protegidas

```java
.authorizeHttpRequests(auth -> {
    // Sempre públicas (sem token):
    auth.requestMatchers(
        "/actuator/health", "/health",
        "/api/v1/auth/**",                        // login, registro, refresh
        "/api/v1/pagamentos/webhooks/asaas"       // webhook: validado por assinatura, não JWT
    ).permitAll();

    // Leitura pública ANTES do login (só GET):
    auth.requestMatchers(HttpMethod.GET, "/api/v1/cidades", "/api/v1/cidades/**").permitAll();
    auth.requestMatchers(HttpMethod.GET, "/api/v1/planos",  "/api/v1/planos/**").permitAll();

    // Todo o resto exige token válido:
    auth.anyRequest().authenticated();
})
```

Aquele `auth.anyRequest().authenticated()` é a linha de ouro: **por padrão tudo é protegido**,
e você **abre exceções** conscientemente. Esquecer de proteger é muito pior do que esquecer de
liberar, então o padrão seguro é "fechado".

### Headers de segurança (HSTS e amigos)

O `SecurityConfig` também carimba **headers defensivos** em toda resposta:

```java
.headers(headers -> headers
    // HSTS: obriga o navegador a usar HTTPS por 1 ano (evita downgrade para HTTP)
    .httpStrictTransportSecurity(hsts -> hsts
        .includeSubDomains(true)
        .maxAgeInSeconds(31536000))
    // Impede embutir a API em <iframe> (proteção contra clickjacking)
    .frameOptions(frame -> frame.deny())
    // Impede o browser "adivinhar" o Content-Type (anti MIME-sniffing)
    .contentTypeOptions(Customizer.withDefaults())
)
```

Você não precisa decorar cada um agora. A ideia é: **segurança são camadas**. O JWT protege
a identidade; esses headers protegem contra ataques do lado do navegador. Defesa em
profundidade.

### O registro do filtro na cadeia

```java
http.addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
```

Isso encaixa o nosso porteiro **antes** do filtro padrão do Spring. A ordem importa: o
contexto de segurança precisa estar populado (o crachá carimbado) **antes** de o Spring
decidir se você tem permissão para a rota.

## Roles e autorização: USER, ADMIN e EMPRESA

Autenticado, entramos na **autorização**. O Descontai tem três papéis (*roles*):

| Role | Quem é | Pode… |
|------|--------|-------|
| `USER` | Usuário comum do app | Ver e resgatar cupons |
| `EMPRESA` | Parceiro que oferta cupons | Gerenciar as ofertas da própria empresa |
| `ADMIN` | Equipe do Descontai | Gerenciar cidades, planos, moderar tudo |

A role vem do banco, e o `CustomUserDetails` a traduz para o formato que o Spring entende —
com o prefixo `ROLE_` que o Spring exige:

```java
@Override
public Collection<? extends GrantedAuthority> getAuthorities() {
    String roleValue = user.getRole();
    // Garante "ROLE_" sem duplicar (aceita "ADMIN" ou já "ROLE_ADMIN" no banco)
    String authority = roleValue != null && roleValue.startsWith("ROLE_")
            ? roleValue
            : "ROLE_" + roleValue;
    return List.of(new SimpleGrantedAuthority(authority));
}
```

Com `@EnableMethodSecurity` ligado no `SecurityConfig`, dá para proteger rotas **por role**
direto no controller com `@PreAuthorize`. Exemplos reais do projeto:

```java
@PreAuthorize("hasRole('ADMIN')")   // só a equipe do Descontai
public class AdminController { ... }

@PreAuthorize("hasRole('EMPRESA')") // só o parceiro logado
public class EmpresaMeController { ... }
```

Se um `USER` comum tentar acessar uma rota `hasRole('ADMIN')`, ele passa na **autenticação**
(o token é válido) mas falha na **autorização** → recebe **403 Forbidden**. Autenticação e
autorização, cada uma no seu momento — exatamente como abrimos a aula.

## Access token curto + refresh token longo

Falta uma peça. Se o access token JWT durasse muito (dias), um token roubado seria um
desastre prolongado. Se durasse pouco e não houvesse mais nada, o usuário faria login a cada
15 minutos — péssimo. A solução do mercado (e do Descontai) são **dois tokens**:

- **Access token (JWT)**: **curto**, ~15 minutos (`jwt.expiration=900000`). É ele que vai em
  toda requisição. Se vazar, o estrago tem prazo de validade curto.
- **Refresh token**: **longo** (30 dias). Serve **só** para pedir um access token novo quando
  o atual expira, sem refazer login.

E aqui vem o detalhe crucial de segurança: o refresh token é guardado no banco como **HASH
SHA-256**, nunca em texto puro. Olha o `RefreshTokenService`:

```java
public String gerarRefreshToken(Usuario usuario) {
    byte[] bytes = new byte[32];
    random.nextBytes(bytes); // 32 bytes verdadeiramente aleatórios
    String rawToken = Base64.getUrlEncoder().withoutPadding().encodeToString(bytes);

    RefreshToken rt = new RefreshToken();
    rt.setUsuario(usuario);
    rt.setTokenHash(sha256(rawToken)); // guarda só o HASH, nunca o valor bruto
    rt.setExpiresAt(LocalDateTime.now().plusDays(refreshExpirationDays));
    repository.save(rt);

    return rawToken; // o valor bruto vai pro cliente e some do servidor
}
```

::callout{type="warning" title="Por que HASH também no refresh token — o mesmo princípio da senha"}
É a **mesma lógica do BCrypt na senha**: se o banco vazar, os hashes dos refresh tokens são
**inúteis** para o atacante — ele não consegue voltar do hash para o token real. A entidade
`RefreshToken` deixa isso claro: só o `tokenHash` (64 caracteres hex) é persistido.

Tem mais: cada uso do refresh **rotaciona** (revoga o atual e emite um novo). Assim, se um
token for interceptado e usado, o dono legítimo percebe (o dele parou de funcionar) e a
janela de abuso é mínima. É defesa em profundidade de novo — desconfie de qualquer sistema
que guarde tokens (ou senhas) recuperáveis.
::

```java
public Usuario validarERotacionar(String rawToken) {
    RefreshToken rt = repository.findByTokenHash(sha256(rawToken))
            .orElseThrow(() -> new BusinessException("Refresh token invalido."));
    if (rt.isRevoked())  throw new BusinessException("Refresh token ja foi utilizado ou revogado.");
    if (rt.getExpiresAt().isBefore(LocalDateTime.now()))
        throw new BusinessException("Refresh token expirado. Faca login novamente.");

    rt.setRevoked(true); // revoga o atual — a rotação emite um novo
    repository.save(rt);
    return rt.getUsuario();
}
```

O desenho todo: token de acesso curto minimiza o dano se vazar; refresh longo evita relogin
constante; guardar só o hash protege o banco; a rotação limita o abuso. Cada decisão paga um
pedaço da conta da segurança.

## Resumindo

- **Autenticação** = provar quem você é (login). **Autorização** = o que você pode fazer
  (roles). Nessa ordem — daí `401` (não autenticado) x `403` (sem permissão).
- Senha **nunca** em texto puro: usamos **BCrypt**, que tem **salt** e é **lento de propósito**.
  É via única — como triturar sem botão de desfazer.
- Um **JWT** tem 3 partes (`header.payload.signature`), é assinado com **HMAC-SHA256** e é
  **stateless**: a validade é matemática, sem sessão no servidor — por isso escala fácil.
- O `JwtAuthenticationFilter` é o **porteiro**: intercepta toda requisição, lê o
  `Authorization: Bearer`, valida e popula o `SecurityContext`.
- O `SecurityConfig` centraliza as regras: **stateless**, CSRF off (com consciência), rotas
  públicas x `anyRequest().authenticated()`, headers (HSTS) e `@EnableMethodSecurity`.
- Roles **USER/EMPRESA/ADMIN** via `CustomUserDetails` + `@PreAuthorize("hasRole(...)")`.
- **Access token curto** (15 min) + **refresh token longo** guardado como **hash** e
  **rotacionado** — cada escolha reduz o dano de um vazamento.

::callout{type="tip" title="Fixando"}
Descreva em voz alta: *"o usuário faz login, recebe um JWT curto e um refresh longo; a cada
requisição o filtro-porteiro valida o JWT e carimba quem ele é; o SecurityConfig e o
@PreAuthorize decidem se ele pode acessar aquela rota."* Se você conseguiu, entendeu a
segurança de uma API de verdade.
::
