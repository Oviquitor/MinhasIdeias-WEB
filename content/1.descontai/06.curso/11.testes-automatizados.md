Até agora você aprendeu a **escrever** o código do backend. Nesta aula você aprende a
**provar que ele funciona** — de um jeito automático, que roda em segundos e não depende de
ninguém clicar num app para conferir. Isso se chama **teste automatizado**, e é uma das coisas
que mais separam um projeto de brincadeira de um projeto profissional.

## Por que testar?

Todo mundo "testa" o código: você roda a aplicação, abre o app, faz um login e vê se deu certo.
O problema é que **isso é manual, lento e você esquece de testar metade das coisas**. Amanhã,
quando mexer em outra parte, vai testar tudo de novo na mão? Não vai.

Teste automatizado é você escrever **um código que testa o seu código**. Uma vez escrito, ele
roda sozinho quantas vezes você quiser. E ele te dá três presentes gigantes:

- **Pega bug cedo.** Um bug encontrado enquanto você escreve custa minutos para arrumar. O mesmo
  bug encontrado por um usuário em produção custa horas, estresse e, às vezes, dinheiro.
- **Dá coragem para mudar o código sem medo.** Precisa refatorar aquele método feio? Se você tem
  testes, é só mudar e rodar: se os testes continuam verdes, você **não quebrou nada**. Sem testes,
  cada mudança é uma roleta-russa.
- **Documenta o comportamento.** Um bom teste é uma frase legível: "quando a senha está errada,
  o login lança um erro de credenciais inválidas". Isso vale mais que um comentário — porque um
  comentário mente, mas um teste que passa **está sempre atualizado**.

::callout{type="info" title="A ideia em uma frase"}
Testar não é perder tempo — é **comprar tempo futuro**. Você paga um pouco agora escrevendo o
teste, e economiza muito depois: menos bugs em produção, menos medo de mexer, menos "será que
isso ainda funciona?".
::

## Os dois tipos que você mais vai usar

No Descontai (e na maioria dos projetos) existem dois tipos de teste que fazem o trabalho pesado:

| Tipo | O que testa | Usa banco de verdade? | Velocidade |
|------|-------------|-----------------------|------------|
| **Unitário** | Uma peça **isolada** (um use case, um service) | ❌ Não — usa **mocks** (dublês) | ⚡ Milissegundos |
| **Integração** | Várias peças **juntas**, de verdade | ✅ Sim — PostgreSQL real no Docker | 🐢 Segundos |

A regra prática: **muitos testes unitários** (rápidos, focados nas regras de negócio) e
**alguns testes de integração** (lentos, para provar que tudo se conecta). Vamos ver os dois.

## Teste unitário: testando uma peça isolada

Um teste **unitário** pega **uma única peça** do código — no nosso caso um *use case* ou um
*service* — e testa **só ela**, sem envolver o banco, a rede ou nenhuma outra parte pesada.

Mas peraí: o `LoginUseCase` precisa do repositório de usuários, do codificador de senha, do
gerador de JWT... como testar "só ele" se ele depende de tudo isso?

### Mock: o dublê de cinema

A resposta é o **mock** (dublê). Pensa no cinema: numa cena perigosa, o ator não pula do prédio
de verdade — entra um **dublê**, que faz exatamente aquele movimento e nada mais. O mock é isso
para o código: um objeto **falso** que você programa para responder o que você quiser.

Em vez de usar o `UsuarioRepository` real (que iria no banco), você usa um dublê e diz:
*"quando alguém te pedir o usuário `joao@email.com`, devolve este usuário aqui que eu inventei"*.
Assim o teste fica **rápido** (não toca o banco) e **controlado** (você decide exatamente o
cenário).

No Descontai usamos a biblioteca **Mockito** para criar esses dublês. Olha o começo do
`LoginUseCaseTest`:

```java
@ExtendWith(MockitoExtension.class)
class LoginUseCaseTest {

    @Mock
    private UsuarioRepository users;
    @Mock
    private PasswordEncoder encoder;
    @Mock
    private JwtTokenProvider jwt;
    @Mock
    private RefreshTokenService refreshTokenService;

    private LoginUseCase useCase;

    @BeforeEach
    void setUp() {
        useCase = new LoginUseCase(users, encoder, jwt, refreshTokenService);
    }
```

Três peças para entender:

::fields
::field{name="@ExtendWith(MockitoExtension.class)"}
Liga o Mockito no teste. É o que faz as anotações `@Mock` funcionarem.
::
::field{name="@Mock"}
Cria um **dublê** daquela dependência. `users` parece um `UsuarioRepository`, mas é falso — não
tem banco nenhum atrás. Só faz o que a gente mandar.
::
::field{name="@BeforeEach setUp()"}
Roda **antes de cada teste**, criando um `LoginUseCase` novinho e injetando os dublês nele
(repare: aqui damos `new` na mão, passando os mocks — é a injeção de dependência que a Aula 3
mostrou, mas com peças falsas).
::
::

### O padrão AAA: Arrange, Act, Assert

Todo bom teste tem três partes, sempre nessa ordem. Chamamos de **AAA**:

::steps
### Arrange (preparar)
Monte o cenário: crie os dados de entrada e **programe os mocks** para responderem o que você
precisa naquele caso.

### Act (agir)
Chame o método que está sendo testado. **Uma única ação** — é o que você quer verificar.

### Assert (verificar)
Confira se o resultado foi o esperado. Se não for, o teste **falha** e aponta o problema.
::

Agora olha o teste de sucesso do login com o AAA marcado:

```java
@Test
void loginValido_retornaTokenCompletoComRefresh() {
    // ARRANGE — monta o usuário e ensina os dublês a responderem
    Usuario u = usuarioComId(1L, "joao@email.com", "$2a$10$hash");
    u.setCidade(cidadeComId(10L, "Sao Paulo"));
    u.setSaldoIndicacao(BigDecimal.TEN);
    u.setIndicador(true);

    when(users.findByEmail("joao@email.com")).thenReturn(Optional.of(u));
    when(encoder.matches("senha123", "$2a$10$hash")).thenReturn(true);
    when(jwt.generateToken("joao@email.com")).thenReturn("access.jwt.token");
    when(refreshTokenService.gerarRefreshToken(u)).thenReturn("refresh-raw-token");

    // ACT — a única ação: executar o login
    TokenResponse resp = useCase.execute(loginRequest("joao@email.com", "senha123"));

    // ASSERT — confere o que voltou
    assertThat(resp.getToken()).isEqualTo("access.jwt.token");
    assertThat(resp.getRefreshToken()).isEqualTo("refresh-raw-token");
    assertThat(resp.getId()).isEqualTo(1L);
    assertThat(resp.getCidadeId()).isEqualTo(10L);
    assertThat(resp.getSaldoIndicacao()).isEqualTo(BigDecimal.TEN);
    assertThat(resp.getIndicador()).isTrue();
}
```

Repara na linha mágica do Mockito:

```java
when(users.findByEmail("joao@email.com")).thenReturn(Optional.of(u));
```

Leia em voz alta: *"**quando** pedirem o usuário por esse email, **então retorne** este usuário"*.
É você dirigindo o dublê. O `LoginUseCase` acha que falou com um repositório de verdade — mas
nenhum banco foi tocado. Por isso o teste roda em **milissegundos**.

E o `assertThat(...).isEqualTo(...)` (da biblioteca **AssertJ**) é a verificação: *"eu **afirmo
que** o token retornado **é igual a** `access.jwt.token`"*. Se não for, o teste falha ali.

### Testando o caminho do erro (o mais importante!)

Um erro clássico de júnior é testar **só o caso feliz**. Mas os bugs quase sempre moram nos
**casos de erro**: senha errada, usuário inexistente, dado inválido. O `LoginUseCaseTest`
cobre isso:

```java
@Test
void senhaErrada_lancaBusinessException() {
    Usuario u = usuarioComId(1L, "joao@email.com", "$2a$10$hash");
    when(users.findByEmail("joao@email.com")).thenReturn(Optional.of(u));
    when(encoder.matches(anyString(), anyString())).thenReturn(false); // senha não bate

    assertThatThrownBy(() -> useCase.execute(loginRequest("joao@email.com", "senhaerrada")))
            .isInstanceOf(BusinessException.class)
            .hasMessage("Credenciais invalidas");
}
```

Aqui a gente afirma que uma **exceção é lançada**: `assertThatThrownBy(...)` diz *"executar isso
deve estourar um erro"*, e completamos verificando que é da classe `BusinessException` com a
mensagem certa. Ou seja: testamos que, quando a senha não bate, o sistema **recusa o login do
jeito correto** — sem tocar em banco nenhum.

::callout{type="tip" title="Mockito também verifica o que NÃO deve acontecer"}
Mock não serve só para responder — ele também **espia** as chamadas. No `EmpresaServiceTest`, a
regra é: não pode apagar uma empresa que já teve cupons resgatados. O teste prova isso com
`verify(...)`:

```java
@Test
void delete_comResgates_bloqueiaEProtegeHistorico() {
    when(resgates.existsByProduto_Empresa_Id(5L)).thenReturn(true);

    assertThatThrownBy(() -> service.delete(5L))
            .isInstanceOf(BusinessException.class);
    verify(companies, never()).deleteById(5L); // garante que NÃO apagou
}
```

O `verify(companies, never()).deleteById(5L)` afirma: *"o repositório **nunca** foi chamado para
apagar"*. Testar o que **não** deve acontecer é tão valioso quanto testar o que deve.
::

## Teste de integração: o ensaio geral

O teste unitário é ótimo, mas ele tem um ponto cego: **tudo em volta é falso**. E se o problema
estiver justamente na conexão com o banco? Numa query SQL errada? Numa migration que não roda?
O mock nunca vai pegar isso, porque não existe banco no teste unitário.

Para isso existe o **teste de integração**: em vez de dublês, ele usa as **peças de verdade**,
conectadas umas nas outras. É o **ensaio geral** da peça de teatro — com os atores reais, o
palco montado e o cenário completo, para ver se tudo funciona junto.

### @SpringBootTest: sobe o contexto inteiro

A anotação `@SpringBootTest` faz o teste **subir a aplicação Spring completa** — exatamente como
a Aula 3 descreveu: ela varre o projeto, cria todos os beans, injeta as dependências reais e
conecta no banco. Não é mais uma peça isolada; é a orquestra inteira afinando.

### Testcontainers: um PostgreSQL de verdade, descartável

Mas subir a aplicação de verdade precisa de um banco de verdade. Usar o banco de produção? Jamais.
Um banco de teste fixo instalado na máquina de cada dev? Frágil e chato de manter.

A solução profissional é o **Testcontainers**: uma biblioteca que **sobe um PostgreSQL real
dentro de um container Docker**, só para o teste, e **joga fora** quando termina. Banco de
verdade, limpo, idêntico ao de produção, e que não deixa sujeira. É por isso que ele está no
`pom.xml`:

```xml
<!-- 🧪 Testes -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>

<!-- 🧱 Testcontainers -->
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>junit-jupiter</artifactId>
    <scope>test</scope>
</dependency>
```

Repare no `<scope>test</scope>`: essas bibliotecas **só existem durante os testes** — não vão
para o pacote de produção. Faz sentido: ninguém precisa de Testcontainers no servidor no ar.

::callout{type="info" title="Mock x banco real — a diferença que importa"}
No **teste unitário**, o repositório é um **dublê**: ele finge devolver dados, e nenhuma linha
de SQL roda. Rápido, mas não prova que o banco aceita aquela query.

No **teste de integração**, o repositório é **real** e conversa com um **PostgreSQL de verdade**
(no container). Se a query estiver errada ou a tabela não existir, o teste **quebra** — que é
exatamente o que queremos descobrir antes de ir para produção.
::

### O teste que prova que a aplicação inteira sobe

O teste de integração mais fundamental do Descontai é curtinho — e é dos mais importantes do
projeto:

```java
@SpringBootTest
class DescontaiApplicationTests {

    @Test
    void contextLoads() {
    }

}
```

O corpo está **vazio** de propósito. O teste é o `@SpringBootTest` em si: se a aplicação
**conseguir subir** — criar todos os beans, injetar tudo, conectar no banco e rodar as
migrations sem estourar nenhum erro — o teste passa. Se **qualquer coisa** estiver quebrada na
inicialização (um bean mal configurado, uma dependência faltando, uma migration inválida), o
contexto **não sobe** e o teste **falha**.

Por isso chamamos de **smoke test**: a analogia vem da eletrônica — você liga o aparelho pela
primeira vez e vê se **sai fumaça**. É o teste mais barato e mais valioso: com uma linha, você
prova que a aplicação **liga**.

::callout{type="tip" title="Gancho com a Aula 9: ele valida as migrations do zero"}
Como o `contextLoads` sobe o Spring conectado a um PostgreSQL **vazio e recém-criado** pelo
Testcontainers, o **Flyway roda todas as migrations do zero** naquele banco limpo. Ou seja: esse
teste também garante que a **sequência inteira de migrations** (Aula 9) aplica sem erro, do banco
vazio até o estado atual. Se alguém escreveu uma migration quebrada, o `contextLoads` acusa.
::

## O que vale a pena testar (e o que não vale)

Você não precisa — nem deve — testar 100% de cada linha. O foco é onde mora o risco:

- ✅ **Regras de negócio.** O login recusa senha errada? Não dá para apagar empresa com resgates?
  A receita é calculada certo? É aqui que os bugs doem — teste com carinho.
- ✅ **Casos de erro e limites.** Entrada inválida, lista vazia, formato não suportado. O
  `RelatorioServiceTest`, por exemplo, testa que um formato `pdf` (não suportado) lança erro e que
  um campo com vírgula é escapado no CSV.
- ✅ **Que a aplicação sobe** (o `contextLoads`).
- ❌ **Getters/setters triviais, código sem lógica.** Testar `getNome()` devolver o nome não pega
  bug nenhum — é só barulho.

::callout{type="warning" title="Teste comportamento, não implementação"}
Um bom teste verifica **o que o código faz** (o resultado, o comportamento visível), não **como
ele faz por dentro**. Assim, quando você refatorar o *como* (melhorar o código sem mudar o
resultado), o teste continua verde — e é justamente essa a "coragem de mudar sem medo" de que
falamos no começo.
::

## Onde os testes rodam de verdade

Rodar os testes na sua máquina é bom, mas gente esquece. Por isso, num projeto profissional, os
testes rodam **automaticamente** a cada mudança de código, num servidor — o **CI** (Integração
Contínua). Toda vez que alguém abre um Pull Request, o robô do CI baixa o código, roda **todos**
os testes (unitários e de integração, com Testcontainers subindo o PostgreSQL lá também) e só
libera o merge se **tudo estiver verde**.

Isso é o assunto da próxima aula — mas guarde a ideia: o teste que você escreve hoje vira um
**porteiro automático** que impede código quebrado de entrar no projeto.

## Resumindo

- **Testar** pega bugs cedo, dá coragem para mudar o código sem medo e documenta o comportamento.
- **Teste unitário** isola **uma peça** (use case/service) e usa **mocks** (dublês) no lugar das
  dependências — rápido e sem banco. Padrão: `@ExtendWith(MockitoExtension.class)`, `@Mock`,
  `when(...).thenReturn(...)`, `assertThat(...)`.
- Estruture cada teste com **AAA**: **Arrange** (preparar e programar os mocks), **Act** (a ação),
  **Assert** (verificar).
- **Teste de integração** usa as peças **de verdade**: `@SpringBootTest` sobe o contexto inteiro
  e o **Testcontainers** sobe um **PostgreSQL real** num container Docker, descartável.
- **Mock** finge dados (nenhum SQL roda); **banco real** prova que as queries e migrations
  funcionam.
- O `DescontaiApplicationTests.contextLoads` é o **smoke test**: prova que a aplicação **liga** —
  e, de quebra, valida as **migrations do zero** (Aula 9).
- Teste **regras de negócio e casos de erro**; ignore getters/setters triviais. E teste
  **comportamento**, não implementação.
