Você já sabe rodar os testes na sua máquina (Aula 11). Mas e o seu colega? E daqui a três
meses, quando ninguém lembra mais de rodar? Nesta aula, a gente ensina uma **máquina** a fazer
isso sozinha, **a cada mudança que chega no projeto** — e conta a história real de como o CI do
Descontai ficou **vermelho por muito tempo** até descobrirmos dois bugs escondidos um atrás do outro.

## O problema: "na minha máquina funciona"

Imagine o cenário. Você termina uma tarefa, roda os testes, tudo verde, faz o `push`. Só que:

- Você esqueceu de comitar um arquivo novo — na máquina do colega, nem compila.
- Você tinha uma variável de ambiente configurada só na sua máquina, que ninguém mais tem.
- Um teste que você nem sabia que existia quebrou, mas você rodou só os "seus".

O código chega no repositório **quebrado**, o próximo colega baixa, e agora ele perde meia hora
descobrindo que o problema não foi ele que causou. Multiplica isso por um time inteiro e você
tem o caos.

A solução tem nome: **CI**.

## O que é CI (Integração Contínua)

**CI** (Continuous Integration, Integração Contínua) é uma ideia simples e poderosa:

> **A cada `push` ou Pull Request, uma máquina neutra baixa o código do zero, compila e roda
> TODOS os testes — automaticamente.**

Se algo quebrar, o time é avisado na hora, com um ✅ verde ou um ❌ vermelho bem visível no
GitHub. Ninguém precisa lembrar de nada.

::callout{type="info" title="A analogia do revisor que nunca dorme"}
Pensa no CI como um **revisor automático que nunca dorme e nunca tem preguiça**. Toda vez que
alguém propõe uma mudança, ele pega uma máquina limpa (sem os "vícios" da sua máquina), monta o
projeto do zero e roda a bateria inteira de testes. Se passar, ele carimba de verde. Se falhar,
ele grita antes que o código estrague o trabalho de todo mundo. É a sua **rede de segurança**.
::

Por que uma máquina **neutra**? Porque a sua máquina está "contaminada": tem arquivos que você
esqueceu de comitar, variáveis que só você configurou, versões de Java diferentes. O CI parte
sempre de um ambiente **limpo e idêntico** — se passa lá, passa em qualquer lugar. É o fim do
"mas na minha máquina funcionava".

## E o CD (Entrega/Deploy Contínuo)

**CD** é o irmão do CI e pode significar duas coisas parecidas:

- **Continuous Delivery** (Entrega Contínua): depois que o CI passou, o sistema **empacota** a
  aplicação pronta para ir para produção (no nosso caso, monta a **imagem Docker** — Aula 13).
- **Continuous Deployment** (Deploy Contínuo): vai um passo além e **coloca no ar** sozinho.

No Descontai, o pipeline faz a parte de CI (compilar + testar) e já monta a imagem Docker
(Delivery). O deploy em si é assunto da Aula 16.

::callout{type="tip" title="A sigla junta: CI/CD"}
Fala-se sempre "CI/CD" como se fosse uma coisa só. É o caminho automático que uma mudança
percorre: **do `push` → compilar → testar → empacotar → (opcionalmente) subir para produção**,
sem ninguém apertar botão no meio.
::

## GitHub Actions: os conceitos

O **GitHub Actions** é a ferramenta do próprio GitHub para montar esse pipeline. Você descreve o
que quer que aconteça num arquivo, e o GitHub executa. Antes de olhar o arquivo real, vamos
aprender o vocabulário — cada termo aqui vai aparecer no `ci.yml` do Descontai.

::fields
::field{name="Workflow" type="o arquivo .yml"}
O **roteiro** de tudo. É um arquivo dentro de `.github/workflows/`. O do Descontai se chama
`ci.yml`. Um workflow reúne um ou mais "jobs".
::
::field{name="Evento / Trigger" type="on:"}
O **gatilho** que dispara o workflow. Ex.: "toda vez que alguém der `push`" ou "toda vez que
abrir um Pull Request". É o `on:` no topo do arquivo.
::
::field{name="Job" type="uma tarefa"}
Uma **tarefa** dentro do workflow (ex.: "compilar e testar", ou "montar o Docker"). Cada job
roda numa máquina própria. Jobs podem depender uns dos outros.
::
::field{name="Step" type="um passo"}
Cada **passo** dentro de um job, executado em ordem: "baixar o código", "instalar o Java",
"rodar os testes". Um step roda um comando ou usa uma ação pronta.
::
::field{name="Runner" type="a máquina"}
A **máquina** onde o job roda. O GitHub te empresta um computador limpo — no nosso caso um
**Ubuntu (Linux)**, definido por `runs-on: ubuntu-latest`. Ele nasce, faz o trabalho e é destruído.
::
::field{name="Services" type="containers de apoio"}
Containers que sobem **junto** só para dar suporte ao job. No Descontai, sobe um **PostgreSQL**
temporário para os testes de integração terem um banco real para conversar.
::
::field{name="Env / Secrets" type="variáveis de ambiente"}
As **variáveis de ambiente** que o job precisa (senhas, tokens, URLs). Lembra da Aula 3:
segredo nunca no código, sempre vem do ambiente. No CI, esse "ambiente" é o bloco `env:`.
::
::

::callout{type="info" title="Por que YAML?"}
O arquivo é escrito em **YAML** (`.yml`). É um formato de configuração baseado em **indentação**
(espaços à esquerda importam!). Duas coisas guardar: use **espaços, nunca Tab**, e a hierarquia
é dada pelo recuo. Um espaço a mais ou a menos e o GitHub reclama.
::

## O pipeline real do Descontai

Agora abrimos o arquivo de verdade: `.github/workflows/ci.yml`. Vamos por partes.

### O gatilho (quando roda?)

```yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
```

Lê-se assim: **este workflow (chamado "CI") roda quando** houver um `push` nas branches `main`
ou `develop`, **ou** quando um Pull Request for aberto/atualizado apontando para a `main`. Ou
seja: nenhum código entra nas branches importantes sem passar pelo revisor automático.

### O job `build-and-test`

Este é o coração do CI. Repare no cabeçalho:

```yaml
jobs:
  build-and-test:
    name: Build & Test
    runs-on: ubuntu-latest
```

`runs-on: ubuntu-latest` é o **runner**: um Linux limpo. **Guarde bem esse detalhe "Linux"** —
ele é o vilão da nossa história de depuração daqui a pouco.

**O banco de dados de mentira (services).** Os testes de integração do Descontai precisam de um
PostgreSQL de verdade. O CI sobe um só para isso:

```yaml
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_DB: descontai_test
          POSTGRES_USER: descontai
          POSTGRES_PASSWORD: descontai_test_pass
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432
```

Isso sobe um container `postgres:16-alpine` **temporário**, com um banco `descontai_test`. O
bloco `--health-cmd pg_isready` é uma checagem de saúde: o GitHub só começa os testes **depois**
que o Postgres estiver de fato pronto para aceitar conexões (senão os testes correriam antes do
banco existir). Quando o job acaba, o container é jogado fora.

**Os passos (steps).** Agora a sequência do que a máquina faz, em ordem:

::steps
### Baixar o código (checkout)
```yaml
      - name: Checkout
        uses: actions/checkout@v4
```
O runner nasce vazio. O primeiro passo é **clonar o repositório** para dentro dele. `uses:` quer
dizer "use uma ação pronta" — aqui, a oficial `actions/checkout`.

### Instalar o Java 17
```yaml
      - name: Setup Java 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: maven
```
Instala exatamente a versão de Java que o projeto usa (17, fixada no `pom.xml` — Aula 3). O
`cache: maven` guarda as dependências baixadas entre execuções, para o build ser mais rápido da
próxima vez.

### Compilar e rodar os testes
```yaml
      - name: Build e testes
        run: ./mvnw clean verify -B
        env:
          DB_URL: jdbc:postgresql://localhost:5432/descontai_test
          DB_USER: descontai
          DB_PASS: descontai_test_pass
          JWT_SECRET: ci-test-secret-must-be-32-chars-long!
          JWT_EXPIRATION: 86400000
          FLYWAY_ENABLED: true
          FLYWAY_BASELINE_ON_MIGRATE: true
          FCM_ENABLED: false
          ASAAS_API_KEY: test
          ASAAS_WEBHOOK_TOKEN: ci-test-webhook-token
          APP_CORS_ALLOWED_ORIGINS: http://localhost:3000
          SWAGGER_ENABLED: "false"
```
O passo principal. `run:` executa um comando de terminal: `./mvnw clean verify` (o Maven Wrapper
limpa o projeto, compila e roda **toda** a bateria de testes; o `-B` é modo "batch", sem enfeites
no log). O bloco `env:` fornece todas as **variáveis de ambiente** que a aplicação precisa —
repare que `DB_URL` aponta para o `localhost:5432`, exatamente o Postgres que subimos ali em cima.

### Verificar que o JAR foi gerado
```yaml
      - name: Verificar JAR gerado
        run: ls -lh target/*.jar
```
Confere que o **artefato final** (o `.jar` executável) realmente foi produzido. Se o build tivesse
falhado silenciosamente, aqui a gente pegaria.
::

::callout{type="tip" title="Repare nas variáveis de teste"}
Os valores em `env:` são **de mentira, só para o CI** — `JWT_SECRET` é um texto qualquer de 32
caracteres, `ASAAS_API_KEY: test`, etc. Nenhum segredo real aparece aqui (isto é um arquivo
público no Git!). Segredos de verdade ficam guardados nos **GitHub Secrets** e são injetados com
`${{ secrets.NOME }}`. Guarde esse detalhe do `ASAAS_WEBHOOK_TOKEN` — ele é o protagonista da
segunda parte da nossa história.
::

### O job `docker` (a parte de CD)

```yaml
  docker:
    name: Docker Build
    runs-on: ubuntu-latest
    needs: build-and-test
    if: github.ref == 'refs/heads/main'

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t descontai-api:${{ github.sha }} .
```

Duas linhas mágicas no topo:

- `needs: build-and-test` — este job **só roda se o `build-and-test` passou**. Não faz sentido
  empacotar um código que nem passou nos testes.
- `if: github.ref == 'refs/heads/main'` — só monta a imagem Docker **quando o push é na `main`**.
  Em Pull Requests e na `develop`, a gente só testa; a imagem "de produção" é montada apenas no
  código que já foi aprovado e chegou na `main`.

Depois, `docker build` monta a imagem (usando a tag `${{ github.sha }}`, o identificador único
daquele commit) e um smoke test sobe o container rapidinho só para ver se ele "de pé" não explode.
É a parte de **CD (Delivery)**: o entregável pronto para ir ao ar.

## A história real: o CI que vivia vermelho

Aqui vem a parte que nenhum tutorial conta. Quando o `ci.yml` foi criado, **ele falhou em todos
os runs por um bom tempo**. E o mais didático: eram **dois bugs encadeados** — um escondia o
outro. Só dava para ver o segundo depois de matar o primeiro.

::steps
### Bug nº 1: "Permission denied" — o `./mvnw` nem começava
O log do CI mostrava algo assim, e nem um teste chegava a rodar:

```bash
/home/runner/work/.../Descontai-API/mvnw: Permission denied
Error: Process completed with exit code 126.
```

**Por que?** O arquivo `mvnw` (o Maven Wrapper) é um **script**, e no Linux um script precisa da
**permissão de executável** para ser rodado com `./mvnw`. O projeto foi criado no **Windows**, que
não tem essa noção de "bit de executável". Quando o `mvnw` foi versionado no Git, ele entrou **sem**
essa permissão. Na sua máquina Windows, tudo bem. Mas o runner é **Linux** (lembra do `ubuntu-latest`?):
ele tentou executar `./mvnw`, viu que não tinha permissão e recusou — o famoso **exit code 126**
("comando encontrado, mas não executável"). O build morria **antes** de compilar qualquer coisa.

**A correção** foi marcar o arquivo como executável no próprio Git (não só no disco):
```bash
git update-index --chmod=+x mvnw
```
Isso grava no Git que o `mvnw` é executável. No próximo `push`, o runner Linux baixou o arquivo já
com a permissão certa e conseguiu rodá-lo.

### Bug nº 2: o contexto do Spring não subia nos testes
Com o `mvnw` finalmente rodando, os testes **enfim começaram** — e aí apareceu o **segundo** erro,
que estava escondido atrás do primeiro:

```bash
Error creating bean ... AsaasWebhookValidator:
Missing required environment variable: ASAAS_WEBHOOK_TOKEN
```

**Por que?** Os testes de integração sobem o contexto do Spring, e — lembra da Aula 3? — o perfil
ativo padrão é o `prod`. Nesse perfil, o Descontai tem uma **trava de segurança fail-fast**: ele se
**recusa a subir** se o `ASAAS_WEBHOOK_TOKEN` (o segredo que valida os avisos de pagamento) não
estiver definido. É de propósito: é melhor a aplicação **não subir** do que subir insegura. Só que
no CI essa variável não existia, então o contexto Spring quebrava e **todos** os testes falhavam de
uma vez.

**A correção** foi fornecer um valor de teste no `ci.yml`, exatamente como já era feito para o
`JWT_SECRET`:
```yaml
        env:
          # ...
          ASAAS_WEBHOOK_TOKEN: ci-test-webhook-token
```
Com o token de mentira presente, a trava de segurança ficou satisfeita, o contexto subiu, os testes
rodaram — e o CI **finalmente ficou verde**. ✅
::

::callout{type="warning" title="A moral da história: ler o LOG é a habilidade nº 1"}
Repare no padrão: **o primeiro erro escondia o segundo**. Enquanto o `./mvnw` dava "Permission
denied", era **impossível** saber que faltava o `ASAAS_WEBHOOK_TOKEN` — os testes nem chegavam a
rodar. Só depois de matar o bug nº 1 o bug nº 2 apareceu.

Isso é o dia a dia de depurar pipeline: você **conserta a camada de cima e a de baixo se revela**.
E a única forma de fazer isso é **lendo o log de erro com atenção** — a mensagem, o `exit code`, o
nome do bean que falhou. O log te diz **exatamente** o que aconteceu; a pressa de "chutar" a correção
é o que trava o júnior. Ler o log é, de longe, a habilidade número um de quem cuida de CI/CD.
::

## Resumindo

- **CI (Integração Contínua)**: a cada `push`/PR, uma máquina neutra compila e roda **todos** os
  testes sozinha. É a sua **rede de segurança** — o fim do "na minha máquina funciona".
- **CD (Entrega/Deploy Contínuo)**: depois do CI verde, empacota (imagem Docker) e, opcionalmente,
  coloca no ar.
- **GitHub Actions** monta isso num arquivo `.yml` em `.github/workflows/`. Vocabulário:
  **workflow** (arquivo), **evento/trigger** (`on:`), **job** (tarefa), **step** (passo),
  **runner** (a máquina Ubuntu), **services** (o Postgres temporário) e **env/secrets** (variáveis).
- O `ci.yml` do Descontai tem dois jobs: `build-and-test` (checkout → Java 17 → `./mvnw clean verify`
  com as envs → verifica o JAR) e `docker`, que **só roda na `main`** e só se os testes passaram.
- **Segredo nunca no código**: no CI usamos valores de teste; os reais ficam nos GitHub Secrets.
- A história real: dois bugs encadeados (`mvnw` sem permissão de executável → `exit 126`, corrigido
  com `git update-index --chmod=+x mvnw`; depois, faltava o `ASAAS_WEBHOOK_TOKEN` no perfil `prod`).
  **Moral: ler o log é a habilidade nº 1 — um erro escondia o outro.**

::callout{type="tip" title="Fixando"}
Antes de seguir, tente responder em voz alta: "por que o CI roda numa máquina **Linux limpa** e
não confia na minha?" e "o que significa um step falhar com `exit code 126`?". Se você souber
dizer *"porque a máquina neutra pega o que eu esqueci de comitar, e 126 é comando encontrado mas
sem permissão de executável"* — você entendeu o coração de CI/CD.
::
