Toda equipe de software já viveu esta cena: o código funciona perfeitamente no computador de
quem escreveu, mas quebra no computador do colega ou no servidor. A frase clássica é **"na minha
máquina funciona"**. Nesta aula você vai entender como o **Docker** acaba com esse problema — e
vamos ler o `Dockerfile` **real** do Descontai, linha a linha.

## O problema: "na minha máquina funciona"

Sua aplicação não roda no vácuo. Para o Descontai subir, o computador precisa ter:

- o **Java 17** instalado (na versão certa),
- todas as **dependências** que o `pom.xml` lista (Spring, PostgreSQL driver, etc.),
- as **variáveis de ambiente** configuradas,
- e o sistema operacional com as ferramentas certas.

Se o seu notebook tem Java 17 mas o servidor tem Java 11, ou se o colega esqueceu de instalar
uma ferramenta, o mesmo código se comporta de forma **diferente**. Cada máquina é um floco de
neve — e isso é uma fábrica de bugs difíceis de reproduzir.

::callout{type="info" title="A ideia central do Docker"}
E se, em vez de mandar só o código e torcer para que a outra máquina esteja configurada igual,
a gente empacotasse a aplicação **junto com tudo que ela precisa para rodar** (Java, dependências,
configuração) numa **caixa lacrada e padronizada**? Essa caixa roda **igualzinho** em qualquer
lugar — seu notebook, o CI, o servidor. É exatamente isso que o Docker faz.
::

## A analogia do contêiner de navio

Antes dos contêineres de navio existirem, cada carga era embarcada de um jeito: sacos, caixas,
barris de tamanhos diferentes. Carregar e descarregar era lento e caótico. Aí padronizaram: uma
**caixa de metal do mesmo tamanho** que qualquer navio, caminhão ou guindaste sabe manusear, **não
importa o que tem dentro**.

O Docker faz o mesmo com software. Ele coloca sua aplicação num **container** padronizado. A
"máquina" que roda esse container (seu PC, o servidor na nuvem) não precisa saber nada sobre Java
ou Spring — ela só sabe rodar containers. O que tem dentro é problema seu, e já vem embalado.

## Container x máquina virtual

Você talvez já tenha ouvido falar em **máquina virtual (VM)** — outra forma de isolar aplicações.
A diferença é importante:

| | Máquina virtual (VM) | Container |
|---|---|---|
| O que virtualiza | Um **computador inteiro** (com sistema operacional próprio) | Só o **processo** da aplicação |
| Peso | Pesada (GBs, minutos para subir) | Leve (MBs, sobe em segundos) |
| Kernel do SO | Cada VM tem o seu | **Compartilham** o kernel do host |
| Isolamento | Total | Isolado, mas mais econômico |

Um **container é apenas um processo isolado** rodando na sua máquina. Ele "acha" que está sozinho
num sistema (tem seu próprio sistema de arquivos, sua própria rede), mas por baixo ele
**compartilha o kernel** do sistema operacional do host. Por isso é tão leve: não precisa carregar
um sistema operacional inteiro para cada aplicação, como uma VM faz.

::callout{type="tip" title="Leve importa de verdade"}
Como containers são leves, o servidor consegue rodar **vários** ao mesmo tempo, e o deploy de uma
nova versão leva **segundos** em vez de minutos. Isso é o que torna prático subir dez cópias da
API para aguentar carga (assunto da Aula 14) e trocar de versão sem derrubar o site.
::

## Imagem x container: o molde e o bolo

Essa dupla é a que mais confunde no começo, então vamos com calma:

- **Imagem**: é a **receita/molde congelado**. É um pacote imutável que contém o sistema de
  arquivos com tudo pronto (Java + o `.jar` da aplicação + config). Você **constrói** uma imagem
  uma vez.
- **Container**: é uma **instância rodando** daquela imagem. É o "bolo" que você assa usando o molde.

Da mesma imagem você pode criar **vários containers** — como fazer vários bolos com o mesmo molde.
A imagem nunca muda; cada container é uma execução viva dela.

::callout{type="info" title="Onde as imagens moram: o Docker Hub"}
Depois de construída, a imagem precisa ficar guardada em algum lugar para o servidor baixar. Esse
lugar é um **registry** — um repositório de imagens, como o GitHub é para código. O mais famoso é
o **Docker Hub**. É de lá que vem a imagem-base do Java que usamos abaixo (`eclipse-temurin`), e é
para um registry que o **CI do Descontai envia** a imagem depois de construí-la (Aula 12).
::

## O Dockerfile: a receita da imagem

O **`Dockerfile`** é um arquivo de texto com as **instruções** para construir a imagem — passo a
passo, de cima para baixo. É a receita escrita. Vamos ler o do Descontai **inteiro** e entender
cada instrução.

Uma coisa importante já de cara: este Dockerfile usa **multi-stage build** (construção em vários
estágios). São **dois** `FROM`, ou seja, duas etapas. Já já você vai entender por que isso é uma
das melhores práticas do mercado.

### Etapa 1 — Build (compilar a aplicação)

```dockerfile
FROM eclipse-temurin:17-jdk AS build

WORKDIR /app

COPY mvnw .
COPY .mvn .mvn
COPY pom.xml .

RUN chmod +x mvnw && ./mvnw dependency:go-offline -B

COPY src src
RUN ./mvnw clean package -DskipTests -B
```

::fields
::field{name="FROM eclipse-temurin:17-jdk AS build"}
Toda imagem parte de uma **imagem-base**. Aqui partimos do `eclipse-temurin:17-jdk`, que já vem
com o **JDK 17** (o kit completo do Java, que sabe **compilar** código). O `AS build` dá um **nome**
a este estágio — vamos precisar dele depois. Ninguém instala Java do zero: a gente "herda" de uma
imagem pronta do Docker Hub.
::
::field{name="WORKDIR /app"}
Define a **pasta de trabalho** dentro do container. A partir daqui, os comandos rodam dentro de
`/app`. É como dar um `cd /app`.
::
::field{name="COPY mvnw . / COPY .mvn .mvn / COPY pom.xml ."}
**Copia** arquivos do projeto para dentro da imagem. Repare que copiamos **primeiro** o Maven
Wrapper (`mvnw`, `.mvn`) e o `pom.xml` — **antes** do código-fonte. Isso é de propósito (veja o
callout sobre cache abaixo).
::
::field{name="RUN chmod +x mvnw && ./mvnw dependency:go-offline -B"}
`RUN` **executa um comando** durante a construção da imagem. Aqui damos permissão de execução ao
`mvnw` e mandamos o Maven **baixar todas as dependências** do `pom.xml`. Como isso vem antes de
copiar o código, o Docker consegue reaproveitar essa camada.
::
::field{name="COPY src src"}
Agora sim copiamos o **código-fonte** (a pasta `src`) para dentro da imagem.
::
::field{name="RUN ./mvnw clean package -DskipTests -B"}
Compila e **empacota** a aplicação num arquivo `.jar` (o "executável" do Java). O `-DskipTests`
pula os testes aqui — porque os testes já rodaram antes, no CI (Aula 11/12); não faz sentido
rodar de novo na hora de empacotar.
::
::

::callout{type="tip" title="Por que copiar o pom.xml antes do código?"}
O Docker constrói a imagem em **camadas** e guarda cada uma em **cache**. Se uma camada não mudou,
ele a reaproveita em vez de refazer. Baixar dependências é **lento**; escrever código muda **toda
hora**. Copiando o `pom.xml` e baixando dependências **primeiro**, enquanto você só mexe no código
(`src`), o Docker reusa a camada de dependências e **não baixa tudo de novo** a cada build. É um
truque de performance que economiza minutos.
::

### Etapa 2 — Runtime (o que de fato vai rodar)

Terminada a compilação, começa o **segundo estágio**. Este é o que vira a imagem **final** — a
que roda em produção.

```dockerfile
FROM eclipse-temurin:17-jre-alpine

# Usuário não-root — reduz superfície de ataque em caso de exploração
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

COPY --from=build /app/target/*.jar app.jar

RUN chown -R appuser:appgroup /app
USER appuser

EXPOSE 8080

ENV SPRING_PROFILES_ACTIVE=prod
ENV JAVA_OPTS=""
```

::fields
::field{name="FROM eclipse-temurin:17-jre-alpine"}
Uma **nova** imagem-base, agora só com o **JRE** (Java Runtime Environment) — o suficiente para
**rodar** Java, sem o compilador. E `alpine` é uma versão de Linux minúscula. Resultado: uma base
bem menor. Este é o segredo do multi-stage (veja o callout).
::
::field{name="RUN addgroup ... && adduser ..."}
Cria um **usuário e grupo comuns** (`appuser`) dentro da imagem. Por padrão, o container rodaria
como **root** (administrador), o que é perigoso.
::
::field{name="COPY --from=build /app/target/*.jar app.jar"}
Aqui está a mágica do multi-stage: `--from=build` copia um arquivo **do estágio anterior** (aquele
que chamamos de `build`). Pegamos **só o `.jar` já compilado** e o trazemos para a imagem enxuta,
renomeando para `app.jar`. Todo o JDK, o Maven e o código-fonte ficam para trás — não vão para a
imagem final.
::
::field{name="RUN chown ... / USER appuser"}
Damos ao `appuser` a posse da pasta `/app` e, a partir do `USER appuser`, **tudo passa a rodar
como esse usuário sem privilégios**, não como root.
::
::field{name="EXPOSE 8080"}
**Documenta** que a aplicação escuta na porta **8080** (a porta padrão do Tomcat embutido do
Spring — Aula 3). É a porta pela qual a API recebe requisições.
::
::field{name="ENV SPRING_PROFILES_ACTIVE=prod / ENV JAVA_OPTS=\"\""}
Define **variáveis de ambiente** padrão. Ativa o perfil `prod` do Spring (Aula 3) e cria um
`JAVA_OPTS` vazio, que serve de "gancho" para passar opções extras à JVM sem editar o Dockerfile.
::
::

::callout{type="warning" title="Por que multi-stage deixa a imagem menor E mais segura"}
A imagem do **JDK** (com compilador, Maven, código-fonte) é grande, e cada ferramenta extra é uma
possível brecha de segurança. Mas nada disso é preciso para **rodar** a aplicação — só o `.jar` e
um Java capaz de executá-lo. O multi-stage compila num estágio "gordo" e **descarta tudo**, levando
apenas o `.jar` para um estágio "magro" com só o JRE. Imagem final **menor** (baixa e sobe mais
rápido) e **mais segura** (menos coisa dentro = menos superfície de ataque). Somado ao **usuário
não-root**, é o padrão de mercado para produção.
::

### O ENTRYPOINT: o comando que sobe a API

```dockerfile
ENTRYPOINT ["sh", "-c", \
  "java \
   -XX:+UseContainerSupport \
   -XX:MaxRAMPercentage=75.0 \
   -Djava.security.egd=file:/dev/./urandom \
   $JAVA_OPTS \
   -jar app.jar"]
```

O **`ENTRYPOINT`** é **o comando que roda quando o container inicia** — a "hora de assar o bolo".
Aqui ele chama o `java -jar app.jar`, ou seja, sobe a aplicação Spring Boot. Mas repare nas duas
flags destacadas:

::fields
::field{name="-XX:+UseContainerSupport"}
Diz à JVM (a máquina virtual do Java) para **perceber que está dentro de um container** e respeitar
os **limites de memória do container**, não os da máquina física inteira.
::
::field{name="-XX:MaxRAMPercentage=75.0"}
Limita o **heap** da JVM a **75% da RAM disponível** para o container. Deixa uma folga para o resto.
::
::

::callout{type="warning" title="Por que a JVM precisa ser avisada do container"}
Historicamente, a JVM olhava a **RAM total da máquina** e alocava memória com base nisso. Num
container que tem só, digamos, 512 MB de limite, a JVM podia achar que tinha 16 GB, alocar demais
e ser **morta** pelo sistema por estourar o limite (o temido `OOMKilled`). O `UseContainerSupport`
faz a JVM **ler o limite real do container**, e o `MaxRAMPercentage` a mantém abaixo desse teto. É
uma configuração pequena que evita crashes misteriosos em produção. (A flag `egd=.../urandom` só
acelera a geração de números aleatórios na inicialização.)
::

### O HEALTHCHECK: "você ainda está viva?"

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1
```

O **`HEALTHCHECK`** ensina o Docker a **checar se a aplicação está saudável**. A cada 30 segundos
ele faz uma requisição ao endpoint `/actuator/health` (o health check do Spring — Aula 3/17). Se a
API responder, está viva; se falhar 3 vezes seguidas, o container é marcado como **unhealthy**.

Os parâmetros: `--interval=30s` (checa a cada 30s), `--timeout=10s` (desiste se demorar mais que
10s), `--start-period=60s` (dá 60s de "graça" no boot, sem contar falhas enquanto a API ainda está
subindo) e `--retries=3` (só considera doente após 3 falhas seguidas).

::callout{type="tip" title="Por que isso importa no deploy"}
Quem gerencia os containers (a nuvem, Aula 16) usa esse sinal para agir sozinho: **não manda
tráfego** para um container ainda não saudável e **reinicia ou substitui** um que ficou doente.
É o que permite o site se **auto-curar** sem alguém acordar de madrugada.
::

## Como isso se encaixa no projeto

O Dockerfile não vive sozinho — ele é a peça que conecta o desenvolvimento à produção:

::steps
### O CI constrói a imagem (Aula 12)
Quando o código é aprovado e mesclado, o pipeline de CI/CD executa este `Dockerfile`, gera a
imagem e a **envia para o registry**. A imagem passa a ser o "artefato" oficial daquela versão.

### O deploy usa essa imagem (Aula 16)
O servidor na nuvem **baixa a imagem** do registry e sobe um (ou vários) **containers** a partir
dela. Como a imagem já traz tudo dentro, o servidor não precisa ter Java nem nada instalado — só
saber rodar containers.

### Roda igual em todo lugar
A **mesma imagem** que passou nos testes é a que vai para produção. Fim do "na minha máquina
funciona": se funciona na imagem, funciona em qualquer lugar que rode aquela imagem.
::

## Resumindo

- O Docker resolve o **"na minha máquina funciona"** empacotando a aplicação **+ tudo que ela
  precisa** numa unidade padronizada.
- Um **container** é um **processo isolado** que compartilha o kernel do host — por isso é **leve**,
  diferente de uma **VM**, que carrega um sistema operacional inteiro.
- **Imagem** é o molde congelado e imutável; **container** é a instância rodando dela. Uma imagem,
  vários containers.
- As imagens ficam guardadas num **registry** (ex.: **Docker Hub**), de onde o CI envia e o deploy baixa.
- O **`Dockerfile`** é a receita. O do Descontai usa **multi-stage**: compila com o JDK e joga fora
  o peso, levando só o `.jar` para uma imagem enxuta com JRE — **menor e mais segura**.
- Boas práticas presentes: **usuário não-root**, JVM ciente do container (**`UseContainerSupport`**
  + **`MaxRAMPercentage`**) e **`HEALTHCHECK`** para auto-cura.
