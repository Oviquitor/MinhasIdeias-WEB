Sua API está no ar, os testes passam, o deploy funciona. Aí chega o dia do lançamento e
**muita gente entra ao mesmo tempo**. A API fica lenta, algumas requisições dão erro… e você
não sabe por quê. Esta aula é sobre isso: **o que acontece por dentro quando a carga aumenta**,
quais são os "canos" que entopem primeiro e como o Descontai foi dimensionado para segurar o
tranco. Tudo com números reais do projeto.

## O que é "carga", afinal?

**Carga** é a quantidade de trabalho que chega na API ao mesmo tempo. Cada usuário que abre o
app e pede a lista de cupons gera uma **requisição** (aquele `GET` da Aula 1). Uma requisição
sozinha é trivial. **Mil ao mesmo tempo** é outra história — porque os recursos do servidor
(CPU, memória, conexões com o banco) são **finitos**.

A pergunta central desta aula é: quando as requisições chegam mais rápido do que a API
consegue processar, **o que acontece com o excesso?** A resposta envolve dois "gargalos"
principais: as **threads** do servidor e o **pool de conexões** do banco.

::callout{type="info" title="A analogia do supermercado"}
Imagine um supermercado. Os **caixas** são as **threads**: cada caixa atende um cliente por vez.
Se há 200 caixas, no máximo 200 clientes são atendidos **simultaneamente**. Os clientes que
chegam quando todos os caixas estão ocupados formam uma **fila** — que é o `accept-count`. E lá
no fundo, o **estoque** (o banco de dados) tem um número limitado de funcionários para buscar
produtos. Guarde essa imagem: ela explica a aula inteira.
::

## Conceito 1: as threads do servidor (Tomcat)

Lembra da Aula 3? O Spring Boot vem com um servidor web embutido chamado **Tomcat**. Quando uma
requisição HTTP chega, o Tomcat **não processa tudo numa fila única** — isso seria lentíssimo.
Em vez disso, ele tem um **pool de threads**.

Uma **thread** é uma "linha de execução": um trabalhador que pega uma requisição, processa do
começo ao fim (consulta o banco, aplica as regras, monta o JSON) e devolve a resposta. Com
várias threads, a API atende **várias requisições ao mesmo tempo** — exatamente como vários
caixas atendendo vários clientes em paralelo.

No Descontai, o `application.properties` define:

```properties
# Tomcat — capacidade de concorrência. threads.max limita o processamento simultâneo;
# accept-count é a fila de conexões aguardando thread livre.
server.tomcat.threads.max=${TOMCAT_THREADS_MAX:200}
server.tomcat.threads.min-spare=${TOMCAT_THREADS_MIN:20}
server.tomcat.accept-count=${TOMCAT_ACCEPT_COUNT:200}
server.tomcat.max-connections=${TOMCAT_MAX_CONNECTIONS:10000}
```

Traduzindo cada linha:

::fields
::field{name="threads.max = 200" type="o número de caixas"}
No máximo **200 requisições** são processadas **ao mesmo tempo** por instância. É o limite de
concorrência real. A requisição 201 não é processada agora — ela **espera**.
::
::field{name="threads.min-spare = 20" type="caixas sempre abertos"}
Mesmo parada, a API mantém **20 threads** prontas. Criar thread custa tempo; ter algumas de
prontidão faz a API responder rápido ao primeiro pico, sem "esquentar".
::
::field{name="accept-count = 200" type="o tamanho da fila"}
Se as 200 threads estão ocupadas, as próximas conexões entram numa **fila de espera** de até
200 lugares. Assim que um caixa (thread) fica livre, ele chama o próximo da fila.
::
::field{name="max-connections = 10000" type="quantos entram no prédio"}
O total de conexões TCP que o Tomcat aceita segurar (em processamento + na fila). É um teto
alto, de proteção: acima disso, o servidor recusa novas conexões em vez de sufocar.
::
::

::callout{type="tip" title="Por que não colocar threads.max = 10000?"}
Parece que "mais threads = aguenta mais gente", mas não é bem assim. Cada thread consome
**memória** e disputa a **CPU**. Se você tem 2 CPUs e 5 mil threads, elas passam mais tempo
brigando pela vez (*context switching*) do que trabalhando — e a API fica **mais lenta**. O
número de threads deve ser proporcional à CPU da máquina e ao **próximo gargalo**: o banco.
::

## Conceito 2: o pool de conexões do banco (HikariCP)

Quase toda requisição do Descontai precisa **falar com o banco** (listar cupons, checar
assinatura, gravar um pagamento). Para conversar com o PostgreSQL, a aplicação abre uma
**conexão** — um canal de rede autenticado com o banco.

Aqui vem a parte importante: **abrir uma conexão nova é caro**. Envolve handshake de rede,
autenticação, alocação de memória no lado do banco… são dezenas de milissegundos. Se cada
requisição abrisse e fechasse a própria conexão, a API gastaria mais tempo **abrindo canos**
do que fazendo trabalho útil.

A solução é o **pool de conexões**: em vez de abrir e fechar, a aplicação abre um punhado de
conexões **uma vez, no início**, e as **reaproveita**. Quando uma requisição precisa do banco,
ela **pega emprestada** uma conexão do pool, usa, e **devolve** para a próxima requisição usar.
O Descontai usa o **HikariCP**, o pool mais rápido do ecossistema Java:

```properties
spring.datasource.hikari.pool-name=DescontaiPool
# Pool default elevado para 20 (era 5) — dimensionar junto ao max_connections do Postgres.
spring.datasource.hikari.maximum-pool-size=${DB_POOL_MAX:20}
spring.datasource.hikari.minimum-idle=${DB_POOL_MIN_IDLE:5}
spring.datasource.hikari.connection-timeout=${DB_POOL_CONN_TIMEOUT:20000}
spring.datasource.hikari.leak-detection-threshold=${DB_POOL_LEAK_THRESHOLD:30000}
```

::callout{type="info" title="A analogia do restaurante"}
O banco de dados é um restaurante com um número **fixo de mesas**. Cada conexão é uma mesa. O
`maximum-pool-size = 20` significa que a API reservou **20 mesas**. Se as 20 estão ocupadas e
chega um 21º pedido, ele **espera** uma mesa vagar (até `connection-timeout = 20000`, ou seja,
20 segundos; passou disso, a requisição falha). Reservar mesa antes (o pool) é muito mais rápido
do que construir uma mesa nova a cada cliente.
::

Repare no detalhe: temos **200 threads** mas só **20 conexões** no pool. Isso é intencional e
nos leva à peça mais importante da aula.

::callout{type="warning" title="O detalhe que salva a produção: leak-detection"}
O `leak-detection-threshold = 30000` faz o Hikari **avisar no log** se alguma requisição segurar
uma conexão por mais de 30 segundos sem devolver. Isso quase sempre é um **bug** (a famosa
"conexão vazada"): a mesa fica ocupada por um cliente que já foi embora, e o pool esvazia sem
motivo. Sem esse alerta, você só descobriria o problema quando a API travasse.
::

## Conceito 3: a regra de ouro do dimensionamento

Aqui está a regra que amarra tudo, e que está escrita **no próprio projeto** (no
`GO-LIVE-10K.md` e nos comentários do `app.yaml`):

::callout{type="warning" title="A REGRA DE OURO"}
**`DB_POOL_MAX × número_de_instâncias  <  max_connections do Postgres`**

O total de conexões que **todas** as suas instâncias podem abrir tem que caber dentro do limite
que o **banco** aceita.
::

Por que isso? Porque o PostgreSQL também tem um teto: o `max_connections`. É o número máximo de
conexões que o **banco inteiro** aceita — de **todas** as aplicações somadas. No restaurante:
não importa quantos garçons (instâncias) você contrate, o salão tem um número fixo de mesas. Se
os garçons tentarem sentar mais gente do que há mesa, o restaurante **recusa** clientes na porta.

Vamos à conta real do Descontai. No `.do/app.yaml`:

```yaml
instance_count: 2
# ...
- key: DB_POOL_MAX
  value: "20"
```

Fazendo as contas:

| Componente | Valor | O que significa |
|-----------|-------|-----------------|
| `DB_POOL_MAX` | **20** | conexões que **cada instância** reserva |
| `instance_count` | **2** | quantas cópias da API rodam em paralelo |
| **Total de conexões da API** | **20 × 2 = 40** | o que a API pode abrir no banco, no pico |
| `max_connections` do Postgres | **≥ ~100** | precisa sobrar folga acima de 40 |

O projeto documenta exatamente isso: *"pool 20 × 2 instâncias = 40 → `max_connections ≥ ~100`"*.

::callout{type="tip" title="Por que ~100 e não exatamente 40?"}
Você **nunca** dimensiona o banco no talo. Precisa de **folga** para: as conexões de
manutenção do próprio Postgres, a ferramenta de backup, você conectando com um cliente SQL para
investigar, um deploy que sobe uma instância nova **antes** de derrubar a antiga (por um instante
existem 3 instâncias, não 2)… Por isso a conta dá 40, mas se pede `max_connections` bem acima.
Estourar o `max_connections` é um dos jeitos mais clássicos de derrubar uma aplicação.
::

## Conceito 4: escalar para cima ou para os lados

Chegou a hora de aguentar mais carga do que uma máquina segura. Existem dois caminhos:

::fields
::field{name="Escala VERTICAL — uma máquina maior" type="scale up"}
Você troca o servidor por um mais **potente**: mais CPU, mais RAM. Continua sendo **uma
instância**, só que mais forte. É simples (não muda nada no código), mas tem **teto físico** — e
máquinas gigantes ficam caríssimas. No `app.yaml`, é subir o `instance_size_slug` (de
`professional-xs` para `-s`, `-m`…).
::
::field{name="Escala HORIZONTAL — mais máquinas" type="scale out"}
Você coloca **mais cópias** da mesma API rodando lado a lado, e um balanceador distribui as
requisições entre elas. É o `instance_count` no `app.yaml`. Escala "infinitamente" e dá
**redundância** (se uma cai, as outras seguem). Mas exige cuidado: cada instância nova soma
conexões no banco (lembra da regra de ouro!) e coisas como cache/rate-limit precisam ser
**compartilhadas** entre instâncias (é por isso que existe o Redis — assunto da próxima aula).
::
::

No Descontai, o `.do/app.yaml` já começa com `instance_count: 2`. O comentário do arquivo explica
o porquê: *"Comece com 2 (redundância / zero-downtime deploy) e aumente conforme o load test."*
Ou seja, **2 instâncias não é (só) sobre carga** — é para que um deploy possa derrubar uma
instância por vez sem a API sair do ar.

::callout{type="info" title="A regra prática de mercado"}
Comece **vertical** (é mais simples) até ficar caro ou bater no teto da máquina. Depois vá
**horizontal**. E lembre: no horizontal, o banco geralmente **não** escala junto tão fácil — ele
costuma ser o gargalo final, o que nos leva ao próximo ponto.
::

## Conceito 5: o que satura primeiro (e como achar o gargalo)

Quando a API começa a sofrer, algo **entope primeiro**. Raramente é a rede ou a CPU da API
diretamente — quase sempre é o **banco/pool**. A cadeia é esta:

::steps
### As threads enchem
As 200 threads ficam todas ocupadas. Novas requisições vão para a fila (`accept-count`).

### O pool esvazia
Mas por que as threads estão presas? Geralmente porque estão **esperando o banco**. Com 200
threads e só 20 conexões, no máximo 20 threads trabalham de verdade — as outras 180 estão
**paradas na fila do pool**, esperando uma conexão vagar.

### O tempo de resposta dispara
Cada requisição agora espera na fila do pool antes de fazer qualquer coisa. O que respondia em
50ms passa a levar segundos. É o famoso **p95 subindo** (o tempo que 95% das requisições
respeitam).

### As requisições começam a falhar
Se a espera passa do `connection-timeout` (20s), a requisição estoura com erro. O usuário vê
"algo deu errado". A API não caiu — ela **saturou**.
::

O gargalo quase nunca é "faltou thread". É "**o banco não dá conta**" — seja porque o pool é
pequeno, seja porque as próprias **consultas SQL estão lentas** (uma query sem índice segura a
conexão por muito tempo, e o pool inteiro trava por causa dela).

::callout{type="tip" title="Como achar o gargalo na prática"}
Você não adivinha — você **mede**. O Descontai expõe métricas via Actuator/Prometheus (Aula 17)
e tem um **teste de carga com k6** (`load-test/k6-descontai.js`) para simular usuários de verdade.
Olhando as métricas sob carga, você vê **qual pool está no talo**: se o pool do Hikari vive em
20/20, o gargalo é o banco → aumente `DB_POOL_MAX` (respeitando a regra de ouro!) ou otimize as
queries. Se a CPU está em 100% e o pool folgado, o gargalo é a máquina → escale vertical. **Mede,
depois ajusta.** Mexer no escuro só troca um problema por outro.
::

## Conceito 6: "10 mil usuários" ≠ "10 mil ao mesmo tempo"

Esse é o erro de dimensionamento **número um** de quem está começando — e o próprio
`GO-LIVE-10K.md` faz questão de deixar isso claro.

Existe uma diferença **enorme** entre:

- **10 mil usuários ativos**: pessoas que usam o app **ao longo do dia**. Elas abrem, olham uns
  cupons, fecham. Espalhadas em 24 horas, a qualquer instante talvez só **algumas dezenas**
  estejam realmente fazendo uma requisição.
- **10 mil usuários simultâneos**: 10 mil requisições **no mesmo segundo**. Isso é um cenário
  bem mais extremo (uma promoção-relâmpago, uma notificação push disparada para todos de uma vez).

Dimensionar para o número errado custa caro dos dois lados: se você provisiona para 10 mil
simultâneos quando na verdade tem 10 mil ativos, **paga por máquinas ociosas**. Se ignora o pico
real, a API **cai** na hora H.

::callout{type="info" title="O dimensionamento real do Descontai"}
O `GO-LIVE-10K.md` diz textualmente: *"**10 mil usuários ativos** (nem todos ao mesmo tempo) é
sustentado por **1 instância bem dimensionada + Postgres gerenciado**. Redis + múltiplas
instâncias só são obrigatórios para **10 mil simultâneos** ou redundância/zero-downtime."*

Ou seja: os defaults do projeto (`DB_POOL_MAX=20`, cache/rate-limit em memória) já seguram os
10 mil **ativos**. Só se o pico simultâneo crescer é que se liga o Redis e sobe o
`instance_count`. **Você dimensiona para o pico real, não para o número de cadastros.**
::

E como você descobre o pico real? De novo: **medindo**. Métricas em produção mostram quantas
requisições por segundo você recebe no horário de maior movimento. É esse número — não o total
de usuários — que decide o tamanho da sua infraestrutura.

## Resumindo

- **Carga** é trabalho chegando ao mesmo tempo; os recursos do servidor são **finitos**, então
  há gargalos.
- **Threads do Tomcat** (`threads.max=200`) = quantas requisições rodam em paralelo por instância;
  o excesso espera na fila (`accept-count=200`). Mais threads não é sempre melhor — disputam CPU.
- **Pool de conexões** (HikariCP, `DB_POOL_MAX=20`) reaproveita conexões porque **abrir conexão é
  caro**; e o banco tem um teto (`max_connections`).
- **Regra de ouro**: `DB_POOL_MAX × instâncias < max_connections`. No Descontai: **20 × 2 = 40**,
  logo `max_connections ≥ ~100` (sempre com folga).
- **Vertical** = máquina maior; **horizontal** = mais instâncias (`instance_count`). Comece
  vertical, escale horizontal — e cuide do banco, que não escala junto.
- **O banco/pool satura primeiro** na maioria dos casos. Não adivinhe o gargalo: **meça** (métricas
  + teste de carga k6).
- **Usuários ativos ≠ simultâneos.** Dimensione para o **pico real**, medido em produção — não
  para o número de cadastros.

::callout{type="tip" title="Fixando"}
Feche os olhos e reconte a história do supermercado com números reais: *"200 caixas (threads),
fila de 200 (accept-count), mas só 20 mesas no restaurante do fundo (pool do banco) — e o
restaurante inteiro só tem ~100 mesas no total (max_connections), então 2 garçons reservando 20
cada dá 40, e cabe."* Se você consegue contar isso, entendeu como uma API aguenta carga.
::
