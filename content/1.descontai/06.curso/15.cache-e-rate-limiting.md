Sua API já funciona. Mas conforme o número de usuários cresce, duas perguntas aparecem:
**"como deixar as respostas mais rápidas?"** e **"como impedir que alguém abuse dela?"**.
As duas respostas moram na mesma pasta do projeto (`infra`) e compartilham uma ideia central —
**guardar coisas na memória** — que nos leva direto a entender o que é o **Redis**. Vamos do zero.

## Parte 1 — O problema que o cache resolve

Pensa na tela de cadastro do Descontai: ela precisa listar as **cidades** disponíveis. Toda vez
que alguém abre essa tela, a API faz `GET /api/v1/cidades`, e por baixo dispara um `SELECT` no
PostgreSQL para buscar a lista.

Só que a lista de cidades **quase nunca muda**. Talvez uma vez por mês alguém adicione uma cidade
nova. Mesmo assim, se 5.000 pessoas abrirem a tela de cadastro hoje, a API vai bater no banco
**5.000 vezes** para devolver exatamente a mesma resposta. Isso é **desperdício**: o banco de
dados é o recurso mais caro e mais lento da aplicação, e a gente está incomodando ele à toa.

::callout{type="info" title="Cache, na analogia do post-it"}
Imagine que alguém te pergunta "qual o telefone da pizzaria?" dez vezes por dia. Na primeira,
você abre a agenda, procura e responde (lento). Aí você **anota num post-it** e cola no monitor.
Nas próximas nove vezes, você só olha o post-it (instantâneo). **Cache** é isso: você guarda o
resultado de uma consulta cara para reaproveitá-lo, em vez de refazer o trabalho toda vez.
::

### A solução: guardar o resultado por um tempo (TTL)

A ideia do cache é simples: na **primeira** vez que alguém pede as cidades, a API busca no banco
**e guarda o resultado numa memória rápida**. Nas próximas vezes, ela devolve o que está guardado,
sem tocar no banco. Muito mais rápido.

Mas tem um detalhe: e se uma cidade nova for cadastrada? O post-it fica **desatualizado**. Por isso
todo cache tem um **TTL** (*Time To Live*, "tempo de vida"): depois de X minutos, o valor guardado
"expira" e é buscado de novo no banco. No Descontai, o TTL é de **10 minutos** — ou seja, uma
mudança demora no máximo 10 minutos para aparecer, o que é aceitável para dados de referência.

## Cache em ação: @Cacheable e @CacheEvict

No Spring, você não escreve a lógica de "olhar o post-it" na mão. Você **anota** o método com
`@Cacheable`, e o Spring cuida do resto:

```java
@Cacheable("cidades")
public List<Cidade> listarTodas() {
    return cidadeRepository.findAll(); // só executa se NÃO estiver no cache
}
```

Na primeira chamada, o método roda e o resultado vai para o cache chamado `"cidades"`. Nas
seguintes (dentro do TTL), o Spring **nem executa o método** — devolve direto o valor guardado.

E quando um admin cadastra uma cidade nova? Aí a gente precisa **rasgar o post-it** para forçar
uma releitura. Isso é o `@CacheEvict`:

```java
@CacheEvict(value = "cidades", allEntries = true)
public Cidade criar(Cidade nova) {
    return cidadeRepository.save(nova); // ao salvar, limpa o cache "cidades"
}
```

O `allEntries = true` esvazia o cache inteiro. Assim, a próxima leitura busca do banco (já com a
cidade nova) e reabastece o post-it.

## O CacheConfig real do Descontai

Quem define **quais** caches existem, o **TTL** e o **backend** é a classe
`infra/config/CacheConfig.java`. Repare nos nomes de cache e no TTL de 10 minutos:

```java
@Configuration
@EnableCaching
public class CacheConfig {

    private static final Duration TTL = Duration.ofMinutes(10);

    public static final String CIDADES = "cidades";
    public static final String PLANOS = "planos";
    public static final String CATEGORIAS = "categorias";
    public static final String TIPOS_PRODUTO = "tiposProduto";

    @Bean
    @ConditionalOnProperty(name = "cache.backend", havingValue = "memory", matchIfMissing = true)
    public CacheManager caffeineCacheManager() {
        CaffeineCacheManager manager = new CaffeineCacheManager(
                CIDADES, PLANOS, CATEGORIAS, TIPOS_PRODUTO);
        manager.setCaffeine(Caffeine.newBuilder()
                .expireAfterWrite(TTL)
                .maximumSize(10_000));
        return manager;
    }
    // ...
}
```

::callout{type="tip" title="Por que só esses quatro caches?"}
`cidades`, `planos`, `categorias` e `tiposProduto` são **dados de referência**: mudam pouco e são
lidos muito (no cadastro, na landing, na montagem do catálogo). São os candidatos perfeitos a cache.
Já os **cupons de um usuário** mudam o tempo todo e são pessoais — cachear isso traria mais dor de
cabeça (dados errados) do que ganho. **A regra: cacheie o que é muito lido e pouco alterado.**
::

## Parte 2 — O que é o Caffeine

Você reparou na anotação `@ConditionalOnProperty(... havingValue = "memory" ...)`. Ela diz: "crie
esse bean **só se** a configuração `cache.backend` for `memory`". E o `memory` usa o **Caffeine**.

**Caffeine** é uma biblioteca de cache **em memória**, dentro da própria aplicação Java. Os dados
ficam guardados na RAM do mesmo processo que roda a API. Isso o torna **extremamente rápido** (não
sai da máquina, não usa rede). É o post-it colado no seu próprio monitor: ninguém mais tem acesso a
ele, mas para você é instantâneo.

## Parte 3 — O que é o Redis

Agora imagine que o post-it não fica colado no **seu** monitor, mas num **quadro branco na parede**
da sala. Qualquer pessoa da equipe pode ler e escrever nele. É **um pouco** mais lento (você
precisa se levantar e ir até a parede), mas **todo mundo enxerga a mesma informação**.

Esse quadro branco compartilhado é o **Redis**.

::callout{type="info" title="Redis, do zero"}
**Redis** é um banco de dados **chave-valor** (você guarda um valor associado a uma chave, tipo
`cidades → [Recife, Olinda]`) que roda **na memória** (RAM), por isso é **super-rápido**. A grande
diferença para o Caffeine: o Redis roda como um **programa separado, FORA da sua aplicação**, e por
isso pode ser **compartilhado por várias instâncias** da API ao mesmo tempo.
::

| | **Caffeine** | **Redis** |
|---|---|---|
| Onde os dados moram | Na RAM da própria aplicação | Num servidor separado |
| Velocidade | Máxima (sem rede) | Muito alta (via rede local) |
| Compartilhado entre instâncias? | **Não** (cada uma tem o seu) | **Sim** (todas veem o mesmo) |
| Precisa instalar algo a mais? | Não (é só uma dependência) | Sim (um servidor Redis) |
| Analogia | Post-it no seu monitor | Quadro branco na parede |

## Parte 4 — O ponto-chave da escala: 1 instância x várias

Aqui está o motivo de tudo isso existir. Uma "instância" é uma cópia da sua API rodando. Quando o
tráfego cresce, você não deixa uma única instância sofrendo: você sobe **várias cópias** e coloca um
balanceador de carga distribuindo os pedidos entre elas (isso é **escala horizontal**, tema da
Aula 16).

**Com 1 instância**, o Caffeine basta. Só existe um post-it, e ele é sempre o mesmo.

**Com várias instâncias**, cada uma tem o **seu próprio** Caffeine, isolado. E aí começam os
problemas:

- **No cache**, é um desperdício menor: cada instância bate no banco uma vez para encher o seu
  post-it. Chato, mas tolerável.
- **No rate limiting**, é um **buraco de segurança**. Se a instância A conta "esse IP já tentou
  login 5 vezes", a instância B não sabe de nada. Um atacante que espalha as tentativas entre as
  instâncias **burla o limite** — cada uma deixa passar 5, e no total passam 15, 20...

::callout{type="warning" title="A regra de ouro"}
Contadores e caches **em memória** são isolados por instância. No momento em que você roda **mais de
uma instância**, qualquer estado que precisa ser **consistente entre elas** (principalmente o
contador do rate limiting) tem que sair da memória local e ir para um lugar **compartilhado** — o
**Redis**.
::

### O padrão: backend selecionável por configuração

O detalhe elegante do Descontai é que **a mesma aplicação** roda com Caffeine ou com Redis — quem
decide é uma **configuração**, não uma mudança de código. Isso é feito com `@ConditionalOnProperty`:
o Spring lê a propriedade e só cria o bean correspondente.

```java
// Cria este bean SÓ se cache.backend=memory (ou se a propriedade não existir)
@ConditionalOnProperty(name = "cache.backend", havingValue = "memory", matchIfMissing = true)
public CacheManager caffeineCacheManager() { ... }

// Cria este outro SÓ se cache.backend=redis
@ConditionalOnProperty(name = "cache.backend", havingValue = "redis")
public CacheManager redisCacheManager(RedisConnectionFactory connectionFactory) { ... }
```

E no `application.properties`, a escolha vem de uma variável de ambiente (lembra do padrão
`${VARIAVEL:default}` da Aula 3?):

```properties
# CACHE / RATE LIMIT (backend selecionável)
# memory (default, Caffeine) ou redis (distribuído entre instâncias)
cache.backend=${CACHE_BACKEND:memory}
rate-limit.backend=${RATE_LIMIT_BACKEND:memory}

# REDIS (usado apenas quando cache.backend/rate-limit.backend = redis)
spring.data.redis.host=${REDIS_HOST:localhost}
spring.data.redis.port=${REDIS_PORT:6379}
spring.data.redis.password=${REDIS_PASSWORD:}
```

::callout{type="tip" title="Por que isso é tão bom?"}
Em desenvolvimento, você roda **1 instância** e o default `memory` funciona sem instalar Redis
nenhum. Em produção com **várias instâncias**, você só define `CACHE_BACKEND=redis` e
`RATE_LIMIT_BACKEND=redis` no ambiente — **sem recompilar nada**. Um único código, dois modos de
operar. É a filosofia "configuração vem do ambiente" levada a sério.
::

## Parte 5 — Rate Limiting

**Rate limiting** é limitar **quantas vezes** alguém pode fazer uma ação num período. O caso clássico
do Descontai: **tentativas de login por IP**. Sem limite, um atacante pode tentar milhares de senhas
por segundo até acertar (ataque de **força bruta**). Com rate limiting, depois de poucas tentativas
erradas, o IP é **bloqueado por um tempo**.

::callout{type="info" title="Rate limit é a catraca"}
Pensa numa **catraca de metrô** que só deixa passar 5 pessoas por vez, depois trava por 10 minutos.
Não importa o quão rápido você empurre — a catraca conta e barra. O rate limiter é essa catraca na
porta dos endpoints sensíveis.
::

### O algoritmo "token bucket" (Bucket4j), simplificado

O algoritmo usado no modo memória se chama **token bucket** ("balde de fichas"), fornecido pela
biblioteca **Bucket4j**. A ideia:

::steps
### Cada IP ganha um balde com fichas
No Descontai, o balde começa com **5 fichas** (a capacidade).

### Cada tentativa gasta 1 ficha
Tentou logar → tira uma ficha do balde. Enquanto houver fichas, passa.

### Sem fichas, a catraca trava
Se o balde zerou, as próximas tentativas são **recusadas**.

### O balde se reabastece
A cada **10 minutos**, o balde volta a ter 5 fichas. A catraca destrava sozinha.
::

### As duas implementações do RateLimiterStore

Assim como no cache, o rate limiter tem uma **abstração** (`RateLimiterStore`) e duas
implementações escolhidas por configuração. Primeiro a interface, que esconde "onde" o contador mora:

```java
public interface RateLimiterStore {
    boolean tryConsume(String key); // true = dentro do limite; false = cota excedida
}
```

Quem chama (o `LoginRateLimiter`) não sabe nem se importa qual backend está ativo — ele só chama
`tryConsume("login:" + ip)`. Isso é a **injeção de dependência** e as **interfaces** da Aula 4 em
ação: trocar o backend não toca em quem usa.

**Backend em memória** (`CaffeineRateLimiterStore`) — padrão, Bucket4j + Caffeine:

```java
@Component
@ConditionalOnProperty(name = "rate-limit.backend", havingValue = "memory", matchIfMissing = true)
public class CaffeineRateLimiterStore implements RateLimiterStore {

    static final int CAPACIDADE = 5;
    static final int JANELA_MINUTOS = 10;

    private final Cache<String, Bucket> buckets = Caffeine.newBuilder()
            .expireAfterAccess(Duration.ofMinutes(JANELA_MINUTOS + 1))
            .maximumSize(100_000)
            .build();

    @Override
    public boolean tryConsume(String key) {
        return buckets.get(key, k -> Bucket.builder()
                .addLimit(Bandwidth.classic(CAPACIDADE, Refill.intervally(CAPACIDADE, JANELA)))
                .build()).tryConsume(1);
    }
}
```

**Backend distribuído** (`RedisRateLimiterStore`) — um contador simples no Redis, com `INCR` +
`EXPIRE`, compartilhado por todas as instâncias:

```java
@Component
@ConditionalOnProperty(name = "rate-limit.backend", havingValue = "redis")
public class RedisRateLimiterStore implements RateLimiterStore {

    static final int CAPACIDADE = 5;
    private static final String PREFIXO = "ratelimit:";

    @Override
    public boolean tryConsume(String key) {
        String redisKey = PREFIXO + key;
        Long count = redis.opsForValue().increment(redisKey); // +1 atômico
        if (count == null) {
            return true; // falha ao falar com o Redis: não bloqueia usuário legítimo (fail-open)
        }
        if (count == 1L) {
            redis.expire(redisKey, JANELA); // primeira vez: liga o TTL da janela
        }
        return count <= CAPACIDADE;
    }
}
```

::callout{type="warning" title="Dois detalhes de gente grande"}
1. **Evicção por tempo no Caffeine** (`expireAfterAccess`): baldes de IPs que pararam de tentar são
apagados sozinhos. Sem isso, guardar um balde por IP para sempre seria um **vazamento de memória**.
2. **Fail-open no Redis**: se o Redis estiver fora do ar, o código devolve `true` (deixa passar) em
vez de barrar todo mundo. A escolha aqui foi **não derrubar o login legítimo** por causa de um
problema de infraestrutura — uma decisão de projeto que vale entender.
::

## Resumindo

- **Cache** guarda resultados de consultas caras (ex.: lista de cidades) para não bater no banco
  toda vez. Cada valor expira após um **TTL** (10 min no Descontai).
- No Spring, `@Cacheable` guarda e `@CacheEvict` limpa. O `CacheConfig` define os caches
  (`cidades`, `planos`, `categorias`, `tiposProduto`).
- **Caffeine** é cache em memória, **dentro** da aplicação (rápido, mas isolado por instância).
- **Redis** é um banco chave-valor na memória que roda **fora** da aplicação e é **compartilhado**
  por várias instâncias.
- **A escala manda:** 1 instância → memória basta; **várias instâncias** → precisa de estado
  **compartilhado** (Redis), senão o cache diverge e o rate limit vira **burlável**.
- O **backend é selecionável por configuração** (`@ConditionalOnProperty` + `cache.backend` /
  `rate-limit.backend`): o mesmo código roda em memory ou redis sem recompilar.
- **Rate limiting** limita tentativas (ex.: login por IP contra força bruta) via **token bucket**
  (Bucket4j), com implementações Caffeine e Redis por trás da interface `RateLimiterStore`.
