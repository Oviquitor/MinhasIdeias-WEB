Até agora o Descontai rodava na sua máquina, na porta 8080, e só você acessava. Nesta aula a
gente **coloca a API no ar** — num servidor na internet, com um endereço público, respondendo
ao app e aos portais de verdade. É o momento em que o projeto sai do "funciona aqui" e vira um
serviço que o mundo consome.

## O que é "fazer deploy"

**Fazer deploy** é pegar a sua aplicação e colocá-la para rodar num computador ligado 24/7 na
internet, com um endereço que qualquer cliente consegue alcançar. Enquanto o código só roda no
seu notebook, ninguém de fora chega nele — feche o notebook e a API "morre". Deploy é o oposto
disso: um servidor que **não desliga**, acessível de qualquer lugar.

Lembra da Aula 1? O app Flutter é o **cliente** e a API é o **servidor**. Deploy é literalmente
o ato de dar ao servidor um lugar fixo na internet para ele ficar esperando os pedidos.

::callout{type="info" title="Onde o Descontai roda"}
A API do Descontai fica hospedada no **DigitalOcean**, num serviço chamado **App Platform**. O
app manda `GET /api/v1/cidades` para um endereço público (ex.: `https://api.descontai.com`) e é
essa máquina na nuvem que responde — não mais o seu `localhost:8080`.
::

## O que é "a nuvem" — e IaaS x PaaS

"Nuvem" parece coisa mágica, mas é simples: são **computadores de outra pessoa** (Amazon,
Google, DigitalOcean…) que você **aluga** pela internet, em vez de comprar um servidor físico e
mantê-lo você mesmo. Você paga pelo que usa e liga/desliga quando quiser.

O que muda bastante é **quanto trabalho** de infraestrutura sobra para você. Aí entram duas
siglas que todo dev ouve:

::fields
::field{name="IaaS — Infraestrutura como Serviço"}
A nuvem te entrega uma **máquina virtual crua** (um servidor Linux vazio). Você instala o Java,
configura o servidor web, cuida de firewall, atualizações de segurança, reinício quando cai…
Máximo controle, máximo trabalho.
::
::field{name="PaaS — Plataforma como Serviço"}
Você entrega o **código (ou a imagem Docker)** e a plataforma cuida do resto: provisiona o
servidor, sobe sua app, reinicia se cair, coloca HTTPS. Menos controle sobre a máquina, muito
menos trabalho.
::
::

::callout{type="tip" title="A analogia do apartamento"}
**IaaS** é comprar um terreno e **construir a casa**: você escolhe tudo, mas encana, fia a luz e
conserta o telhado sozinho. **PaaS** é **alugar um apartamento mobiliado**: já vem com água, luz,
portaria e manutenção — você só chega com suas coisas e mora. O Descontai mora num apartamento
mobiliado: o **App Platform** é um **PaaS**. A gente entrega a imagem Docker (da Aula 13) e ele
cuida do servidor.
::

## O spec: `.do/app.yaml`

No App Platform, em vez de clicar em mil botões no painel, a gente descreve **toda a
infraestrutura num arquivo** — o `.do/app.yaml`. Isso se chama *infraestrutura como código*: o
mesmo arquivo versionado no Git descreve exatamente como a app roda. Vamos percorrer o arquivo
real do Descontai, bloco por bloco.

### Cabeçalho: nome e região

```yaml
name: descontai-api
region: nyc
```

`name` é o nome da app dentro do App Platform. `region` é o **data center físico** onde ela vai
rodar — `nyc` é Nova York. Quanto mais perto dos usuários, menor a latência.

### `services`: a nossa API

```yaml
services:
  - name: api
    github:
      repo: Koomit-Software/Descontai-API
      branch: main
      deploy_on_push: true
    dockerfile_path: Dockerfile
```

Um **service** é um processo que fica rodando e atende requisições — no nosso caso, a API. Repare
em três coisas importantes:

::fields
::field{name="github + deploy_on_push: true"}
Aponta para o repositório e a branch `main`. Com `deploy_on_push: true`, **todo push na `main`
dispara um deploy automático**. Você empurra o código → a plataforma reconstrói e sobe a nova
versão sozinha. Isso é o famoso **CD (Continuous Deployment)** da Aula 12: o CI testa, e o CD
publica sem ninguém apertar botão.
::
::field{name="dockerfile_path: Dockerfile"}
Diz para a plataforma **empacotar a app usando o Dockerfile** — exatamente a imagem que você
construiu na Aula 13. A nuvem não precisa saber que é Java: ela só constrói a imagem e roda o
container. Mesma imagem que roda na sua máquina roda em produção.
::
::

### Escala: `instance_count` e `instance_size_slug`

```yaml
    instance_count: 2
    instance_size_slug: professional-xs
    http_port: 8080
```

Aqui está a **escala** (o tema da Aula 14):

- `instance_count: 2` → sobem **duas cópias** da API rodando em paralelo. Duas cópias trazem
  **redundância**: se uma cai ou está sendo atualizada, a outra continua atendendo.
- `instance_size_slug: professional-xs` → o **tamanho** de cada cópia (CPU e memória). Como diz o
  próprio comentário do arquivo, você "sobe para -s/-m conforme o p95 do load test" (Aula 17).
- `http_port: 8080` → a porta em que a API escuta dentro do container (a mesma do Spring Boot).

::callout{type="info" title="Escala horizontal x vertical"}
Aumentar `instance_count` (mais cópias) é **escala horizontal**. Aumentar `instance_size_slug`
(cópias maiores) é **escala vertical**. O `.do/app.yaml` deixa os dois a um ajuste de distância —
você decide com base no teste de carga, não no chute.
::

### `health_check`: a plataforma vigia a API

```yaml
    health_check:
      http_path: /actuator/health
      initial_delay_seconds: 60
      period_seconds: 15
      timeout_seconds: 5
      success_threshold: 1
      failure_threshold: 3
```

O App Platform fica **batendo periodicamente** na rota `/actuator/health` (a mesma que o Spring
Boot Actuator expõe — Aula 17) para saber se a instância está saudável. Lendo os campos:

| Campo | O que significa |
|-------|-----------------|
| `http_path` | Qual rota checar: `/actuator/health` |
| `initial_delay_seconds: 60` | Espera 60s após subir antes de começar a checar (dá tempo do Spring iniciar) |
| `period_seconds: 15` | Checa a cada 15 segundos |
| `timeout_seconds: 5` | Se não responder em 5s, conta como falha |
| `failure_threshold: 3` | 3 falhas seguidas → a instância é considerada doente |

Se uma instância falhar 3 checagens seguidas, a plataforma **mata e reinicia** essa instância
automaticamente. Você não precisa acordar 3h da manhã para reiniciar nada — a nuvem faz isso.

### `envs`: variáveis de ambiente (e os segredos)

Lembra da Aula 3: o `application.properties` lê tudo de **variáveis de ambiente**
(`${DB_URL}`, `${JWT_SECRET}`…), nunca deixa segredo escrito no código. É aqui, no bloco `envs`,
que essas variáveis são definidas para produção.

```yaml
    envs:
      - key: SPRING_PROFILES_ACTIVE
        value: prod
      - key: DB_URL
        value: ${db.JDBC_URL}
      - key: DB_USER
        value: ${db.USERNAME}
      - key: DB_PASS
        value: ${db.PASSWORD}
      - key: DB_POOL_MAX
        value: "20"
      - key: FLYWAY_ENABLED
        value: "true"
      - key: SWAGGER_ENABLED
        value: "false"
      - key: APP_CORS_ALLOWED_ORIGINS
        value: https://empresas.descontai.com,https://admin.descontai.com
```

Repare em valores importantes: `SPRING_PROFILES_ACTIVE=prod` liga o perfil de produção;
`SWAGGER_ENABLED=false` esconde a documentação interna do público; e `APP_CORS_ALLOWED_ORIGINS`
lista **exatamente** os portais que podem chamar a API pelo navegador (Empresas e Admin) — nunca
`*`, como o `DEPLOY.md` avisa.

Agora a parte crítica. Alguns valores **não têm `value`** — têm `type: SECRET`:

```yaml
      - key: JWT_SECRET
        scope: RUN_TIME
        type: SECRET
      - key: ASAAS_API_KEY
        scope: RUN_TIME
        type: SECRET
      - key: ASAAS_WEBHOOK_TOKEN
        scope: RUN_TIME
        type: SECRET
      - key: MAIL_PASSWORD
        scope: RUN_TIME
        type: SECRET
```

Esses são os **segredos**: a chave que assina os tokens JWT, as credenciais do Asaas (pagamento)
e a senha do e-mail. Eles aparecem no arquivo **só como nome** — o valor real é digitado à parte,
no painel do App Platform, e a plataforma o guarda **criptografado**. Assim o segredo entra na app
como variável de ambiente em tempo de execução (`scope: RUN_TIME`), mas **nunca é escrito no
repositório**.

::callout{type="warning" title="Segredo NÃO vai para o Git — nunca"}
O próprio topo do `.do/app.yaml` avisa: *"valores sensíveis (JWT_SECRET, ASAAS_\*, MAIL_\*) NÃO
ficam aqui"*. Se você commitar uma senha ou uma chave de API, ela fica no **histórico do Git para
sempre** — qualquer um com acesso ao repositório a vê, e trocar a chave depois é dor de cabeça.
Por isso segredo é sempre `type: SECRET` (definido no painel) e nunca `value: ...` no arquivo. O
`GO-LIVE-10K.md` lista esses quatro como obrigatórios: sem eles, a API **se recusa a subir** em
`prod` — uma trava de segurança de propósito.
::

### `databases`: o banco gerenciado

```yaml
databases:
  - name: db
    engine: PG
    version: "16"
    production: true
```

Este bloco pede à DigitalOcean um **PostgreSQL 16 gerenciado**. "Gerenciado" (*managed*) é a
palavra-chave: a nuvem cuida de **backup automático, alta disponibilidade e atualizações** do
banco. Você não instala Postgres nem faz backup na mão — é o apartamento mobiliado de novo,
agora para o banco de dados.

E como a API descobre o endereço e a senha do banco? Repare lá em cima, no `envs`:

```yaml
      - key: DB_URL
        value: ${db.JDBC_URL}
      - key: DB_USER
        value: ${db.USERNAME}
      - key: DB_PASS
        value: ${db.PASSWORD}
```

O `${db.JDBC_URL}` é uma **referência ao componente de banco** chamado `db`. A plataforma
**injeta sozinha** a URL, o usuário e a senha do Postgres que ela criou. Você nunca digita a
credencial do banco em lugar nenhum: ela nasce no App Platform e é entregue à app na hora.

::callout{type="info" title="A regra do pool x max_connections (Aula 14)"}
Cada instância abre um **pool** de conexões com o banco (`DB_POOL_MAX: "20"`). Com 2 instâncias,
são até `20 × 2 = 40` conexões. A regra de ouro do `DEPLOY.md` é:
**`DB_POOL_MAX × instance_count < max_connections do Postgres`**. Se o banco aguenta ~100
conexões, 40 estão folgadas. Estoure isso e novas instâncias não conseguem conectar — por isso o
número de instâncias e o pool andam juntos.
::

### O Redis opcional (comentado)

No fim do `envs` e do `databases` há blocos **comentados** de Redis:

```yaml
      # - key: CACHE_BACKEND
      #   value: redis
      # - key: RATE_LIMIT_BACKEND
      #   value: redis
      # - key: REDIS_HOST
      #   value: ${redis.HOSTNAME}
```

Estão desligados de propósito. Com **1 instância**, cache e rate limit em memória (Caffeine)
bastam. Mas com **múltiplas instâncias**, cada cópia teria seu próprio contador de rate limit e
seu próprio cache — inconsistente. Aí você provisiona um **Redis gerenciado** (compartilhado
entre as instâncias) e descomenta esses blocos. Deixar pronto e comentado é uma cortesia com o
seu "eu do futuro": quando precisar escalar, é só tirar o `#`.

## Rollback e zero-downtime

Duas ideias que só fazem sentido quando você tem mais de uma instância (`instance_count: 2`):

- **Zero-downtime (sem tempo fora do ar)**: ao subir uma versão nova, a plataforma não derruba as
  duas cópias de uma vez. Ela sobe a nova, confirma no `health_check` que está saudável, **só
  então** tira a antiga — e faz isso uma instância de cada vez. Enquanto uma atualiza, a outra
  atende. O usuário não percebe o deploy acontecendo.
- **Rollback (voltar atrás)**: se um deploy novo apresentar problema, o App Platform guarda a
  versão anterior e permite **reverter** para ela com um clique. Como cada deploy é uma imagem
  Docker imutável (Aula 13), "voltar" é literalmente subir a imagem antiga de novo — rápido e
  previsível.

::callout{type="tip" title="Por que 2 instâncias, então?"}
Não é só sobre aguentar mais tráfego. Duas instâncias dão **redundância** (se uma morre, a outra
segura) e **deploy sem downtime** (atualiza uma de cada vez). Por isso o comentário do arquivo diz
"comece com 2 (redundância / zero-downtime deploy)". Segundo o `GO-LIVE-10K.md`, 10 mil usuários
*ativos* aguentam até com 1 instância bem dimensionada + Postgres gerenciado — múltiplas
instâncias entram para redundância e picos de *simultâneos*.
::

## Resumindo

- **Fazer deploy** é colocar a API para rodar 24/7 num servidor na internet, com endereço público.
- **Nuvem** é alugar computadores de outra pessoa. **IaaS** te dá a máquina crua (você faz tudo);
  **PaaS** cuida do servidor por você — o Descontai usa o **App Platform (PaaS)** da DigitalOcean.
- O `.do/app.yaml` descreve a infra como código: `services` (a API), `github` com
  `deploy_on_push: true` (**CD** — push na `main` faz deploy sozinho), `dockerfile_path` (a imagem
  da Aula 13), `instance_count`/`instance_size_slug` (escala) e `health_check` (a plataforma
  reinicia a instância se `/actuator/health` falhar).
- O bloco `envs` define variáveis de ambiente; segredos (`JWT_SECRET`, `ASAAS_*`, `MAIL_PASSWORD`)
  são `type: SECRET` — definidos no painel, **nunca no Git**.
- O bloco `databases` provisiona um **Postgres 16 gerenciado** (backup + alta disponibilidade),
  com credenciais injetadas via `${db.JDBC_URL}`. Lembre da regra
  `DB_POOL_MAX × instance_count < max_connections`.
- Com múltiplas instâncias você ganha **zero-downtime** (atualiza uma de cada vez) e **rollback**
  fácil (subir a imagem anterior). O Redis fica comentado, pronto para quando escalar.
