## Parte 1 — Os padrões de mercado que o curso ensinou

Um "padrão" (pattern) é uma **solução conhecida para um problema que se repete**. Você não
inventa do zero: você reconhece o problema e aplica a receita que o mercado já validou.
Repare que **nenhum** dos padrões abaixo é enfeite — cada um resolve uma dor concreta.

### Clean Architecture / camadas
Separar o código em **domínio, aplicação, infraestrutura e API** (como na Aula 4) evita que
tudo vire uma bola de espaguete onde mexer numa coisa quebra outra. A regra de ouro: as
camadas de dentro (o domínio) **não sabem** nada sobre as de fora (banco, HTTP). Isso deixa a
regra de negócio testável e independente de framework.

### O padrão UseCase
Cada ação do sistema ("criar cupom", "renovar assinatura", "fazer login") vira uma classe de
**caso de uso** (Aula 6). Como diz o próprio `package-info` do projeto: *"o controller chama o
use case; o use case coordena services, repositories e mappers"*. Uma classe = uma ação. Fica
óbvio onde a regra vive e fácil de testar isolado.

### O padrão Repository
Você **nunca** escreve SQL solto pelo código. Toda conversa com o banco passa por uma interface
de repositório (Aula 5). Isso esconde o "como buscar" e deixa o resto do código pedindo em
alto nível: `usuarioRepository.findByEmail(...)`. Trocar o banco ou otimizar uma query não
vaza para quem usa.

### DTOs na borda
Os objetos que entram e saem pela API (**DTOs**, Aula 6) não são as entidades do banco. Isso
protege o domínio: você escolhe exatamente o que expõe (nunca o `passwordHash`, por exemplo) e
valida o que entra **na borda**, antes de chegar na regra de negócio.

### Gateway para serviços externos
Toda integração com terceiros (pagamento Asaas, push do Firebase — Aula 8) passa por um
**gateway**. O `package-info` do projeto explica o porquê: *"para evitar espalhar detalhes de
autenticação, URLs e payloads externos pelo domínio"*. Se o fornecedor mudar, você troca **um**
arquivo.

### Configuração via ambiente + segredos fora do código
Senhas, tokens e URLs vêm de **variáveis de ambiente**, nunca escritos no código (Aula 3 e 16).
Lembra do `${DB_URL}`? É o princípio [12-Factor](https://12factor.net/config): o mesmo código
roda em qualquer lugar, só muda o ambiente. **Segredo no Git é vazamento garantido.**

### Fail-fast na inicialização
Se falta uma configuração crítica (ex.: o segredo do webhook de pagamento em produção), a API
**se recusa a subir** — em vez de subir quebrada e falhar só quando um cliente paga (Aula 16).
Falhar cedo e alto é muito melhor do que falhar tarde e escondido.

### Migrations aditivas
A estrutura do banco é versionada com Flyway (Aula 9), e cada mudança é um arquivo novo que
**soma** ao schema — nunca reescreve o passado. Isso deixa o banco reconstruível do zero e
seguro para times: ninguém "esquece" de aplicar uma alteração.

### Testes em pirâmide
Muitos testes unitários rápidos (Mockito) na base, menos testes de integração (Testcontainers)
no topo (Aula 11). A pirâmide dá **confiança sem lentidão**: você mexe no código sabendo na
hora se quebrou algo.

### CI/CD como rede de segurança
Cada mudança passa por um pipeline no GitHub Actions que compila, testa e valida antes de ir
para produção (Aula 12). É a rede embaixo do trapezista: você arrisca mais porque sabe que, se
cair, o pipeline te segura antes de o usuário sentir.

### Backend selecionável (memory/redis)
Cache e rate limit funcionam com Caffeine na memória **ou** Redis distribuído, escolhido por
configuração (Aula 15). Um dev sobe a API no notebook sem instalar Redis; em produção, liga o
Redis para escalar. **Mesmo código, dois modos** — decidido pelo ambiente, não por `if` no meio
da lógica.

::callout{type="info" title="Por que memorizar os nomes importa"}
Numa entrevista ou num PR, dizer *"isolei a integração num gateway"* ou *"validei no DTO da
borda"* mostra que você pensa em padrões, não em gambiarra. Os nomes são o **vocabulário
comum** do backend — eles fazem você ser entendido por qualquer time.
::

## Parte 2 — As boas práticas de código do projeto

Padrão é a arquitetura; **boa prática** é o cuidado no dia a dia da escrita. Foi isso que fez o
código do Descontai ser agradável de ler ao longo do curso.

::fields
::field{name="Nomes claros"}
`findByEmail`, `LoginUseCase`, `saldoIndicacao`. O nome diz o que a coisa faz **sem precisar
abrir**. Nome bom é documentação que nunca fica desatualizada.
::
::field{name="Comentários que explicam o PORQUÊ"}
Repare no `BaseEntity`: o comentário não diz *"este é o id"* (óbvio) — ele explica *"usamos
IDENTITY para delegar ao PostgreSQL, evitando a tabela de sequência extra"*. O código já mostra
o **o quê**; o comentário guarda a **decisão**.
::
::field{name="Conventional Commits"}
Os commits seguem um padrão: `feat(api): ...`, `fix(api): ...`, `refactor(api): ...`,
`test(api): ...`, `docs: ...`. O histórico vira uma linha do tempo legível — dá para saber o
que mudou e por quê só lendo os títulos.
::
::field{name="Código pequeno e coeso"}
Cada classe tem **uma** responsabilidade. Um use case faz uma ação; um gateway fala com um
serviço; uma entidade guarda dados. Arquivo pequeno é arquivo fácil de entender e testar.
::
::field{name="package-info documentando cada pacote"}
Cada pasta tem um `package-info.java` que explica, em português, **o papel daquele pacote**.
Quem chega no projeto lê o `package-info` e já sabe onde as coisas moram — sem precisar caçar.
::
::

::callout{type="warning" title="A regra do comentário"}
O erro clássico do júnior é comentar o óbvio: `// soma 1 ao contador` em cima de `contador++`.
Isso é ruído. Comente quando a resposta para *"por que assim e não do jeito mais simples?"* não
está no código — como o comentário do `@Version` no `Usuario`, que explica que existe para
evitar duas atualizações concorrentes se sobrescreverem em silêncio.
::

## Parte 3 — Roteiro: construa a SUA API do zero à produção

Chegou a hora de virar autor. Aqui está o caminho completo, aplicando **tudo** que você
aprendeu. Não tente fazer tudo num dia — cada passo é um marco. E sempre que travar, **volte na
aula correspondente**: ela tem o código real do Descontai como referência.

::callout{type="tip" title="Antes de começar"}
Escolha um domínio **pequeno e que você entenda**. Nada de "clone do Instagram". Pense em algo
como: uma lista de tarefas, um catálogo de livros pessoais, um controle de gastos, um cadastro
de plantas com lembrete de rega. Simples de propósito — o objetivo é **aplicar a arquitetura**,
não impressionar.
::

::steps
### Escolha um domínio simples
Defina 1 ou 2 entidades principais e o que o app faz com elas. Ex.: `Tarefa` (título, feita,
prazo) e `Usuario`. Escreva em uma frase: *"uma API onde cada usuário cria e marca tarefas
como feitas"*. Ter o domínio claro na cabeça é meio caminho (como na Aula 1, pensando em
recursos).

### Crie o projeto Spring Boot
Use o [start.spring.io](https://start.spring.io) e adicione os starters: `web`, `data-jpa`,
`postgresql`, `security`, `validation`, `flyway`. Rode `./mvnw spring-boot:run` e confirme que
sobe na porta 8080 (como na Aula 3). Já organize as pastas em camadas: `api`, `application`,
`domain`, `infra`, `security` (como na Aula 4).

### Modele as entidades + migrations
Crie sua `BaseEntity` com id/createdAt/updatedAt e suas entidades JPA (como na Aula 5). Para
cada tabela, escreva uma **migration Flyway** aditiva em `db/migration` (como na Aula 9). Suba
a API com o banco vazio e veja o Flyway criar tudo — a sensação de "subir do zero" é ótima.

### Crie os repositórios
Para cada entidade, uma interface que estende `JpaRepository` (como na Aula 5). Adicione as
buscas de que precisa (`findByUsuarioId`, etc.) só declarando o método — o Spring implementa.
Nada de SQL solto no resto do código.

### Monte um CRUD completo (controller → usecase → service → dto)
Escolha uma entidade e faça o fluxo inteiro (como na Aula 6): DTOs de entrada/saída com
validação (`@NotBlank`, etc.), um use case por ação (criar, listar, atualizar, apagar), o
service com a regra, e o controller REST expondo `GET/POST/PUT/DELETE`. Teste tudo no Postman
(como na Aula 2).

### Adicione autenticação JWT
Cadastro com senha em **BCrypt**, login que devolve um **token JWT**, e um filtro que protege
as rotas privadas (como na Aula 7). Faça o dono só enxergar as próprias tarefas — é o seu
primeiro gostinho de autorização de verdade.

### Escreva os testes
Testes unitários dos use cases com **Mockito** (mockando o repositório) e um teste de
integração de uma rota com **Testcontainers** subindo um Postgres real (como na Aula 11). Não
precisa de 100% — cubra o caminho feliz e os erros importantes. Monte a pirâmide.

### Escreva o Dockerfile
Empacote a aplicação num container com um `Dockerfile` (como na Aula 13). Rode localmente com
`docker build` + `docker run` e acesse a API do container. Aqui morre o "na minha máquina
funciona".

### Configure o CI no GitHub Actions
Suba o projeto no GitHub e crie um `.github/workflows/ci.yml` que, a cada push, compila e roda
os testes (como na Aula 12). Faça um PR de propósito com um teste quebrado e veja o CI barrar —
essa é a rede de segurança funcionando.

### Faça o deploy num PaaS
Publique num PaaS (DigitalOcean App Platform, Render, Railway…) com um banco gerenciado (como
na Aula 16). Configure os **segredos como variáveis de ambiente** (nunca no código!) e um
**health check**. Quando a URL pública responder, sua API está **no ar de verdade**. 🚀

### Adicione cache e rate limiting
Coloque cache numa rota de leitura pesada (Caffeine) e um rate limit para proteger o login de
abuso (como na Aula 15). Deixe o backend **selecionável**: memória no dev, Redis quando
precisar escalar.

### Meça com um teste de carga
Escreva um script **k6** e dispare carga na sua API (como na Aula 17). Olhe as métricas do
Actuator, veja onde satura primeiro (como na Aula 14) e ajuste o pool de conexões. Provar
quantos usuários sua API aguenta — **com número, não com achismo** — fecha o ciclo.
::

::callout{type="info" title="Não precisa ser linear e perfeito"}
No mundo real ninguém faz esses 12 passos em sequência limpa. Você vai voltar, refatorar,
quebrar e consertar. **Isso é o processo, não um erro.** O importante é ter feito cada peça pelo
menos uma vez com as próprias mãos — depois disso, elas viram naturais.
::

## O kit do dev backend

Guarde esta tabela. É o seu checklist de "uma API está pronta para o mundo?". Passou por todos
os itens? Então você não fez um brinquedo — fez um produto.

| Peça do kit | O que garante | Aula |
|-------------|---------------|------|
| Arquitetura em camadas | Código organizado e testável, sem espaguete | 4 |
| Entidades + Repository | Dados modelados e acesso ao banco isolado | 5 |
| Rota ponta a ponta (Controller→UseCase→Service→DTO) | Fluxo claro e regra no lugar certo | 6 |
| Autenticação JWT + BCrypt | Só entra quem pode; senha nunca exposta | 7 |
| Tratamento de erro + integrações via gateway | Erros previsíveis; terceiros isolados | 8 |
| Migrations (Flyway) | Banco versionado e reconstruível | 9 |
| Índices e cuidado com N+1 | Banco que não derruba o servidor | 10 |
| Testes em pirâmide | Confiança para mudar sem medo | 11 |
| CI/CD (GitHub Actions) | Rede de segurança a cada commit | 12 |
| Docker | "Funciona em qualquer máquina" | 13 |
| Deploy + segredos no ambiente | API no ar, config segura | 16 |
| Cache + rate limit | Rápida e protegida contra abuso | 15 |
| Observabilidade + teste de carga | Enxergar produção e provar a escala | 17 |
