Agora a gente abre o projeto de verdade. Assim como na Aula 3 do curso da API a gente dissecou a
estrutura de um projeto Spring Boot, aqui vamos entender **como um projeto Flutter é organizado**, o
que é cada pasta e arquivo, e **como o app "sobe"** quando você abre.

## O mapa do projeto

Na raiz do `Descontai-APP`, os itens que importam:

```text
Descontai-APP/
├── lib/                 # >>> TODO o código Dart do app mora aqui <<<
├── android/             # projeto nativo Android (gerado; você quase não mexe)
├── ios/                 # projeto nativo iOS (gerado; você quase não mexe)
├── test/                # testes automatizados
├── pubspec.yaml         # dependências, assets, fontes e versão do app
├── pubspec.lock         # versões exatas travadas (gerado — não edite na mão)
├── analysis_options.yaml# regras de lint/analyzer
└── README.md
```

::callout{type="info" title="Por que existem as pastas android/ e ios/?"}
Lembra que o Flutter vira app **nativo** nas duas plataformas? As pastas `android/` e `ios/` são os
"projetos hospedeiros" de cada sistema — é onde ficam o ícone do app, as permissões, o
`google-services.json` (Firebase Android) e o `GoogleService-Info.plist` (Firebase iOS). O Flutter
**gera** boa parte disso; você só mexe para configurar assinatura, permissões e Firebase. **O seu
dia a dia é 95% dentro de `lib/`.**
::

## O pubspec.yaml: a identidade e as dependências

O `pubspec.yaml` é para o Flutter o que o `pom.xml` é para o Spring (curso da API): é onde você
declara o **nome**, a **versão** e — principalmente — as **dependências** (bibliotecas de terceiros).
Olha um recorte real do Descontai:

```yaml
name: descontai
description: "Descontai — plataforma de descontos e cupons."
version: 1.0.0+1            # versão 1.0.0, build número 1

environment:
  sdk: ^3.9.2              # versão mínima do Dart

dependencies:
  flutter:
    sdk: flutter
  dio: ^5.5.0             # cliente HTTP (falar com a API)
  go_router: ^17.0.0     # navegação entre telas
  provider: ^6.1.5+1     # gerenciamento de estado
  firebase_messaging: ^15.0.0   # notificações push
  firebase_crashlytics: ^4.1.0  # relatório de crashes
  flutter_secure_storage: ^9.2.2 # guardar o token com segurança
  local_auth: ^2.3.0     # biometria (digital/face)
  qr_flutter: ^4.1.0     # gerar QR Code
  mobile_scanner: ^3.2.0 # ler QR Code (câmera)
  cached_network_image: ^3.3.1  # carregar e cachear imagens
  shimmer: ^3.0.0        # efeito "esqueleto" de carregamento
```

::callout{type="tip" title="Leia o pubspec como um raio-x do app"}
Só de olhar as dependências, você **já sabe o que o app faz**: tem `dio` (fala com API), `go_router`
(navega), `provider` (estado), `firebase_messaging` (push), `local_auth` (biometria), `qr_flutter` +
`mobile_scanner` (QR de cupom). Cada dependência que aparecer no curso, volte aqui: ela foi escolhida
para resolver **um** problema. Repare que cada uma vira uma aula mais pra frente.
::

Aquele `^5.5.0` significa "a versão 5.5.0 **ou** uma mais nova compatível". Você declara o que quer;
o comando `flutter pub get` baixa tudo e trava as versões exatas no `pubspec.lock` (que garante que
seu colega e o CI usem **exatamente** as mesmas versões — mesma ideia do curso da API).

## A pasta lib/: onde o app vive

Todo o código Dart mora em `lib/`. E — que beleza — o Descontai organiza `lib/` com a **mesma
filosofia de camadas** que você viu na API:

```text
lib/
├── main.dart            # ponto de partida do app (o "main")
├── core/                # infraestrutura compartilhada
│   ├── config/          # endpoints da API e cache de sessão
│   ├── network/         # o cliente HTTP (Dio)
│   ├── data/
│   │   ├── models/      # objetos que espelham o JSON da API
│   │   └── services/    # chamadas HTTP por assunto (auth, home, pagamento)
│   ├── result/          # tipo Result (sucesso/falha)
│   ├── router/          # rotas (GoRouter)
│   ├── theme/           # tema Material
│   └── utils/           # utilitários (tratamento de erro, strings)
├── features/            # cada FUNCIONALIDADE do app, isolada
│   ├── auth/            # login, cadastro, splash, biometria
│   ├── home/            # feed principal de cupons
│   ├── search/          # busca
│   ├── product_detail/  # detalhe e uso do cupom
│   ├── historico/       # histórico de economia
│   └── profile/         # perfil, assinatura, indicações
└── shared/              # widgets e utilitários reutilizáveis (botão, campo, aviso)
```

::callout{type="info" title="core, features e shared — a organização por funcionalidade"}
- **`core/`**: a "casa de máquinas" compartilhada por todo o app — rede, sessão, rotas, modelos.
- **`features/`**: cada funcionalidade num pacote próprio (auth, home, profile...). Tudo de uma tela
  — a página, o controller, os widgets, os casos de uso — mora junto, na sua pasta.
- **`shared/`**: peças visuais reutilizáveis sem regra de negócio (um botão, um campo de texto).

Essa divisão **por funcionalidade** (chamada *feature-first*) é ótima para times: mexer no "login"
significa abrir **uma** pasta (`features/auth`), sem caçar arquivos espalhados. É o mesmo espírito de
"cada coisa no seu lugar" da arquitetura da API.
::

## Código gerado: o build_runner

Você vai notar arquivos com nomes tipo `algo.g.dart` ou `algo.freezed.dart`. Esses **não são escritos
à mão** — são **gerados** por uma ferramenta chamada `build_runner`, a partir de anotações que você
coloca no seu código. O Descontai usa isso para **transformar JSON em objetos** sem escrever o parse
repetitivo na mão (vamos ver na Aula 10).

```bash
# gera o código a partir das anotações (models, etc.)
dart run build_runner build --delete-conflicting-outputs
```

::callout{type="tip" title="Regra do código gerado"}
Nunca edite um arquivo `.g.dart` ou `.freezed.dart` na mão — sua mudança some na próxima geração.
Você edita o arquivo **fonte** (com as anotações) e roda o `build_runner` de novo. É como uma
"migration" do código: a fonte da verdade é a anotação, não o gerado.
::

## Como o app sobe: o main()

Todo app Flutter começa na função `main()` do `lib/main.dart` — o equivalente ao método `main` da
aplicação Java. É lá que o app se prepara **antes** de mostrar a primeira tela. Olha o esqueleto real
do Descontai:

```dart
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized(); // prepara o Flutter
  await Firebase.initializeApp();            // liga o Firebase (push, crash)
  _configureCrashReporting();                // captura crashes → Crashlytics
  await SessionCache.restore();              // restaura a sessão salva (token)
  await _configureFcm();                     // configura notificações push
  await initializeDateFormatting('pt_BR', null); // formata datas em português

  runApp(
    MultiProvider(                           // registra o estado global...
      providers: [
        ChangeNotifierProvider(create: (_) => HomeController()),
        ChangeNotifierProvider(create: (_) => PesquisaController()),
        ChangeNotifierProvider(create: (_) => UserController()),
      ],
      child: const MyApp(),                  // ...e sobe o app
    ),
  );
}
```

Leia como uma receita de inicialização: **prepara o Flutter → liga o Firebase → restaura a sessão →
configura push → sobe o app**. O `runApp(...)` é o que efetivamente coloca a primeira tela no ar. E
aquele `MultiProvider` registra o **estado global** (os controllers) — algo que a Aula 7 vai destrinchar.

::callout{type="warning" title="Por que fazer tudo isso ANTES de mostrar a tela?"}
Repare que a sessão é restaurada (`SessionCache.restore()`) **antes** do `runApp`. Isso é de
propósito: quando a primeira tela aparecer, o app **já sabe** se o usuário está logado — e manda ele
direto para a Home ou para o login, sem "piscar" telas. Preparar o essencial antes de desenhar é o
que faz o app abrir já no lugar certo.
::

## Resumindo

- Um projeto Flutter tem `lib/` (todo o código Dart), `android/` e `ios/` (projetos nativos gerados),
  `pubspec.yaml` (dependências e versão) e `test/`.
- O **`pubspec.yaml`** é o "pom.xml do Flutter": declara dependências como `dio`, `go_router`,
  `provider`, `firebase_*`, `local_auth`. Ele é um **raio-x** do que o app faz.
- **`lib/`** é organizado em camadas — `core/` (infra), `features/` (funcionalidades isoladas) e
  `shared/` (peças reutilizáveis). É a mesma filosofia de organização da API.
- Arquivos `.g.dart`/`.freezed.dart` são **gerados** pelo `build_runner` — você edita a fonte, nunca
  o gerado.
- O app sobe pela função **`main()`**: prepara o Flutter, liga o Firebase, **restaura a sessão**,
  configura push e chama `runApp(...)` para mostrar a primeira tela.

::callout{type="tip" title="Fixando"}
Abra o `pubspec.yaml` do Descontai-APP e tente adivinhar, só pelo nome de cada dependência, **que
parte do app** ela sustenta. Depois abra o `main.dart` e leia a sequência do `main()` como uma
receita. Se você entende "o que o app carrega antes de aparecer", você entende como um app nasce.
::
