Na Aula 2 você aprendeu que **tudo é widget**. Agora vamos ao detalhe que separa quem "copia layout"
de quem **entende** Flutter: existem **dois tipos** de widget — os que **não mudam** e os que
**mudam**. Saber quando usar cada um é a habilidade mais importante da interface.

## O método build: a receita da tela

Todo widget tem um método `build`. Ele **retorna a árvore de widgets** que deve aparecer na tela.
Pense no `build` como uma **receita**: "para me desenhar, monte um card, com uma coluna dentro, com
dois textos e um botão". O Flutter chama esse `build`, segue a receita e pinta a tela.

```dart
Widget build(BuildContext context) {
  return Text('Olá, mundo'); // a receita: "desenhe este texto"
}
```

A grande sacada do Flutter: quando algo muda, ele **chama o `build` de novo** e **compara** o
resultado novo com o antigo, redesenhando **só o que mudou**. Você nunca manda "apague este texto e
escreva outro" — você só descreve **como a tela deve estar agora**, e o Flutter cuida da diferença.
Isso se chama **UI declarativa**.

::callout{type="info" title="Declarativo, na analogia do pedido"}
Num restaurante **imperativo**, você diria ao garçom cada passo: "pegue o prato, coloque arroz,
agora feijão, agora tire o arroz...". Cansativo e fácil de errar. No **declarativo**, você só diz
**como quer o prato final**: "quero arroz e feijão". Se mudar de ideia, você descreve o novo prato
inteiro, e a cozinha se vira para chegar lá. O `build` é você descrevendo **o prato final** (a tela
como deve estar) — o Flutter faz o resto.
::

## Widget Stateless: a tela que não muda sozinha

Um **StatelessWidget** ("sem estado") é um widget que, **depois de desenhado, não muda por conta
própria**. Ele recebe uns dados prontos e os mostra. Se os dados forem os mesmos, a tela é sempre a
mesma.

Um card de cupom é um bom exemplo: ele recebe um produto e o exibe. Ele **não** tem botão que muda a
si mesmo — só mostra.

```dart
class ProdutoCard extends StatelessWidget {
  const ProdutoCard({super.key, required this.nome, required this.desconto});

  final String nome;      // dados que chegam de fora (imutáveis)
  final String desconto;

  @override
  Widget build(BuildContext context) {
    return Card(
      child: Column(
        children: [
          Text(nome),
          Text(desconto),
        ],
      ),
    );
  }
}
```

::callout{type="tip" title="A regra do Stateless"}
Use **StatelessWidget** quando o widget só **mostra** o que recebe e **não muda sozinho** depois de
desenhado. É o caso da maioria dos componentes visuais: cards, títulos, ícones, o logo. No Descontai,
o `ProdutoCard`, o `CategoriaChip` e o `Logo` são stateless — recebem dados e os exibem. **Prefira
sempre Stateless quando possível**: é mais simples e mais leve.
::

## Widget Stateful: a tela que muda

Um **StatefulWidget** ("com estado") é um widget que **pode mudar ao longo do tempo** por conta de
alguma interação: um campo de texto que o usuário digita, um checkbox que marca/desmarca, um
formulário. Ele guarda um **estado** (dados que mudam) e, quando esse estado muda, **redesenha**.

O gatilho para redesenhar é o `setState`:

```dart
class ContadorCupons extends StatefulWidget {
  const ContadorCupons({super.key});
  @override
  State<ContadorCupons> createState() => _ContadorCuponsState();
}

class _ContadorCuponsState extends State<ContadorCupons> {
  int _usados = 0; // ESTADO: um dado que muda

  @override
  Widget build(BuildContext context) {
    return Column(
      children: [
        Text('Cupons usados: $_usados'),
        ElevatedButton(
          onPressed: () {
            setState(() {   // avisa o Flutter: "o estado mudou, redesenhe"
              _usados++;    // muda o dado
            });
          },
          child: const Text('Usei um cupom'),
        ),
      ],
    );
  }
}
```

::callout{type="warning" title="O papel do setState (e o erro clássico)"}
Mudar `_usados++` **sozinho não atualiza a tela** — você precisa envolver a mudança em
`setState(() { ... })`. É o `setState` que avisa o Flutter "ei, meu estado mudou, chame meu `build`
de novo". O erro nº 1 do iniciante é mudar a variável **fora** do `setState` e não entender por que a
tela não muda. **Mudou algo que a tela mostra? Envolva em `setState`.**
::

## Como escolher: Stateless ou Stateful?

A pergunta é uma só: **"este widget precisa mudar depois de aparecer, por conta de uma interação
interna dele?"**

::fields
::field{name="Não muda sozinho → Stateless"}
Cards, títulos, ícones, o logo, um item de lista. Recebe dados, mostra, fim. É a **maioria** dos
widgets. Mais simples e mais leve.
::
::field{name="Muda por interação interna → Stateful"}
Formulários, campos de texto, checkbox, um botão que troca de estado, uma animação. Guarda dados que
mudam e usa `setState` para redesenhar.
::
::

::callout{type="info" title="E o estado que a TELA INTEIRA compartilha?"}
Tem um terceiro caso: e quando **vários** widgets, ou telas diferentes, precisam do **mesmo** dado —
como a lista de cupons que aparece na Home **e** na Busca? O `setState` só serve para o estado
**local** de um widget. Para estado **compartilhado**, o Descontai usa o **Provider** com
`ChangeNotifier` — e é exatamente por isso que existe o `HomeController`. Esse é o tema da Aula 7:
guarde a pergunta "onde mora o estado?" que a gente responde lá.
::

## No Descontai

Olhando o app real:

- **Stateless**: `ProdutoCard`, `CategoriaChip`, `Logo`, `Button` (o botão compartilhado) — só exibem.
- **Stateful**: as páginas com formulário e interação, como `LoginPage` (campos, checkbox "lembrar"),
  `RegisterPage`, telas que controlam um estado local de UI (um loading de botão, um toggle).
- **Estado compartilhado** (nem um nem outro — Provider): `HomeController`, `UserController`,
  `PesquisaController` — o assunto da Parte 3.

## Resumindo

- Todo widget tem um método **`build`** que **retorna a árvore** a ser desenhada. O Flutter é
  **declarativo**: você descreve **como a tela deve estar**, e ele calcula a diferença e redesenha só
  o que mudou.
- **StatelessWidget**: não muda sozinho depois de desenhado. Recebe dados e mostra. É a maioria dos
  componentes (cards, títulos). Prefira sempre que der.
- **StatefulWidget**: muda por interação interna (formulários, checkbox). Guarda um **estado** e usa
  **`setState`** para avisar o Flutter que precisa redesenhar.
- **`setState`** é o gatilho do redesenho local — mudar a variável sem ele não atualiza a tela.
- Estado **compartilhado** entre telas não é `setState`: é **Provider/ChangeNotifier** (Aula 7).

::callout{type="tip" title="Fixando"}
Para cada tela do Descontai que você imaginar, pergunte: *"o que aqui muda por interação, e o que só
é exibido?"* O que muza localmente pede Stateful + `setState`; o que só mostra é Stateless; o que é
compartilhado entre telas pede Provider. Saber **onde cada estado mora** é o pulo do gato do Flutter.
::
