Um app tem dezenas de telas, e o usuário anda entre elas o tempo todo: da Home para o detalhe do
cupom, do perfil para a assinatura, do login para a Home. Quem organiza esse trânsito é o
**roteamento**. O Descontai usa o **GoRouter** — e nesta aula você vai ler o router **de verdade** do
app, incluindo a parte mais importante: os **guards** que barram telas que exigem login.

## O problema: navegar sem virar bagunça

Você **poderia** empurrar telas na mão, uma sobre a outra. Mas num app real isso vira caos: como
saber o caminho atual? Como impedir que alguém sem login abra a tela de perfil digitando a rota? Como
passar o "id do produto" para a tela de detalhe? A solução é **centralizar as rotas** num só lugar,
com nomes e regras. Esse lugar é o **GoRouter**.

::callout{type="info" title="GoRouter = o mapa de endereços do app"}
Assim como uma API REST tem rotas (`/produtos`, `/produtos/1`), o app tem **rotas internas**
(`/base`, `/produto/:id`, `/assinatura`). O GoRouter é o **mapa**: ele diz qual **tela** corresponde
a cada **caminho**, como passar parâmetros e **quem pode entrar** em cada endereço. Você navega por
**nome de rota**, nunca digitando caminhos soltos pelo código.
::

## As rotas: caminho → tela

No Descontai, cada rota liga um **caminho** a uma **tela** (uma page). Veja um recorte real do
`lib/core/router/app_router.dart`:

```dart
routes: [
  GoRoute(path: AppRoutes.splash,  builder: (_, __) => const SplashPage()),
  GoRoute(path: AppRoutes.welcome, builder: (_, __) => const WelcomePage()),
  GoRoute(path: AppRoutes.login,   builder: (_, __) => const LoginPage()),
  GoRoute(path: AppRoutes.base,    builder: (_, __) => const BasePage()),
  GoRoute(path: AppRoutes.profile, builder: (_, __) => const ProfilePage()),
  // ...
]
```

Cada `GoRoute` diz: "quando o caminho for **X**, construa a tela **Y**". Repare que os caminhos vêm
de constantes em `AppRoutes` (ex.: `AppRoutes.login`), nunca de strings soltas — assim ninguém digita
`"/logni"` errado e o compilador ajuda.

Para **navegar**, uma tela chama:

```dart
context.go(AppRoutes.base);      // vai para a Home (substitui a pilha)
context.push(AppRoutes.profile); // empilha o perfil (dá para "voltar")
```

::callout{type="tip" title="go x push, sem decoreba"}
`go` **troca** para a nova tela (bom para "entrou → vá para a Home"). `push` **empilha** a nova tela
por cima (bom para abrir um detalhe e depois **voltar** com a seta). Escolha pela pergunta: *"faz
sentido o usuário voltar para a tela anterior?"* Se sim, `push`; se é uma troca de contexto, `go`.
::

## Parâmetros: passar dados pela rota

A tela de detalhe precisa saber **qual** produto mostrar. Isso vai na própria rota, como um
parâmetro. No Descontai:

```dart
GoRoute(
  path: '/produto/:id',   // ":id" é um parâmetro de caminho
  builder: (_, state) => ProductDetailPage(
    produtoId: state.pathParameters['id']!,   // lê o id da URL
    produto: state.extra is ProdutosResponse  // e um objeto extra opcional
        ? state.extra as ProdutosResponse
        : null,
  ),
),
```

Para abrir o produto 42, a Home chama `context.push('/produto/42', extra: produto)`. Duas formas de
passar dados aparecem aqui:

::fields
::field{name="pathParameters" type="vai na URL (:id)"}
O `:id` no caminho vira `state.pathParameters['id']`. Bom para identificadores simples (um id). Fica
visível na "URL" da navegação.
::
::field{name="extra" type="objeto passado direto"}
`state.extra` carrega um **objeto Dart inteiro** (aqui, o `ProdutosResponse` já carregado). Isso
evita buscar o produto de novo na API se a Home já o tem. Repare no cuidado: o código checa
`is ProdutosResponse` antes de usar — se vier vazio, cai para `null` e a tela busca sozinha.
::
::

## Guards: quem pode abrir cada tela

Aqui está a parte mais importante — e a que conecta direto com a segurança que você viu no curso da
API. Um **guard** é uma regra que, **antes** de abrir uma tela, decide se o usuário **pode** entrar,
e para onde redirecioná-lo se não puder. No GoRouter, isso é o `redirect`. Olha o guard global real
do Descontai:

```dart
redirect: (context, state) {
  if (state.fullPath == AppRoutes.splash) return null; // splash sempre passa

  final token = SessionCache.token;
  final isLogged = token != null && token.isNotEmpty;

  final isAuthRoute = const {
    AppRoutes.login, AppRoutes.welcome, AppRoutes.register, AppRoutes.redefinirSenha,
  }.contains(state.fullPath);

  // Não logado tentando entrar numa tela protegida → manda para o welcome
  if (!isLogged && !isAuthRoute) return AppRoutes.welcome;

  // Já logado tentando abrir login/cadastro → manda para a Home
  if (isLogged && isAuthRoute) return AppRoutes.base;

  return null; // pode seguir para a rota pedida
}
```

Leia como duas frases:

1. **Sem token e tentando abrir tela protegida?** Volta para o `welcome` (não deixa entrar).
2. **Com token e tentando abrir login/cadastro?** Vai direto para a `base` (não faz sentido logar de
   novo).

::callout{type="info" title="O guard usa o SessionCache que você viu no main()"}
Repare que o guard olha `SessionCache.token` — o mesmo cache de sessão restaurado no `main()` (Aula
3). Por isso, quando o app abre, ele **já sabe** se você está logado e te leva ao lugar certo, sem
piscar telas. E como a rota do detalhe/perfil está "atrás" do guard, digitar a rota na mão **não**
fura a segurança do app. (Lembrando da Aula 1: a segurança **de verdade** ainda é a API — o guard é
só a boa experiência do cliente.)
::

### Guard por tela: a regra da indicação

Além do guard global, uma rota pode ter o **seu próprio** guard. A tela de indicações só faz sentido
para quem é indicador:

```dart
GoRoute(
  path: AppRoutes.indicacoes,
  redirect: (_, __) =>
      SessionCache.indicador == true ? null : AppRoutes.profile,
  builder: (_, __) => const IndicacoesSaldoPage(),
),
```

Traduzindo: "só entra na tela de indicações quem tem `indicador == true`; senão, volta para o
perfil". É a mesma ideia de **autorização** da API (papéis/roles), agora do lado do app para dar uma
experiência coerente.

## A mágica do refreshListenable

Uma última peça esperta. O router tem `refreshListenable: SessionCache.listenable`. Isso faz o
GoRouter **reavaliar os guards automaticamente** sempre que a sessão mudar. Ou seja: quando o usuário
faz **logout**, o `SessionCache` avisa, o router reavalia e — como agora não há token — o guard
**expulsa** o usuário das telas protegidas para o welcome, sem você escrever isso à mão em cada tela.

::callout{type="tip" title="Por que isso é elegante"}
Sem o `refreshListenable`, depois de um logout você teria que lembrar de mandar o usuário para o
login **em todo lugar**. Com ele, a sessão é a **fonte da verdade**: mudou a sessão → o router
reavalia → o usuário vai parar no lugar certo. Um único ponto de controle em vez de regra espalhada.
É o mesmo princípio do "atualize a sessão primeiro" que guia o app inteiro.
::

## Resumindo

- **Navegação** é o trânsito entre telas; o **GoRouter** centraliza as rotas (caminho → tela) num só
  lugar, com nomes em `AppRoutes` (nunca strings soltas).
- Navegue com **`go`** (troca de tela) ou **`push`** (empilha, permite voltar).
- Passe dados por **`pathParameters`** (`:id` na rota) ou por **`extra`** (um objeto Dart inteiro,
  com checagem de tipo antes de usar).
- **Guards** (`redirect`) decidem quem pode abrir cada tela: o guard **global** barra não-logados e
  redireciona logados para fora do login; guards **por rota** aplicam regras específicas (ex.:
  `indicador == true`).
- **`refreshListenable: SessionCache.listenable`** faz o router reavaliar os guards quando a sessão
  muda — logout expulsa das telas protegidas automaticamente.

::callout{type="tip" title="Fixando"}
Explique em voz alta: *"o que impede alguém sem login de abrir a tela de perfil no Descontai?"*.
Resposta: *"o guard global do GoRouter — ele vê que `SessionCache.token` está vazio e redireciona
para o welcome, antes de a tela abrir"*. E complete: *"mas a proteção real dos dados é a API, o guard
é só a experiência"*. Se você amarrou as duas pontas, entendeu navegação **e** segurança de cliente.
::
