Na Aula 4 ficou uma pergunta no ar: *e quando várias telas precisam do mesmo dado?* O `setState`
resolve o estado **local** de um widget, mas não serve quando a lista de cupons precisa aparecer na
Home **e** na Busca, ou quando o nome do usuário aparece em vários cantos. A resposta do Descontai é
**Provider + ChangeNotifier** — o coração da reatividade do app. Vamos do zero.

## O que é "estado", afinal

**Estado** é qualquer dado que muda enquanto o app roda e que a tela precisa mostrar: a lista de
cupons, se está carregando, se deu erro, o nome do usuário, quantos cupons estão disponíveis. Quando
o estado muda, a tela precisa **se atualizar** para refletir o novo dado.

O desafio: **onde esse estado mora?**

::fields
::field{name="Estado local (setState)"}
Vive **dentro de um widget** e só interessa a ele: o texto que está sendo digitado num campo, um
checkbox marcado. Resolvido com `setState` (Aula 4).
::
::field{name="Estado compartilhado (Provider)"}
Precisa ser visto por **várias telas/widgets**: a lista de cupons, o usuário logado, os favoritos.
Não pode viver preso a um widget — precisa de uma "caixa" acima das telas, que várias possam ler.
Esse é o trabalho do **Provider**.
::
::

## ChangeNotifier: a caixa de estado que avisa

Um **ChangeNotifier** é uma classe que **guarda estado** e sabe **avisar** quando esse estado muda.
Ela tem um método mágico: `notifyListeners()`. Quem estiver "ouvindo" essa classe é notificado e
**se redesenha**.

::callout{type="info" title="ChangeNotifier = rádio; widgets = ouvintes"}
Pense numa **estação de rádio**. O `ChangeNotifier` é a estação: ela guarda a "música atual" (o
estado) e, quando troca de música, **transmite** (`notifyListeners()`). Os widgets são os
**rádios ligados** nela: quando a transmissão chega, eles se atualizam para tocar a nova música (mostrar
o novo estado). Você não vai em cada rádio dizer "toque isso" — a estação transmite e todos ouvem.
::

## O HomeController de verdade

O `HomeController` do Descontai é um `ChangeNotifier`. Ele guarda tudo que a Home precisa e sabe
carregar isso da API. Olha o esqueleto real (`lib/features/home/.../home_controller.dart`):

```dart
class HomeController extends ChangeNotifier {
  // O ESTADO que a Home mostra:
  List<ProdutosResponse> produtos = [];
  List<ProdutosResponse> top5 = [];
  List<CategoriaResponse> categorias = [];
  List<ProdutosUsuarioResponse> historico = [];
  bool isLoading = false;
  String? errorMessage;

  // Um valor DERIVADO do estado (calculado, não guardado):
  int get cuponsDisponiveis =>
      produtos.where((p) => p.disponivelParaUso).length;

  Future<void> carregarDados() async {
    isLoading = true;         // 1) entra em "carregando"
    errorMessage = null;
    notifyListeners();        //    → avisa a tela: mostre o loading

    final result = await _useCase(usuarioId: ..., cidadeId: ...); // 2) busca na API

    switch (result) {         // 3) trata sucesso ou falha (Result — Aula 11)
      case Success(:final data):
        produtos = data.produtos;
        categorias = data.categorias;
        historico = data.historico;
        _buildDerivedLists();
      case Failure(:final message):
        errorMessage = message;
    }

    isLoading = false;        // 4) saiu do "carregando"
    notifyListeners();        //    → avisa a tela: mostre os dados (ou o erro)
  }
}
```

Leia o fluxo: **liga o loading e avisa → busca na API → guarda o resultado (ou o erro) → desliga o
loading e avisa**. Cada `notifyListeners()` faz a Home se redesenhar com o estado novo — primeiro
mostrando o "carregando", depois os cupons.

::callout{type="tip" title="Por que o estado da Home vive num controller separado?"}
Repare que **nenhuma regra de tela** ("qual é o texto do botão", "que cor usar") está aqui — o
controller só cuida de **dados e estado**. Isso separa **o que mostrar** (o controller) de **como
mostrar** (a página). A página fica burra e enxuta; o controller concentra a lógica de carregar,
guardar e derivar. É a mesma separação de responsabilidades que você viu na arquitetura da API.
::

### Valores derivados: calcule, não guarde

Repare no `cuponsDisponiveis`: ele **não é uma variável guardada** — é um `get` que **calcula** na
hora, contando os produtos disponíveis. Isso evita o clássico bug de "dois dados que deveriam
combinar e saem de sincronia". Se a lista de produtos muda, o número de disponíveis **sempre** está
certo, porque é calculado a partir dela. **Guarde a fonte; derive o resto.**

## Provider: entregando o controller às telas

O `ChangeNotifier` guarda e avisa. O **Provider** é quem **entrega** esse controller para as telas e
faz a "escuta" funcionar. Lembra do `main.dart` (Aula 3)?

```dart
MultiProvider(
  providers: [
    ChangeNotifierProvider(create: (_) => HomeController()),
    ChangeNotifierProvider(create: (_) => PesquisaController()),
    ChangeNotifierProvider(create: (_) => UserController()),
  ],
  child: const MyApp(),
)
```

Isso cria os controllers **uma vez**, no topo do app, e os disponibiliza para **qualquer** tela
abaixo. Aí, dentro de uma página, você "sintoniza o rádio":

```dart
// Ouve o controller: quando ele chamar notifyListeners(), este widget se redesenha
final home = context.watch<HomeController>();

if (home.isLoading) return const CircularProgressIndicator(); // mostra loading
if (home.errorMessage != null) return Text(home.errorMessage!); // mostra erro
return ListView(                                                // mostra os cupons
  children: home.produtos.map((p) => ProdutoCard(produto: p)).toList(),
);
```

::callout{type="info" title="watch x read, sem decoreba"}
- **`context.watch<HomeController>()`** — *ouve* o controller: o widget se redesenha quando o estado
  muda. Use no `build`, para **mostrar** dados.
- **`context.read<HomeController>()`** — só *pega* o controller uma vez, sem ouvir. Use para **chamar
  uma ação** (ex.: `context.read<HomeController>().recarregar()` no pull-to-refresh), sem querer
  redesenhar por isso.

A regra prática: **watch para exibir, read para agir.**
::

## Os três controllers do Descontai

O app tem três "estações de rádio" globais, cada uma dona de um pedaço do estado:

| Controller | Cuida de |
|---|---|
| `HomeController` | Produtos, top 5, categorias, histórico, loading e erro da Home |
| `UserController` | Usuário logado (`/usuarios/me`), validade da assinatura e favoritos |
| `PesquisaController` | Busca por texto, filtros e favoritos na tela de Busca |

Como todos são globais, o `UserController` pode atualizar o nome do usuário e **todas** as telas que
o ouvem se atualizam juntas — sem você sincronizar nada na mão. É o poder do estado compartilhado.

## Resumindo

- **Estado** é o dado que muda e a tela mostra. **Local** (de um widget) usa `setState`;
  **compartilhado** (entre telas) usa **Provider**.
- **`ChangeNotifier`** é a "caixa de estado que avisa": guarda os dados e chama **`notifyListeners()`**
  quando mudam — como uma **estação de rádio** transmitindo para os ouvintes.
- O **`HomeController`** real segue o fluxo: liga o loading e avisa → busca na API → guarda resultado
  ou erro → desliga o loading e avisa. Ele cuida de **dados**, não de aparência.
- Valores **derivados** (ex.: `cuponsDisponiveis`) são **calculados**, não guardados — evitam
  dessincronia.
- O **Provider** cria os controllers no topo (`MultiProvider` no `main`) e os entrega às telas:
  **`watch`** para exibir (redesenha), **`read`** para agir (só chama ação).

::callout{type="tip" title="Fixando"}
Descreva o ciclo de uma tela reativa: *"a página **ouve** (`watch`) o `HomeController`; quando o
controller **carrega os produtos** e chama **`notifyListeners()`**, a página **se redesenha** mostrando
os cupons"*. Se esse "ouviu → avisou → redesenhou" ficou claro, você domina o coração do Flutter
moderno.
::
