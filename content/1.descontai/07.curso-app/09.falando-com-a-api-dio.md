Chegou a hora de olhar a peça que **realmente conversa com a API**: o **Dio**, o cliente HTTP do app.
Toda requisição do Descontai — login, cupons, pagamento — passa por aqui. E o Dio traz um recurso
poderoso, os **interceptors**, que resolvem de forma elegante dois problemas que apareceriam em
**toda** chamada: colocar o token e tratar o erro.

## O que é o Dio

O **Dio** é uma biblioteca para fazer requisições HTTP em Dart. Ele é a versão "com esteroides" do
básico: você diz "faça um POST para `/auth/login` com esse corpo", e ele cuida de montar a requisição,
enviar, esperar a resposta e te entregar o resultado — com suporte a timeout, headers, logs e
interceptors.

```dart
final response = await _dio.post(
  '/auth/login',
  data: {'email': email, 'senha': senha}, // vira o body JSON
);
final json = response.data; // a resposta já parseada
```

## Uma instância só: o singleton

Um detalhe importante: o app usa **um único** Dio compartilhado por todos os services. Isso se chama
**singleton** ("instância única"). Assim, a configuração (endereço base, timeout, token) é definida
**uma vez** e vale para todas as chamadas. No `lib/core/network/dio_client.dart`:

```dart
class DioClient {
  DioClient._internal() { /* configura tudo aqui */ }

  static final DioClient _instance = DioClient._internal(); // a única instância
  final Dio _dio = Dio();

  static Dio get instance => _instance._dio; // todos pegam O MESMO Dio
}
```

::callout{type="info" title="Por que uma instância única?"}
Se cada service criasse o **seu** Dio, você teria que repetir "endereço base, timeout, colocar token"
em cada um — e mudar isso viraria um pesadelo. Com um singleton, há **um lugar** que configura a rede
para o app inteiro. É o mesmo motivo de a API ter um pool de conexões único: recurso compartilhado se
configura **uma vez**.
::

## A configuração base: endereço e timeouts

Quando o DioClient nasce, ele define as opções que valem para todas as chamadas:

```dart
_dio.options = BaseOptions(
  baseUrl: AppEndpoints.baseUrl,               // ex.: https://descontai-api...
  connectTimeout: const Duration(seconds: 15), // 15s para CONECTAR
  receiveTimeout: const Duration(seconds: 30), // 30s para RECEBER a resposta
  headers: {'Content-Type': 'application/json'},
);
```

::callout{type="tip" title="Por que timeout é obrigatório no mobile"}
Lembra da Aula 1: no celular, a internet é imprevisível (metrô, elevador, sinal fraco). **Sem
timeout**, uma requisição poderia ficar "pendurada" para sempre, e a tela travaria mostrando
"carregando" eternamente. O timeout diz "se não respondeu em 15/30 segundos, **desista** e me avise o
erro" — para o app poder mostrar "sem conexão, tente de novo" em vez de congelar. Rede que pode
falhar **precisa** de tempo-limite.
::

E note o `_validateBaseUrl()`: em build de produção (`kReleaseMode`), o app **se recusa a subir** se
a URL não for HTTPS. É o mesmo *fail-fast* da API — melhor quebrar na hora do que enviar o token do
usuário por uma conexão insegura.

## Interceptors: o segredo do Dio

Aqui está o recurso que torna o Dio tão bom. Um **interceptor** é um "porteiro" que **intercepta**
toda requisição **antes** de ela sair, e toda resposta/erro **antes** de chegar em quem pediu. Isso
permite fazer, num lugar só, o que teria que ser repetido em todas as chamadas.

::callout{type="info" title="Interceptor = a inspeção antes do embarque"}
Pense no interceptor como a **inspeção de segurança do aeroporto**: **toda** mala passa por ali antes
de embarcar (na ida) e há um balcão para problemas na volta. Você não coloca uma inspeção em cada
portão de embarque — coloca **uma**, no caminho de todos. O interceptor é essa passagem obrigatória
por onde todas as requisições passam.
::

### Interceptor de requisição: colocar o token

Toda rota protegida da API precisa do header `Authorization: Bearer <token>` (você viu isso no curso
da API). Em vez de colocar isso em cada chamada, o Descontai faz **uma vez**, no interceptor:

```dart
onRequest: (options, handler) {
  // Em rotas privadas, injeta o token guardado no SessionCache
  if (!_isPublicAuthEndpoint(options.path)) {
    final token = SessionCache.token;
    if (token != null && token.isNotEmpty) {
      options.headers['Authorization'] = 'Bearer $token';
    }
  }
  return handler.next(options); // segue a requisição
}
```

Traduzindo: "antes de **qualquer** requisição sair, se ela **não** for uma rota pública de auth
(login, cadastro, esqueci-senha), pegue o token do `SessionCache` e cole no header". Assim, nenhum
service precisa lembrar de colocar o token — o interceptor faz por todos.

::callout{type="tip" title="O interceptor conecta as pontas do curso"}
Repare: o app pega o token do `SessionCache` (Aula 3/7) e o envia no header `Bearer` que a **API**
espera e valida no `JwtAuthenticationFilter` (que você estudou no curso da API!). É literalmente a
mesma chave sendo entregue de um lado e conferida do outro. Ver o token nascer no login, morar no
`SessionCache`, ser injetado pelo interceptor e validado pela API — é enxergar o sistema **inteiro**.
::

### Interceptor de erro: tratar 401 num lugar só

E quando a API responde **401** (token inválido/expirado)? Sem um lugar central, cada tela teria que
tratar isso. O interceptor `onError` resolve para todas:

```dart
onError: (error, handler) async {
  final status = error.response?.statusCode;
  // ... (tenta renovar o token — veja a Aula 14) ...

  // 401/403 sem como renovar: encerra a sessão (token morto)
  if ((status == 401 || status == 403) && !isAuthEndpoint) {
    await SessionCache.clearAll();
  }
  return handler.next(error);
}
```

Quando o token não vale mais e não dá para renovar, o app **limpa a sessão** (`clearAll()`). Lembra do
`refreshListenable` da Aula 6? Ao limpar a sessão, o GoRouter reavalia e **expulsa** o usuário para o
login — automaticamente. Tudo isso disparado por **um** ponto: o interceptor.

::callout{type="warning" title="Tem uma peça mais esperta aqui"}
O código real **não** desloga de cara no 401 — primeiro ele tenta **renovar** o token com o refresh
token, e só encerra a sessão se a renovação falhar. Essa lógica de renovação (e um detalhe elegante
chamado *single-flight*) é importante o bastante para ter a **Aula 14 inteira**. Por ora, guarde: o
interceptor de erro é o lugar onde "o token expirou" é tratado **uma vez** para o app todo.
::

## Logs seguros

Em modo debug, o Dio loga as requisições para ajudar a depurar — mas repare que o Descontai
**desliga** o log de body e headers:

```dart
if (kDebugMode) {
  _dio.interceptors.add(LogInterceptor(
    requestBody: false, requestHeader: false, // não loga corpo nem headers
    responseBody: false,
  ));
}
```

::callout{type="danger" title="Por que NÃO logar body e headers"}
O corpo do login tem a **senha**; os headers têm o **token**. Se isso fosse para o log, senha e token
poderiam vazar em ferramentas de crash/analytics. A regra de segurança do app é explícita: *"não
logar headers, body de login, token, cartão ou CVV"*. Log ajuda a depurar — mas **nunca** às custas
de vazar dado sensível. É a mesma disciplina de segredos que a API tem.
::

## Resumindo

- O **Dio** é o cliente HTTP do app: você diz "POST para tal rota com tal corpo" e ele cuida de
  enviar, esperar e entregar a resposta.
- Há **uma única instância** (singleton): a configuração de rede (endereço base, timeout) é definida
  **uma vez** para o app todo.
- **Timeouts** são obrigatórios no mobile — a internet pode falhar, e sem timeout a tela travaria
  para sempre. Em produção, o app exige **HTTPS** (fail-fast).
- **Interceptors** são "porteiros" por onde toda requisição/erro passa. O de **requisição** injeta o
  `Authorization: Bearer <token>` do `SessionCache`; o de **erro** trata 401/403 num lugar só
  (limpando a sessão quando o token morre).
- Em debug, o log **não** inclui body nem headers — para **nunca** vazar senha, token, cartão ou CVV.

::callout{type="tip" title="Fixando"}
Explique o caminho do token: *"nasce no login, é guardado no `SessionCache`, o **interceptor** do Dio
o injeta em toda requisição privada como `Bearer`, e a **API** o valida"*. Se você amarrou esse fio
do app até a API, você entendeu como um app autenticado funciona por dentro.
::
