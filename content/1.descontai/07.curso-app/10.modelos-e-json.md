O Dio te trouxe a resposta da API — mas ela chega como **JSON**: um monte de texto com chaves e
colchetes (lembra do curso da API?). A tela não sabe lidar com texto bruto; ela quer um **objeto Dart**
com campos tipados: `produto.nome`, `produto.valorDesconto`. Transformar um no outro é o papel dos
**modelos** (models). E aqui mora uma das lições mais valiosas do mobile: **parse defensivo**.

## O problema: JSON é "sem forma"

Quando o Dio devolve `response.data`, você recebe um `Map<String, dynamic>` — um dicionário onde os
valores são `dynamic` ("qualquer coisa"). Você **poderia** usar direto:

```dart
final nome = response.data['produto']; // dynamic — pode ser String, null, número...
```

Mas isso é uma bomba-relógio. E se a API mandar `null`? E se o campo vier como número em vez de texto?
E se o nome da chave mudar? O app **quebraria na cara do usuário**. A solução é criar uma **classe
modelo** que faz a tradução **num lugar só**, com cuidado.

::callout{type="info" title="Model = o tradutor entre a API e o app"}
Um **model** é uma classe Dart que **espelha** um pedaço da resposta da API. Ele tem um método
`fromJson` que recebe o `Map` cru e devolve um objeto tipado e seguro. A partir daí, o resto do app
trabalha com `ProdutosResponse` (com campos de verdade), e **ninguém mais toca no JSON cru**. O model
é a fronteira: JSON bagunçado entra de um lado, objeto limpo sai do outro.
::

## O fromJson na prática

Olha o model de login real (`lib/core/data/models/login_response.dart`), simplificado:

```dart
class LoginResponse {
  final int? id;
  final String? accessToken;
  final String? refreshToken;
  final String? cidade;
  final bool indicador;

  const LoginResponse({ this.id, this.accessToken, /* ... */ this.indicador = false });

  factory LoginResponse.fromJson(Map<String, dynamic> json) {
    final token = json['accessToken'] ?? json['token']; // aceita os dois nomes
    return LoginResponse(
      id: _parseInt(json['id']),
      accessToken: token is String && token.isNotEmpty ? token : null,
      refreshToken: /* ... */,
      cidade: json['cidade'] as String?,
      indicador: _parseBool(json['indicador']),
    );
  }
}
```

Duas coisas para notar:

1. **`factory ... fromJson`** é o "construtor tradutor": recebe o `Map` e monta o objeto.
2. Os campos são **anuláveis** (`int?`, `String?`) porque a API pode não mandar todos — o
   **null safety** do Dart (Aula 2) obriga o app a lidar com isso.

## Parse defensivo: o coração da aula

Aqui está a lição que separa um app frágil de um robusto. A API **pode** mandar um dado num formato
inesperado: um número como texto (`"42"` em vez de `42`), um campo faltando, um booleano como `"true"`
em string. Um parse ingênuo (`json['id'] as int`) **explodiria** nesses casos. O Descontai usa
**helpers defensivos** que "dão um jeito" em vez de quebrar. Olha o `_parseInt` real:

```dart
static int _parseInt(dynamic value) {
  if (value is int) return value;             // já é int? ótimo
  if (value is num) return value.toInt();     // é número (double)? converte
  if (value is String) return int.tryParse(value) ?? 0; // é texto? tenta; senão 0
  return 0;                                   // null ou qualquer outra coisa? 0
}
```

E o `_parseBool`, que aceita `true`, `1`, `"true"`, `"1"` e trata o resto como falso:

```dart
static bool _parseBool(dynamic value, {bool defaultValue = false}) {
  if (value == null) return defaultValue;
  if (value is bool) return value;
  if (value is num) return value != 0;
  if (value is String) {
    final lower = value.toLowerCase().trim();
    if (lower == 'true' || lower == '1') return true;
    if (lower == 'false' || lower == '0') return false;
  }
  return defaultValue;
}
```

::callout{type="tip" title="Por que parse defensivo é ouro no mobile"}
No mobile, você **não controla** o que chega: a API pode evoluir, um campo pode faltar numa versão
antiga do app, um tipo pode mudar. Parse defensivo garante que, **no pior caso**, você tem um valor
padrão seguro (`0`, `''`, `false`) em vez de um **crash** na tela do usuário. É a diferença entre "um
card ficou sem foto" e "o app fechou". **Tolerar o inesperado é resiliência** — o mesmo espírito de
"a rede vai falhar, prepare-se" que guia o app inteiro.
::

### Detalhe de gente grande: compatibilidade e defaults seguros

Dois cuidados reais no `ProdutosResponse` que valem estudar:

::fields
::field{name="Aceitar nomes de chave diferentes"}
`json['fotoUrl'] ?? json['imagemUrl']` e `json['accessToken'] ?? json['token']`: o model aceita **mais
de um nome** para o mesmo dado. Isso deixa o app **compatível** com versões diferentes da API sem
quebrar — uma ponte entre contratos que evoluíram.
::
::field{name="Default seguro no que importa"}
Repare em `disponivelParaUso`: se a informação **não vier**, o default é `false` — ou seja, na dúvida,
o cupom **não** é tratado como disponível. É a escolha **segura**: melhor esconder um cupom por engano
do que liberar um uso indevido. O default "erra para o lado seguro".
::
::

E o `StatusCupom.fromApi(...)` traduz o texto de status da API (`"DISPONIVEL"`, `"USADO"`...) para um
**enum** Dart, com um *fallback* para versões antigas da API que não mandavam esse campo — de novo,
priorizando não quebrar.

## Objetos aninhados

O JSON de um produto tem objetos **dentro** dele (a empresa, a categoria). O model reflete isso: o
`ProdutosResponse` tem um campo `Empresa empresa`, e o `fromJson` chama o `Empresa.fromJson(...)` do
sub-objeto. Cada pedaço do JSON vira sua própria classe, aninhada como o JSON é aninhado.

```dart
final Empresa empresa = Empresa.fromJson(json['empresa'] ?? <String, dynamic>{});
//                                        ↑ se 'empresa' faltar, usa um mapa vazio
//                                          (e o Empresa.fromJson devolve tudo com defaults)
```

## E o código gerado (json_serializable)?

Você viu no `pubspec.yaml` as dependências `json_serializable` e `freezed`, e na Aula 3 o
`build_runner`. Elas **geram** o `fromJson` automaticamente a partir de anotações, poupando esse código
repetitivo. O Descontai tem essa ferramenta disponível, mas os models principais usam **`fromJson`
escrito à mão** — de propósito.

::callout{type="info" title="Por que escrever fromJson à mão aqui?"}
O código gerado é ótimo quando o JSON é **previsível e bem-comportado**. Mas ele costuma ser **rígido**:
espera o tipo exato e pode quebrar se a API mandar um número como string ou omitir um campo. Escrevendo
à mão com os helpers defensivos, o Descontai ganha **controle total** sobre "o que fazer quando o dado
vem estranho" — aceitar nomes alternativos, aplicar defaults seguros, tolerar versões antigas. É uma
troca consciente: **mais código, mais robustez.** Ambas as abordagens são válidas — o importante é
entender o *porquê* da escolha.
::

## Resumindo

- A API responde **JSON**, que chega como `Map<String, dynamic>` (valores `dynamic`). Usar isso direto
  é frágil.
- Um **model** é uma classe que espelha a resposta e tem um **`factory fromJson`** que traduz o `Map`
  cru num objeto tipado — a **fronteira** onde o JSON bagunçado vira objeto limpo.
- Campos são **anuláveis** (`int?`, `String?`) porque a API pode não mandar tudo (null safety).
- **Parse defensivo** (helpers como `_parseInt`, `_parseBool`, `_asString`) evita crashes: no pior
  caso, retorna um **default seguro** em vez de quebrar. Aceita nomes de chave alternativos e trata
  versões antigas da API.
- **Defaults erram para o lado seguro** (ex.: cupom sem info de disponibilidade → indisponível).
- Objetos **aninhados** viram models aninhados (`Empresa.fromJson` dentro de `ProdutosResponse`).
- `json_serializable`/`freezed` **geram** `fromJson`; o Descontai prefere escrever à mão nos models
  principais para ter **parse defensivo** — uma troca consciente de mais código por mais robustez.

::callout{type="tip" title="Fixando"}
Pense: *"o que acontece no Descontai se a API mandar `valorDesconto` como a string `\"15.5\"` em vez do
número `15.5`?"*. Com parse ingênuo, **crash**. Com `_parseDouble`, ele detecta que é String, faz
`double.tryParse` e devolve `15.5` — a tela nem percebe. Se você entendeu por que isso importa,
entendeu resiliência de cliente.
::
