Autenticação é a porta de entrada do app. Nesta aula você vai seguir o **login de ponta a ponta**: o
usuário digita e-mail e senha, e a gente acompanha esses dados até virarem uma **sessão ativa** que
abre a Home. É o momento em que tudo que você aprendeu (Dio, models, SessionCache, controllers) se
junta num fluxo real.

## Relembrando a outra ponta

No curso da API, você viu o `AuthController` e o `LoginUseCase`: eles recebem e-mail e senha, conferem
a senha com BCrypt e devolvem um **access token** (JWT) + um **refresh token**. Aqui, a gente está do
lado de **quem chama** esse endpoint e **guarda** o que ele devolve. As duas pontas do login,
finalmente juntas.

## O fluxo de login em passos

Segundo o documento de arquitetura, o login faz exatamente isto:

::steps
### A tela chama o serviço
A `LoginPage` (um StatefulWidget, com os campos e o checkbox "lembrar") captura e-mail/senha e chama
`AuthService.login(email, senha, remember)`.

### O serviço faz o POST na API
O `AuthService` manda `POST /auth/login` pelo Dio e recebe o JSON com token, refresh token e dados do
usuário.

### Guarda a sessão
`SessionCache.saveAuth(...)` guarda o token (com segurança) e os dados na memória e no armazenamento.

### Registra o token de push
`NotificationService.registerToken()` avisa a API o token FCM deste aparelho (para receber
notificações).

### Sincroniza o usuário
`UserController.sincronizarUsuario()` busca `/usuarios/me` para garantir cidade, indicador e validade
da assinatura corretos.

### Carrega a Home e navega
`HomeController.carregarDados()` busca os cupons e o app vai para `/base`.
::

::callout{type="warning" title="A ordem importa: sincronizar ANTES da Home"}
Repare no passo 5. O `UserController.sincronizarUsuario()` vem **antes** da Home — e isso é uma regra
crítica do projeto. Por quê? Porque a Home precisa da **cidade** do usuário para buscar os cupons
certos (`/produtos?cidadeId=...`). Se a Home carregasse antes de a cidade estar resolvida, viria vazia
ou errada. Ordem de operações não é detalhe — é o que faz o app abrir **certo**.
::

## O AuthService de verdade

Olha o coração do `login` real (`lib/core/data/services/auth_service.dart`), comentado:

```dart
Future<LoginResult> login(String email, String password, bool remember) async {
  try {
    final response = await _dio.post(
      AppEndpoints.login,
      data: {'email': email, 'senha': password}, // o corpo que a API espera
    );

    // 1) traduz o JSON num objeto seguro (Aula 10)
    final loginResponse = LoginResponse.fromJson(response.data as Map<String, dynamic>);
    final token = loginResponse.accessToken;

    // 2) sem token? trata como falha amigável
    if (token == null || token.isEmpty) {
      return const LoginResult.error('Falha no login. Verifique seus dados.');
    }

    // 3) guarda a sessão inteira (token, refresh, dados do usuário)
    await SessionCache.saveAuth(
      tokenValue: token,
      refreshTokenValue: loginResponse.refreshToken,
      rememberValue: remember,
      email: loginResponse.userEmail ?? email,
      name: loginResponse.userName,
      id: '${loginResponse.id ?? ''}',
      cidadeId: loginResponse.cidadeId,
      cidade: loginResponse.cidade,
      codigoIndicacao: loginResponse.codigoIndicacao,
      saldoIndicacao: loginResponse.saldoIndicacao,
      indicador: loginResponse.indicador,
    );

    await _registerFcmToken();     // 4) registra push
    return const LoginResult.ok(); // 5) sucesso
  } on DioException catch (e) {
    return LoginResult.error(_formatDioError(e, context: _AuthContext.login)); // erro amigável
  }
}
```

Repare como tudo se encaixa: o **Dio** (Aula 9) faz o POST, o **model** (Aula 10) traduz a resposta, o
**SessionCache** guarda, o **AppError** (Aula 11) transforma a falha em mensagem. O login é o "encontro"
de todas as peças do curso.

::callout{type="tip" title="Por que checar se o token veio vazio?"}
No passo 2, mesmo com a API respondendo 200, o app confere se o token **realmente** veio. É parse
defensivo (Aula 10) aplicado à segurança: **nunca** assuma que a resposta está perfeita. Se o token
faltar, o app mostra "verifique seus dados" em vez de guardar uma sessão quebrada que falharia na
próxima tela. Desconfiar da resposta é maturidade.
::

## O token vira a sessão: SessionCache.saveAuth

O `saveAuth` é onde a resposta do login **vira sessão**. Ele faz três coisas:

::fields
::field{name="Guarda em memória"}
Coloca token, `userId`, `cidadeId`, `indicador` etc. nos campos estáticos do `SessionCache`, para
leitura instantânea (o interceptor do Dio e o guard do router leem daqui, sem tocar no disco).
::
::field{name="Persiste os dados não sensíveis"}
Grava `id`, `cidade`, `codigoIndicacao` etc. no `SharedPreferences`, para sobreviver ao app fechar.
::
::field{name="Persiste o token com segurança (se remember)"}
Se o usuário marcou "lembrar de mim", grava o token no **secure storage** (criptografado) — nunca no
SharedPreferences. Isso é o tema da próxima aula.
::
::

E, ao final, `saveAuth` chama `_notifyChanged()` — que dispara o `SessionCache.listenable`. Lembra do
`refreshListenable` do router (Aula 6)? É este aviso que faz o GoRouter perceber "agora há token!" e
liberar as telas protegidas. Login e navegação, conectados por um único sinal.

## Cadastro e reset: o mesmo padrão

O `register` e o `resetPassword` seguem a mesma receita: montam o corpo, chamam o Dio no endpoint
certo (`/auth/register`, `/auth/redefinir-senha`), tratam a resposta e devolvem um resultado
amigável. O cadastro inclusive já faz **auto-login** ao final — o usuário sai do cadastro direto para
dentro do app, sem precisar logar de novo. Menos atrito, melhor experiência.

::callout{type="info" title="Mensagens de erro por contexto"}
Repare no `AuthService`: um 401 no **login** vira *"E-mail ou senha inválidos"*; um 409 no **cadastro**
vira *"E-mail já cadastrado"*. O mesmo código de status ganha uma mensagem diferente conforme o fluxo.
Isso é cuidado com o usuário: a mensagem fala a língua da tela em que ele está.
::

## Resumindo

- O login liga todas as peças: **tela → `AuthService` (Dio) → `LoginResponse.fromJson` → `SessionCache.
  saveAuth` → registro de push → sincroniza usuário → carrega Home → navega**.
- **Ordem crítica:** sincronizar o usuário (`/usuarios/me`) vem **antes** da Home, porque a Home precisa
  da **cidade** para buscar os cupons certos.
- O `AuthService` faz **parse defensivo** (confere se o token veio), guarda a sessão e converte falhas
  em **mensagens amigáveis** por contexto (login vs cadastro).
- `SessionCache.saveAuth` transforma a resposta em sessão: memória (leitura rápida), SharedPreferences
  (dados não sensíveis) e secure storage (token, se "lembrar"). Ao final, **avisa** o router via
  `listenable`.
- Cadastro e reset seguem o mesmo padrão; o cadastro já faz **auto-login**.

::callout{type="tip" title="Fixando"}
Trace o token no login: *"a API devolve o token → `LoginResponse.fromJson` o extrai → `saveAuth` o
guarda na memória e no secure storage → o `listenable` avisa o router → o guard vê o token e libera a
Home"*. Se você seguiu o token do JSON até a tela liberada, entendeu autenticação no cliente.
::
