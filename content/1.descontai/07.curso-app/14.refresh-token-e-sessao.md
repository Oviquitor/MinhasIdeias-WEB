No curso da API você viu que o **access token expira rápido** (uns 15 minutos) de propósito — se
vazar, o estrago é limitado. Mas isso cria um problema no app: o usuário está lá, usando tranquilo, e
de repente o token vence. Jogá-lo para o login no meio de uma ação seria péssimo. A solução é o
**refresh token**, e a forma como o Descontai o implementa é uma das partes mais elegantes do código.

## O par de tokens (relembrando)

O login devolve **dois** tokens:

::fields
::field{name="Access token" type="curto, ~15 min"}
O que vai em toda requisição (`Bearer`). Expira rápido para limitar o dano se for roubado.
::
::field{name="Refresh token" type="longo, dias"}
Uma credencial guardada com segurança (secure storage, Aula 13) que serve **só** para pedir um novo
access token quando o antigo expira — sem exigir e-mail e senha de novo.
::
::

::callout{type="info" title="A analogia da pulseira do show"}
O **access token** é como a pulseira de papel de um show que **vale só o dia**: prova que você pagou,
mas descolou/venceu rápido. O **refresh token** é o seu **comprovante de compra**: com ele, você vai
na bilheteria e pega uma pulseira nova sem comprar outro ingresso. O app faz exatamente isso: quando a
pulseira (access) vence, ele usa o comprovante (refresh) para pegar uma nova — em silêncio, sem
incomodar você.
::

## O gatilho: o 401 no interceptor

Lembra do interceptor de erro do Dio (Aula 9)? É ali que a renovação começa. Quando uma requisição
volta com **401** (token expirado) numa rota protegida, o app **não** desloga na hora — ele tenta
renovar primeiro:

```dart
onError: (error, handler) async {
  final status = error.response?.statusCode;
  final alreadyRetried = requestOptions.extra[_retriedFlag] == true;
  final hasRefreshToken = SessionCache.refreshToken != null &&
      SessionCache.refreshToken!.isNotEmpty;

  // 401 em rota protegida, ainda não tentei renovar, e tenho refresh token:
  if (status == 401 && !isAuthEndpoint && !alreadyRetried && hasRefreshToken) {
    final refreshed = await _refreshSession();      // 1) tenta renovar
    if (refreshed) {
      try {
        final retried = await _retry(requestOptions); // 2) repete a requisição original
        return handler.resolve(retried);              //    e devolve o resultado
      } catch (_) { /* falhou de novo → encerra abaixo */ }
    }
    await SessionCache.clearAll();  // 3) não deu: encerra a sessão (vai pro login)
  }
  // ...
}
```

Leia como uma tentativa de resgate: **renova o token → repete a requisição que falhou → se der certo,
o usuário nem percebe**. Só se a renovação falhar (refresh também expirou/revogado) é que o app
encerra a sessão e manda para o login. Na maioria das vezes, o vencimento do token é **invisível**
para o usuário.

::callout{type="tip" title="Duas travas espertas nesse if"}
Repare em `!alreadyRetried` e no `_retriedFlag`. Quando o app repete a requisição, ele a **marca** com
uma flag. Se ela falhar com 401 **de novo**, a flag impede uma nova tentativa de refresh — cortando um
possível **loop infinito** de "expirou → renova → expirou → renova...". É o tipo de detalhe que separa
código que "funciona no happy path" de código que aguenta o mundo real.
::

## O problema do refresh: a corrida

Agora imagine a Home abrindo: ela dispara **várias** requisições ao mesmo tempo (produtos, categorias,
histórico). Se o token expirou, **todas** voltam 401 **quase juntas**. Sem cuidado, cada uma tentaria
renovar o token — cinco renovações simultâneas do **mesmo** token! Isso é desperdício e pode até dar
conflito (a segunda renovação invalidaria a primeira).

::callout{type="info" title="O padrão single-flight"}
**Single-flight** ("voo único") resolve isso: quando várias chamadas precisam da **mesma** coisa cara
ao mesmo tempo, só a **primeira** executa; as outras **esperam o mesmo resultado**. É como cinco
pessoas pedindo "me traz água" ao mesmo tempo — o garçom traz **uma** jarra e serve todo mundo, em vez
de fazer cinco viagens à cozinha.
::

## Single-flight no código real

Olha como o Descontai garante uma renovação **única** compartilhada:

```dart
Future<bool>? _refreshOperation; // guarda a renovação EM ANDAMENTO

Future<bool> _refreshSession() {
  // Se já existe uma renovação rolando, devolve A MESMA para todos.
  // Só cria uma nova se não houver nenhuma em andamento.
  return _refreshOperation ??=
      _performRefresh().whenComplete(() => _refreshOperation = null);
}
```

O `??=` é a mágica: "se `_refreshOperation` for nulo, crie a renovação; senão, use a que já existe". A
primeira requisição que chega no 401 inicia o refresh; as outras quatro que chegam logo atrás **pegam
a mesma operação** e esperam o mesmo resultado. Uma renovação, cinco requisições servidas. Ao terminar,
o `whenComplete` limpa a variável para a próxima vez.

## O Dio isolado: evitando recursão

Um último detalhe fino. A renovação em si é uma chamada HTTP (`POST /auth/refresh`). Se ela usasse o
**mesmo** Dio com o **mesmo** interceptor, e ela também falhasse com 401... o interceptor tentaria
renovar de novo, gerando **recursão**. Por isso o `_performRefresh` cria um **Dio separado, sem
interceptors**:

```dart
// Dio isolado (sem interceptors) para evitar recursão de refresh.
final client = Dio(BaseOptions(baseUrl: AppEndpoints.baseUrl, /* timeouts */));
final response = await client.post(AppEndpoints.refresh, data: {'refreshToken': refresh});
```

Se a renovação der certo, o app guarda o novo par de tokens com `SessionCache.updateTokens(...)`
(coerente com a política de remember/biometria da Aula 13) e volta `true`. Se der errado, volta
`false` — e o interceptor encerra a sessão.

::callout{type="warning" title="Por que tanto cuidado por um token?"}
Pode parecer exagero: flag anti-loop, single-flight, Dio isolado. Mas o refresh acontece **no meio da
experiência do usuário**, com várias requisições concorrentes, em rede instável. Cada uma dessas
proteções evita um bug real e chato de reproduzir: loop infinito, tela deslogando sozinha, cinco
renovações brigando. É o tipo de robustez que o usuário nunca vê — e é justamente por isso que ele
nunca tem o problema.
::

## Resumindo

- O **access token** é curto (~15 min); o **refresh token** é longo e serve **só** para pedir um novo
  access token — como uma pulseira de show (vence) e o comprovante de compra (renova).
- No **interceptor de erro**, um 401 dispara: **renova → repete a requisição original → o usuário nem
  percebe**. Só encerra a sessão se a renovação falhar.
- Uma **flag** (`_retriedFlag`) evita **loop infinito** de refresh quando a repetição também falha.
- **Single-flight**: várias requisições que estouram 401 juntas compartilham **uma única** renovação
  (via `_refreshOperation ??= ...`), em vez de renovar o token várias vezes.
- A renovação usa um **Dio isolado sem interceptors** para não cair em **recursão**.

::callout{type="tip" title="Fixando"}
Explique o single-flight com o exemplo da Home: *"a Home dispara 4 requisições, o token expirou, as 4
voltam 401 quase juntas; sem single-flight seriam 4 renovações; com ele, a 1ª renova e as outras 3
esperam o mesmo resultado"*. Se você entendeu por que uma renovação só é suficiente (e melhor),
entendeu o padrão.
::

recurso que traz o usuário **de volta** ao app: as **notificações push** com Firebase.
