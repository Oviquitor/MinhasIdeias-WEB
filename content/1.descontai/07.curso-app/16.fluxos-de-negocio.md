Você já tem todas as peças técnicas. Nesta aula, a gente as junta nos **fluxos que dão sentido ao
app**: mostrar os cupons da cidade certa, usar um cupom no balcão, e assinar/pagar. São as regras de

## Cidade: a chave que destrava os cupons

Tudo na Home gira em torno de uma pergunta: **em qual cidade o usuário está?** Porque os cupons são
por cidade (`GET /produtos?cidadeId=...`). Se a cidade estiver errada ou vazia, a Home vem vazia. Por
isso o app trata a resolução da cidade com muito cuidado. O `HomeController` real resolve o `cidadeId`
numa **ordem de fallback**:

```dart
// 1) tenta o cidadeId em memória (SessionCache)
var cidadeId = SessionCache.cidadeId;
// 2) se não houver, tenta o salvo em disco (SharedPreferences)
cidadeId ??= prefs.getInt('cidadeId');
// 3) último recurso: descobrir o id pelo NOME da cidade em cache
cidadeId ??= await _resolverCidadeIdPorNome(prefs);

// só então busca os cupons
final result = await _useCase(usuarioId: ..., cidadeId: cidadeId);
```

::callout{type="warning" title="A regra de ouro da Home: cidade primeiro"}
O documento de arquitetura é enfático: *"A Home deve carregar produtos somente depois de existir
cidade resolvida."* É por isso que, no login (Aula 12), a gente **sincroniza o usuário antes** da Home
— para a cidade já estar lá. E é por isso que essa cascata de fallback existe: o app tenta de tudo
(memória → disco → nome) para nunca abrir a Home "cega". Fluxos que dependem de um dado precisam
**garantir** esse dado antes de agir.
::

O usuário também pode **trocar a cidade** manualmente na barra do topo (`CustomLocationAppBar`). Quando
troca, o app grava a nova cidade no SharedPreferences e no `SessionCache`, e chama
`HomeController.recarregar()` — e a Home busca os cupons da cidade nova. Um dado muda, a tela reflete.

## As seções da Home: derivar, não pedir de novo

A Home mostra "Top 5", "Gastronomia", "Serviços". O app **não** faz uma chamada separada para cada
seção — ele busca a lista **uma vez** e **deriva** as seções dela na memória (lembra dos valores
derivados da Aula 7?):

```dart
void _buildDerivedLists() {
  // Top 5: produtos com posicaoTop > 0, ordenados
  top5 = produtos.where((p) => p.posicaoTop > 0).toList()
      ..sort((a, b) => a.posicaoTop.compareTo(b.posicaoTop));

  // Gastronomia: categoria que contém "gastronom"
  gastronomia = produtos
      .where((p) => p.categoria.toLowerCase().contains('gastronom'))
      .toList();

  // Serviços: categoria que contém "servico"/"serviço"
  servicos = produtos
      .where((p) => p.categoria.toLowerCase().contains('servico') ||
                    p.categoria.toLowerCase().contains('serviço'))
      .toList();
}
```

::callout{type="tip" title="Uma busca, várias visões"}
Buscar tudo de uma vez e **filtrar na memória** é mais rápido e mais leve que fazer 4 chamadas à API —
menos rede, menos espera, menos coisa que pode falhar. É a mesma lógica de "não bater no servidor à
toa" que você viu no cache da API. As seções são só **visões diferentes** da mesma lista.
::

## Usar o cupom: código ou QR

O momento-chave do produto: o usuário está no balcão e vai **usar** o cupom. O app oferece duas formas
de identificar o cupom, e ambas caem no mesmo endpoint da API:

::fields
::field{name="Por código" type="digitar"}
O usuário informa o código do cupom, e o app chama `POST /cupons/usar`. Simples.
::
::field{name="Por QR Code" type="escanear (mobile_scanner)"}
O app abre a câmera (`mobile_scanner`), lê o QR do estabelecimento e extrai o código — sem digitação.
Depois, o mesmo `POST /cupons/usar`.
::
::

```text
ProductDetailPage
  → usuário toca "Usar cupom"
  → informa código OU escaneia o QR
  → HomeService.usarCupom()  → POST /cupons/usar
  → HomeController.recarregar()  (atualiza o estado do cupom)
```

::callout{type="info" title="Quem DECIDE se pode usar? A API."}
O app mostra o botão "usar", mas quem **valida** — assinatura ativa? cupom já usado neste ciclo? — é a
**API** (você viu essa regra no `UsuarioProdutoService` do curso da API). O app manda o pedido e
**mostra a resposta**: sucesso ou a mensagem de erro que a API devolveu. É a regra da Aula 1 na
prática: **o cliente é a vitrine; a API é o segurança**. Nunca confie só no app para liberar um uso.
::

## Assinatura: a regra do "ativo"

Muita coisa no app depende de o usuário ter **assinatura ativa**. A regra, definida no `UserController`:

> *"A assinatura é ativa se a data de validade for maior que o fim do dia atual."*

- Usuário **com** assinatura ativa → pode usar cupom.
- Usuário **sem** assinatura ativa → é direcionado para `/assinatura` (a tela de planos).

O app guarda a `validade` (vinda de `/usuarios/me` e do pagamento) e compara com a data de hoje. É a
mesma ideia de "validade e ciclo" que a API controla — o app só **reflete** esse estado para dar a
experiência certa (mostrar "assine para usar" ou liberar o cupom).

## Pagamento: Pix e cartão

Para assinar, o usuário escolhe um plano (`GET /planos`) e paga. Dois caminhos, ambos via Asaas (que
você viu na API):

```text
SubscriptionPlansPage → escolhe plano → SubscriptionPaymentPage
  → Pix:    POST /pagamentos/asaas/pix     → mostra QR/código para copiar
  → Cartão: POST /pagamentos/asaas/cartao  → envia os dados do cartão
```

::callout{type="danger" title="Cuidado extra com o cartão"}
Cartão é o dado mais sensível do app. As regras do Descontai são explícitas:

- **CVV e campos sensíveis são limpos da memória após a tentativa** de pagamento — não ficam
  "pendurados" no formulário.
- A validade `MM/AA` é **normalizada** para ano completo antes de enviar.
- **Cartão expirado é bloqueado no próprio app**, antes mesmo de chamar a API — validação de borda.

E lembre do que a própria doc do projeto alerta: em produção, o ideal é **tokenizar** o cartão no
provedor de pagamento (padrão PCI), para o dado sensível trafegar o mínimo possível. O app já toma o
cuidado de **não reter** o CVV — a mesma disciplina de "não logar cartão/CVV" que vimos na Aula 9.
::

## O mapa dos fluxos

Juntando tudo o que você aprendeu, o app inteiro é uma coreografia dessas peças:

::steps
### Entrar
Login/biometria → token no secure storage → sincroniza usuário (cidade!) → Home.

### Descobrir
Home resolve a cidade → busca cupons → deriva Top 5 / Gastronomia / Serviços → usuário navega e busca.

### Usar
Abre o detalhe → usa o cupom (código ou QR) → API valida → estado atualiza.

### Assinar
Sem assinatura ativa → tela de planos → paga por Pix ou cartão → validade atualiza → libera os cupons.
::

Cada seta dessas passou por uma aula deste curso. Você já viu o app inteiro por dentro.

## Resumindo

- **Cidade** é a chave da Home: o `HomeController` resolve o `cidadeId` numa cascata (memória → disco
  → nome) e **só busca cupons depois de tê-la** — por isso o usuário é sincronizado antes da Home.
- As **seções** (Top 5, Gastronomia, Serviços) são **derivadas** de uma única lista de produtos na
  memória — uma busca, várias visões.
- **Usar cupom** funciona por **código** ou **QR** (`mobile_scanner`), ambos via `POST /cupons/usar`;
  quem **valida** é a API, não o app.
- **Assinatura ativa** = validade maior que o fim do dia atual; sem ela, o usuário vai para os planos.
- **Pagamento** por Pix ou cartão via Asaas; o cartão tem cuidados extras: **CVV limpo após a
  tentativa**, validade normalizada e cartão expirado bloqueado no app.

::callout{type="tip" title="Fixando"}
Pegue "o usuário abre o app e usa um cupom" e narre o caminho inteiro com o vocabulário do curso:
*"biometria destrava o token → sincroniza usuário (cidade) → Home resolve cidadeId e busca cupons →
deriva as seções → detalhe → usar cupom por QR → API valida → estado atualiza"*. Se você consegue
contar essa história ponta a ponta, você domina o app.
::
