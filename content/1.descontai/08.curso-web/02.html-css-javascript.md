Antes de Vue e Nuxt, você precisa da base sobre a qual **tudo** na web é feito: **HTML, CSS e
JavaScript**. Não vamos esgotar cada um (são mundos), mas dar o essencial para você **entender o que
o Vue faz por baixo**. Porque o Vue não substitui esses três — ele os organiza melhor.

## Os três pilares, com uma analogia

::callout{type="info" title="A casa: estrutura, pintura e eletricidade"}
Pensa numa **casa**. O **HTML** são as **paredes e cômodos** — a estrutura: aqui tem um título, ali um
parágrafo, acolá um botão. O **CSS** é a **pintura e a decoração** — cor, tamanho, espaçamento: deixa a
estrutura bonita. O **JavaScript** é a **eletricidade** — o que dá vida: quando você aperta o
interruptor (clica no botão), algo acontece. Três camadas, três papéis. Todo site é isso.
::

## HTML: a estrutura

**HTML** (HyperText Markup Language) descreve **o que tem** na página, usando **tags** — etiquetas
entre `< >`. Cada tag tem um significado:

```html
<h1>Meus cupons</h1>              <!-- um título principal -->
<p>Gerencie suas ofertas.</p>      <!-- um parágrafo -->
<button>Novo cupom</button>        <!-- um botão -->
<input placeholder="Buscar…" />    <!-- um campo de texto -->
```

As tags se aninham (uma dentro da outra), formando uma **árvore** — a mesma ideia de árvore de widgets
do Flutter, se você fez o curso do app. Um `<div>` (uma "caixa" genérica) pode conter um `<h1>` e um
`<button>`, e assim por diante.

::callout{type="tip" title="Semântica importa"}
Use a tag que **significa** o que você quer: `<button>` para um botão (não um `<div>` clicável),
`<nav>` para navegação, `<form>` para formulário. Isso ajuda acessibilidade (leitores de tela) e SEO.
Repare no portal: a sidebar usa `<nav>` e `<aside>`, o login usa `<form>`. HTML semântico é código que
se explica.
::

## CSS: a aparência

**CSS** (Cascading Style Sheets) diz **como as coisas aparecem**: cor, tamanho, posição, espaçamento.
Você escreve **regras** que selecionam elementos e aplicam estilos:

```css
.title {
  font-size: 14.5px;
  font-weight: 600;
  color: #14110e;
}
.dc-btn--primary {
  background: #e83d22;   /* o laranja da marca Descontai */
  color: #fff;
  border-radius: 12px;
}
```

O `.title` é um **seletor de classe**: ele aplica o estilo a todo elemento com `class="title"`. No
portal, você vê classes assim o tempo todo (`dc-btn`, `dc-card`, `kpi`).

::callout{type="info" title="Variáveis CSS: a base do design system"}
Repare no portal que as cores não são escritas cruas em cada lugar — elas usam `var(--brand)`,
`var(--text)`, `var(--border)`. Essas são **variáveis CSS**: você define a paleta **uma vez** (em
`assets/css/main.css`) e usa por toda parte. Muda a cor da marca? Um lugar só. É o mesmo princípio de
"centralizar o visual" que vimos no app — e é o coração de um **design system**. Voltamos nisso na
Aula 7.
::

## JavaScript: o comportamento

**JavaScript** (JS) é a linguagem que dá **vida** à página: reage a cliques, muda o conteúdo, chama a
API. Se você já viu qualquer linguagem, reconhece a estrutura:

```js
let contador = 0                       // variável
function usarCupom() {                 // função
  contador = contador + 1
  console.log('Cupons usados:', contador)
}
// quando o botão for clicado, chama a função:
botao.addEventListener('click', usarCupom)
```

JS é o que transforma HTML+CSS estáticos num app interativo. **O Vue é escrito em cima do
JavaScript** — ele não substitui o JS, ele te dá um jeito muito melhor de organizá-lo.

## TypeScript: JavaScript com segurança

O portal usa **TypeScript** (TS), que é **JavaScript + tipos**. Em JS puro, uma variável pode ser
qualquer coisa e você só descobre o erro rodando. Em TS, você **declara os tipos** e o editor te avisa
o erro **enquanto você escreve**:

```ts
function saudar(nome: string): string {   // 'nome' é texto, retorna texto
  return `Olá, ${nome}`
}
saudar(42)  // ❌ ERRO na hora: 42 não é string
```

::callout{type="tip" title="Por que TypeScript vale a pena"}
Lembra do `null safety` do Dart (curso do app)? TypeScript traz a mesma ideia para a web: ele pega uma
montanha de bugs **antes** de o código rodar, e o autocompletar do editor fica muito melhor (ele
"sabe" os campos de cada objeto). No portal, os tipos das respostas da API ficam em `app/types/api.ts`
— assim o editor conhece o formato de um `ProdutoResponse` e te avisa se você errar um campo. Mais
código, muito menos bug.
::

## Como o Vue melhora tudo isso

Em JS puro, para atualizar a tela você teria que "achar" o elemento e mexer nele na mão
(`elemento.textContent = novoValor`) — trabalhoso e fácil de errar em telas grandes. O **Vue** resolve
isso com **reatividade**: você liga um dado à tela, e **quando o dado muda, a tela se atualiza
sozinha**. Você descreve **o que** mostrar, não **como** mexer em cada elemento. É a mesma UI
declarativa do Flutter — e o assunto central das próximas aulas.

## Resumindo

- Todo site tem três pilares: **HTML** (estrutura — tags aninhadas em árvore), **CSS** (aparência —
  regras de estilo) e **JavaScript** (comportamento — reage e muda a página).
- **Variáveis CSS** (`var(--brand)`) centralizam o visual — a base de um design system.
- **TypeScript** é JavaScript **com tipos**: pega bugs enquanto você escreve e melhora o autocompletar
  (os tipos da API ficam em `app/types/api.ts`).
- O **Vue** é escrito em cima do JS e traz **reatividade**: liga o dado à tela e a atualiza sozinho —
  você descreve o "o quê", não o "como".

::callout{type="tip" title="Fixando"}
Sabendo os três pilares, olhe qualquer tela do portal e separe as camadas: *"esse card é HTML
(estrutura), a cor laranja e o arredondado são CSS (aparência), e o clique no 'remover' que abre o
modal é JavaScript (comportamento)"*. Quem enxerga as três camadas entende a web por baixo.
::
