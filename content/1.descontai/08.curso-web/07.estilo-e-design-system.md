O portal não usa nenhuma biblioteca de UI (Aula 4) — todo o visual é feito à mão, e mesmo assim fica
**consistente e bonito**. Como? Com duas ideias: **CSS escopado** (para o estilo não vazar) e um
**design system de variáveis** (para o visual vir de um lugar só). Esta aula fecha a Parte 2.

## O problema do CSS que vaza

CSS tem uma armadilha clássica: ele é **global** por padrão. Se você escreve uma regra para `.title`
num arquivo, ela afeta **todo** `.title` do site — inclusive o de outra tela que você nem lembrava. Em
projetos grandes, isso vira um caos: mexer no estilo de uma tela quebra outra.

## A solução do Vue: style scoped

Lembra do SFC (Aula 3)? A terceira seção é `<style scoped>`. Aquele **`scoped`** é a mágica: ele faz o
CSS daquele componente afetar **só** aquele componente — nunca vaza para o resto. Olha o `DcKpi.vue`:

```vue
<style scoped>
.value {
  font-weight: 800;
  font-size: 32px;
}
</style>
```

Esse `.value` só estiliza o `.value` **dentro do `DcKpi`**. Se outra tela tiver um `.value`, ele não é
afetado. O Vue faz isso automaticamente (adicionando um marcador único a cada elemento do componente).

::callout{type="tip" title="scoped = cada peça cuida do seu visual"}
Com `scoped`, cada componente é **dono** do seu estilo, sem medo de colidir com os outros. Você mexe no
CSS do `DcKpi` com a certeza de que **só** o `DcKpi` muda. Isso deixa o estilo **local** e seguro — a
mesma tranquilidade que o `setState` local dá no estado. É o que torna viável ter dezenas de
componentes sem um "efeito dominó" de CSS.
::

## O design system: variáveis CSS

Se cada componente tem seu CSS escopado, como garantir que **todos** usem o mesmo laranja, a mesma
fonte, o mesmo arredondado? A resposta são as **variáveis CSS** — definidas **uma vez**, globalmente
(em `app/assets/css/main.css`), e usadas por todo componente:

```css
/* em main.css — a paleta e os tokens do design system */
:root {
  --brand: #e83d22;          /* o laranja da marca */
  --text: #14110e;           /* cor de texto principal */
  --muted: #8a8175;          /* texto secundário */
  --border: #e7e2d9;         /* bordas */
  --surface: #ffffff;        /* fundo de cards */
  --font-display: 'Sora', sans-serif;
}
```

E os componentes **consomem** essas variáveis com `var(...)`:

```css
.dc-btn--primary { background: var(--brand); }   /* usa o laranja central */
.title { color: var(--text); font-family: var(--font-display); }
```

::callout{type="info" title="Variáveis CSS = a fonte única da verdade do visual"}
Repare que **nenhum** componente escreve `#e83d22` cru — todos usam `var(--brand)`. Isso significa que
mudar a cor da marca é **um lugar só**: altera `--brand` no `main.css` e **todo** o portal muda junto.
É o mesmo princípio do `app_colors.dart` do app e do `docTypes.ts` do portal de docs. Um design system
é isso: **decisões visuais centralizadas** em tokens (cor, espaço, fonte) que todo o resto referencia.
::

### Design tokens: nomear em vez de "chutar números"

Repare que as variáveis têm **nomes com significado** (`--brand`, `--muted`, `--border`), não valores
soltos. Isso são **design tokens**: em vez de "cor cinza `#8a8175`", você pensa "cor de texto
secundário". Se amanhã esse cinza mudar, o **nome** continua o mesmo e o valor muda num lugar. Você
programa com **conceitos**, não com números mágicos.

## Classes utilitárias

Além das variáveis, o portal tem **classes reutilizáveis** para padrões que se repetem — botões
(`dc-btn`, `dc-btn--primary`, `dc-btn--ghost`), cards (`dc-card`), campos (`dc-input`), rótulos
(`dc-label`). Elas moram no CSS global e são usadas em qualquer template:

```vue
<button class="dc-btn dc-btn--primary">Novo cupom</button>
<div class="dc-card">...</div>
```

::callout{type="tip" title="Componentes x classes utilitárias"}
Duas ferramentas de reuso visual: um **componente** (`DcModal`) encapsula estrutura + estilo +
comportamento; uma **classe utilitária** (`dc-btn`) é só um pacotinho de CSS reaproveitável. Regra
prática: se tem **lógica** ou estrutura repetida, faça um **componente**; se é só um **conjunto de
estilos** que se repete (um botão padrão), uma **classe** basta. O portal usa os dois na medida certa.
::

## Responsividade: media queries

Um portal roda em telas grandes (desktop) e pequenas (celular). O CSS se adapta com **media queries** —
regras que só valem abaixo de um certo tamanho. Você viu isso no layout real (a sidebar que vira menu
deslizante) e na tela de cupons:

```css
.content { padding: 32px; }

@media (max-width: 880px) {   /* só em telas até 880px de largura */
  .content { padding: 20px 18px 80px; }  /* menos padding no celular */
  .search { display: none; }             /* esconde a busca para poupar espaço */
}
```

::callout{type="info" title="Pense em telas, não em pixels fixos"}
A media query permite **reorganizar** a interface conforme o espaço: no desktop, a sidebar fica fixa à
esquerda; no celular, ela sai da tela e desliza por cima ao tocar num botão (você viu isso no
`default.vue`). Não é "encolher tudo" — é **adaptar o layout** ao contexto. Um portal profissional
funciona bem no notebook do gestor **e** no celular dele no corredor.
::

## Resumindo

- CSS é **global** por padrão e vaza. O **`<style scoped>`** do SFC faz o estilo afetar **só** aquele
  componente — visual local e seguro.
- O **design system** vive em **variáveis CSS** (`--brand`, `--text`…) definidas uma vez no
  `main.css`. Componentes usam **`var(--brand)`** — mudar o visual é **um lugar só**.
- **Design tokens** são variáveis com **nome semântico** (`--muted` em vez de um cinza cru): você
  programa com conceitos.
- **Classes utilitárias** (`dc-btn`, `dc-card`) reaproveitam pacotes de estilo; use **componente** para
  estrutura+lógica, **classe** para só estilo.
- **Media queries** (`@media (max-width: 880px)`) adaptam o layout a telas menores — reorganizam, não
  só encolhem.

::callout{type="tip" title="Fixando"}
Explique como o portal fica consistente sem biblioteca de UI: *"cada componente tem CSS `scoped` (não
vaza), e todos consomem as **mesmas variáveis** do design system (`var(--brand)` etc.), definidas num
lugar só. Mudar a marca é mudar um token"*. Se você entendeu escopo + tokens, entendeu como se mantém
um visual coeso em escala.
::
