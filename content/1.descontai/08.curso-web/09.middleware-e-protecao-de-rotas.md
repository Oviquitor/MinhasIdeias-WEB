As telas do portal existem, mas nem todas podem ser abertas por qualquer um: quem não fez login **não**
pode ver os cupons. Quem faz esse controle é o **middleware** — o "porteiro" que roda **antes de cada
navegação**. Nesta aula você lê o guard real do portal e vê como ele conecta com a segurança que você
já conhece.

## O que é um middleware de rota

Um **middleware** é uma função que o Nuxt executa **antes** de mostrar uma rota, e que pode **deixar
passar** ou **redirecionar** para outro lugar. É o mesmo conceito do guard do GoRouter (curso do app) e
do filtro de segurança da API (curso da API): um ponto por onde toda navegação passa e é avaliada.

::callout{type="info" title="Middleware = o porteiro do prédio"}
Pense no **porteiro** de um prédio. Toda pessoa que quer subir passa por ele **antes** de pegar o
elevador. Morador com chave? Pode subir. Visitante sem autorização? É barrado e mandado para a
recepção. O middleware é esse porteiro das rotas: antes de te deixar entrar numa tela, ele checa se
você **pode**.
::

## O sufixo .global: aplica a todas as telas

O portal tem o arquivo `app/middleware/auth.global.ts`. Aquele **`.global`** no nome é uma convenção do
Nuxt: um middleware global roda **automaticamente antes de toda navegação**, sem você precisar declarar
em cada página. Um arquivo, proteção para o portal inteiro.

## O guard real, linha a linha

Olha o `auth.global.ts` completo — é curtinho e faz três decisões:

```ts
export default defineNuxtRouteMiddleware((to) => {
  // 1) No servidor (SSR) não há localStorage; sai cedo e deixa o cliente decidir.
  if (import.meta.server) return

  const { isAuthenticated, hydrate } = useAuth()
  // 2) Garante que a sessão foi lida do localStorage antes de checar.
  hydrate()

  const isLogin = to.path === '/login'

  // 3a) Não logado tentando tela interna → vai para o login.
  if (!isAuthenticated.value && !isLogin) {
    return navigateTo('/login')
  }
  // 3b) Já logado tentando abrir o login → vai para a home.
  if (isAuthenticated.value && isLogin) {
    return navigateTo('/')
  }
  // Demais casos: deixa passar (retorno vazio).
})
```

Leia as três decisões como frases:

::steps
### Está no servidor? Sai cedo.
`import.meta.server` é verdadeiro durante a renderização no servidor. Como a sessão vive no
**localStorage** (que só existe no navegador), no servidor não dá para saber se está logado — então o
guard **não decide nada** ali e deixa o cliente resolver. (No portal, que é SPA, isso é uma proteção
extra.)

### Garante a sessão carregada (hydrate).
O `hydrate()` lê a sessão do localStorage para a memória, **antes** de checar `isAuthenticated`. Isso
cobre o caso de o usuário digitar uma URL direto — a sessão precisa estar pronta na hora da checagem.

### Decide: barra ou redireciona.
Não logado numa tela interna → **manda para `/login`**. Já logado tentando ver o `/login` → **manda
para `/`** (não faz sentido relogar). Qualquer outro caso → **deixa passar**.
::

::callout{type="tip" title="As duas regras espelham o guard do app"}
Repare que são exatamente as duas regras do GoRouter no curso do app: "sem token e tela protegida →
login" e "com token e tela de login → home". A **ideia de autorização de cliente é universal** — muda a
sintaxe (Nuxt aqui, GoRouter lá), não o raciocínio. Quem entendeu num, entende no outro.
::

## De onde vem o "está logado?"

O guard pergunta `isAuthenticated.value` — mas quem responde isso? O composable **`useAuth`** (que a
Aula 12 detalha). Por ora, o essencial: `isAuthenticated` é um valor reativo que é `true` quando existe
uma sessão salva. O guard só **consulta** essa verdade; quem a **mantém** é o `useAuth`. Separação de
responsabilidades: o porteiro checa o crachá, mas não é ele quem emite o crachá.

## A regra de ouro (de novo)

::callout{type="warning" title="O middleware é UX, não segurança de verdade"}
Assim como no app: este guard impede o usuário **comum** de ver uma tela sem login — é **boa
experiência**. Mas ele roda no **navegador**, que o usuário controla. Alguém mal-intencionado pode
burlar o front-end e chamar a API direto. Por isso, **a segurança real está na API**: cada rota
protegida exige o `Bearer` token e é validada no servidor (curso da API). O middleware embeleza; a API
protege. **Nunca** troque uma pela outra.
::

## Middleware por página (quando precisar)

Além do `.global`, você pode ter middlewares que rodam **só em páginas específicas** (sem o `.global` no
nome, declarados via `definePageMeta({ middleware: 'nome' })`). Serve para regras pontuais — por
exemplo, "só admin abre esta tela". No portal de Empresas, o guard global basta; mas é bom saber que a
granularidade existe, como o guard por-rota do GoRouter (`indicador == true`) no app.

## Resumindo

- Um **middleware de rota** roda **antes de cada navegação** e pode **deixar passar** ou
  **redirecionar** — o "porteiro" das telas.
- O sufixo **`.global`** faz o middleware valer para **todas** as páginas automaticamente.
- O `auth.global.ts` faz três coisas: **sai cedo no servidor** (sem localStorage), **hidrata** a sessão,
  e decide: **não logado → `/login`**, **logado no `/login` → `/`**, senão passa.
- Quem responde "está logado?" é o composable **`useAuth`** (Aula 12); o guard só **consulta**.
- **Regra de ouro:** o middleware é **UX** (roda no navegador, burlável); a **segurança real está na
  API**, que valida o token no servidor.

::callout{type="tip" title="Fixando"}
Explique o que acontece se um usuário deslogado digitar `/cupons` na barra do navegador: *"o middleware
global roda antes da tela, hidrata a sessão, vê que `isAuthenticated` é falso e a rota não é `/login`,
então redireciona para `/login`"*. E complete: *"mas mesmo que ele burlasse isso, a API barraria a
requisição sem token"*. Se você amarrou as duas camadas, entendeu proteção de rotas.
::
