O guard (Aula 9) perguntava "está logado?" e o `useApi` (Aula 11) injetava "o token da sessão". Agora
você conhece quem responde a isso: o **`useAuth`**, o composable que gerencia a **sessão** da empresa
logada. Vamos ver o login de ponta a ponta e como a sessão sobrevive a um refresh da página.

## O que é a sessão aqui

No portal, a **sessão** é um objeto com o essencial da empresa logada:

```ts
interface Session {
  token: string              // o JWT — vai no Bearer de cada requisição
  refreshToken: string | null
  empresa: EmpresaResponse   // dados da empresa (nome, cidade…)
}
```

Esse é o "crachá" do usuário: enquanto existir uma sessão, ele está logado; quando ela some, ele volta
para o login.

## Onde a sessão vive: useState + localStorage

A sessão precisa de duas coisas: estar **disponível para todo o portal** (o guard, o `useApi`, a
sidebar) e **sobreviver a um refresh** da página (senão o usuário deslogava ao apertar F5). O `useAuth`
combina duas ferramentas:

::fields
::field{name="useState" type="disponível em todo o portal"}
A sessão vive num `useState('dc-session')` (Aula 10) — estado global compartilhado. Assim o guard, o
`useApi` e a sidebar leem a **mesma** sessão.
::
::field{name="localStorage" type="sobrevive ao refresh"}
O `localStorage` é uma "gavetinha" do navegador que **persiste** mesmo fechando a aba. A sessão é
salva ali para, ao reabrir/atualizar, ser **restaurada**.
::
::

```ts
const STORAGE_KEY = 'descontai_empresa_session'

function persist(s: Session | null) {
  session.value = s                                    // atualiza o estado global
  if (!import.meta.client) return
  if (s) localStorage.setItem(STORAGE_KEY, JSON.stringify(s)) // salva no navegador
  else localStorage.removeItem(STORAGE_KEY)             // ou limpa (logout)
}

function hydrate() {
  if (!import.meta.client || session.value) return
  const raw = localStorage.getItem(STORAGE_KEY)         // lê do navegador
  if (raw) session.value = JSON.parse(raw)              // restaura para o estado
}
```

O `hydrate()` é o que o guard chamava (Aula 9): ele **lê a sessão do localStorage para a memória**, para
o portal saber que você está logado logo ao abrir.

::callout{type="warning" title="localStorage x cookie httpOnly — o trade-off honesto"}
O próprio código comenta: *"Em produção, considere cookies httpOnly emitidos pela API"*. Por quê?
Token em `localStorage` é **acessível por JavaScript** — se o site tiver uma falha de XSS (script
malicioso injetado), o token pode vazar. Um cookie **httpOnly** não é acessível por JS, o que é mais
seguro. O portal usa localStorage pela **simplicidade** (funciona sem mudanças no back), mas registra o
caminho mais seguro. É a mesma honestidade dos outros cursos: **documentar o trade-off**, não fingir
que não existe.
::

## O login de ponta a ponta

A tela `login.vue` (que usa o layout `auth`) coleta e-mail/senha e chama `useAuth().login()`. Olha o
handler real da tela:

```ts
async function onSubmit() {
  erro.value = ''
  loading.value = true
  try {
    await login({ email: email.value, senha: senha.value })  // chama o composable
    toast.success('Bem-vindo de volta!')
    await navigateTo('/')                                     // entra no portal
  } catch (e: any) {
    erro.value = e?.message || 'Não foi possível entrar. Verifique suas credenciais.'
  } finally {
    loading.value = false
  }
}
```

E o `login()` do `useAuth`, no modo real, faz o `POST /auth/login` (o mesmo endpoint do curso da API) e
guarda a sessão:

```ts
async function login(credentials: LoginRequest) {
  if (config.public.mockMode === 'true') {
    // Modo demonstração: qualquer credencial entra como a empresa de exemplo.
    persist({ token: 'mock-token', refreshToken: 'mock-refresh', empresa: MOCK_EMPRESA })
    return
  }
  // === API real ===
  const res = await $fetch('/auth/login', { baseURL: config.public.apiBase, method: 'POST', body: credentials })
  // ...busca os dados da empresa vinculada...
  persist({ token: res.token, refreshToken: res.refreshToken, empresa })
}
```

::callout{type="info" title="O mesmo login da API, visto do cliente"}
Repare: é o mesmo `POST /auth/login` que devolve o `TokenResponse` (curso da API). O portal pega o
`token`, monta a `Session` e a **persiste** com `persist(...)` — que salva no estado global **e** no
localStorage. A partir daí, o `useApi` injeta esse token em toda chamada (Aula 11) e o guard vê que você
está logado (Aula 9). Uma peça alimenta as outras.
::

## O que o useAuth expõe

No final, o `useAuth` devolve uma API limpa que o resto do portal consome:

```ts
return {
  session,
  empresa: computed(() => session.value?.empresa ?? null),   // dados da empresa
  token: computed(() => session.value?.token ?? null),       // o token (usado pelo useApi)
  isAuthenticated: computed(() => !!session.value),           // está logado? (usado pelo guard)
  hydrate, login, logout, updateEmpresa,
}
```

Repare nos **`computed`** (Aula 6): `isAuthenticated` é `true` sempre que existe sessão; `token` e
`empresa` derivam da sessão. Quem consome não mexe na sessão crua — usa esses valores derivados,
prontos e reativos. E o **`logout`** é o simétrico do login: `persist(null)` (limpa estado e
localStorage) + `navigateTo('/login')`.

::callout{type="tip" title="Um composable, muitos consumidores"}
Veja como o `useAuth` amarra o portal: a **tela de login** chama `login`; o **guard** lê
`isAuthenticated`; o **`useApi`** lê `token`; a **sidebar** lê `empresa` e chama `logout`. Todos falam
com a **mesma** sessão (é `useState`, global). Mudou a sessão (login/logout) → todos reagem juntos. Esse
é o poder do estado compartilhado da Aula 10, aplicado à autenticação.
::

## Resumindo

- A **sessão** é o "crachá": `{ token, refreshToken, empresa }`. Existe sessão → logado.
- Ela vive em **`useState`** (global, todo o portal lê a mesma) **+ `localStorage`** (persiste, sobrevive
  ao refresh). O **`hydrate()`** restaura do localStorage para a memória ao abrir.
- **Trade-off honesto:** localStorage é simples mas exposto a XSS; cookie **httpOnly** é mais seguro (o
  código registra isso como evolução).
- O **login** chama `POST /auth/login` (o mesmo da API), monta a `Session` e a **persiste**; o
  **logout** limpa tudo e volta ao `/login`.
- O `useAuth` expõe **`computed`** derivados: `isAuthenticated` (guard), `token` (useApi), `empresa`
  (sidebar) — todos reativos sobre a mesma sessão.

::callout{type="tip" title="Fixando"}
Trace o token no portal: *"o login faz `POST /auth/login`, guarda o token na `Session` (useState +
localStorage); o `useApi` lê esse token e o injeta como `Bearer`; o guard lê `isAuthenticated` para
liberar as telas; ao dar F5, `hydrate()` restaura tudo do localStorage"*. Se você amarrou login →
sessão → uso → persistência, dominou a autenticação no front.
::
