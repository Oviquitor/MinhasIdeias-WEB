Um portal de gestão vive de **CRUD**: **C**reate, **R**ead, **U**pdate, **D**elete — criar, ler,
editar e remover. Nesta aula você junta tudo (rotas, componentes, `useApi`, toasts) no fluxo completo
dos cupons, e aprende o padrão mais importante de comunicação entre componentes: **props descem,
eventos sobem**.

## As quatro operações e suas telas

O CRUD de cupons se espalha por três arquivos de página, cada um uma rota (Aula 8):

| Operação | Rota | Arquivo | Método do useApi |
|----------|------|---------|------------------|
| **Read** (listar) | `/cupons` | `cupons/index.vue` | `listCupons()` |
| **Create** (criar) | `/cupons/novo` | `cupons/novo.vue` | `createCupom(body)` |
| **Update** (editar) | `/cupons/:id` | `cupons/[id].vue` | `getCupom(id)` + `updateCupom(id, body)` |
| **Delete** (remover) | (na lista) | `cupons/index.vue` | `deleteCupom(id)` |

Cada operação chama um método do `useApi` (Aula 11) — a tela nunca vê a URL.

## Read: listar (revisão)

Você já viu na Aula 6: `onMounted(load)` busca a lista, guarda em `cupons` (um `ref`), e o template usa
`v-for` para renderizar os cards, com estados de carregando/erro/vazio. É a base — as outras operações
partem dela.

## Delete: remover com confirmação

Remover é **destrutivo** — apagar sem confirmar é um convite ao acidente. O padrão do portal: um clique
no lixeira **marca** o cupom para exclusão, o que abre um **modal de confirmação**; só o "Remover" do
modal executa de fato. Olha o real:

```ts
const confirmDelete = ref<ProdutoResponse | null>(null)  // cupom a remover (ou null)
const deleting = ref(false)

async function doDelete() {
  if (!confirmDelete.value) return
  deleting.value = true
  try {
    await api.deleteCupom(confirmDelete.value.id)
    toast.success('Cupom removido.')
    confirmDelete.value = null   // fecha o modal
    await load()                 // recarrega a lista
  } catch (e: any) {
    toast.error(e?.message || 'Erro ao remover.')
  } finally {
    deleting.value = false
  }
}
```

E no template, o botão só **abre** o modal, e o modal (reativo a `confirmDelete`) faz o resto:

```vue
<button class="icon-btn danger" @click="confirmDelete = c">🗑</button>

<DcModal :open="!!confirmDelete" title="Remover cupom" @close="confirmDelete = null">
  <p>Remover <b>{{ confirmDelete?.produto }}</b>? Isso tira o cupom do app.</p>
  <template #footer>
    <button class="dc-btn dc-btn--ghost" @click="confirmDelete = null">Cancelar</button>
    <button class="dc-btn dc-btn--danger" :disabled="deleting" @click="doDelete">
      {{ deleting ? 'Removendo…' : 'Remover' }}
    </button>
  </template>
</DcModal>
```

::callout{type="tip" title="Três detalhes de qualidade nesse fluxo"}
1. **`:open="!!confirmDelete"`** — o modal abre sozinho quando há um cupom marcado (reatividade!).
2. **`:disabled="deleting"`** — o botão trava durante a remoção, impedindo **duplo clique** (evita
remover duas vezes).
3. **`await load()`** — depois de remover, recarrega a lista, para a tela refletir a verdade do
servidor. Confirmar → travar → recarregar: o roteiro de toda ação destrutiva bem-feita.
::

## Create: criar com um componente de formulário

A criação delega o formulário a um componente reutilizável, o `<CupomForm>`. A página `novo.vue` é
enxuta — ela só reage quando o formulário diz "pronto, aqui está o cupom":

```ts
// app/pages/cupons/novo.vue
async function onSubmit(body: ProdutoRequest) {
  saving.value = true
  try {
    await api.createCupom(body)
    toast.success('Cupom criado com sucesso!')
    await navigateTo('/cupons')      // volta para a lista
  } catch (e: any) {
    toast.error(e?.message || 'Erro ao criar o cupom.')
  } finally {
    saving.value = false
  }
}
```

```vue
<!-- Sem prop `initial` → o formulário entra em modo "criação" -->
<CupomForm submit-label="Criar cupom" :loading="saving" @submit="onSubmit" @cancel="navigateTo('/cupons')" />
```

## O padrão-chave: props descem, eventos sobem

Aqui está o conceito mais importante da aula — como um componente **filho** (`CupomForm`) conversa com
a **página** (o pai):

::fields
::field{name="Props descem" type="pai → filho"}
A página passa **dados para dentro** do formulário via props: `submit-label` (o texto do botão),
`:loading` (se está salvando), e — na edição — `:initial` (o cupom a editar). Dados fluem **para baixo**.
::
::field{name="Eventos sobem" type="filho → pai"}
O formulário **avisa** a página quando algo acontece via **eventos**: `@submit="onSubmit"` (o usuário
enviou, aqui está o corpo pronto) e `@cancel` (desistiu). Ações fluem **para cima**.
::
::

::callout{type="info" title="Por que não deixar o formulário chamar a API direto?"}
Porque assim o `CupomForm` fica **reutilizável**: ele só sabe **montar e validar** o formulário — não
sabe se é criação ou edição, nem qual método da API chamar. Ele **emite** o corpo pronto e deixa a
página decidir (`novo.vue` chama `createCupom`; `[id].vue` chama `updateCupom`). O **mesmo** formulário
serve para criar **e** editar. Separar "montar o form" (filho) de "o que fazer com ele" (pai) é o que
torna o componente reaproveitável. É a regra de ouro da comunicação entre componentes.
::

Dentro do `CupomForm`, o "avisar o pai" é feito com `defineEmits` e `emit`:

```ts
// dentro de CupomForm.vue (esquema)
const emit = defineEmits<{ submit: [body: ProdutoRequest]; cancel: [] }>()
// quando o usuário envia e a validação passa:
emit('submit', corpoMontado)   // dispara o @submit que a página escuta
```

## Update: criar e editar com o mesmo formulário

A edição (`cupons/[id].vue`) reaproveita **tudo**: ela busca o cupom (`getCupom(id)`), passa ele ao
`<CupomForm :initial="cupom">` (agora o form nasce **preenchido**), e no `@submit` chama
`updateCupom(id, body)`. Repare: **o mesmo `CupomForm`** monta a tela de criação (sem `initial`) e a de
edição (com `initial`). Uma peça, dois usos — graças ao padrão props/eventos.

::callout{type="tip" title="O ciclo de toda mutação"}
Repare no padrão que se repete em criar, editar e remover: **dispara a ação → mostra estado de
'salvando' → em sucesso, avisa (toast) e navega/recarrega → em erro, mostra a mensagem → sempre desliga
o 'salvando' (`finally`)**. É o mesmo ciclo do `load()` (Aula 6) e do app. Toda operação que fala com a
API segue essa coreografia — decore-a e você escreve qualquer CRUD.
::

## Resumindo

- **CRUD** = criar, ler, editar, remover. No portal, se espalha por `cupons/index.vue` (listar +
  remover), `cupons/novo.vue` (criar) e `cupons/[id].vue` (editar) — cada um chamando um método do
  `useApi`.
- **Delete** usa um **modal de confirmação** (aberto reativamente por `confirmDelete`), trava o botão
  durante a ação (`:disabled`) e **recarrega** a lista depois.
- **Create/Update** delegam o formulário a um componente reutilizável (`CupomForm`).
- **Padrão-chave:** **props descem** (pai passa dados ao filho) e **eventos sobem** (filho avisa o pai
  via `defineEmits`/`emit`). Isso deixa o `CupomForm` reutilizável para criar **e** editar.
- Toda mutação segue o mesmo **ciclo**: dispara → estado de salvando → sucesso (toast + navega) / erro
  (mensagem) → sempre desliga o salvando.

::callout{type="tip" title="Fixando"}
Explique por que o mesmo `CupomForm` serve para criar e editar: *"porque ele só monta e valida o
formulário e **emite** o corpo pronto; quem decide o que fazer é a página — `novo.vue` chama
`createCupom`, `[id].vue` chama `updateCupom`. Props descem (o `initial`), eventos sobem (o `submit`)"*.
Se você entendeu essa divisão, domina a comunicação entre componentes.
::

que faz o portal parecer **profissional**. Na próxima aula, a gente sistematiza: **UX, estados de tela e
