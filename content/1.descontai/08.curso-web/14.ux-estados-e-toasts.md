Um portal pode ter todas as telas certas e ainda parecer amador — se ele congela ao carregar, mostra
uma tela em branco quando não há dados, ou não avisa se uma ação deu certo. A diferença está na **UX**:
como o portal se comporta nos momentos que não são o "caminho feliz". Esta aula sistematiza isso e fecha
a Parte 4.

## Toda tela tem quatro estados

O erro clássico do iniciante é programar só o sucesso. Mas toda tela que busca dados da API tem, na
verdade, **quatro estados** — e você precisa desenhar os quatro:

::fields
::field{name="Carregando" type="loading = true"}
A requisição está a caminho. Mostre um indicador (de preferência um skeleton) — nunca uma tela vazia
que parece travada.
::
::field{name="Erro" type="erro != ''"}
Deu ruim (sem internet, erro do servidor). Mostre a mensagem amigável (do `useApi`, Aula 11).
::
::field{name="Vazio" type="lista.length === 0"}
A busca deu certo, mas não há dados. Mostre um "estado vazio" que **convida à ação** — não confunda com
erro.
::
::field{name="Sucesso" type="dados prontos"}
Chegou. Mostre os dados.
::
::

A tela de cupons (Aula 6) implementa os quatro com diretivas `v-if`/`v-else` encadeadas:

```vue
<div v-if="erro" class="erro">{{ erro }}</div>          <!-- 1. erro -->
<div v-else-if="loading">...skeletons...</div>           <!-- 2. carregando -->
<DcEmpty v-else-if="!filtrados.length" ... />            <!-- 3. vazio -->
<div v-else>...grade de cupons...</div>                  <!-- 4. sucesso -->
```

::callout{type="warning" title="O 'caminho feliz' é uma armadilha"}
Se você só programa o sucesso, o portal parece ótimo no seu wi-fi rápido com dados de teste — e péssimo
na vida real: uma tela branca quando a empresa ainda não tem cupons, um congelamento quando a API
demora. **A maior parte da qualidade percebida mora nos estados de carregando, erro e vazio.** Para
cada tela, pergunte sempre: *"e se estiver carregando? e se der erro? e se vier vazio?"*.
::

## Skeletons: carregar sem parecer travado

Em vez de um spinner girando no nada, o portal mostra **skeletons** — retângulos cinza no formato dos
cards que vão aparecer. Você viu no template de cupons:

```vue
<div v-if="loading" class="dc-grid">
  <div v-for="i in 6" :key="i" class="dc-card">
    <div class="dc-skeleton" style="height: 120px" />   <!-- a "foto" fantasma -->
    <div class="dc-skeleton" style="height: 16px; width: 60%" /> <!-- o "título" fantasma -->
  </div>
</div>
```

::callout{type="info" title="Por que skeleton é melhor que spinner"}
Um esqueleto com o **formato** do conteúdo diz ao cérebro "a informação está vindo, vai aparecer aqui" —
a espera parece mais curta e o portal parece mais rápido, mesmo levando o mesmo tempo. É a mesma técnica
do `shimmer` do app (curso do app, Aula 17). O `DcKpi` também tem um `:loading` que troca o valor por um
skeleton. Percepção de velocidade é parte da UX.
::

## O estado vazio inteligente

Um detalhe que separa o bom do ótimo: o portal **distingue** "você ainda não tem cupons" de "sua busca
não encontrou nada". São situações diferentes, e a mensagem certa muda tudo. Olha o `DcEmpty` real:

```vue
<DcEmpty
  v-else-if="!filtrados.length"
  icon="ticket"
  :title="busca ? 'Nenhum cupom encontrado' : 'Você ainda não tem cupons'"
  :subtitle="busca ? 'Tente outro termo de busca.' : 'Crie seu primeiro cupom e comece a atrair assinantes.'"
>
  <NuxtLink v-if="!busca" to="/cupons/novo" class="dc-btn dc-btn--primary">Criar cupom</NuxtLink>
</DcEmpty>
```

::callout{type="tip" title="Vazio não é erro — é uma oportunidade"}
Repare: se **há busca**, mostra "nenhum encontrado, tente outro termo". Se **não há** (a empresa é
nova), mostra "você ainda não tem cupons" **com um botão de criar**. Um estado vazio genérico ("nada
aqui") deixaria o usuário perdido; um estado vazio **inteligente** o **guia para o próximo passo**. É
empatia virando interface — e um dos detalhes que mais elevam a percepção de qualidade.
::

## Toasts: o feedback de cada ação

Todo toque merece resposta. Quando o usuário cria, edita ou remove um cupom, o portal avisa com um
**toast** — aquele aviso flutuante que aparece e some. Você viu o `useToast` por dentro na Aula 10;
aqui, o uso:

```ts
toast.success('Cupom criado com sucesso!')   // verde
toast.error('Erro ao remover.')              // vermelho
```

O `<DcToasts>` (montado uma vez no `app.vue`) lê a lista de toasts (o `useState` global) e os renderiza,
cada um sumindo após 3,5s. Uma linha na tela, e o usuário **sabe** que a ação funcionou.

::callout{type="info" title="Sem feedback, o usuário duvida (e clica de novo)"}
Imagine remover um cupom e **nada** acontecer na tela por um segundo. O usuário pensa "será que
cliquei?" e clica de novo — podendo causar bug (duplo clique). O toast + o botão que trava
(`:disabled="deleting"`) + a lista que recarrega formam uma conversa clara: *"recebi seu clique, estou
fazendo, pronto, deu certo"*. Feedback é o que faz a interface parecer **viva e confiável**.
::

## O checklist de UX de uma tela

Junte tudo num roteiro. Para **qualquer** tela do portal, garanta:

::steps
### Estado de carregando
Skeleton no formato do conteúdo (não um spinner no vazio).

### Estado de erro
Mensagem amigável (vinda do `useApi`) e, se possível, um jeito de tentar de novo.

### Estado vazio inteligente
Distinga "sem dados ainda" (com convite à ação) de "busca sem resultado" (sugira outro termo).

### Feedback nas ações
Toast de sucesso/erro, botões que travam durante a operação, e a tela refletindo a verdade do servidor
(recarga após mutação).
::

## Resumindo

- Toda tela que busca dados tem **quatro estados**: **carregando**, **erro**, **vazio** e **sucesso** —
  implementados com `v-if`/`v-else` encadeados. Programar só o sucesso é o erro clássico.
- **Skeletons** (retângulos no formato do conteúdo) fazem o carregamento parecer mais rápido que um
  spinner no vazio.
- O **estado vazio inteligente** distingue "sem dados ainda" (com botão de ação) de "busca sem
  resultado" (sugere outro termo) — guia o usuário em vez de deixá-lo perdido.
- **Toasts** dão feedback de cada ação (sucesso/erro); junto com botões que **travam** e a **recarga**
  após mutação, formam uma conversa clara com o usuário.

::callout{type="tip" title="Fixando"}
Para qualquer tela, responda as quatro perguntas: *"e se carregando? (skeleton) e se erro? (mensagem) e
se vazio? (convite à ação) e a ação, dá feedback? (toast + botão travado + recarga)"*. Quem responde as
quatro constrói portais que parecem — e são — profissionais.
::
