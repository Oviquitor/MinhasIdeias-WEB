Você reparou que o portal funciona **sem a API no ar** — você roda `npm run dev` e tudo aparece,
navega, cria, remove. Isso é o **mock mode**: um modo de demonstração com dados de exemplo. Longe de ser
só um truque, é uma técnica **profissional** que destrava o desenvolvimento. Esta aula é sobre ela.

## O problema: o back nem sempre está pronto

Imagine que você é o dev do front e precisa construir a tela de faturamento. Só que o endpoint
`/empresas/me/faturamento` **ainda não existe** na API — o time de back vai fazer semana que vem. O que
você faz? Fica parado esperando?

::callout{type="info" title="Mock = um dublê do back-end"}
Um **mock** é um **dublê**: no lugar da API real, o front usa **dados de exemplo fixos** que imitam o
que a API vai devolver. Como um dublê de cinema faz a cena perigosa no lugar do ator, o mock "faz o
papel" do back-end enquanto ele não está pronto. Você constrói a tela **inteira** — layout, estados,
interações — usando o dublê, e depois só troca o dublê pelo ator (a API real).
::

## Como o portal liga/desliga o mock

Lembra do `runtimeConfig` (Aula 4 e 11)? A chave `mockMode` controla tudo:

```ts
// nuxt.config.ts
runtimeConfig: { public: {
  apiBase: 'https://descontai-api-.../api/v1',
  mockMode: 'true',   // 'true' = dados de exemplo; 'false' = API real
} }
```

E dentro do `useApi`, **cada método** verifica esse modo antes de decidir de onde vêm os dados:

```ts
const mock = () => config.public.mockMode === 'true'

async listCupons(): Promise<ProdutoResponse[]> {
  if (mock()) {
    await delay()             // simula a demora da rede
    return clone(MOCK_CUPONS) // devolve os dados de exemplo
  }
  return api<ProdutoResponse[]>('/empresas/me/produtos')  // ou a API real
},
```

Repare no padrão: **`if (mock())` devolve o exemplo; senão, chama a API real**. O resto do portal — a
tela, os estados, o CRUD — **não sabe nem se importa** se os dados são reais ou de mentira. Ele só chama
`api.listCupons()`. Isso é o poder da camada única (Aula 11): trocar a fonte dos dados não toca nas
telas.

## Os dados de exemplo e a simulação

Os dados fixos moram em `app/utils/mock.ts` (`MOCK_CUPONS`, `MOCK_EMPRESA`, `MOCK_DASHBOARD`…). E dois
detalhes tornam o mock **realista**:

::fields
::field{name="delay() — simula a rede"}
`await delay()` espera ~280ms antes de responder, imitando a demora de uma requisição real. Assim você
**vê os skeletons** (Aula 14) funcionando, em vez de os dados aparecerem instantaneamente e esconderem
o estado de carregamento.
::
::field{name="clone() — evita bagunça"}
`clone(MOCK_CUPONS)` devolve uma **cópia** dos dados, não o original. Assim, se a tela mexer no
resultado, não corrompe o mock para a próxima chamada. E o `createCupom` no mock chega a **adicionar**
ao array de exemplo, para a criação "funcionar de verdade" na demonstração.
::
::

::callout{type="tip" title="Um mock bom imita até os defeitos da rede"}
Repare o cuidado: o mock não devolve os dados **na hora** — ele **espera** (delay) para simular a rede.
Um mock que responde instantaneamente esconderia bugs de carregamento que só aparecem com a rede real. O
bom dublê imita **tudo**, inclusive a demora. É isso que faz você conseguir testar os quatro estados da
tela (Aula 14) sem sequer ter a API.
::

## Contract-first: combinar o contrato antes

O detalhe mais profissional aqui: repare nos comentários `[PROPOSTO]` no `useApi`:

```ts
// [PROPOSTO] GET /empresas/me/dashboard
// [PROPOSTO] GET /empresas/me/resgates
```

Esses são endpoints que **ainda não existem** na API — e estão documentados num arquivo
`API_REQUIREMENTS.md`. Isso é a técnica **contract-first** ("contrato primeiro"):

::callout{type="info" title="Contract-first, na prática"}
Antes de o back-end existir, os times **combinam o contrato**: "o endpoint vai ser `GET
/empresas/me/faturamento` e vai devolver um JSON com este formato". Com o contrato acordado, o dev do
**front** constrói a tela usando o **mock** (que segue esse formato), e o dev do **back** implementa o
endpoint — **em paralelo**, sem um esperar o outro. Quando os dois terminam, é só virar a chave
(`mockMode: false`) e conectar. Os tipos em `app/types/api.ts` (Aula 11) **são** esse contrato escrito
em código. É assim que times de verdade trabalham rápido sem pisar no pé um do outro.
::

## Ligando a API real

Quando os endpoints estiverem prontos, virar a chave é só configuração — **sem mudar código**:

```bash
# .env
NUXT_PUBLIC_MOCK_MODE=false
NUXT_PUBLIC_API_BASE=http://localhost:8080/api/v1
```

O `if (mock())` de cada método passa a cair no `else`, chamando a API real. As telas continuam
idênticas — elas nunca souberam a diferença. É a recompensa de ter centralizado a fonte dos dados.

::callout{type="warning" title="O mock não substitui testar com a API real"}
Cuidado com uma armadilha: o mock devolve dados **perfeitos e previsíveis**. A API real pode devolver um
campo faltando, um erro 500, uma lista paginada. **Antes de ir para produção, teste com a API de
verdade** (`mockMode: false`) — é aí que aparecem os casos que o mock não previu. O mock acelera o
desenvolvimento; ele não dispensa a validação com o back real.
::

## Resumindo

- **Mock mode** faz o portal rodar com **dados de exemplo**, sem a API no ar — um "dublê" do back-end.
- Controlado por **`runtimeConfig.mockMode`**: cada método do `useApi` faz `if (mock())` devolve o
  exemplo, `else` chama a API real. As **telas não sabem a diferença** (camada única).
- Os dados fixos (`app/utils/mock.ts`) usam **`delay()`** (simula a rede, para ver os skeletons) e
  **`clone()`** (evita corromper o exemplo). Realismo importa.
- **Contract-first**: os times combinam o **contrato** (endpoints + formato, em `API_REQUIREMENTS.md` e
  nos tipos) antes; front (com mock) e back (implementando) trabalham **em paralelo**. Os `[PROPOSTO]`
  marcam o que ainda falta.
- Ligar a real é **só configuração** (`NUXT_PUBLIC_MOCK_MODE=false`) — mas **teste com a API real antes
  de produção**, pois o mock esconde imperfeições.

::callout{type="tip" title="Fixando"}
Explique por que o mock destrava o time: *"o front constrói as telas usando dados de exemplo que seguem
o contrato acordado, enquanto o back implementa os endpoints em paralelo; quando ambos terminam, viram a
chave `mockMode: false` e conectam — ninguém ficou esperando ninguém"*. Se você entendeu contrato +
paralelismo, entendeu como times entregam rápido.
::
