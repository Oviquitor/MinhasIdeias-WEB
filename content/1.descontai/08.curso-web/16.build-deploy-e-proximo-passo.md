## SPA x SSR: onde a página é montada

Antes do deploy, uma decisão que o `nuxt.config.ts` já tomou (Aula 4): **onde o HTML é montado?**

::fields
::field{name="SPA (ssr: false)" type="monta no navegador"}
O servidor entrega um HTML quase vazio + o JavaScript; o **navegador** monta a tela. Rápido para
navegar, ideal para **painéis atrás de login** (sem necessidade de SEO). É o que o Portal de Empresas é.
::
::field{name="SSR (ssr: true)" type="monta no servidor"}
O **servidor** monta o HTML pronto e o envia; o navegador só o "hidrata". Melhor para **SEO** e para o
primeiro carregamento aparecer rápido. Ideal para sites **públicos** (como o Descontai-WEB).
::
::

::callout{type="info" title="Por que o portal escolheu SPA"}
O `nuxt.config.ts` tem `ssr: false`. Faz sentido: é um **painel de gestão atrás de login** — o Google
não precisa indexar a tela de faturamento de uma empresa. SPA dá navegação fluida e um deploy mais
simples (arquivos estáticos). Já um site de marketing prefere **SSR/SSG** para aparecer no Google. **A
mesma stack (Nuxt) faz os dois** — muda uma linha de config. Cada front escolhe o modelo que serve seu
objetivo (lembra da Aula 1).
::

## O build de produção

Durante o desenvolvimento, `npm run dev` roda o portal com hot reload — ótimo para trabalhar, mas não
publicável. Para a nuvem, você gera o **build de produção**, otimizado e compacto:

```bash
npm run build     # gera a versão de produção (na pasta .output)
npm run preview   # testa esse build localmente antes de publicar
```

O build minifica o código, otimiza os assets e produz os arquivos que o servidor de hospedagem vai
servir. Como o `nuxt.config` tem `nitro: { preset: 'cloudflare-pages' }`, ele gera a saída no formato
que a **Cloudflare Pages** entende.

## O deploy: Cloudflare Pages

O portal é publicado na **Cloudflare Pages** — uma hospedagem **gratuita** para sites/SPAs. O fluxo
típico é conectado ao Git (a mesma ideia de CI/CD do curso da API):

::steps
### Conecta o repositório
Você liga o repositório do GitHub à Cloudflare Pages. Ela passa a observar a branch principal.

### Configura o build
Informa o comando (`npm run build`) e a pasta de saída. A Cloudflare roda isso na nuvem a cada push.

### Deploy automático a cada push
Você dá `git push`, a Cloudflare **builda e publica** sozinha. Em minutos, a nova versão está no ar,
numa URL pública. É **entrega contínua** — o mesmo conceito do curso da API, aplicado ao front.
::

## Variáveis de ambiente: a config da nuvem

Lembra do `runtimeConfig` (Aula 4, 11, 15)? No deploy, você **não** publica com `mockMode: true` e a URL
de teste. Você configura, no painel da Cloudflare, as variáveis de ambiente de produção:

```bash
NUXT_PUBLIC_MOCK_MODE=false                 # usa a API real, não o mock
NUXT_PUBLIC_API_BASE=https://api.descontai.com/api/v1   # a API de produção
```

::callout{type="tip" title="Config vem do ambiente — pela última vez"}
Este é o mesmo princípio que apareceu nos **três** cursos: API (`${DB_URL}`), app (`--dart-define`),
portal (`runtimeConfig` + env). O **mesmo build** roda apontando para a API de teste (com mock) ou para
a de produção (real), só mudando as variáveis do ambiente — **nunca** o código. Segredos e endereços
vêm de fora. Se você levar **uma** ideia de arquitetura dos três cursos, que seja esta.
::

## Roteiro: construa o SEU portal do zero à nuvem

Hora de virar autor. Aqui está o caminho, aplicando **tudo** que você aprendeu. Não faça tudo num dia —
cada passo é um marco. Travou? **Volte na aula correspondente**, com o código do portal como referência.

::callout{type="tip" title="Antes de começar"}
Escolha um painel **pequeno e que você entenda**. Nada de "clone do Notion". Pense em: um gerenciador
de tarefas, um catálogo de produtos de uma loja, um painel de despesas. Simples de propósito — o
objetivo é **aplicar a arquitetura**, não impressionar.
::

::steps
### Crie o projeto Nuxt
`npx nuxi init meu-portal`. Rode `npm run dev` e veja subir em `localhost:3000`. Configure o
`nuxt.config.ts` com `ssr: false` e um `runtimeConfig` com `apiBase` e `mockMode` (Aula 4).

### Monte o design system
Um `main.css` com as variáveis (`--brand`, `--text`…) e algumas classes utilitárias (`dc-btn`,
`dc-card`) (Aula 7). Tudo o mais vai consumir esses tokens.

### Crie os componentes base
Um `DcKpi`, um `DcModal`, um `DcTopbar` com props e slots (Aula 5). Peças pequenas e reutilizáveis.

### Configure rotas e layouts
Crie páginas em `app/pages/` (viram rotas sozinhas) e um `layouts/default.vue` com a moldura + `<slot />`
(Aula 8). Navegue com `NuxtLink`.

### Adicione o guard de rotas
Um `middleware/auth.global.ts` que barra telas sem login e redireciona (Aula 9).

### Crie a camada de API e o mock
Um `useApi` que envelopa o `$fetch` com baseURL + token + erro, e um `mock.ts` com dados de exemplo.
Ligue/desligue por `mockMode` (Aulas 11 e 15). Comece **tudo no mock**.

### Implemente autenticação
Um `useAuth` com a sessão em `useState` + `localStorage`, uma tela de login, e o `logout` (Aula 12).

### Construa um CRUD completo
Escolha uma entidade e faça listar/criar/editar/remover, com um componente de formulário reutilizável
(props descem, eventos sobem) e um modal de confirmação (Aula 13).

### Capriche na UX
Os quatro estados (carregando/erro/vazio/sucesso), skeletons, estado vazio inteligente e toasts (Aula
14). É o que faz parecer profissional.

### Conecte a API real
Combine o **contrato** com quem faz o back (ou use a sua própria API do curso da API!), vire
`mockMode: false` e teste com dados reais (Aula 15).

### Faça o deploy
Suba na Cloudflare Pages (ou Vercel/Netlify), configure as variáveis de ambiente de produção, e dê o
push. Quando a URL pública responder, seu portal está **no ar de verdade**. 🚀
::

::callout{type="info" title="Não precisa ser linear e perfeito"}
No mundo real ninguém faz esses passos numa sequência limpa. Você vai voltar, refatorar, quebrar e
consertar. **Isso é o processo, não um erro.** O importante é ter feito cada peça pelo menos uma vez
com as próprias mãos — depois disso, elas viram naturais.
::

## O kit do dev front-end

Guarde esta tabela. É o seu checklist de "meu portal está pronto para o mundo?".

| Peça | O que garante | Aula |
|------|---------------|------|
| Componentes + props/slots | Interface reutilizável | 5 |
| Reatividade (ref/computed) + diretivas | Telas que se atualizam sozinhas | 6 |
| CSS scoped + design system | Visual consistente e sustentável | 7 |
| Rotas por arquivo + layouts | Navegação e molduras | 8 |
| Middleware (guard) | Telas protegidas | 9 |
| Composables + useState | Lógica e estado compartilhados | 10 |
| useApi ($fetch + token + erro) | Comunicação com a API centralizada | 11 |
| useAuth (sessão + localStorage) | Login e sessão persistente | 12 |
| CRUD (props descem, eventos sobem) | Gestão de dados completa | 13 |
| Estados de tela + toasts | UX profissional | 14 |
| Mock mode + contract-first | Desenvolver em paralelo com o back | 15 |
| Build + deploy + env | Portal no ar, config segura | 16 |

::callout{type="note" title="As três pontas do sistema"}
Se você fez os **três** cursos — a [API](../curso), o [app](../curso-app) e este —, você tem algo
raro: enxerga o **sistema inteiro**. Sabe como o token nasce no back-end, viaja para o app e para o
portal, e volta validado; como uma regra vive na API e é refletida em duas interfaces diferentes; como o
mesmo `POST /auth/login` alimenta um `SessionCache` no Flutter e um `useAuth` no Nuxt. Pouquíssimos devs
têm essa visão das três frentes tão cedo. Ela é o que te faz entender **por que** as coisas são como
são — e construir melhor por causa disso.
::
