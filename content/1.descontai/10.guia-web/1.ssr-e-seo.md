No curso web, o Portal de Empresas tinha `ssr: false` — era uma **SPA** (renderiza no navegador). Este
site institucional faz o **oposto**: mantém **SSR/SSG**. Entender por quê é entender a diferença mais
importante entre um painel e um site público.

## Relembrando: SPA x SSR

::fields
::field{name="SPA (ssr: false)" type="monta no navegador"}
O servidor manda um HTML quase vazio + JavaScript; o **navegador** monta a tela. Ótimo para painéis
atrás de login, onde ninguém precisa "achar" a página no Google.
::
::field{name="SSR/SSG (ssr: true)" type="monta no servidor"}
O **servidor** monta o HTML **já pronto, com o conteúdo dentro**, e envia. O navegador mostra na hora e
depois "hidrata" (liga a interatividade). Essencial para sites **públicos**.
::
::

O `nuxt.config.ts` deste site deixa isso explícito no comentário: *"Site de marketing: mantém
SSR/pré-renderização para SEO."* — enquanto o portal desligava o SSR. Mesma stack, decisão oposta,
motivo claro.

## O que é SEO (e por que a SPA atrapalha)

**SEO** (Search Engine Optimization) é o conjunto de práticas para o site **aparecer bem no Google**.
E o Google precisa **ler o conteúdo** da página para indexá-la.

::callout{type="info" title="A analogia do robô que lê a página"}
Imagine um **robô do Google** visitando seu site. Numa **SPA**, o que ele recebe primeiro é uma página
**quase vazia** ("carregando…") — o conteúdo só aparece depois que o JavaScript roda. O robô pode não
esperar, e indexar uma página vazia. Numa página **SSR**, o robô recebe o **HTML já cheio de conteúdo**
(títulos, textos, os benefícios do clube) — ele lê tudo na hora e indexa direito. Para um site que
**vive de ser encontrado no Google**, isso é a diferença entre existir e não existir na busca.
::

Por isso: **painel atrás de login → SPA** (o Google não precisa vê-lo); **site de marketing → SSR/SSG**
(o Google **precisa** vê-lo para trazer visitantes). A escolha do modo de renderização **serve o
objetivo** do site.

## SSG: pré-renderizar de vez

Como as páginas deste site (Sobre, Para você) são praticamente **as mesmas para todo mundo**, dá para ir
além do SSR: **pré-renderizar** o HTML **na hora do build** (SSG — Static Site Generation). O resultado
é um HTML pronto servido como arquivo estático — **rápido e barato** (é o que a Cloudflare Pages
hospeda de graça).

::callout{type="tip" title="SSR x SSG, sem decoreba"}
**SSR** monta a página **a cada visita** (bom quando o conteúdo muda por usuário/hora). **SSG** monta
**uma vez, no build** (bom quando o conteúdo é o mesmo para todos — como um site institucional). Para
marketing, SSG costuma ser ideal: máxima velocidade e SEO, custo mínimo. O Nuxt faz os dois com a mesma
base; você escolhe conforme a página.
::

## LCP: performance também é conversão

Há uma segunda razão para SSR/SSG num site de marketing: **velocidade de exibição**. A métrica que
importa se chama **LCP** (Largest Contentful Paint) — quanto tempo até o maior conteúdo da tela
aparecer. Um site que demora a mostrar algo **perde visitantes** (e o Google penaliza no ranking).

Com SSR/SSG, o conteúdo principal já vem no HTML — aparece **imediatamente**, sem esperar o JavaScript.
Numa SPA, o usuário veria uma tela em branco até o JS montar tudo. Para converter, cada milissegundo do
primeiro paint conta.

::callout{type="info" title="O cuidado com a hidratação"}
Depois que o HTML do servidor aparece, o Vue "**hidrata**" a página no navegador — liga os cliques e as
animações ao HTML que já estava lá. Um detalhe fino: o HTML do servidor e o do cliente precisam
**bater** (senão o Vue reclama de "hydration mismatch" e pode redesenhar). Você vai ver, na próxima
página, como a animação de entrada do site foi construída com cuidado **exatamente** para não quebrar
essa hidratação — um sinal de código maduro.
::

## Resumindo

- Painéis (Empresas, Admin) são **SPA** (`ssr: false`) — o Google não precisa vê-los. Este site de
  marketing é **SSR/SSG** — ele **precisa** aparecer na busca.
- **SEO** é aparecer bem no Google; o robô do Google lê melhor um **HTML já cheio** (SSR) do que uma
  SPA "vazia" que só se preenche via JavaScript.
- **SSG** pré-renderiza as páginas **no build** (ideal para conteúdo igual para todos) — rápido e barato
  (Cloudflare Pages).
- **LCP** (tempo até o conteúdo aparecer) importa para conversão e ranking; SSR/SSG mostra o conteúdo
  **na hora**. A **hidratação** liga a interatividade depois, e exige que servidor e cliente batam.
