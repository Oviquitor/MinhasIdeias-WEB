# 9. Tipos de oferta

| Tipo | Exemplo | Campos críticos |
| --- | --- | --- |
| Percentual | 20% off | Percentual, limite máximo, valor mínimo, validade |
| Fixo | R$ 10 off | Valor fixo, valor mínimo, limite |
| 2x1 | Pague 1 leve 2 | Produto principal, item grátis, regra menor valor |
| Combo | Combo casal | Itens, preço cheio, preço promocional |
| Brinde | Ganhe sobremesa | Condição, brinde, limite |
| Primeira visita | 30% na primeira compra | Uso único por usuário na empresa |
| Recorrente | Toda terça | Dias, horários, quantidade |
| Relâmpago | Só hoje | Início, fim, estoque, push |


## 9.1 Regras obrigatórias de qualquer oferta

Toda oferta precisa ter:

- Título claro.
- Descrição curta.
- Como usar.
- Regras.
- Dias válidos.
- Horários válidos.
- Validade.
- Limite por usuário.
- Limite total, se aplicável.
- Se acumula ou não com outras promoções.
- Se precisa consumir no local.
- Se exige reserva.
- Se vale em feriados.

## 9.2 Evitar oferta enganosa

O Admin deve revisar:

- Desconto real.
- Regras escondidas.
- Título exagerado.
- Imagem incompatível.
- Benefício impossível de cumprir.
- Margem muito arriscada para parceiro.


# 10. Cupons e regras de resgate

| Status | Significado |
| --- | --- |
| DISPONIVEL | Usuário pode usar |
| GERADO | Código/QR gerado |
| RESERVADO | Uso temporariamente reservado |
| USADO | Cupom validado |
| EXPIRADO | Passou da validade |
| CANCELADO | Cancelado por regra/Admin |
| RECUSADO | Parceiro recusou com motivo |
| SUSPEITO | Marcado por antifraude |


## 10.1 Cupom como entidade de negócio

Um cupom deve registrar:

- Usuário.
- Oferta.
- Parceiro.
- Código.
- QR/token.
- Status.
- Data de geração.
- Data de uso.
- Validade.
- Economia.
- Método de validação.
- Auditoria.
- Score de risco.

## 10.2 Validação de cupom

Antes de validar, o backend deve checar:

- Usuário autenticado.
- Assinatura ativa.
- Oferta ativa.
- Parceiro ativo.
- Cidade compatível.
- Horário permitido.
- Limite por usuário.
- Limite total.
- Código válido.
- Cupom não usado.
- Cupom não expirado.

## 10.3 Economia

Armazenar sempre que possível:

- Valor cheio.
- Valor pago.
- Valor economizado.
- Tipo de economia.
- Se é estimada ou confirmada.

Regra importante: não inflar economia. Quando não houver valor real, mostrar “economia estimada”.


# 11. Antifraude

## 11.1 Fraudes possíveis

- Usuário usando cupom várias vezes.
- Compartilhamento indevido de código.
- QR fixo copiado.
- Parceiro validando uso falso.
- Funcionário validando sem consumo.
- Indicação fraudulenta.
- Pagamento contestado.
- Conta compartilhada abusivamente.

## 11.2 Medidas recomendadas

- QR dinâmico com expiração.
- Código temporário.
- Rate limit.
- Auditoria de uso.
- Idempotência.
- Score de risco.
- Revisão manual.
- Bloqueio progressivo.
- Limites por oferta.
- Limites por usuário.
- Webhook idempotente.

## 11.3 QR dinâmico

O QR ideal deve conter ou representar:

- ID do cupom.
- ID da oferta.
- ID do usuário.
- Timestamp.
- Expiração curta.
- Assinatura criptográfica.
- Nonce.

O QR não deve funcionar indefinidamente.

## 11.4 Score de risco

Faixas:

- 0 a 20: baixo risco.
- 21 a 50: observar.
- 51 a 80: validação extra.
- 81 a 100: bloquear e revisar.

Sinais:

- Muitos usos em pouco tempo.
- Muitos aparelhos por usuário.
- Muitos usuários no mesmo aparelho.
- Resgates fora do padrão.
- Indicações em massa.
- Webhook duplicado com divergência.
- Tentativas repetidas de código inválido.


# 12. Assinaturas e pagamentos

## 12.1 Estados de pagamento

| Status | Significado |
| --- | --- |
| CREATED | Cobrança criada |
| PENDING | Aguardando pagamento |
| CONFIRMED | Pagamento confirmado |
| RECEIVED | Valor recebido |
| OVERDUE | Vencido |
| FAILED | Falhou |
| CANCELLED | Cancelado |
| REFUNDED | Reembolsado |
| CHARGEBACK | Contestado |

## 12.2 Estados de assinatura

| Status | Significado |
| --- | --- |
| INACTIVE | Nunca assinou ou está sem plano |
| ACTIVE | Assinatura válida |
| PAST_DUE | Pagamento pendente/vencido |
| EXPIRED | Validade terminou |
| CANCELLED | Cancelada |
| SUSPENDED | Suspensa por regra |
| REFUNDED | Pagamento reembolsado |


## 12.3 Arquitetura correta

```text
App Flutter
↓
Backend Descontai
↓
Gateway de pagamento
↓
Webhook no backend
↓
Backend atualiza pagamento e assinatura
↓
App sincroniza usuário
```

O app nunca deve ativar assinatura sozinho. A ativação deve vir do backend após confirmação real do provedor.

## 12.4 Pix

Fluxo:

```text
Usuário escolhe plano
↓
Backend cria cobrança Pix
↓
Gateway retorna QR e copia-e-cola
↓
App exibe Pix
↓
Usuário paga no banco
↓
Webhook confirma
↓
Backend ativa assinatura
↓
App libera cupons
```

## 12.5 Cartão

Regras:

- Não armazenar número de cartão.
- Não armazenar CVV.
- Preferir tokenização.
- Registrar status interno.
- Aguardar confirmação quando necessário.
- Lidar com recusas de forma clara.

## 12.6 Webhooks

Webhook deve:

- Validar origem/assinatura/token.
- Ser idempotente.
- Registrar payload bruto.
- Mapear status externo para interno.
- Atualizar pagamento.
- Ativar/renovar assinatura.
- Processar indicação.
- Registrar erro se falhar.

## 12.7 Provedores

- **Asaas:** bom para Pix, cartão, cobranças e operação brasileira.
- **Mercado Pago:** bom para Pix, cartão, checkout e assinaturas.
- **Stripe:** bom para arquitetura global, billing e cartões; validar disponibilidade comercial e requisitos no Brasil antes de escolher como principal.

Recomendação MVP: escolher um provedor principal e isolar em uma interface `PaymentGateway` para permitir troca futura.


# 13. Sistema de indicações

## 13.1 Objetivo

Gerar crescimento orgânico com rastreabilidade e controle.

## 13.2 Fluxo

```text
Indicador compartilha código
↓
Indicado cadastra usando código
↓
Indicado assina
↓
Pagamento confirma
↓
Sistema gera crédito
↓
Crédito fica pendente
↓
Prazo de segurança passa
↓
Saldo fica disponível
```

## 13.3 Estados

- CREATED.
- PENDING_PAYMENT.
- PENDING_RELEASE.
- AVAILABLE.
- BLOCKED.
- CANCELLED.
- PAID_OUT.

## 13.4 Regras

- Usuário não pode usar o próprio código.
- Crédito só libera após pagamento confirmado.
- Reembolso cancela crédito.
- Chargeback bloqueia crédito.
- Admin pode bloquear código.
- Deve existir auditoria.


# 14. IA no Descontai

## 14.1 Princípio

A IA deve gerar valor prático, não ser enfeite. Ela precisa reduzir trabalho, melhorar decisão, aumentar relevância e proteger margem.

## 14.2 IA para usuários

Funcionalidades:

- Recomendações de ofertas por histórico, cidade, categoria, horário e economia.
- Busca conversacional: “onde tem desconto para jantar hoje?”.
- Planejador de saída econômica: orçamento, pessoas, categoria e horário.
- Explicação simples das regras.
- Alertas de ofertas favoritas vencendo.
- Sugestão de oferta que ajuda a “pagar” a assinatura.

Exemplos de perguntas:

- “Qual cupom mais compensa hoje?”
- “Tem algo aberto agora?”
- “Quero sair com minha esposa gastando pouco.”
- “Quanto eu já economizei?”

## 14.3 IA para parceiros

Funcionalidades:

- Gerar título e descrição de oferta.
- Sugerir regras claras.
- Sugerir limites para proteger margem.
- Analisar campanha.
- Criar texto para Instagram/WhatsApp.
- Identificar ofertas com muitos cliques e poucos resgates.
- Alertar desconto agressivo demais.

## 14.4 IA para Admin

Funcionalidades:

- Detectar anomalias.
- Classificar tickets.
- Revisar oferta confusa.
- Sugerir categorias.
- Gerar resumo de cidade.
- Priorizar parceiros com problema.

## 14.5 Segurança da IA

A IA não deve:

- Expor dados pessoais de outros usuários.
- Criar promessa de venda garantida.
- Alterar campanha sem confirmação.
- Aprovar parceiro automaticamente.
- Inventar regra de oferta.
- Inflar economia sem base.

## 14.6 Prompt base para oferta

```text
Você é um consultor de campanhas locais do Descontai.
Crie uma oferta clara, honesta e atrativa.
Não prometa resultado garantido.
Proteja a margem do parceiro.
Use linguagem simples.
Retorne título, descrição, regras, dias válidos, limite sugerido e alerta de margem.
```

## 14.7 Prompt base para explicar regra

```text
Explique as regras desta oferta em linguagem simples.
Não altere nenhuma regra.
Não adicione benefício que não existe.
Se houver ambiguidade, diga que o usuário deve confirmar com o estabelecimento.
```
