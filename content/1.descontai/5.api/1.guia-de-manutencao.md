# Guia de manutencao

Este documento complementa os comentarios no codigo. Ele explica a organizacao
do projeto e os fluxos que exigem mais cuidado em manutencao.

## Arquitetura

O projeto esta separado em camadas:

- `api/controller`: entrada HTTP. Controllers recebem requests, leem usuario
  autenticado e delegam para use cases. Regra de negocio nao deve ficar aqui.
- `api/dto`: contratos JSON da API. Requests e responses evitam expor entidades
  JPA diretamente ao aplicativo.
- `application/usecase`: uma acao do sistema por classe, como login, criar
  produto, processar webhook ou usar cupom.
- `application/service`: regras compartilhadas por varios use cases, como
  pagamento, autenticacao, notificacao e cadastros basicos.
- `domain/model`: entidades JPA persistidas no banco.
- `infra/repository`: consultas Spring Data JPA.
- `infra/mapper`: conversao entre entidades e DTOs.
- `infra/gateway`: integracao com sistemas externos, como Asaas e FCM.
- `security`: configuracao de seguranca, JWT e adaptadores para Spring Security.

## Fluxo HTTP padrao

1. Controller recebe a requisicao.
2. Controller monta a entrada do use case.
3. Use case chama services, repositories e mappers.
4. Service aplica regra de negocio e abre transacao quando precisa.
5. Repository persiste ou consulta entidades.
6. Mapper transforma entidade em response DTO.
7. Controller devolve `ResponseEntity` ou DTO.

## Autenticacao

Arquivos principais:

- `security/config/SecurityConfig.java`: rotas publicas, filtros, headers e beans.
- `security/jwt/JwtTokenProvider.java`: cria, valida e le dados de JWT.
- `security/jwt/JwtAuthenticationFilter.java`: extrai Bearer token e popula o
  contexto de seguranca.
- `application/service/AuthService.java`: cadastro e reset de senha.
- `application/usecase/auth`: login, registro, refresh, logout e reset.

Pontos de cuidado:

- Senha nunca deve ser salva em texto puro; use sempre `PasswordEncoder`.
- Token de reset de senha e salvo como hash, nao em texto claro.
- Rotas novas devem ser avaliadas em `SecurityConfig`: publica ou autenticada.

## Pagamentos e assinatura

Arquivos principais:

- `api/controller/PagamentoController.java`: endpoints de pagamento e webhook.
- `application/usecase/pagamento`: checkout, registro, historico e webhook.
- `application/service/PagamentoService.java`: aplica pagamento, validade,
  ciclo de assinatura e indicacao.
- `infra/gateway/AsaasGateway.java`: chamadas HTTP ao Asaas.
- `infra/gateway/AsaasWebhookValidator.java`: validacao do webhook.

Regras importantes:

- `paymentId` externo garante idempotencia. Webhook reenviado nao pode duplicar
  pagamento nem renovar assinatura duas vezes.
- `assinaturaCiclo` aumenta a cada pagamento confirmado.
- O ciclo e usado para impedir reuso do mesmo cupom dentro da mesma assinatura.
- Codigo de indicacao so deve gerar bonus no primeiro pagamento do indicado.

## Cupons

Arquivos principais:

- `api/controller/UsuarioProdutoController.java`: endpoints de usar/listar/avaliar.
- `application/service/UsuarioProdutoService.java`: valida assinatura, periodo do
  cupom, duplicidade por ciclo e avaliacao.
- `infra/repository/UsuarioProdutoRepository.java`: consultas e agregados de uso.
- `domain/model/UsuarioProduto.java`: historico de uso do cupom pelo usuario.

Regras importantes:

- Usuario precisa ter assinatura ativa para usar cupom.
- O mesmo cupom nao pode ser usado duas vezes no mesmo ciclo.
- A avaliacao so pode ser feita pelo dono do uso e apenas uma vez.

## Notificacoes

Arquivos principais:

- `api/controller/NotificacaoController.java`
- `application/service/NotificacaoService.java`
- `application/usecase/notificacao`
- `infra/gateway/FcmGateway.java`

Pontos de cuidado:

- Device token pertence ao usuario autenticado.
- Falha de push nao deve quebrar regra principal como pagamento ou uso de cupom,
  a menos que o requisito mude explicitamente.

## Cadastros basicos

Categorias, cidades, empresas, planos, produtos, sugestoes e tipos de produto
seguem o mesmo padrao:

- Controller recebe HTTP.
- Use case representa a acao.
- Service aplica regra simples de CRUD.
- Repository acessa banco.
- Mapper monta DTO de resposta.

Ao criar um novo cadastro, copie esse padrao para manter consistencia.

## Erros

Use:

- `BusinessException`: regra de negocio violada.
- `NotFoundException`: recurso inexistente.
- `GlobalExceptionHandler`: conversao de excecoes para resposta HTTP.

Evite retornar `null` em regras novas, exceto em fluxos onde isso ja e parte do
contrato, como webhook defensivo.

## Banco e migracoes

As migracoes ficam em `src/main/resources/db/migration`.

Cuidados:

- Nunca altere uma migration antiga que ja rodou em ambiente compartilhado.
- Crie sempre uma nova migration `V{numero}__descricao.sql`.
- Se uma regra depende de unicidade no banco, mantenha tambem validacao na
  aplicacao para erro mais claro ao usuario.

## Checklist antes de alterar

- A mudanca pertence ao controller, use case, service, repository ou gateway?
- Existe DTO para nao expor entidade diretamente?
- A rota precisa de JWT ou deve ser publica?
- A regra precisa de transacao?
- Existe risco de duplicidade por concorrencia ou webhook reenviado?
- O erro deve ser `BusinessException`, `NotFoundException` ou outro tratamento?
- Ha teste cobrindo o fluxo principal e pelo menos um erro relevante?
