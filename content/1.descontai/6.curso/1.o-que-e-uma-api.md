Antes de escrever uma linha de código, você precisa entender **o que estamos construindo
e por que**. Esta aula é 100% conceito — e é a base de tudo. Sem pressa aqui.

## O problema que uma API resolve

Pensa no Descontai. Existem **vários programas diferentes** que precisam dos mesmos dados:

- O **app** no celular (feito em Flutter) mostra os cupons.
- O **portal das Empresas** (site) deixa o parceiro cadastrar ofertas.
- O **portal Admin** (site) deixa a equipe gerenciar tudo.

Todos eles precisam saber, por exemplo, "quais cupons existem na cidade do usuário?".
Seria uma loucura cada um ter sua própria cópia do banco de dados e sua própria regra de
"quem pode ver o quê". Ia dar divergência, bug e caos.

A solução é ter **um cérebro central** que:

- guarda os dados (usuários, cupons, pagamentos…),
- aplica as regras (só quem pagou vê cupom premium),
- e responde às perguntas de todo mundo de um jeito padronizado.

Esse cérebro central é a **API**. O app e os portais não falam com o banco direto — eles
**pedem** as coisas para a API, e a API responde.

::callout{type="info" title="API, em uma frase"}
**API** (Application Programming Interface) é um programa que fica rodando num servidor
esperando **pedidos** de outros programas, e devolve **respostas**. É um garçom: você (o app)
não entra na cozinha (o banco de dados) — você faz o pedido ao garçom (a API), e ele traz o
prato pronto seguindo as regras da casa.
::

## Cliente e servidor

Essa é a dupla mais importante de toda a web:

- **Cliente**: quem faz o pedido. No Descontai, o cliente é o app Flutter ou o portal web.
  (Também chamamos o navegador de "cliente".)
- **Servidor**: quem recebe o pedido, processa e responde. É onde a nossa API roda —
  um computador ligado 24/7 na nuvem.

O cliente e o servidor conversam pela internet. E essa conversa segue um **protocolo**
(um conjunto de regras combinadas) chamado **HTTP**.

## HTTP: a língua da web

Toda vez que você abre um site ou o app carrega os cupons, acontece um **HTTP request**
(pedido) seguido de um **HTTP response** (resposta). É sempre esse par: pediu → respondeu.

Um request tem 4 partes principais:

::fields
::field{name="Método (verbo)" type="GET, POST, PUT, DELETE"}
O que você quer **fazer**. `GET` = ler/buscar. `POST` = criar. `PUT` = atualizar.
`DELETE` = apagar. É como o verbo da frase: "**BUSCAR** os cupons".
::
::field{name="URL (endereço)" type="texto"}
**Onde** está o recurso. Ex.: `https://api.descontai.com/api/v1/cidades`.
É como o endereço do restaurante + a mesa.
::
::field{name="Headers (cabeçalhos)" type="pares chave-valor"}
Informações extras sobre o pedido: quem está pedindo (o token de autenticação), qual formato
de resposta você aceita, etc. É o "recadinho" junto ao pedido.
::
::field{name="Body (corpo)" type="geralmente JSON"}
Os **dados** que você está enviando (só em `POST`/`PUT`). Ex.: ao criar uma conta, o body
leva o email e a senha. Um `GET` normalmente não tem body.
::
::

E a resposta (response) tem 3 partes:

- **Status code**: um número que diz se deu certo (veremos abaixo).
- **Headers**: informações sobre a resposta.
- **Body**: o conteúdo em si (geralmente JSON).

::callout{type="tip" title="Exemplo concreto no Descontai"}
Quando o app quer a lista de cidades para a tela de cadastro, ele manda:

```http
GET /api/v1/cidades HTTP/1.1
Host: api.descontai.com
```

E a API responde:

```http
HTTP/1.1 200 OK
Content-Type: application/json

[
  { "id": 1, "nome": "Recife" },
  { "id": 2, "nome": "Olinda" }
]
```

Simples assim: o app **perguntou** e a API **respondeu** com dados.
::

## JSON: o formato dos dados

Repare que a resposta acima está num formato de texto com chaves e colchetes. Isso é **JSON**
(JavaScript Object Notation). É o jeito mais comum de trocar dados entre cliente e servidor
porque é **legível para humanos** e **fácil para máquinas**.

```json
{
  "id": 42,
  "nome": "Pizzaria do Zé",
  "ativo": true,
  "cidade": { "id": 1, "nome": "Recife" },
  "cupons": ["PIZZA30", "COMBO2X1"]
}
```

Você tem: textos entre aspas, números, `true`/`false`, objetos (`{ }`) e listas (`[ ]`).
Só isso. O app recebe esse texto e transforma em telas; a API recebe JSON do app e transforma
em objetos no código. É a "moeda de troca" da conversa.

## Status codes: o "deu certo?" em números

Todo response vem com um número de status. Você **precisa** conhecer as famílias:

| Faixa | Significado | Exemplos |
|------|-------------|----------|
| **2xx** | Deu certo ✅ | `200 OK`, `201 Created` (criou algo), `202 Accepted` (aceitei, vou processar) |
| **3xx** | Redirecionamento | `301 Moved` |
| **4xx** | Erro **do cliente** ❌ | `400 Bad Request` (pedido malformado), `401 Unauthorized` (sem login), `403 Forbidden` (logado, mas sem permissão), `404 Not Found` |
| **5xx** | Erro **do servidor** 💥 | `500 Internal Server Error` (bug na API) |

::callout{type="warning" title="A diferença que confunde todo iniciante"}
`401` x `403`: **401** é "você **não disse quem é** (não mandou token válido)". **403** é
"eu sei quem você é, mas você **não tem permissão** para isso" (ex.: um usuário comum tentando
acessar rota de admin). E **4xx é culpa do cliente**, **5xx é culpa da API** — saber de quem é
a culpa é meio caminho para achar o bug.
::

## REST: o jeito organizado de desenhar as rotas

Cada endereço que a API atende é uma **rota** (ou *endpoint*). Poderíamos inventar nomes
aleatórios, mas o mercado adota um estilo chamado **REST**, que traz **previsibilidade**.

A ideia central do REST: você pensa em **recursos** (coisas: cidade, cupom, pagamento) e usa
os **verbos HTTP** para agir sobre eles. Olha como fica limpo e previsível no Descontai:

| Quero… | Método + Rota |
|--------|---------------|
| Listar as cidades | `GET /api/v1/cidades` |
| Ver a cidade 1 | `GET /api/v1/cidades/1` |
| Criar uma cidade | `POST /api/v1/cidades` |
| Atualizar a cidade 1 | `PUT /api/v1/cidades/1` |
| Apagar a cidade 1 | `DELETE /api/v1/cidades/1` |

Repara no padrão: **o mesmo endereço** (`/cidades/1`) faz coisas diferentes dependendo do
**verbo**. Isso é REST. Qualquer dev que chega no projeto já adivinha as rotas — não precisa
decorar.

::callout{type="info" title="E o /api/v1?"}
Todas as rotas do Descontai começam com `/api/v1`. O `v1` é a **versão** da API. Se um dia
for preciso mudar algo de forma incompatível, cria-se um `/api/v2` **sem quebrar** os apps
que ainda usam o `v1`. É uma prática de mercado para não deixar clientes antigos na mão.
::

## O que decide quem pode fazer o quê

Nem toda rota é aberta. No Descontai:

- `GET /api/v1/cidades` é **público** (a tela de cadastro precisa listar cidades **antes** do login).
- `POST /api/v1/cidades` (criar cidade) exige ser **ADMIN**.
- Ver seus cupons exige estar **logado**.

Como a API sabe quem é você e o que você pode fazer? Através de **autenticação** (provar quem
você é) e **autorização** (o que você pode). Isso é assunto sério e tem uma aula inteira
(Aula 7), onde a gente vê o **token JWT**. Por ora, guarde a ideia: a API é um porteiro.

## Resumindo

- Uma **API** é o cérebro central que guarda dados, aplica regras e responde a vários clientes.
- **Cliente** pede, **servidor** responde, e eles falam **HTTP**.
- Um pedido tem **método + URL + headers + body**; a resposta tem **status + headers + body**.
- Os dados trafegam em **JSON**.
- **Status codes** dizem se deu certo (2xx), se o cliente errou (4xx) ou se a API quebrou (5xx).
- **REST** organiza as rotas por recurso + verbo, deixando tudo previsível.
