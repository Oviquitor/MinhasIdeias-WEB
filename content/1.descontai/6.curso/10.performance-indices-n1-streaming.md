Até agora a gente se preocupou em fazer a API **funcionar**. Nesta aula, o foco muda: fazer ela
**funcionar rápido, mesmo com muita gente usando**. Uma API pode estar 100% correta e ainda assim
ficar lenta — ou pior, **cair** — quando o volume de dados cresce. Três armadilhas clássicas causam
90% desses problemas: falta de **índice**, o problema **N+1** e carregar **tudo na memória** de uma
vez. Vamos ver os três, cada um com o código real do Descontai.

## Parte 1 — Índices: o "índice do livro" do banco

Imagine um livro de 800 páginas e você quer achar tudo sobre "cupom". Você tem duas opções:

- **Sem índice**: virar página por página, da primeira à última, lendo cada linha. Se a palavra
  aparece na página 790, você leu 789 páginas à toa.
- **Com índice**: ir ao final do livro, no **índice remissivo**, achar "cupom → páginas 12, 340, 790"
  e ir direto. Segundos em vez de horas.

O banco de dados funciona igualzinho. Uma tabela é um monte de linhas. Quando você faz uma busca
como "me dá todos os resgates do usuário 42", o banco tem duas formas de responder:

::fields
::field{name="Sequential Scan (varredura sequencial)" type="sem índice"}
O banco lê a tabela **inteira**, linha por linha, comparando o `usuario_id` de cada uma com o `42`.
Com 10 linhas, tudo bem. Com **1 milhão** de linhas, ele lê 1 milhão para te devolver talvez 5.
É o "virar página por página".
::
::field{name="Index Scan (busca por índice)" type="com índice"}
O banco consulta uma estrutura auxiliar (o índice) que já sabe **onde** estão as linhas do usuário
42, e vai direto nelas. É o "índice remissivo do livro".
::
::

Um **índice** é, então, uma estrutura extra que o banco mantém para uma ou mais colunas, deixando a
busca por aquelas colunas muito mais rápida. No Descontai, quem cria os índices é uma **migration**
do Flyway (você viu migrations na Aula 9). Este é o arquivo real `V7__performance_indexes.sql`:

```sql
-- usuario_produto (tabela de uso de cupons — maior volume de leitura)
CREATE INDEX IF NOT EXISTS idx_uproduto_usuario
    ON usuario_produto(usuario_id);

CREATE INDEX IF NOT EXISTS idx_uproduto_produto
    ON usuario_produto(produto_id);

-- pagamento
CREATE INDEX IF NOT EXISTS idx_pagamento_usuario
    ON pagamento(usuario_id);

-- produtos (filtros de listagem)
CREATE INDEX IF NOT EXISTS idx_produto_empresa
    ON produtos(empresa_id);
```

Leia a primeira linha em voz alta: "cria um índice chamado `idx_uproduto_usuario` na coluna
`usuario_id` da tabela `usuario_produto`". Agora, quando a API pergunta "quais cupons o usuário 42
resgatou?", o banco usa esse índice e responde na hora, sem varrer a tabela toda.

### Quando criar um índice?

Você não sai criando índice em toda coluna (índice tem um custo, veremos abaixo). A regra prática,
que os índices da V7 seguem, é criar índice em:

| Tipo de coluna | Por quê | Exemplo na V7 |
|----------------|---------|---------------|
| **Chaves estrangeiras** (os `_id` que ligam tabelas) | São usadas o tempo todo para juntar tabelas e filtrar "os X deste Y" | `usuario_id`, `empresa_id`, `categoria_id` |
| **Colunas de filtro frequente** (o que vai no `WHERE`) | O banco vai buscar por elas repetidamente | `codigo_indicacao` (busca por código de indicação) |
| **Combinações usadas juntas** (índice composto) | Filtra por duas colunas ao mesmo tempo | `notificacoes(usuario_id, lida)` — "notificações não lidas do usuário" |

::callout{type="info" title="Índice composto: a ordem importa"}
Repare em `idx_notificacao_lida ON notificacoes(usuario_id, lida)`. Ele cobre duas colunas de uma
vez. É como um índice de livro organizado primeiro por autor, depois por título: ótimo para achar
"todos os livros do autor X" **e** "o livro Y do autor X" — mas não ajuda a buscar só por título.
Por isso a coluna mais "seletiva"/mais usada vem primeiro.
::

::callout{type="warning" title="Índice não é de graça"}
Todo índice ocupa espaço e precisa ser **atualizado** a cada `INSERT`/`UPDATE`/`DELETE` — afinal, o
"índice do livro" tem que continuar correto quando você adiciona páginas. Ou seja: índice acelera a
**leitura** e desacelera um pouquinho a **escrita**. Por isso você indexa o que é buscado com
frequência, não tudo. Uma tabela de leitura pesada como `usuario_produto` compensa demais; uma
coluna que ninguém filtra, não.
::

::callout{type="tip" title="Detalhe de produção que está no próprio arquivo"}
No topo da V7 há um comentário importante: criar índice trava (`lock`) a tabela enquanto é
construído. Em um banco com tráfego real, o PostgreSQL oferece `CREATE INDEX CONCURRENTLY`, que
constrói sem travar — mas ele **não roda dentro de uma transação**, e o Flyway usa transação por
padrão. Por isso a migration usa `CREATE INDEX` simples e o comentário orienta rodar como
`CONCURRENTLY` manualmente em produção. Você não precisa decorar isso agora; só saiba que "criar
índice em produção com cuidado" é um assunto real.
::

## Parte 2 — O problema N+1: uma query vira mil

Este é o bug de performance mais comum em quem usa JPA/Hibernate (que você viu na Aula 5), e o mais
traiçoeiro: **o código parece inocente e funciona perfeitamente nos testes com poucos dados**.

O cenário: você lista os resgates de uma empresa e, para cada resgate, quer mostrar o **nome do
produto** e o **nome da empresa**. Um código ingênuo faz assim:

```java
// Pega a lista de resgates (1 query)
List<UsuarioProduto> resgates = repository.findByEmpresa(empresaId);

for (UsuarioProduto up : resgates) {
    // Cada acesso a um relacionamento "lazy" dispara OUTRA query!
    String produto = up.getProduto().getProduto();      // query!
    String empresa = up.getProduto().getEmpresa().getNome(); // query!
}
```

O que acontece por baixo dos panos:

::steps
### 1 query para a lista
`findByEmpresa` roda **uma** query e traz, digamos, 300 resgates.

### +N queries, uma por item
Como produto e empresa são carregados "sob demanda" (*lazy*), no primeiro `up.getProduto()` de cada
resgate o Hibernate dispara **uma nova query** para buscar aquele produto. Com 300 resgates, são
**300 queries extras**.

### Total: 1 + N
Uma query virou **301 queries**. Daí o nome **N+1**: 1 (a lista) + N (uma por item). Com poucos
dados ninguém percebe. Com muitos, a API fica lentíssima e o banco sofre.
::

::callout{type="warning" title="Por que passa despercebido"}
Nos testes locais você tem 5 registros — 6 queries, roda em milissegundos, "tá rápido". Em produção,
com milhares de registros, viram milhares de queries e a tela demora segundos. O N+1 é um problema
que **só aparece na escala**. Por isso a gente aprende a evitá-lo por princípio, não só quando dói.
::

### A solução: `join fetch`

A ideia é simples: em vez de buscar a lista e depois correr atrás dos relacionamentos um a um, você
manda o banco trazer **tudo junto numa query só** — a lista e os relacionamentos, num único
"apanhado". Em JPQL (a linguagem de query do JPA), isso é o `join fetch`.

Este é o método real do `UsuarioProdutoRepository` que alimenta o dashboard de empresas:

```java
/**
 * Lista os resgates (usos de cupom) de todos os produtos de uma empresa.
 * Carrega produto e usuário em uma única query para o serviço montar os DTOs
 * e as agregações sem incorrer em N+1.
 */
@Query("""
        select up from UsuarioProduto up
        join fetch up.produto p
        join fetch up.usuario u
        where p.empresa.id = :empresaId
        order by up.usedAt desc
        """)
List<UsuarioProduto> findResgatesByEmpresa(@Param("empresaId") Long empresaId);
```

O `join fetch up.produto` diz: "ao buscar os resgates, **já traga o produto de cada um na mesma
query**". Idem para o usuário. Resultado: **1 query em vez de 301**. Quando o serviço faz
`up.getProduto().getProduto()`, o dado já está carregado na memória — nenhuma query extra é
disparada.

Repare também neste outro método do mesmo repositório, com um `join fetch` mais profundo:

```java
@Query("""
        select up from UsuarioProduto up
        join fetch up.produto p
        join fetch p.empresa e
        left join fetch e.cidade
        join fetch p.categoria c
        join fetch p.tipoProduto t
        where up.usuario.id = :usuarioId
        """)
List<UsuarioProduto> findByUsuarioIdComRelacoes(@Param("usuarioId") Long usuarioId, ...);
```

Aqui o resgate traz o produto, o produto traz a empresa **e** a categoria **e** o tipo, e a empresa
traz a cidade — tudo de uma vez. Sem o `join fetch`, montar a tela "meus cupons" de um usuário com
50 cupons poderia disparar centenas de queries. Com ele, é uma só.

::callout{type="tip" title="O nome do próprio método já avisa a intenção"}
Note como os métodos se chamam `...ComRelacoes` e `...ComProdutoEmpresa`, e os comentários dizem
literalmente "para evitar N+1". Isso é uma prática boa: deixar explícito que aquele método existe
**justamente** para carregar tudo de uma vez. Quem for usar sabe que pode acessar os relacionamentos
sem medo.
::

## Parte 3 — Streaming: o relatório que quase derrubava o servidor

Agora o caso mais sério — e mais recente — do projeto. É o tipo de problema que não deixa a API só
lenta: pode **derrubar** a instância inteira que atende **todos** os usuários.

### O que estava errado

O Portal Admin tem uma tela de **exportar relatório em CSV** (lista de pagamentos, de resgates,
etc.). A versão antiga fazia, em essência, isto:

```java
// VERSÃO ANTIGA (problemática) — NÃO faça assim com tabelas grandes
List<Pagamento> todos = pagamentoRepository.findAll(); // carrega a tabela INTEIRA na memória

StringBuilder csv = new StringBuilder();                 // monta o CSV INTEIRO na memória
for (Pagamento p : todos) {
    csv.append(p.getId()).append(",").append(p.getValor()).append("\n");
}
return csv.toString(); // só então devolve
```

Parece razoável, e funciona com 100 pagamentos. O problema aparece quando a tabela cresce:

::steps
### findAll traz tudo para a memória
`findAll()` carrega **todos** os registros de uma vez. Com 500 mil pagamentos, são 500 mil objetos
Java vivos na memória ao mesmo tempo.

### O StringBuilder duplica o problema
Além dos objetos, o CSV completo é montado como **um texto gigante** na memória antes de ser enviado.
Agora você tem os dados **duas vezes** ocupando heap.

### A memória estoura (OOM)
A instância tem um limite de memória (heap). Passou do limite → `OutOfMemoryError`. A JVM
**derruba a aplicação**. E aqui está o pior: **não é só o admin que exportou que sofre — a instância
que caiu era a mesma que atendia todos os apps e portais**. Um clique de um admin derruba o serviço
inteiro.
::

::callout{type="warning" title="Memória constante vs. proporcional ao volume"}
A raiz do problema é que o uso de memória era **proporcional ao volume de dados**: dobrou o número de
registros, dobrou a memória. Isso **nunca** é seguro, porque a tabela só cresce — um dia ela passa do
limite. O objetivo é ter uso de memória **constante**: exportar 1 mil ou 1 milhão de linhas deve
consumir praticamente a mesma memória. A diferença fica só no **tempo**, não no risco de cair.
::

### A correção: escrever direto no OutputStream, lendo em cursor

A ideia de **streaming** é não guardar nada inteiro. Em vez de "carrega tudo → monta tudo → devolve
tudo", você faz "lê **um** registro do banco → escreve **essa** linha na resposta → descarta → lê o
próximo". Como só há um punhado de linhas na memória a cada instante, o consumo é constante.

Duas peças fazem isso funcionar no Descontai.

**Peça 1 — o controller devolve um `StreamingResponseBody`** em vez de um texto pronto. Este é o
`AdminRelatorioController` real:

```java
@GetMapping("/{tipo}")
public ResponseEntity<StreamingResponseBody> gerar(@PathVariable String tipo, ...) {
    // Valida ANTES do stream: entrada inválida vira 400 (depois de começar, o status já foi 200).
    String t = service.validar(tipo, formato);
    String nome = "relatorio-" + t + "-" + LocalDate.now() + ".csv";

    // O corpo é uma função que escreve direto no OutputStream da resposta:
    StreamingResponseBody body = out -> service.escrever(t, out);

    HttpHeaders headers = new HttpHeaders();
    headers.setContentType(MediaType.parseMediaType("text/csv; charset=UTF-8"));
    headers.setContentDisposition(ContentDisposition.attachment().filename(nome).build());
    return new ResponseEntity<>(body, headers, HttpStatus.OK);
}
```

`StreamingResponseBody` é do Spring: em vez de você retornar o CSV pronto, você entrega uma **função**
que recebe o `OutputStream` (o "cano" da resposta que vai para o cliente) e escreve nele aos poucos.
O Spring vai enviando para o navegador conforme você escreve.

**Peça 2 — o repositório entrega um `Stream` com cursor**, não uma `List`. Este é o método real do
`PagamentoRepository`:

```java
/**
 * Retornar Stream + fetchSize faz o PostgreSQL usar um cursor: os registros
 * chegam em lotes em vez de todos de uma vez, mantendo a memória constante
 * mesmo com centenas de milhares de linhas.
 */
@QueryHints(@QueryHint(name = "org.hibernate.fetchSize", value = "200"))
@Query("select p from Pagamento p join fetch p.usuario order by p.id")
Stream<Pagamento> streamTodosComUsuario();
```

Três detalhes que fazem toda a diferença:

- **`Stream<Pagamento>`** (não `List`): o `Stream` entrega os registros **um a um, sob demanda**. A
  lista inteira nunca existe na memória.
- **`fetchSize = 200`**: instrui o PostgreSQL a usar um **cursor** e trazer os dados em **lotes de
  200**, em vez de mandar tudo de uma vez pela rede. É a memória constante na prática.
- **`join fetch p.usuario`**: repare que a lição da Parte 2 volta aqui — o usuário vem junto, para
  não cair em N+1 ao montar cada linha do CSV.

E o serviço junta as duas peças, lendo do `Stream` e escrevendo linha por linha no `Writer` (que
escreve no `OutputStream`). Trecho real do `RelatorioService`:

```java
private void csvPagamentos(Writer w) throws IOException {
    w.write("id,usuarioId,usuario,valor,metodo,status,validadeGerada,criadoEm\n");
    try (Stream<Pagamento> stream = pagamentos.streamTodosComUsuario()) { // try-with-resources fecha o cursor
        for (var it = stream.iterator(); it.hasNext(); ) {
            Pagamento p = it.next();          // lê UM
            Usuario u = p.getUsuario();
            linha(w, p.getId(), ..., p.getValor(), p.getStatus(), ...); // escreve a linha
        }                                     // o objeto sai de escopo e é descartado; lê o próximo
    }
}
```

::callout{type="info" title="Por que o @Transactional(readOnly = true) e o try-with-resources"}
Um cursor de banco precisa da **conexão/sessão aberta** enquanto você lê os registros aos poucos.
Por isso o método `escrever(...)` do serviço é `@Transactional(readOnly = true)`: a transação
mantém a sessão viva durante toda a escrita. E o `try (Stream ...)` (try-with-resources) garante que
o cursor seja **fechado** ao final, liberando a conexão. Esquecer de fechar vaza recursos do banco.
::

### Bônus: às vezes a melhor query é a que nem traz linhas

O relatório de **receita** antes carregava todos os pagamentos aprovados só para **somar** os valores
por mês na aplicação. Mas somar é justamente o que o banco faz de melhor. A correção foi empurrar a
conta para o banco com `GROUP BY` — ele devolve **~1 linha por mês** já somada:

```java
@Query("""
        select year(p.createdAt) as ano, month(p.createdAt) as mes, sum(p.valor) as total
        from Pagamento p
        where p.status = :status and p.createdAt is not null and p.valor is not null
        group by year(p.createdAt), month(p.createdAt)
        order by year(p.createdAt), month(p.createdAt)
        """)
List<ReceitaMensalProjection> somarReceitaPorMes(@Param("status") String status);
```

Em vez de trazer 500 mil linhas para somar no Java, o banco devolve uma dúzia de linhas (uma por
mês) já prontas. **A query mais rápida costuma ser a que traz menos dados** — deixe o banco fazer o
trabalho pesado (somar, contar, agrupar) sempre que der.

## Resumindo

- **Índice** é o "índice remissivo" do banco: sem ele, uma busca **varre a tabela inteira**
  (*sequential scan*); com ele, o banco vai direto às linhas (*index scan*). Crie índices em
  **chaves estrangeiras** (`usuario_id`, `empresa_id`) e em **colunas de filtro frequente** — não em
  tudo, porque índice pesa na escrita.
- **N+1** é quando 1 query (a lista) dispara **+N queries** (uma por item) para carregar
  relacionamentos *lazy*. Resolve-se com **`join fetch`**, trazendo a lista e os relacionamentos numa
  **única query**. Cuidado: só aparece na escala.
- **Streaming** resolve o relatório que carregava tudo na memória (`findAll` + `StringBuilder`) e
  podia estourar o heap (**OOM**) e derrubar a instância. A correção: `StreamingResponseBody`
  escrevendo direto no `OutputStream` + repositório retornando `Stream` com `fetchSize` (cursor) =
  **memória constante**, não proporcional ao volume.
- E o truque de ouro: **deixe o banco somar/contar/agrupar** (`GROUP BY`) em vez de trazer tudo para
  a aplicação.

::callout{type="tip" title="Fixando"}
Se você conseguir explicar em voz alta *"por que exportar 1 milhão de linhas com `findAll` derruba o
servidor, mas com `Stream` + `fetchSize` não"*, você entendeu o coração desta aula: a diferença entre
memória **proporcional ao volume** e memória **constante**.
::
