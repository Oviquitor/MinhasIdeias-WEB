A API está no ar, o deploy funcionou, os testes passaram. Aí, numa terça à noite, um usuário
manda mensagem: "o app tá lento". E agora? Está lento pra todo mundo ou só pra ele? É o banco?
A memória estourou? Um endpoint específico? **Se você só tem o print do WhatsApp do usuário,
você está no escuro.**

Esta aula é sobre acender a luz. Primeiro **observabilidade** (enxergar o que acontece lá dentro
em tempo real) e depois **teste de carga** (descobrir o limite da API **antes** dos usuários reais
descobrirem por você).

## Por que observabilidade importa

Existe uma frase que resume tudo:

::callout{type="info" title="A regra de ouro"}
**Você não conserta o que não consegue enxergar.** Sem instrumentar a aplicação, um bug em
produção vira adivinhação. Com observabilidade, você abre um painel e o próprio sistema te conta
onde dói.
::

Observabilidade se apoia em **três pilares**. A gente vai usar os dois primeiros de verdade nesta
aula e só mencionar o terceiro:

::fields
::field{name="Logs" type="o diário"}
O **diário** do que aconteceu: linha a linha, "às 20h03 o usuário 42 fez login", "às 20h04 deu
erro ao salvar o pagamento". Ótimo pra investigar **um caso específico** depois que já deu ruim.
::
::field{name="Métricas" type="os números ao longo do tempo"}
**Números medidos ao longo do tempo**: quantas requisições por segundo, quanta memória em uso,
quantos erros 5xx nos últimos 5 minutos. É o que te avisa que **algo está degradando** — de
preferência antes de virar incêndio.
::
::field{name="Traces" type="o raio-x de uma requisição"}
O **raio-x de uma única requisição** atravessando o sistema: quanto tempo gastou no controller,
no banco, na chamada externa ao Asaas. Útil em sistemas grandes/distribuídos. No Descontai a gente
só **menciona** — fica pra um momento futuro.
::
::

A analogia que vale a aula inteira: **métricas são o painel do carro**. Você não abre o motor a
cada semáforo — você olha o velocímetro, o marcador de temperatura e a luz de óleo. Quando uma luz
acende, **aí** você vai investigar (nos logs). Observar é dirigir olhando o painel.

## Health check: os sinais vitais da API

O jeito mais básico de "estar vivo" é conseguir responder uma pergunta simples: **"você está de
pé?"**. Isso é o **health check**. No Descontai ele vem pronto pelo **Spring Boot Actuator** — uma
biblioteca que expõe informações operacionais da aplicação.

Olha como o Actuator está ligado no `application.properties` real do projeto:

```properties
# ===============================
# = ACTUATOR
# ===============================
management.endpoints.web.exposure.include=health,info,prometheus
management.endpoint.health.show-details=when_authorized
management.endpoint.health.probes.enabled=true
management.endpoints.web.base-path=/actuator
management.prometheus.metrics.export.enabled=true
# Health do Redis desligado por padrão para não reportar DOWN quando não há Redis.
management.health.redis.enabled=${REDIS_HEALTH_ENABLED:false}
```

Lendo isso de cima pra baixo:

- `exposure.include=health,info,prometheus` — só expomos **três** endpoints. O Actuator tem
  vários outros (dá pra ver variáveis de ambiente, threads etc.), mas expor tudo é risco de
  segurança. Expomos o mínimo.
- `show-details=when_authorized` — os **detalhes** do health (banco OK? e-mail OK?) só aparecem
  pra quem está autenticado. Pra internet aberta, o `/actuator/health` só diz `UP` ou `DOWN`, sem
  entregar a intimidade da infra.
- `health.probes.enabled=true` — **liga liveness e readiness** (o coração desta seção, logo abaixo).

Você chama `GET /actuator/health` e recebe algo assim:

```json
{
  "status": "UP",
  "components": {
    "api": { "status": "UP", "details": { "api": "ok" } },
    "db": { "status": "UP" },
    "diskSpace": { "status": "UP" }
  }
}
```

### Health indicators customizados

O status geral (`UP`) é o **agregado** de vários "sensores" chamados **health indicators**. O
Spring já traz alguns de fábrica (espaço em disco, e — quando o SMTP está configurado — um
indicador `mail` que testa o servidor de e-mail; por isso, sem SMTP, o health do projeto pode ficar
`DOWN`). E o Descontai tem **os seus próprios**, na pasta `src/main/java/com/descontai/infra/health/`:

```java
@Component
public class DatabaseHealthIndicator implements HealthIndicator {
    private final JdbcTemplate jdbc;
    public DatabaseHealthIndicator(JdbcTemplate jdbc){ this.jdbc = jdbc; }

    /** Faz um select 1 para checar se o banco responde. */
    @Override public Health health(){
        try {
            jdbc.queryForObject("select 1", Integer.class);
            return Health.up().build();
        } catch (Exception e) {
            return Health.down(e).build();
        }
    }
}
```

Simples e genial: ele roda um `select 1` no Postgres. Se o banco responde, o componente é `UP`;
se a conexão caiu, é `DOWN` — e como o health é o agregado, a API inteira reporta `DOWN`. Existe
também um `ApiHealthIndicator` (um "ok" básico) e um `CustomHealthEndpoint` que reexpõe tudo isso
numa rota `/health` na raiz.

::callout{type="tip" title="Por que um select 1 e não uma query de verdade?"}
Porque o health check roda **o tempo todo** (a nuvem bate nele a cada poucos segundos). Ele
precisa ser **baratíssimo**. `select 1` só confirma "o banco atende e o pool tem conexão livre" —
sem ler dado nenhum. Health check pesado é health check que vira problema.
::

### Liveness vs Readiness — a parte que a nuvem usa

Aquele `health.probes.enabled=true` desmembra o health em **dois tipos de pergunta** bem
diferentes:

| Prova | A pergunta | Rota |
|-------|-----------|------|
| **Liveness** | "O processo está **vivo** (não travado)?" | `/actuator/health/liveness` |
| **Readiness** | "Está **pronto para receber tráfego** agora?" | `/actuator/health/readiness` |

A diferença parece sutil mas é enorme. Uma app pode estar **viva mas não pronta**: acabou de subir
e ainda está rodando as migrations do Flyway e abrindo o pool de conexões. Nesses segundos ela está
respirando (liveness `UP`), mas mandar tráfego pra ela seria erro (readiness ainda `DOWN`).

E é aqui que a nuvem entra — gancho direto pra **Aula 16 (deploy)**:

- Se **liveness** falha, o orquestrador entende "esse processo travou" e **reinicia** o container.
- Se **readiness** falha, ele entende "ainda não pode atender" e **para de rotear** requisições
  pra essa instância (mas não a mata) — volta a mandar tráfego quando ela ficar `UP`.

::callout{type="info" title="Sinais vitais, na prática"}
Pensa num paciente no hospital. **Liveness** é "o coração está batendo?" — se parar, chama o time
de emergência (reinicia). **Readiness** é "ele já pode voltar a trabalhar?" — pode estar vivo mas
ainda de repouso. A nuvem é a enfermeira que fica olhando os dois monitores e agindo sozinha. Por
isso o `/actuator/health` está registrado no `.do/app.yaml` do deploy.
::

## Métricas: o painel do carro

Health check responde sim/não. **Métricas** respondem **"quanto?"** ao longo do tempo — e é o que
te deixa perceber uma degradação antes dela virar queda.

A cadeia no Descontai é: **Actuator + Micrometer + Prometheus**.

::fields
::field{name="Micrometer"}
A biblioteca que **coleta** os números dentro da aplicação (requisições, latência, memória, pool
de conexões). É tipo o SLF4J, mas pra métricas: uma fachada padrão.
::
::field{name="Prometheus"}
O formato/sistema que **lê e armazena** essas métricas ao longo do tempo. O Actuator expõe tudo
num endpoint que o Prometheus vem "raspar" (fazer *scraping*) de tempos em tempos.
::
::

Com `management.prometheus.metrics.export.enabled=true`, a API expõe `GET /actuator/prometheus`,
que devolve um texto enorme parecido com isto:

```bash
# HELP http_server_requests_seconds Duration of HTTP server request handling
http_server_requests_seconds_count{method="GET",status="200",uri="/api/v1/produtos"} 1834
http_server_requests_seconds_sum{method="GET",status="200",uri="/api/v1/produtos"} 41.2
jvm_memory_used_bytes{area="heap"} 2.68435456E8
hikaricp_connections_active{pool="DescontaiPool"} 7
```

Você não lê esse texto direto — o Prometheus lê, guarda, e uma ferramenta como o Grafana desenha
os gráficos. **O que importa pra você é saber o que vigiar e o que disparar alerta.** As quatro
métricas que valem ouro:

| O que medir | Por que importa | Alerta quando… |
|-------------|-----------------|----------------|
| **CPU / memória** | Recurso esgotado = app lenta ou reiniciando | CPU sustentada > 80%, memória perto do limite |
| **Taxa de erros 5xx** | 5xx é **culpa da API** (Aula 1) — bug/queda | % de 5xx sobe acima do normal |
| **Latência p95** | A experiência real de quase todo mundo | p95 passa de **1s** |
| **Saturação do pool** | Sem conexão livre, tudo espera o banco | `hikaricp_connections_active` bate no `DB_POOL_MAX` |

### O que é "p95" (sem susto)

**p95** é o **percentil 95**. Traduzindo: "**95% das requisições foram mais rápidas que este
número; só os 5% piores foram mais lentos**".

Por que não usar a **média**? Porque a média mente. Imagine 100 requisições: 99 respondem em 100ms
e 1 trava em 10 segundos. A **média** dá ~200ms (parece ótimo!), mas teve gente esperando 10s. O
**p95** olha justamente pra ponta ruim e revela a dor real dos usuários. Por isso a meta do
projeto é medida em **p95, não em média**.

::callout{type="warning" title="Actuator não é pra internet aberta"}
Repare no `MANAGEMENT_PORT` do projeto e na regra do `DEPLOY.md`: o `/actuator/prometheus` **exige
autenticação** e deve ser exposto só na rede interna (ou numa porta de management isolada). Essas
métricas contam demais sobre a sua infra — não é coisa pra deixar pública.
::

### Logs, rapidamente

O terceiro elemento operacional são os **logs** — o diário. No Descontai eles são configurados via
`logging.config=classpath:logback-spring.xml`. Em produção o combinado é: **logs centralizados**
(todos num lugar só, pesquisáveis) com **retenção e alertas** — como pede a seção de observabilidade
do `GO-LIVE-10K.md`. A dinâmica no dia a dia: a **métrica acende a luz** ("subiu o 5xx!"), e você
**vai ao log** achar a exceção exata que causou.

## Teste de carga: o simulado antes da prova

Observabilidade te mostra o presente. Mas antes de abrir pro público, você precisa responder a
pergunta do milhão: **a API aguenta os 10 mil usuários?** Descobrir isso com usuários reais é a
pior forma possível. A boa forma é o **teste de carga**.

::callout{type="info" title="O que é teste de carga"}
É **simular muitos usuários virtuais** batendo na API ao mesmo tempo, de forma controlada, pra
achar o **limite** dela **antes** dos usuários reais chegarem. É o **simulado antes da prova**:
você erra no ambiente seguro (staging), corrige, e chega no dia real preparado.
::

A ferramenta do projeto é o **k6**: você escreve o roteiro de um usuário em JavaScript e o k6 roda
esse roteiro com milhares de cópias simultâneas (os **VUs**, *virtual users*). O script real está
em `load-test/k6-descontai.js` e simula o fluxo de um assinante: **login → refresh de token →
navegar no catálogo → ler dados públicos** (planos e cidades).

### As metas reais do projeto

O k6 não só gera carga — ele **decide se passou ou não** com base em *thresholds*. Estes são os do
Descontai, direto do script:

```javascript
thresholds: {
  http_req_failed: ['rate<0.02'],          // < 2% de falhas HTTP
  http_req_duration: ['p(95)<1000'],       // p95 < 1s no pico
  erros_negocio: ['rate<0.05'],
}
```

Traduzindo as duas metas que mandam: **p95 abaixo de 1 segundo** e **menos de 2% de requisições
falhando**. Se qualquer threshold estourar, o k6 termina com **erro** — ótimo pra travar um
pipeline. É objetivo: passou ou não passou.

### A rampa até 10 mil

O teste não joga 10k de uma vez — ele **sobe aos poucos** (rampa), porque assim você vê **em que
ponto** a coisa começa a degradar:

```javascript
scenarios: {
  rampa_10k: {
    executor: 'ramping-vus',
    startVUs: 0,
    stages: [
      { duration: '2m', target: 1000 },   // aquecimento
      { duration: '3m', target: 5000 },    // meia carga
      { duration: '5m', target: 10000 },   // alvo: 10k simultâneos
      { duration: '5m', target: 10000 },   // sustenta o pico
      { duration: '2m', target: 0 },       // desaceleração
    ],
  },
}
```

E cada checagem dentro do roteiro marca sucesso/erro, como no login:

```javascript
const res = http.post(`${API}/auth/login`,
  JSON.stringify({ email: EMAIL, senha: SENHA }),
  { headers: { 'Content-Type': 'application/json' } });
const ok = check(res, { 'login 200': (r) => r.status === 200 });
erros.add(!ok);
```

### Como rodar

::steps
### Fumaça primeiro (smoke test)
Antes do teste pesado, um teste rápido com poucos VUs pra ver se o roteiro **funciona** — nada
pior que descobrir um bug no script depois de 10 minutos de rampa:

```bash
k6 run -e BASE_URL=http://localhost:8080 -e STAGE=smoke \
       -e EMAIL=carga@descontai.com -e SENHA=senha123 \
       load-test/k6-descontai.js
```

### O teste de verdade, contra STAGING
Nunca contra produção — você estaria atacando seus próprios usuários. Aponte pra **staging**:

```bash
k6 run -e BASE_URL=https://staging.descontai.com \
       -e EMAIL=carga@descontai.com -e SENHA=senha123 \
       load-test/k6-descontai.js
```

### Leia o resultado e ajuste
Olhe se `p(95)` ficou abaixo de 1s e `http_req_failed` abaixo de 2%. Se não, é hora de mexer nos
parâmetros de escala.
::

### O ciclo de ajuste

Quando o teste **não** passa, o `GO-LIVE-10K.md` e o `DEPLOY.md` dizem exatamente quais botões
girar. É um ciclo: **rodar → medir → ajustar → rodar de novo**, mexendo em dois parâmetros da
Aula 14/16:

- **`DB_POOL_MAX`** — o tamanho do pool de conexões com o Postgres. Se a métrica de saturação do
  pool bate no teto durante o teste, as requisições ficam **na fila esperando conexão** e o p95
  dispara. Sobe o pool. **Cuidado com a regra:** `DB_POOL_MAX × instance_count < max_connections`
  do Postgres — senão o banco recusa conexão.
- **`instance_count`** — o número de instâncias da API (escala horizontal). Mais instâncias =
  mais requisições em paralelo. Lembrando: pra várias instâncias com folga, o projeto liga o
  **Redis** (`CACHE_BACKEND=redis`, `RATE_LIMIT_BACKEND=redis`) pra cache e rate limit ficarem
  consistentes entre elas.

::callout{type="tip" title="O teste de carga fecha o ciclo"}
Repara como tudo se conecta: o teste de carga **provoca** a degradação, a **observabilidade** te
mostra **onde** dói (pool saturado? CPU no talo?), e os **parâmetros de escala** são o ajuste. Sem
os três juntos, dimensionar seria chutar. Com eles, é engenharia.
::

## Resumindo

- **Você não conserta o que não consegue enxergar.** Observabilidade tem 3 pilares: **logs** (o
  diário), **métricas** (números ao longo do tempo) e **traces** (raio-x de uma requisição).
- O **health check** (`/actuator/health`, via Actuator) diz se a API está de pé. Indicadores
  customizados como o `DatabaseHealthIndicator` (`select 1`) compõem o status agregado.
- **Liveness** ("está vivo?") faz a nuvem **reiniciar**; **readiness** ("pronto pra tráfego?") faz
  a nuvem **rotear ou parar de rotear** — sem matar a instância.
- **Métricas** (Micrometer → `/actuator/prometheus`) são o painel do carro. Vigie **CPU/memória**,
  **taxa de 5xx**, **latência p95** e **saturação do pool**. **p95** = 95% foram mais rápidas que
  esse número (a média engana).
- **Teste de carga** (k6) é o simulado: simula milhares de VUs contra **staging** pra achar o
  limite antes dos usuários reais. Metas do projeto: **p95 < 1s** e **http_req_failed < 2%**.
- Não passou? Ajuste **`DB_POOL_MAX`** e **`instance_count`** (respeitando `pool × instâncias <
  max_connections`) e rode de novo.

Você fechou o ciclo completo: construir, entregar, escalar e **enxergar** em produção. Na próxima
aula, a gente dá um passo atrás e olha o todo — **padrões, boas práticas** e por onde continuar
crescendo como dev backend. 👉 [Aula 18](18.padroes-boas-praticas-proximo-passo)
