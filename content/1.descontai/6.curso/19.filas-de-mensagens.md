Você já viu, na Aula 8, que o Descontai não trava a requisição do admin quando ele dispara uma
notificação para milhares de usuários — esse trabalho pesado vai para segundo plano com
`@Async`. Nesta aula a gente parte daí para entender uma peça que todo sistema grande acaba
encontrando: a **fila de mensagens**.

## O problema: trabalho que não cabe na requisição

Lembra do cenário da Aula 8. O admin clica em "enviar notificação para todos". Se a API tentasse
percorrer 10 mil usuários e chamar o Firebase para cada um **antes** de responder, o admin ficaria
30 segundos com a tela travada — e o navegador provavelmente estouraria o timeout no meio.

A solução do Descontai é responder na hora (`202 Accepted`, "aceitei, vou processar") e fazer o
envio **em segundo plano**. Olha o código real que faz isso — o `NotificacaoBroadcastService`:

```java
// application/service/NotificacaoBroadcastService.java
@Async("notificationExecutor")   // roda FORA do thread da requisição HTTP
public void enviarParaTodos(String titulo, String mensagem, String payload) {
    int pagina = 0;
    Page<Usuario> page;
    do {
        page = usuarios.findAll(PageRequest.of(pagina, TAMANHO_LOTE)); // lotes de 500
        for (Usuario usuario : page.getContent()) {
            notificacaoService.criarEEnviar(usuario, titulo, mensagem, payload);
        }
        pagina++;
    } while (page.hasNext());
    log.info("Broadcast (todos) concluido...");
}
```

E o pool de threads que executa isso é declarado no `AsyncConfig`:

```java
// infra/config/AsyncConfig.java
@Bean(name = "notificationExecutor")
public Executor notificationExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(2);
    executor.setMaxPoolSize(4);
    executor.setQueueCapacity(100);   // ← repare: já existe uma "fila" aqui dentro
    executor.setThreadNamePrefix("notif-");
    executor.initialize();
    return executor;
}
```

::callout{type="info" title="Repare: o @Async JÁ é uma mini-fila"}
Aquele `setQueueCapacity(100)` é literalmente uma fila **dentro da memória da aplicação**: as
tarefas entram numa fila e 2 a 4 threads vão puxando uma por vez para processar. Ou seja, você já
entende o conceito de fila — o que muda numa "fila de mensagens de verdade" é **onde** essa fila
mora e **quais garantias** ela oferece. É esse pulo que a aula vai te mostrar.
::

## O que é uma fila de mensagens

Uma **fila de mensagens** (*message queue*) é um sistema **separado da sua aplicação** que guarda
"recados" (mensagens) numa ordem, para serem processados depois. Quem coloca o recado se chama
**produtor** (*producer*); quem tira e processa se chama **consumidor** (*consumer*).

::callout{type="info" title="A analogia da senha da padaria"}
Você chega numa padaria movimentada e tira uma **senha**. Você não fica parado esperando o
atendente te olhar — você tirou a senha (deixou o "recado" de que quer ser atendido) e pode fazer
outra coisa. O atendente vai chamando as senhas na ordem, no ritmo dele. A **fila de senhas** é o
sistema de mensagens: **desacopla** quem pede (você) de quem atende (o balconista). Se chegar muita
gente de uma vez, ninguém é perdido — todos ficam na fila esperando a vez.
::

O poder da fila está em três palavras: **desacoplar**, **absorver pico** e **não perder trabalho**.

::fields
::field{name="Desacoplar"}
Quem produz a tarefa **não precisa esperar** quem a executa. A API "solta o recado" e segue livre.
Se o consumidor estiver lento ou até caído, a mensagem **fica guardada** esperando.
::
::field{name="Absorver picos"}
Se 100 mil eventos chegam de uma vez (uma promoção-relâmpago), a fila **empilha** e o consumidor
processa no ritmo que aguenta, sem derrubar nada. É um amortecedor entre a entrada e o processamento.
::
::field{name="Não perder trabalho"}
Uma fila de verdade **persiste** as mensagens (no disco, replicadas). Se a aplicação reiniciar ou
cair no meio, as mensagens **continuam lá** quando ela voltar. Nada se perde.
::
::

## Os nomes que você vai ouvir

| Ferramenta | Em uma frase | Costuma aparecer quando |
|---|---|---|
| **RabbitMQ** | Fila clássica de "tarefas": produtor manda, consumidor processa e confirma | Você quer distribuir trabalho (e-mails, PDFs, push) entre workers |
| **Apache Kafka** | Um "rio de eventos" persistente e de altíssimo volume, que vários sistemas leem | Você tem muitos eventos e vários times/serviços querem reagir a eles |
| **AWS SQS** | Fila gerenciada da AWS — você não administra servidor nenhum | Você já está na AWS e quer fila sem dor de operação |
| **Redis (streams/lists)** | O Redis que você já conhece (Aula 15) também serve de fila simples | Você já tem Redis e a necessidade é modesta |

::callout{type="tip" title="Fila x tópico, sem decoreba"}
Há dois estilos. **Fila (queue):** cada mensagem é processada por **um** consumidor — bom para
"dividir tarefas" (RabbitMQ, SQS). **Tópico / stream (Kafka):** a mensagem é um **evento** que
**vários** interessados podem ler independentemente — bom para "avisar o mundo que algo aconteceu".
Ex.: "PAGAMENTO_CONFIRMADO" poderia ser lido ao mesmo tempo pelo módulo de assinatura, pelo de
e-mail e por um de analytics, cada um no seu ritmo.
::

## Onde uma fila brilharia no Descontai

Pega o webhook de pagamento do Asaas (Aula 8). Hoje, quando o Asaas confirma um pagamento, a API
processa tudo ali mesmo: confirma a assinatura, credita indicação, cria notificação. Funciona bem.
Mas imagine que amanhã, quando um pagamento é confirmado, a gente queira **também**: mandar e-mail
de boas-vindas, atualizar um painel de BI, avisar o parceiro. Fazer tudo isso dentro do webhook o
deixaria lento e frágil (se o e-mail falhar, o webhook inteiro falha?).

Com uma fila, o webhook faria só o essencial e **publicaria um evento** `PAGAMENTO_CONFIRMADO`.
Cada consumidor (e-mail, BI, parceiro) reagiria no seu tempo, isolado dos outros. Um falhar não
derruba os demais.

## Então por que o Descontai NÃO usa fila (ainda)?

Aqui está a lição mais importante da aula. O Descontai **não precisa** — e adicionar uma fila
agora seria **complexidade que só traria custo**:

::fields
::field{name="O volume não pede"}
O único trabalho pesado assíncrono hoje é o broadcast de push. Para isso, um pool de threads com
uma fila em memória (o `@Async` + `queueCapacity`) **resolve com folga**. Fila externa aqui seria
usar um guindaste para levantar uma sacola.
::
::field{name="Fila é mais uma peça para operar"}
RabbitMQ ou Kafka são **servidores a mais** para instalar, monitorar, atualizar, escalar e pagar.
Cada peça nova de infra é uma nova fonte de falha e de conta no fim do mês.
::
::field{name="Perda tolerável"}
Se a API reiniciar no meio de um broadcast, o pior caso é alguns usuários não receberem **uma**
notificação — que o admin pode reenviar. Não é um pagamento perdido. O `@Async` em memória é
**aceitável** para esse risco. Para dinheiro, a conversa mudaria.
::
::

::callout{type="warning" title="O critério honesto: quando trocar @Async por fila de verdade"}
Você **passa** do `@Async` em memória para uma fila externa quando **qualquer** destes for verdade:

1. **Não pode perder a tarefa** se a aplicação cair (ex.: processar um pagamento, emitir nota fiscal).
2. Você roda **várias instâncias** e precisa que **uma só** pegue cada tarefa (fila em memória é por
   instância — todas pegariam a mesma).
3. O volume de tarefas **cresceu** a ponto de a fila em memória encher e começar a rejeitar trabalho.
4. **Vários sistemas diferentes** precisam reagir ao mesmo evento (aí Kafka faz sentido).

Enquanto **nenhum** desses for verdade, `@Async` é a escolha **certa** — mais simples, sem infra
extra. Adicionar Kafka "porque é moderno" é o erro clássico de over-engineering.
::

## Resumindo

- Uma **fila de mensagens** é um sistema separado que guarda "recados" para serem processados
  depois, **desacoplando** quem produz de quem consome.
- Ela resolve três dores: **desacoplar**, **absorver picos** e **não perder trabalho** (persistência).
- Nomes de mercado: **RabbitMQ** e **SQS** (filas de tarefa), **Kafka** (rio de eventos), Redis
  para casos simples.
- O `@Async` do Descontai **já é uma mini-fila em memória** — ótimo para o broadcast de push, que
  tolera perda e roda numa instância.
- **Troque para fila externa quando**: não pode perder a tarefa, roda várias instâncias disputando o
  trabalho, o volume estourou, ou vários sistemas precisam reagir ao mesmo evento. Antes disso,
  fila é peso morto.

::callout{type="tip" title="Fixando"}
Responda em voz alta: *"por que o broadcast de notificação do Descontai pode viver com `@Async` em
memória, mas o processamento de um pagamento — se fosse desenhado assíncrono — exigiria uma fila
persistente?"* Se você respondeu **"porque perder uma notificação é tolerável, perder um pagamento
não é"**, você entendeu o coração da decisão.
::
