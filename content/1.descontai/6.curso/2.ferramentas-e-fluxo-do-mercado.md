Saber programar é só metade do jogo. A outra metade — a que quase nenhum curso ensina — é
**como o trabalho acontece num time**: as ferramentas, o versionamento do código e o ritual
de revisar mudanças antes de irem para produção. É isso que separa o "sei fazer funcionar"
do "sei trabalhar como profissional".

## As ferramentas que constroem a API

O Descontai é escrito em **Java** e usa o framework **Spring Boot**. Para trabalhar nele,
você usa este kit:

::fields
::field{name="Java (JDK 17)" type="linguagem + runtime"}
A linguagem em que a API é escrita. O **JDK** é o "kit de desenvolvimento" que compila e roda
Java. O Descontai fixa a **versão 17** (uma versão LTS, de suporte longo) — todo mundo no time
e o servidor usam a mesma, para não ter surpresa de "funciona na minha máquina".
::
::field{name="Maven" type="gerenciador de build e dependências"}
Todo projeto usa código de terceiros (bibliotecas). O Maven baixa essas bibliotecas, compila
o projeto e gera o pacote final. A "lista de compras" fica num arquivo chamado `pom.xml`
(veremos na Aula 3). Comando típico: `./mvnw clean verify` (limpa, compila e roda os testes).
::
::field{name="IDE (IntelliJ / VS Code)" type="editor de código"}
Onde você escreve o código. Uma IDE dá autocompletar, mostra erros na hora, ajuda a navegar
entre arquivos e a rodar/depurar. Não é obrigatório, mas acelera **muito**.
::
::field{name="Postman / Insomnia" type="cliente HTTP"}
Um programa para **testar a API à mão**: você monta um `GET`/`POST`, manda para a rota e vê a
resposta — sem precisar do app. Essencial para desenvolver e depurar endpoints.
::
::

::callout{type="info" title="Por que \"mvnw\" e não \"mvn\"?"}
O `mvnw` é o **Maven Wrapper**: um script que vem junto no projeto e baixa a versão *exata*
do Maven que o projeto precisa. Assim, quem clona o repositório roda `./mvnw` e usa a mesma
versão de todo mundo — sem precisar instalar o Maven na mão. (Guarde esse `mvnw`: ele vai ter
um papel surpresa lá na aula de CI/CD 😉.)
::

## O problema que o Git resolve

Imagina programar sem uma forma de guardar o histórico. Você muda um arquivo, quebra tudo, e
agora não lembra como estava antes. Ou duas pessoas mexem no mesmo arquivo e uma apaga o
trabalho da outra. Um caos.

O **Git** é um **sistema de controle de versão**: ele fotografa o seu projeto a cada mudança
importante, guarda todo o histórico e permite que várias pessoas trabalhem juntas sem se
atropelar. É a ferramenta mais importante do dia a dia — mais até que a linguagem.

Os conceitos que você **precisa** dominar:

::fields
::field{name="Repositório (repo)"}
A pasta do projeto + todo o seu histórico versionado pelo Git. O do Descontai fica hospedado
no GitHub: `Koomit-Software/Descontai-API`.
::
::field{name="Commit"}
Uma "foto" do projeto num momento, com uma mensagem explicando o que mudou. Ex.:
`feat(db): migration de baseline (V0) para subir schema do zero`. Cada commit tem um código
único (hash), tipo `af6a033`.
::
::field{name="Branch (ramo)"}
Uma linha do tempo paralela. Você cria um branch para trabalhar numa mudança **sem afetar** o
código principal. O branch principal costuma se chamar `main`.
::
::field{name="Merge"}
Juntar o trabalho de um branch de volta ao `main`, unificando as mudanças.
::
::field{name="Push / Pull"}
`push` = enviar seus commits para o servidor (GitHub). `pull` = trazer os commits que os
outros enviaram. É como sincronizar sua cópia com a do time.
::
::

::callout{type="tip" title="Anatomia de uma boa mensagem de commit"}
O Descontai usa o padrão **Conventional Commits**: `tipo(escopo): descrição`. Os tipos mais
comuns: `feat` (nova funcionalidade), `fix` (correção de bug), `refactor` (melhoria sem mudar
comportamento), `test` (testes), `docs` (documentação), `ci` (pipeline). Isso deixa o histórico
legível e até permite gerar changelogs automáticos. Exemplo real:
`perf(relatorios): exporta CSV em streaming (memoria constante)`.
::

## GitHub e o fluxo de trabalho de um time

O **Git** é a ferramenta; o **GitHub** é o site que hospeda os repositórios na nuvem e adiciona
colaboração: revisão de código, automações (CI/CD, que veremos), issues, etc.

Num time profissional, **ninguém sai commitando direto no `main`**. O fluxo é assim:

::steps
### Cria um branch
Você parte do `main` atualizado e cria um branch para a sua tarefa. Ex.:
`git checkout -b feat/baseline-migration`.

### Faz os commits
Trabalha no branch, fazendo commits pequenos e com boas mensagens conforme avança.

### Abre um Pull Request (PR)
Envia o branch para o GitHub e abre um **Pull Request**: um pedido de "por favor, revisem e
juntem minha mudança no `main`". O PR mostra exatamente o que mudou (o **diff**), linha por linha.

### Passa pela revisão (code review)
Outra pessoa do time **lê o seu código** e comenta: "aqui pode dar bug", "esse nome tá confuso",
"faltou um teste". Você ajusta. Esse ritual pega erros cedo e espalha conhecimento pelo time.

### CI roda automaticamente
Ao abrir o PR, uma automação **compila e roda os testes** sozinha. Se algo quebra, o PR fica
marcado com ❌ e não deve ser juntado. (Isso é o **CI/CD** — Aula 12.)

### Merge no main
Com a revisão aprovada e o CI verde, o PR é **mesclado** no `main`. A mudança agora é oficial.
::

::callout{type="warning" title="Por que todo esse ritual?"}
Parece burocracia, mas é o que mantém um produto em produção **estável**. O `main` é sagrado:
o que está nele pode ir para produção a qualquer momento. Branch + PR + review + CI formam uma
**rede de proteção** em camadas: humano (review) e máquina (CI). Empresas sérias **exigem** isso.
::

::callout{type="note" title="E quando é só você?"}
Mesmo trabalhando sozinho no seu projeto, adote o hábito: branches, commits com boas mensagens
e PRs. Primeiro porque organiza o pensamento; segundo porque é **exatamente** o que você vai
fazer quando entrar num time. Treine como joga.
::

## Onde o Descontai roda

Só para fechar o mapa mental: o código vive no **GitHub**; quando é aprovado, uma automação o
**empacota** (com Docker, Aula 13) e o coloca num servidor na **DigitalOcean** (uma provedora
de nuvem, Aula 16). O banco de dados (PostgreSQL) e o cache (Redis) também são serviços
gerenciados na nuvem. Não se preocupe com os nomes agora — cada um tem sua aula.

## Resumindo

- O kit do dev backend: **Java (JDK 17)**, **Maven** (build + dependências, via `mvnw`), uma
  **IDE** e o **Postman** para testar a API.
- **Git** versiona o código e guarda o histórico; **GitHub** hospeda e adiciona colaboração.
- Conceitos-chave: **repositório, commit, branch, merge, push/pull**.
- O fluxo de time: **branch → commits → Pull Request → code review → CI → merge**.
- O `main` é sagrado — ele reflete o que pode ir para produção.
