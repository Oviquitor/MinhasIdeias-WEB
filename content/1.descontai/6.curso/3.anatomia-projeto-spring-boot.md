Agora que você entende o que é uma API e como um time trabalha, vamos **abrir o projeto** do
Descontai e entender como um backend Java/Spring Boot é montado por dentro. No fim desta aula,
a estrutura de pastas vai deixar de ser um monte de arquivos assustadores e virar um mapa que
você sabe ler.

## O que é o Spring Boot (sem enrolação)

**Spring** é um framework: um conjunto enorme de ferramentas prontas para construir aplicações
Java (banco de dados, segurança, web, etc.). O problema é que configurar tudo isso na mão é
trabalhoso. O **Spring Boot** resolve isso: ele já vem com **tudo pré-configurado** de um jeito
sensato, e você só ajusta o que precisa.

::callout{type="info" title="A analogia da casa"}
Spring puro é comprar o terreno e cada tijolo separadamente. Spring **Boot** é a casa pré-fabricada
que já vem com encanamento, luz e portas — você só decora do seu jeito. Por isso ele é o padrão
para começar rápido e ainda ter qualidade profissional.
::

Três superpoderes do Spring Boot que você vai usar o tempo todo:

- **Injeção de dependência**: você não cria os objetos na mão com `new`. Você declara "preciso de
  um `UsuarioRepository`" e o Spring te entrega um pronto. (Veremos isso em ação já já.)
- **Servidor embutido**: a aplicação já vem com um servidor web (Tomcat) dentro. Você roda um
  comando e a API está no ar na porta 8080. Não precisa instalar servidor separado.
- **Auto-configuração**: viu uma biblioteca de banco no projeto? O Spring Boot já configura a
  conexão sozinho, lendo suas propriedades.

## A "lista de compras": pom.xml

Todo projeto Maven tem um `pom.xml`. É onde você declara as **dependências** (bibliotecas de
terceiros) que o projeto usa. O Maven lê esse arquivo, baixa tudo e monta o projeto.

Olha as dependências mais importantes do Descontai e para que cada uma serve:

::fields
::field{name="spring-boot-starter-web"}
O coração: permite criar controllers, rotas HTTP e já traz o servidor Tomcat embutido.
::
::field{name="spring-boot-starter-data-jpa"}
Acesso a banco de dados com JPA/Hibernate — transforma tabelas em objetos Java (Aula 5).
::
::field{name="postgresql"}
O driver do **PostgreSQL**, o banco de dados que o Descontai usa.
::
::field{name="spring-boot-starter-security + jjwt"}
Segurança e autenticação com **JWT** (Aula 7).
::
::field{name="flyway-core"}
**Migrations**: versiona a estrutura do banco de dados (Aula 9).
::
::field{name="caffeine + spring-boot-starter-data-redis + bucket4j"}
**Cache** (deixa a API rápida) e **rate limiting** (protege contra abuso) — Caffeine na memória
ou Redis distribuído (Aula 15).
::
::field{name="actuator + micrometer-registry-prometheus"}
**Observabilidade**: expõe health check e métricas (Aula 17).
::
::field{name="spring-boot-starter-test + testcontainers"}
**Testes** unitários e de integração (Aula 11).
::
::

::callout{type="tip" title="Starters: kits prontos"}
Repare no padrão `spring-boot-starter-*`. São "kits": ao adicionar `starter-web`, você ganha de
uma vez tudo que precisa para uma API web (Tomcat, Jackson para JSON, etc.). Isso é a filosofia
do Boot — juntar peças que sempre andam juntas.
::

O `pom.xml` também fixa a **versão do Java (17)** e a versão do **Spring Boot (3.5.4)**. Fixar
versões é o que garante que todo mundo — seu colega, o CI, o servidor — construa o projeto
exatamente igual.

## A estrutura de pastas

Num projeto Maven/Spring Boot, a estrutura padrão é:

```
Descontai-API/
├── pom.xml                      # dependências e build
├── Dockerfile                   # como empacotar (Aula 13)
├── mvnw                         # o Maven Wrapper
├── .github/workflows/ci.yml     # o pipeline de CI/CD (Aula 12)
├── .do/app.yaml                 # config de deploy (Aula 16)
└── src/
    ├── main/
    │   ├── java/com/descontai/  # TODO o código da aplicação
    │   └── resources/
    │       ├── application.properties   # configurações
    │       └── db/migration/            # as migrations do Flyway
    └── test/
        └── java/com/descontai/  # os testes
```

Duas divisões sagradas:

- **`src/main`** é a aplicação de verdade; **`src/test`** são os testes (não vão para produção).
- Dentro de `main`, **`java`** é o código e **`resources`** são os arquivos de configuração e recursos.

Dentro de `src/main/java/com/descontai/` o código é organizado em **camadas** (`api`, `application`,
`domain`, `infra`, `security`…). Essa organização é o assunto da próxima aula — é o que separa um
projeto profissional de uma bagunça. Por ora, só saiba que **cada pasta tem um papel bem definido**.

## As configurações: application.properties

Se o `pom.xml` diz *quais* ferramentas usar, o `application.properties` diz *como* configurá-las.
É um arquivo de pares `chave=valor`. Alguns exemplos reais do Descontai:

```properties
# Banco de dados
spring.datasource.url=${DB_URL}
spring.datasource.username=${DB_USER}
spring.datasource.password=${DB_PASS}

# Tamanho do pool de conexões (Aula 14)
spring.datasource.hikari.maximum-pool-size=${DB_POOL_MAX:20}

# JWT
jwt.secret=${JWT_SECRET}
jwt.expiration=${JWT_EXPIRATION:900000}

# Qual "perfil" está ativo (dev, prod...)
spring.profiles.active=${SPRING_PROFILES_ACTIVE:prod}
```

::callout{type="warning" title="O padrão ${VARIAVEL:default} — e por que segredos NÃO ficam no código"}
Repare em `${DB_URL}` e `${DB_POOL_MAX:20}`. Isso significa: "pegue o valor da **variável de
ambiente** `DB_URL`; se não existir, use o default (ex.: `20`)". Por que isso importa?

Porque **senhas e segredos nunca podem estar escritos no código** (senão vazam no Git para todo
mundo ver). Eles ficam em **variáveis de ambiente**, definidas no servidor. O mesmo código roda
em qualquer lugar — só muda o ambiente. Isso é um princípio de mercado chamado
[12-Factor App](https://12factor.net/config). Guarde: **configuração vem do ambiente, segredo nunca no código.**
::

::callout{type="info" title="Perfis (profiles)"}
`spring.profiles.active` define o "modo" da aplicação. Em produção é `prod`; em desenvolvimento
pode ser `dev`. Isso permite ter comportamentos diferentes por ambiente (ex.: em `prod`, a API
**exige** o token do webhook de pagamento e recusa subir sem ele — uma trava de segurança).
::

## Como a aplicação "sobe"

Todo projeto Spring Boot tem uma classe com o método `main` — o ponto de partida:

```java
@SpringBootApplication
public class DescontaiApplication {
    public static void main(String[] args) {
        SpringApplication.run(DescontaiApplication.class, args);
    }
}
```

Quando você roda `./mvnw spring-boot:run` (ou o `.jar` em produção), essa linha dispara uma
sequência:

::steps
### O Spring "varre" o projeto
Ele procura todas as classes marcadas com anotações como `@RestController`, `@Service`,
`@Component`, `@Configuration` e cria um objeto (um **bean**) de cada uma.

### Monta o grafo de dependências
Se o `AuthController` precisa de um `LoginUseCase`, que precisa de um `UsuarioRepository`, o
Spring cria tudo na ordem certa e **injeta** um no outro. Você nunca deu `new` em nada — é a
**injeção de dependência** em ação.

### Conecta no banco e roda as migrations
Lê as propriedades, abre o pool de conexões com o PostgreSQL e o **Flyway** garante que a
estrutura do banco está atualizada (Aula 9).

### Sobe o servidor web
O Tomcat embutido começa a escutar na porta 8080. A partir daqui, a API está **no ar** e
pronta para receber requisições.
::

::callout{type="tip" title="Por que \"injeção de dependência\" é tão importante?"}
Porque desacopla o código. O `AuthController` não sabe *como* construir um `LoginUseCase` — ele
só declara que precisa de um. Isso deixa o código **testável** (nos testes, você injeta uma versão
falsa/mock — Aula 11) e **flexível** (dá para trocar a implementação sem mexer em quem usa).
É um dos pilares de todo projeto Java profissional.
::

## Resumindo

- **Spring Boot** é o framework que já vem "montado" — servidor embutido, auto-configuração e
  **injeção de dependência**.
- O **`pom.xml`** lista as dependências (starters) e fixa as versões (Java 17, Spring Boot 3.5.4).
- A estrutura separa **`src/main`** (aplicação) de **`src/test`** (testes), e o código Java das
  configurações (`resources`).
- O **`application.properties`** configura tudo, lendo **variáveis de ambiente** — segredos nunca
  no código.
- Ao subir, o Spring cria os **beans**, injeta as dependências, conecta no banco, roda as migrations
  e sobe o servidor.
