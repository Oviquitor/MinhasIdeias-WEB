Na aula passada você abriu o projeto e viu que dentro de `src/main/java/com/descontai/` o código
não fica todo jogado num lugar só — ele está dividido em pastas com nomes como `api`, `application`,
`domain` e `infra`. Essa divisão **não é frescura**. É o que separa um projeto que dá pra manter por
anos de uma bagunça impossível de mexer. Esta aula é sobre entender **por que** essas paredes existem.

## O problema: código sem paredes

Imagina que a gente jogasse tudo num arquivo só: o código que recebe o pedido HTTP, o que decide as
regras ("nome de categoria não pode repetir"), o que salva no banco de dados — tudo grudado.

Funciona? Funciona. Por uns dias. O problema aparece quando você precisa **mudar** algo:

- Trocar o banco de PostgreSQL para outro? Você mexe em tudo, porque a regra de negócio está grudada no banco.
- Testar a regra "nome não pode repetir" sem subir um banco de verdade? Impossível, estão colados.
- Um dev novo entra no time e pergunta "onde fica a regra de X?". Ninguém sabe, está espalhado.

A solução que o mercado adotou é **separar o código em camadas**, cada uma com um papel único, e com
paredes bem definidas entre elas. Esse estilo tem um nome: **Clean Architecture** (Arquitetura Limpa).

::callout{type="info" title="O que é uma \"camada\", afinal?"}
Uma **camada** é só um grupo de código com **uma responsabilidade bem definida**. Uma camada cuida de
"conversar HTTP", outra cuida de "aplicar as regras do negócio", outra cuida de "falar com o banco".
Cada uma faz **uma coisa** e confia nas outras para o resto — como setores de uma empresa.
::

## A analogia do restaurante

Pensa num restaurante. Ninguém funciona bem se todo mundo faz tudo. Existem setores separados:

::fields
::field{name="Salão (garçom)" type="camada api"}
Recebe o cliente, anota o pedido, entrega o prato. **Não cozinha.** É a porta de entrada — no nosso
projeto, é quem recebe o pedido HTTP e devolve a resposta.
::
::field{name="Cozinha (chef)" type="camada application"}
Onde o prato é **preparado** seguindo as regras da casa ("a massa fica 8 minutos", "não servimos isso
sem reserva"). É o cérebro que **orquestra** e decide. Não atende cliente nem vai ao mercado.
::
::field{name="A receita e os ingredientes" type="camada domain"}
O que é uma "lasanha", quais ingredientes ela tem, o que a define. É o **conhecimento puro** do negócio.
Não sabe nada de garçom nem de fornecedor — só sabe o que é o prato.
::
::field{name="Estoque e fornecedores" type="camada infra"}
De onde vêm os ingredientes: o mercado, a geladeira, o caminhão de entrega. São os **detalhes técnicos**.
No projeto, é o banco de dados, APIs externas, o framework.
::
::

Repare numa coisa: o **garçom depende da cozinha** (ele precisa do prato pronto), a **cozinha depende
da receita**. Mas a **receita não depende de ninguém** — uma lasanha é uma lasanha independente de qual
garçom serve ou de qual mercado veio o queijo. Guarde essa ideia, porque é o coração de tudo.

## As camadas do Descontai

No Descontai, dentro de `src/main/java/com/descontai/`, cada pasta é uma camada:

| Camada | Pasta | Papel | No restaurante |
|--------|-------|-------|----------------|
| **API** | `api/` (controller, dto) | A porta HTTP: recebe requisições, devolve respostas | Salão / garçom |
| **Application** | `application/` (usecase, service) | Orquestra e aplica as regras de negócio | Cozinha / chef |
| **Domain** | `domain/` (model) | As entidades e regras puras do negócio | Receita / ingredientes |
| **Infra** | `infra/` (repository, gateway, config) | Detalhes técnicos: banco, APIs externas, framework | Estoque / fornecedores |
| **Security** | `security/` | Quem pode fazer o quê (autenticação/autorização) | Segurança / portaria |
| **Common** | `common/` | Peças compartilhadas (exceções, utilitários) | Coisas de uso geral da casa |

::callout{type="tip" title="DTO, UseCase, Service, Repository, Entidade — o que é cada um"}
Palavras novas que vão aparecer o tempo todo:

- **Controller**: a classe que atende uma rota HTTP (recebe o pedido).
- **DTO** (*Data Transfer Object*): um objeto "de transporte" — o formato dos dados que entram e saem pela API.
- **UseCase** (caso de uso): representa **uma ação** do sistema ("criar categoria", "listar cupons").
- **Service**: onde ficam as **regras de negócio** reutilizáveis ("nome não pode repetir").
- **Repository**: quem **fala com o banco de dados** (salvar, buscar, apagar).
- **Entidade** (*model*): o objeto que representa uma coisa do mundo real (uma `Categoria`, um `Usuario`).

Não precisa decorar agora — vamos ver cada um em código já já.
::

## A regra de ouro: a dependência aponta pra dentro

Essa é a regra mais importante da aula. Desenhe as camadas como círculos, uma dentro da outra:

```text
        ┌─────────────────────────────────────┐
        │  api  (controllers, DTOs) ── HTTP    │   ← mais "de fora"
        │   ┌─────────────────────────────┐    │
        │   │ application (usecase, service)│   │
        │   │   ┌─────────────────────┐    │    │
        │   │   │   domain  (model)   │    │    │   ← mais "de dentro"
        │   │   │  regras puras       │    │    │
        │   │   └─────────────────────┘    │    │
        │   └─────────────────────────────┘    │
        │  infra (repository, gateway, config) │
        └─────────────────────────────────────┘

  A dependência SEMPRE aponta de fora → para dentro.
  api → application → domain.   domain NÃO conhece ninguém de fora.
```

**As camadas de fora dependem das de dentro, nunca o contrário.** Ou seja:

- A `api` pode usar a `application`. ✅
- A `application` pode usar o `domain`. ✅
- O `domain` **não conhece** a `api`, não conhece o banco, não conhece HTTP. ❌ (ao contrário)

::callout{type="warning" title="Por que o domínio não pode conhecer o \"de fora\"?"}
Porque o **domínio é o coração do negócio** — e o negócio não muda por causa de detalhes técnicos.
Uma `Categoria` continua sendo uma categoria se você trocar o PostgreSQL por outro banco, se trocar
HTTP por outro protocolo, ou se jogar o Spring fora amanhã. Ao proibir o domínio de depender desses
detalhes, você garante que **mudar um detalhe técnico não quebra a regra de negócio**. É exatamente
isso que dá liberdade pra mudar sem medo.
::

## Uma funcionalidade passando pelas camadas

Chega de teoria. Vamos seguir **um pedido de verdade** do Descontai: *criar uma categoria*
(POST /api/v1/categorias). Ele atravessa as camadas em ordem: **Controller → UseCase → Service →
Repository → Entidade**. Olha o código real de cada passo.

::steps

### 1. API — o Controller recebe o pedido HTTP

O `CategoriaController` é o **garçom**. Ele só anota o pedido e repassa. Repare que ele **não tem
nenhuma regra de negócio** — só recebe, delega e devolve a resposta.

```java
// api/controller/CategoriaController.java
@RestController
@RequestMapping("/api/v1/categorias")
public class CategoriaController {

    private final CreateCategoriaUseCase createCategoria;
    // ...outros use cases...

    @PostMapping
    @PreAuthorize("hasRole('ADMIN')")
    public ResponseEntity<CategoriaResponse> create(@RequestBody @Valid CategoriaRequest req) {
        return ResponseEntity.ok(createCategoria.execute(req));
    }
}
```

O que ele faz: recebe o JSON do corpo do pedido (`@RequestBody`) já validado (`@Valid`), transforma
num objeto `CategoriaRequest` (um **DTO**), e chama `createCategoria.execute(req)`. Só isso. Toda a
"cozinha" acontece no `UseCase`. O `@PreAuthorize("hasRole('ADMIN')")` é a camada de **security**
garantindo que só admin cria categoria.

### 2. Application — o UseCase orquestra

O `CreateCategoriaUseCase` é o **passo a passo da ação** "criar categoria". Ele traduz o DTO (mundo
da API) para a entidade de domínio, e delega a regra pro Service.

```java
// application/usecase/categoria/CreateCategoriaUseCase.java
@Component
public class CreateCategoriaUseCase implements UseCase<CategoriaRequest, CategoriaResponse> {

    private final CategoriaService categorias;

    public CreateCategoriaUseCase(CategoriaService categorias) {
        this.categorias = categorias;
    }

    @Override
    public CategoriaResponse execute(CategoriaRequest input) {
        Categoria categoria = new Categoria();          // cria a entidade de DOMÍNIO
        categoria.setNome(input.getNome());
        categoria.setDescricao(input.getDescricao());
        return CategoriaMapper.toResponse(categorias.create(categoria));
    }
}
```

Repare no `implements UseCase<CategoriaRequest, CategoriaResponse>`: todo caso de uso do projeto segue
o mesmo **contrato** — um método `execute(input)`. É um padrão do projeto:

```java
// application/usecase/UseCase.java
public interface UseCase<I, O> {
    O execute(I input);
}
```

::callout{type="note" title="Por que traduzir DTO ↔ entidade?"}
O `CategoriaRequest` é o formato que **entra pela API** (o mundo de fora). A `Categoria` é a **entidade
do domínio** (o coração). O UseCase faz a ponte: converte um no outro. Isso protege o domínio — se
amanhã o formato do JSON mudar, só o DTO e essa tradução mudam; a entidade `Categoria` continua intacta.
::

### 3. Application — o Service aplica a regra de negócio

O `CategoriaService` é onde mora a **regra de verdade**: "não pode existir categoria com nome repetido".
Essa regra vale sempre, não importa se veio da API, de um import em lote, do que for.

```java
// application/service/CategoriaService.java
@Service
public class CategoriaService {

    private final CategoriaRepository categorias;

    public CategoriaService(CategoriaRepository categorias) {
        this.categorias = categorias;
    }

    @Transactional
    @CacheEvict(value = CacheConfig.CATEGORIAS, allEntries = true)
    public Categoria create(Categoria categoria) {
        validarNomeUnico(categoria.getNome(), null);   // A REGRA de negócio
        return categorias.save(categoria);             // delega o "salvar" ao repository
    }

    private void validarNomeUnico(String nome, Long ignoreId) {
        Optional<Categoria> existente = categorias.findByNomeIgnoreCase(nome);
        if (existente.isPresent() && (ignoreId == null || !existente.get().getId().equals(ignoreId))) {
            throw new BusinessException("Categoria ja cadastrada: " + nome);
        }
    }
}
```

O Service **decide** e **valida**, mas não sabe *como* os dados são salvos — ele só pede pro
`categorias.save(...)`. Quem sabe salvar é o Repository (camada de infra).

### 4. Infra — o Repository fala com o banco

O `CategoriaRepository` é a ponte com o **banco de dados**. E olha a mágica: é só uma **interface**,
quase sem código. O Spring Data JPA implementa tudo sozinho.

```java
// infra/repository/CategoriaRepository.java
public interface CategoriaRepository extends JpaRepository<Categoria, Long> {

    Optional<Categoria> findByNomeIgnoreCase(String nome);
}
```

Ao estender `JpaRepository`, ele **já ganha** `save`, `findById`, `findAll`, `deleteById`, etc. de
graça. E ainda podemos declarar buscas customizadas só pelo nome do método: `findByNomeIgnoreCase`
vira automaticamente um "buscar por nome ignorando maiúsculas/minúsculas". Esse é o **detalhe técnico**
que fica escondido na camada de infra — o Service nem precisa saber que por baixo é PostgreSQL.

### 5. Domain — a Entidade, o coração do negócio

No fundo de tudo está a `Categoria`: uma entidade **pura de negócio**. Ela representa o que uma
categoria *é* — nome, descrição, e a regra de que o nome é único.

```java
// domain/model/Categoria.java
@Entity
@Table(name = "categorias")
public class Categoria extends BaseEntity {

    @Column(nullable = false, unique = true)
    private String nome;

    private String descricao;

    // getters e setters...
}
```

Ela não sabe o que é HTTP, não sabe o que é um Controller, não sabe orquestrar nada. Ela só **é**
uma categoria. É o círculo mais de dentro — e o que menos muda ao longo da vida do projeto.

::

## O caminho de volta

Depois que o Repository salva e devolve a `Categoria` (agora com o `id` gerado pelo banco), a resposta
volta pelo mesmo caminho, ao contrário: o Service devolve pro UseCase, que usa o `CategoriaMapper` pra
transformar a entidade num `CategoriaResponse` (o DTO de saída), e o Controller devolve esse DTO como
JSON. O cliente recebe a resposta e nunca soube que existiu banco, service ou use case. Só viu a "porta".

```text
Pedido:    Controller → UseCase → Service → Repository → (Banco)
                (api)   (application) (application) (infra)
Resposta:  Controller ← UseCase ← Service ← Repository ← (Banco)
```

## Por que tudo isso deixa o código melhor

Voltando às dores lá do começo — agora com a resposta:

::fields
::field{name="Testável"}
Como o Service depende de uma **interface** (`CategoriaRepository`), nos testes a gente injeta uma
versão **falsa** (mock) dela. Dá pra testar a regra "nome não repete" **sem subir banco nenhum** —
rápido e isolado. Isso só é possível porque as camadas não estão grudadas.
::
::field{name="Fácil de mudar"}
Trocar de banco? Só a camada `infra` muda; Service, UseCase e domínio nem ficam sabendo. Mudar o
formato do JSON? Só o DTO na `api`. Cada mudança fica **contida numa camada**.
::
::field{name="Fácil de achar as coisas"}
"Onde está a regra de categoria?" → `application/service`. "Onde a rota é atendida?" → `api/controller`.
"Como salva no banco?" → `infra/repository`. Todo dev novo já sabe onde procurar.
::
::field{name="Injeção de dependência faz a cola"}
Lembra da injeção de dependência da Aula 3? É ela que liga as camadas: o Controller **recebe** o UseCase
pronto, o UseCase **recebe** o Service, o Service **recebe** o Repository. Ninguém dá `new` em nada —
o Spring monta o grafo. Por isso é fácil trocar peças por versões falsas nos testes.
::
::

::callout{type="danger" title="O erro clássico do júnior"}
O erro mais comum é **colocar regra de negócio no Controller** (ou fazer o Controller falar direto com o
banco). Isso "funciona", mas fura a arquitetura: a regra fica presa ao HTTP, impossível de testar isolada
e impossível de reusar. Se você se pegar escrevendo um `if` de negócio dentro de um `@PostMapping`, pare:
essa lógica pertence a um **Service**. O Controller só recebe e delega.
::

## Resumindo

- Projetos profissionais separam o código em **camadas**, cada uma com **um papel único**.
- No Descontai: **api** (porta HTTP), **application** (orquestração e regras), **domain** (entidades e
  regras puras), **infra** (banco, APIs externas, framework), além de **security** e **common**.
- **Regra de ouro**: a dependência aponta **de fora pra dentro** — `api → application → domain`. O
  domínio **não conhece** HTTP nem banco.
- Uma funcionalidade atravessa as camadas: **Controller → UseCase → Service → Repository → Entidade**.
- Isso deixa o código **testável** (mocka a infra), **fácil de mudar** (mudança fica contida numa camada)
  e **fácil de navegar** (cada coisa no seu lugar).

::callout{type="tip" title="Fixando"}
Pega qualquer funcionalidade do Descontai (ex.: criar uma cidade) e tente dizer em voz alta por onde ela
passa: *"o Controller recebe o POST, chama o UseCase, que chama o Service com a regra, que usa o Repository
pra salvar a Entidade"*. Se você consegue mapear isso, você entendeu a espinha dorsal do projeto.
::
