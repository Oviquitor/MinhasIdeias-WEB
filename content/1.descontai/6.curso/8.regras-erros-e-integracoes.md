Até aqui você viu como a requisição entra (controller) e como os dados são salvos (banco). Agora
vamos ao **miolo** de um backend de verdade: onde ficam as **regras do negócio**, como tratar
**erros** sem virar bagunça, e como a nossa API **conversa com o mundo lá fora** — cobrar um
pagamento, receber um aviso de confirmação e disparar milhares de notificações sem travar.

## O que é "regra de negócio" e onde ela mora

**Regra de negócio** é toda decisão que o **negócio** (o Descontai) toma, não o computador.
Exemplos:

- "Um cupom só pode ser resgatado se ainda estiver dentro da validade."
- "Ao confirmar um pagamento, a assinatura do usuário é estendida conforme o plano."
- "Uma empresa com cupons já resgatados **não pode** ser excluída."

Repare: nenhuma dessas frases fala de HTTP, JSON ou banco. São **regras do mundo real**. E é
justamente por isso que elas **não moram no controller**.

::callout{type="info" title="A regra de ouro das camadas"}
O **controller** é o garçom: ele recebe o pedido, entrega para a cozinha e traz o prato. Ele
**não cozinha**. Quem cozinha (aplica as regras) é o **service** ou o **use case** (a camada
`application`, da Aula 4). O controller só traduz HTTP ↔ chamada de método. Se você vir um `if`
de regra de negócio dentro de um controller, é um cheiro de que a regra está no lugar errado.
::

No `PagamentoController`, olha como o método é **magro** — ele só delega:

```java
@PostMapping
public ResponseEntity<PagamentoResponse> pagar(@AuthenticationPrincipal CustomUserDetails me,
        @RequestBody @Valid PagamentoRequest req) {
    PagamentoResponse response = registrarPagamentoUseCase.execute(
            new RegistrarPagamentoUseCase.Input(me.getDomainUser(), req.getValor(), /* ... */));
    return ResponseEntity.ok(response);
}
```

Toda a lógica ("valida o valor, incrementa o ciclo de assinatura, estende a validade") está
**dentro do use case**. O controller só chama `.execute(...)`. Isso deixa a regra **testável**
(dá para testar o use case sem subir um servidor web) e **reaproveitável** (o mesmo use case
serve o app e o portal admin).

## Tratamento de erro profissional

Quando uma regra é violada, o que acontece? A resposta amadora é encher tudo de `try/catch`. A
resposta profissional tem duas peças: **exceções semânticas** + **um handler central**.

### Peça 1: exceções que "significam" algo

O Descontai tem exceções que descrevem o **tipo** de problema, não o detalhe técnico:

```java
public class BusinessException extends RuntimeException {
    public BusinessException(String message) {
        super(message);
    }
}
```

E uma irmã, `NotFoundException`, para "o recurso não existe". Quando um use case detecta uma
violação, ele simplesmente **lança**:

```java
if (cupom.isExpirado()) throw new BusinessException("Cupom expirado.");
```

Note que o use case **não sabe** que existe HTTP. Ele só grita "isso viola uma regra!". Quem
traduz esse grito em status HTTP é a próxima peça.

### Peça 2: o handler global (`@RestControllerAdvice`)

Aqui mora a mágica. Em vez de cada controller ter um `try/catch`, existe **uma classe só** que
intercepta qualquer exceção da aplicação e a transforma numa resposta JSON padronizada:

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(NotFoundException.class)
    public ResponseEntity<ApiError> handleNotFound(NotFoundException ex, HttpServletRequest req) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(new ApiError(404, "Not Found", ex.getMessage(), req.getRequestURI()));
    }

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiError> handleBusiness(BusinessException ex, HttpServletRequest req) {
        return ResponseEntity.status(HttpStatus.BAD_REQUEST)
                .body(new ApiError(400, "Bad Request", ex.getMessage(), req.getRequestURI()));
    }
    // ... outros handlers ...
}
```

O `@RestControllerAdvice` é um "vigia" que fica **em volta de todos os controllers**. Toda vez
que uma exceção escapa de um controller, ele a pega antes do Spring, olha o **tipo** e devolve o
status certo com um corpo `ApiError` bonitinho.

Cada tipo de erro vira um status HTTP específico:

| Situação | Exceção / gatilho | Status HTTP |
|----------|-------------------|-------------|
| Recurso não existe | `NotFoundException` | **404** Not Found |
| Regra de negócio violada | `BusinessException` | **400** Bad Request |
| Campo inválido (`@Valid`) | `MethodArgumentNotValidException` | **400** Validation Error |
| JSON malformado / body ausente | `HttpMessageNotReadableException` | **400** Bad Request |
| Verbo errado na rota | `HttpRequestMethodNotSupportedException` | **405** Method Not Allowed |
| Qualquer bug inesperado | `Exception` (catch-all) | **500** Internal Server Error |

::callout{type="tip" title="Por que centralizar em vez de try/catch em todo canto?"}
Imagine 40 controllers, cada um com o mesmo `try/catch` copiado. Se você quiser mudar o formato
do erro, precisa editar 40 lugares (e vai esquecer um). Com o handler central, o formato do erro
é definido **num lugar só**. É o princípio **DRY** (Don't Repeat Yourself). Bônus: seus
controllers ficam limpos, só com a "linha feliz" do fluxo.
::

O handler catch-all (o que pega qualquer `Exception`) tem um cuidado extra de **segurança** que
vale ouro:

```java
@ExceptionHandler(Exception.class)
public ResponseEntity<ApiError> handleGeneric(Exception ex, HttpServletRequest req) {
    String traceId = UUID.randomUUID().toString();
    log.error("[traceId={}] Erro inesperado em {}: {}", traceId, req.getRequestURI(), ex.getMessage(), ex);
    String msg = "Erro interno. Codigo de rastreamento: " + traceId;
    return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(new ApiError(500, "Internal Server Error", msg, req.getRequestURI()));
}
```

::callout{type="warning" title="Nunca vaze stack trace para o cliente"}
Um erro inesperado (500) pode conter nomes de tabelas, queries SQL, caminhos internos — um prato
cheio para um atacante. Por isso o cliente recebe **só um código de rastreamento** (um UUID
aleatório). Os detalhes técnicos vão **para o log**, junto do mesmo UUID. Se um usuário reclama
"deu erro, código abc-123", o suporte busca `abc-123` no log e acha a causa exata. Depuração
completa, **zero vazamento**.
::

### Enums de status: chega de "strings mágicas"

Um pagamento tem status: aprovado, recusado, estornado... Um jeito iniciante é espalhar
`"APPROVED"` como texto solto pelo código. O problema? Se você digitar `"APROVED"` (faltou um P),
**o código compila normalmente** e falha calado em produção.

A solução é um **enum** — uma lista fechada de valores válidos:

```java
public enum StatusPagamento {
    APPROVED,   // confirmado/recebido — libera a assinatura
    REJECTED,   // recusado
    REFUNDED,   // estornado/devolvido
    PENDING     // aguardando confirmação
}
```

Agora, em vez de `"APPROVED"`, o código usa `StatusPagamento.APPROVED.name()`. Se você digitar
errado o nome do enum, **o compilador reclama na hora** — o erro de digitação vira erro de
compilação, e não um bug silencioso. É a "fonte única de verdade" para os estados possíveis.

## Integrações com serviços externos

O Descontai não processa cartão de crédito sozinho (isso exige certificação pesada). Ele delega
para o **Asaas**, um gateway de pagamento. Nossa API precisa **falar com a API do Asaas** pela
internet — uma requisição HTTP de saída.

### O padrão Gateway

Toda a conversa com o Asaas fica trancada numa classe só: o `AsaasGateway`. O resto da aplicação
chama métodos de negócio (`criarPagamentoPix(...)`) e **não vê** URLs, headers ou JSON do Asaas.

```java
@Component
public class AsaasGateway {
    private final RestTemplate restTemplate;

    public AsaasGateway(RestTemplateBuilder builder, @Value("${asaas.api-key}") String apiKey, /* ... */) {
        if (apiKey == null || apiKey.isBlank()) {
            throw new IllegalStateException("asaas.api-key nao configurada.");
        }
        this.restTemplate = builder
                .setConnectTimeout(Duration.ofSeconds(10))  // tempo p/ CONECTAR
                .setReadTimeout(Duration.ofSeconds(20))     // tempo p/ RESPONDER
                .build();
        // ...
    }
}
```

O `RestTemplate` é a ferramenta do Spring para fazer chamadas HTTP de saída (a API vira
**cliente** de outra API). Duas coisas críticas aqui:

::fields
::field{name="Timeouts" type="Duration"}
`connectTimeout` (10s) é quanto esperamos para **estabelecer a conexão**; `readTimeout` (20s) é
quanto esperamos pela **resposta**. Sem timeout, se o Asaas travar, **nossa** requisição fica
pendurada para sempre, segurando um thread — e uma pilha de requisições travadas derruba a API.
Timeout é autopreservação.
::
::field{name="Fail-fast na subida" type="IllegalStateException"}
Se a chave da API não estiver configurada, a aplicação **nem sobe**. Melhor falhar no start
(óbvio, na hora do deploy) do que descobrir no meio de um checkout de um cliente real.
::
::

E toda chamada trata a falha convertendo o erro técnico numa exceção de negócio amigável:

```java
private AsaasPayment criarPagamento(Map<String, Object> payload) {
    try {
        ResponseEntity<AsaasPayment> response = restTemplate.exchange(
                baseUrl + "/payments", HttpMethod.POST,
                new HttpEntity<>(payload, defaultHeaders()), AsaasPayment.class);
        return response.getBody();
    } catch (RestClientException e) {
        throw new BusinessException("Falha ao criar pagamento no Asaas: " + e.getMessage());
    }
}
```

Se o Asaas estiver fora do ar, o `RestClientException` vira uma `BusinessException` — que o
handler global já sabe transformar em 400. Tudo se conecta.

## Webhooks: quando a API externa AVISA a gente

Um pagamento Pix não confirma na hora — o usuário paga minutos depois. Como sabemos que pagou?
Duas opções:

- **Perguntar toda hora** ("já pagou? já pagou? já pagou?") — ineficiente, chamado *polling*.
- **Ser avisado**: a gente deixa um "endereço de recado" e o Asaas nos **liga** quando o
  pagamento confirma. Isso é um **webhook**.

::callout{type="info" title="Webhook, em uma frase"}
Um **webhook** é o inverso de uma chamada normal. Em vez de a nossa API pedir dados ao Asaas, é o
**Asaas** que faz uma requisição HTTP **para a nossa API** para nos avisar de um evento. É como
deixar seu telefone numa loja: em vez de você ligar toda hora perguntando se o produto chegou,
**eles** te ligam quando chega.
::

O endpoint que recebe esse "telefonema" é o `POST /webhooks/asaas`:

```java
@PostMapping("/webhooks/asaas")
public ResponseEntity<String> webhookAsaas(
        @RequestHeader(value = "asaas-access-token", required = false) String webhookToken,
        @RequestBody(required = false) AsaasWebhookRequest body) {
    if (!webhookValidator.isValid(webhookToken)) {
        log.warn("Webhook Asaas rejeitado: token invalido ou ausente");
        return ResponseEntity.status(HttpStatus.UNAUTHORIZED).body("Unauthorized");
    }
    processarWebhookAsaasUseCase.execute(new ProcessarWebhookAsaasUseCase.Input(body));
    return ResponseEntity.ok("ok");
}
```

### Por que validar o token? (senão alguém forja pagamento)

Esse endpoint é **público** — tem que ser, porque o Asaas precisa alcançá-lo sem login. Mas isso
cria um perigo enorme: **qualquer pessoa na internet** poderia mandar um POST fingindo ser o
Asaas dizendo "pagamento confirmado!" e ganhar assinatura de graça.

A defesa é um **token secreto** combinado entre nós e o Asaas. Todo webhook legítimo traz esse
token no header. O `AsaasWebhookValidator` confere:

```java
public boolean isValid(String receivedToken) {
    if (webhookToken == null || webhookToken.isBlank()) {
        return true;  // só possível fora de producao (em prod o boot falha sem token)
    }
    if (receivedToken == null || receivedToken.isBlank()) {
        return false;
    }
    return MessageDigest.isEqual(
            receivedToken.getBytes(StandardCharsets.UTF_8),
            webhookToken.getBytes(StandardCharsets.UTF_8));
}
```

::callout{type="warning" title="Dois detalhes de segurança que separam júnior de sênior"}
**1. `MessageDigest.isEqual` em vez de `==`.** Uma comparação normal de strings retorna `false`
já no primeiro caractere diferente. Medindo o **tempo** de resposta, um atacante poderia adivinhar
o token letra por letra (*timing attack*). O `MessageDigest.isEqual` compara em **tempo
constante**, sempre demorando o mesmo — não vaza dica nenhuma.

**2. Fail-fast em produção.** No perfil `prod`, se o token não estiver configurado, a aplicação
**se recusa a subir**. Rodar o webhook sem token em produção = deixar a porta dos fundos aberta.
Melhor não subir do que subir vulnerável.
::

### Idempotência: não processar duas vezes

O Asaas pode reenviar o mesmo webhook (a rede falha, ele tenta de novo). Se processarmos o mesmo
pagamento **duas vezes**, o usuário ganharia o **dobro** de assinatura. Errado.

A defesa se chama **idempotência**: processar a mesma mensagem várias vezes tem o **mesmo efeito**
que processar uma vez só. Na prática, o use case checa "eu já registrei este `paymentId`?" antes
de creditar — se já existe, ignora e responde 200 mesmo assim (para o Asaas parar de reenviar).

::callout{type="info" title="Idempotência, a analogia do elevador"}
Apertar o botão do elevador 5 vezes não faz o elevador vir 5 vezes — ele já foi "chamado" na
primeira. O botão é **idempotente**. Seu webhook precisa ser assim: receber o mesmo aviso 5 vezes
credita o pagamento **uma vez só**.
::

## Trabalho assíncrono: responder rápido, processar depois

Última peça. Imagine o admin disparar uma notificação push para **todos os usuários**. Se a API
tentar enviar os milhares de pushes **durante a requisição HTTP**, o admin ficaria olhando a tela
girar por minutos — e o timeout estouraria antes de terminar.

### Síncrono vs. assíncrono

::callout{type="info" title="A analogia do café"}
**Síncrono** é pedir um café e **ficar parado no balcão** até ele ficar pronto, sem fazer mais
nada. **Assíncrono** é pedir o café, pegar uma senha, **ir sentar e trabalhar**, e ser chamado
quando estiver pronto. No assíncrono você **não fica bloqueado** esperando — libera para fazer
outra coisa. É exatamente o que queremos: liberar o thread da requisição em vez de segurá-lo.
::

A solução: o endpoint **aceita** o pedido, responde **202 Accepted** ("recebi, vou processar")
na hora, e o envio pesado roda **em segundo plano**.

```java
@PostMapping("/broadcast")
public ResponseEntity<BroadcastResultResponse> broadcast(@RequestBody BroadcastRequest req) {
    BroadcastResultResponse result =
            service.broadcast(req.titulo(), req.mensagem(), req.segmento(), req.payload());
    return ResponseEntity.accepted().body(result);  // 202: aceito, processando em background
}
```

### `@EnableAsync` + o pool de threads

Para rodar algo em background, o Spring precisa de duas coisas: ligar o recurso (`@EnableAsync`)
e ter um **pool de threads** (um time de "trabalhadores" prontos) dedicado a isso.

```java
@Configuration
@EnableAsync
@EnableScheduling
public class AsyncConfig {

    @Bean(name = "notificationExecutor")
    public Executor notificationExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(2);      // trabalhadores fixos
        executor.setMaxPoolSize(4);       // no máximo
        executor.setQueueCapacity(100);   // fila para absorver picos
        executor.setThreadNamePrefix("notif-");
        executor.initialize();
        return executor;
    }
}
```

Aí o serviço de broadcast marca o método com `@Async`, apontando para esse pool:

```java
@Async("notificationExecutor")
public void enviarParaTodos(String titulo, String mensagem, String payload) {
    // percorre a base em lotes de 500 e dispara os pushes...
    // roda FORA do thread da requisição; ao final, só loga o resultado
}
```

::callout{type="tip" title="Por que só 2–4 threads? Não seria mais rápido com 100?"}
Não. O **gargalo** é o serviço externo (o FCM do Google). Mais threads não fazem o FCM responder
mais rápido — só criam threads brigando por conexões de banco. O pool é pequeno **de propósito**,
e a **fila** (capacidade 100) segura as campanhas disparadas em sequência. Dimensionar pool é
entender **onde está o gargalo real**.
::

::callout{type="warning" title="A pegadinha do @Async: self-invocation"}
`@Async` funciona via **proxy** — o Spring embrulha o bean para interceptar a chamada. Se um
método de um bean chamar **outro método do mesmo bean**, a chamada **não passa pelo proxy** e roda
**síncrona** (bloqueando!), como se o `@Async` não existisse. Por isso o broadcast fica num bean
**separado** (`NotificacaoBroadcastService`), chamado **de fora**. É um erro clássico que pega
todo júnior — e alguns sêniores.
::

E o envio em si, no `FcmGateway`, também trata falha por token: se um push falha (token expirado
de um app desinstalado), ele **conta a falha e segue** para o próximo — um destinatário ruim não
derruba a campanha inteira.

## Resumindo

- **Regra de negócio** mora no **service/use case**, nunca no controller. O controller só delega.
- Erros esperados viram exceções semânticas (`BusinessException`, `NotFoundException`); um
  **`@RestControllerAdvice`** central as traduz em JSON padronizado com o status certo
  (400/404/409/500). Centralizar é **DRY** e evita vazar stack trace.
- **Enums de status** eliminam "strings mágicas": erro de digitação vira erro de **compilação**.
- Integrações externas ficam num **Gateway** (ex.: `AsaasGateway`) com **timeouts** e tratamento
  de falha — o resto da app não vê HTTP.
- Um **webhook** é a API externa nos **avisando** de um evento. Sempre **valide o token** (senão
  forjam pagamento) e garanta **idempotência** (não processar duas vezes).
- Trabalho pesado roda **assíncrono** (`@Async` + pool de threads): o endpoint responde **202** na
  hora e processa em background. Cuidado com **self-invocation**.
