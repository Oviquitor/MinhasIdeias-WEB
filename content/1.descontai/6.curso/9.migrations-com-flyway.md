Você já sabe que o código muda o tempo todo — e que o Git guarda cada mudança em ordem, para
qualquer pessoa reconstruir o projeto. Agora um segredo que iniciante quase sempre esquece:
**o banco de dados também muda**. Toda vez que uma funcionalidade nova pede uma coluna, uma
tabela ou um índice, a estrutura do banco precisa evoluir junto. Esta aula é sobre como fazer
isso de um jeito controlado, sem sustos — com **migrations** e o **Flyway**.

## O problema: o banco também muda (e não dá pra fazer na mão)

Imagina que o Descontai já está no ar e a equipe decide lançar o **Portal de Empresas**. Para
isso, o parceiro precisa de um login vinculado à sua empresa. No código, isso vira uma coluna
nova na tabela `usuario`: `empresa_id`. Simples, né? O problema é: **onde** essa coluna precisa
aparecer?

- No banco do **seu** computador (para você testar).
- No banco do **colega** de time.
- No banco do **CI** (o robô que roda os testes).
- No banco de **staging** (ambiente de homologação).
- No banco de **produção** (onde estão os usuários de verdade).

Se cada dev abrir o banco e digitar `alter table usuario add column empresa_id ...` na mão, o
caos é garantido: alguém esquece, alguém digita diferente, alguém aplica duas vezes, e ninguém
sabe mais qual banco está com qual estrutura. Em produção, um comando errado pode derrubar tudo.

::callout{type="info" title="A ideia em uma frase"}
**Migration** é um arquivo com um pedaço de SQL que descreve **uma mudança** na estrutura do
banco. Em vez de alterar o banco na mão, você escreve a mudança num arquivo, comita no Git, e
uma ferramenta aplica esse arquivo **automaticamente**, na ordem certa, em **todo** ambiente.
Essa ferramenta, no Descontai, é o **Flyway**.
::

## A solução: migrations são o "Git do banco"

Guarda essa analogia, porque ela explica quase tudo:

> **Migrations são o Git do banco de dados.**

No Git, cada commit é uma mudança pequena, aplicada em cima da anterior, formando uma linha do
tempo. Você nunca reescreve um commit antigo que já foi para o repositório de todo mundo — você
faz um commit **novo** por cima. As migrations funcionam igual:

- Cada migration é um arquivo numerado: `V1`, `V2`, `V3`… (o `V` é de *version*).
- O Flyway aplica os arquivos **em ordem crescente**, do menor número para o maior.
- Cada migration roda **uma única vez** por banco. Depois de aplicada, o Flyway a considera
  "feita" e nunca mais executa.
- Para mudar algo, você **não edita** uma migration antiga — cria uma **nova**.

No Descontai, elas vivem numa pasta específica (você viu na Aula 3):

```
src/main/resources/db/migration/
├── V0__baseline_schema.sql
├── V1__assinatura_ciclo.sql
├── ...
├── V9__empresa_login_e_status_produto.sql
└── V10__portal_admin.sql
```

::callout{type="info" title="Anatomia do nome do arquivo"}
O nome `V9__empresa_login_e_status_produto.sql` não é aleatório — é um padrão que o Flyway
entende:

- **`V`** → é uma migration versionada (roda uma vez, na ordem).
- **`9`** → o número da versão (define a ordem de execução).
- **`__`** → **dois** underscores separam a versão da descrição.
- **`empresa_login_e_status_produto`** → uma descrição legível do que a migration faz.
- **`.sql`** → o conteúdo é SQL puro.
::

## Como o Flyway sabe o que já rodou: `flyway_schema_history`

"Mas se a migration roda uma única vez, como o Flyway lembra o que já foi feito?" Ótima
pergunta. Ele cria uma **tabela de histórico** dentro do próprio banco, chamada
`flyway_schema_history`. Cada linha é uma migration já aplicada, com a versão, a descrição, a
data e um *checksum* (uma "impressão digital" do conteúdo do arquivo).

Toda vez que a aplicação sobe, o Flyway faz um ritual simples:

::steps
### Lê a tabela de histórico
Descobre qual foi a **última versão aplicada** naquele banco (ex.: `V7`).

### Olha a pasta de migrations
Lista os arquivos disponíveis (`V0` até `V10`) e vê quais ainda **não** estão no histórico.

### Aplica só o que falta, em ordem
Roda `V8`, `V9`, `V10`, uma por uma, e registra cada uma na `flyway_schema_history`.

### Confere que nada mudou no passado
Se o *checksum* de uma migration já aplicada **não bater** com o arquivo atual, o Flyway
**para com erro**. Isso é uma trava de segurança: você não pode alterar o passado.
::

::callout{type="warning" title="A regra de ouro: migration aplicada é IMUTÁVEL"}
Depois que uma migration rodou em qualquer ambiente compartilhado, **nunca mais edite o
arquivo**. Se você mudar o SQL de uma `V5` que já rodou em produção, o *checksum* muda, o
Flyway detecta a divergência e recusa subir. E, mesmo que forçasse, o banco de produção **não**
seria corrigido (a migration já rodou lá). Precisa de mais uma mudança? Crie uma `V11`. Igual
ao Git: você não reescreve o commit dos outros, você faz um commit novo.
::

## Migrations são ADITIVAS: só somam, nunca refazem

Repare no ponto mais importante do modelo. Cada migration parte do estado que a anterior
deixou e **acrescenta** algo. Ninguém recria o banco inteiro a cada versão. No Descontai, a
`V1` assume que a tabela `usuario` já existe e só adiciona uma coluna nela:

```sql
-- V1__assinatura_ciclo.sql
alter table usuario
    add column if not exists assinatura_ciclo integer;

update usuario
set assinatura_ciclo = 0
where assinatura_ciclo is null;

alter table usuario
    alter column assinatura_ciclo set default 0;

alter table usuario
    alter column assinatura_ciclo set not null;
```

Olha o cuidado real aqui (isso é ouro para você aprender): a migration não só adiciona a
coluna — ela **preenche as linhas antigas** com `0` (o `update`) **antes** de exigir que a
coluna seja `not null`. Se ela marcasse `not null` de cara, as linhas de usuários já existentes
(que estariam com a coluna vazia) quebrariam a regra e o banco recusaria a mudança. Esse
passo de preencher dados existentes tem um nome de mercado: **backfill**.

Pulando para a `V9`, quando o Portal de Empresas nasceu, o padrão se repete — adicionar, sem
destruir:

```sql
-- V9__empresa_login_e_status_produto.sql

-- 1. Vínculo usuário -> empresa
alter table usuario
    add column if not exists empresa_id bigint;

alter table usuario
    add constraint fk_usuario_empresa
        foreign key (empresa_id) references empresas (id);

create index if not exists idx_usuario_empresa on usuario (empresa_id);

-- 2. Status do cupom
alter table produtos
    add column if not exists status varchar(30) not null default 'ATIVO';
```

Duas coisas dignas de nota, tiradas dos próprios comentários do arquivo:

- A coluna `empresa_id` é **nullable** (aceita vazio) de propósito: usuários comuns e admins
  continuam **sem** empresa vinculada. A migration não força ninguém a ter empresa.
- O `status` do cupom nasce com `default 'ATIVO'` — assim, todos os cupons que **já existiam**
  automaticamente viram `ATIVO`, sem precisar de um `update` manual. Escolher um default
  esperto é uma forma elegante de fazer backfill.

E a `V10` (Portal Admin) segue exatamente a mesma filosofia — o próprio comentário do arquivo
avisa: *"Tudo é aditivo: nenhuma coluna/linha existente é removida."* Ela adiciona colunas de
moderação e **cria tabelas novas** (chamados de suporte, feature flags, etc.):

```sql
-- V10__portal_admin.sql (trecho)
alter table usuario
    add column if not exists bloqueado boolean not null default false;

alter table empresas
    add column if not exists status varchar(20) not null default 'ATIVA';

create table if not exists feature_flag (
    id          bigserial primary key,
    chave       varchar(60) not null unique,
    ativo       boolean not null default true,
    descricao   varchar(200),
    created_at  timestamp not null default now(),
    updated_at  timestamp
);
```

::callout{type="tip" title="Por que aditivo é tão importante?"}
Porque em produção existem **dados reais e um app publicado**. Se uma migration apagasse uma
coluna que o app ainda usa, você quebraria o aplicativo de milhares de usuários no ar. Ao só
**adicionar**, você evolui o banco sem derrubar quem já depende dele. Remoções existem, mas são
raras, arriscadas e feitas com muito cuidado — o dia a dia é somar.
::

## E o Hibernate? `ddl-auto=validate`

Você viu na Aula 5 que o Hibernate transforma tabelas em objetos Java (as *entities*). O
Hibernate **é capaz** de criar e alterar tabelas sozinho a partir das entidades — mas fazer
isso em produção é perigoso: ele decide as mudanças "por conta própria", sem histórico e sem
controle. No Descontai, quem manda no banco é o **Flyway**, não o Hibernate. Isso está
configurado numa linha do `application.properties`:

```properties
# JPA / HIBERNATE
spring.jpa.hibernate.ddl-auto=validate
```

::callout{type="info" title="O que \"validate\" faz (e o que NÃO faz)"}
`validate` diz ao Hibernate: **"não crie nem altere nada no banco. Apenas confira, ao subir,
se o banco bate com as minhas entidades."** Se uma entidade tem um campo `bloqueado` mas a
coluna não existe no banco, a aplicação **recusa subir** com um erro claro. É uma rede de
segurança: quem **cria** a estrutura é o Flyway (via migrations); o Hibernate só **valida** que
o resultado ficou coerente com o código. Um confere o outro.
::

Ou seja, a divisão de trabalho fica assim:

| Ferramenta | Papel no banco |
|-----------|----------------|
| **Flyway** | **Cria e altera** a estrutura, via migrations versionadas. |
| **Hibernate** (`validate`) | **Só confere** que o banco corresponde às entidades. Não toca em nada. |

## A história real: fazer a API subir do zero (a migration V0)

Agora a parte mais instrutiva desta aula — um problema **de verdade**, recente, documentado no
`docs/GO-LIVE-10K.md` como o **bloqueador nº 1** para colocar o Descontai em produção. É o tipo
de coisa que só se aprende apanhando, e você vai aprender de graça aqui.

**O sintoma:** as migrations começavam na `V1`. Mas a `V1`, como você viu, já faz
`alter table usuario ...` — ou seja, ela **assume** que a tabela `usuario` (e várias outras) já
existe. E existiam mesmo? Sim, mas por um motivo frágil: no começo do projeto, essas tabelas
tinham sido criadas **pelo Hibernate**, não por uma migration. Em produção estava tudo lá, então
ninguém notou o problema.

Até que alguém tentou subir a API num banco **Postgres totalmente vazio** (um ambiente novo, ou
o CI). Aí o Flyway tentava rodar a `V1`, batia num `alter table usuario` e… **a tabela `usuario`
não existia**. Boom: o boot falhava. **Um banco novo simplesmente não subia.**

::callout{type="warning" title="A pegadinha conceitual"}
As migrations contavam a história do banco **a partir da V1**, mas o "capítulo 0" (as tabelas
base) nunca tinha sido escrito como migration — ele vivia só na cabeça do Hibernate. Em produção
o capítulo 0 já estava aplicado por sorte; num banco vazio, faltava. A linha do tempo tinha um
buraco no começo.
::

**A solução:** foi criada a `V0__baseline_schema.sql` — uma migration que recria o schema base
**exatamente como ele existia antes da V1**. Ela cria `usuario`, `empresas`, `produtos`,
`cidades`, e todas as tabelas fundamentais. Veja como o próprio cabeçalho do arquivo explica a
estratégia:

```sql
-- =============================================================================
-- V0: BASELINE DO SCHEMA (permite subir um banco Postgres VAZIO do zero)
-- =============================================================================
-- Contexto: as migrations V1+ são *aditivas* — V1 já faz "alter table usuario ..."
-- assumindo que as tabelas base existem. Antes desta migration, um banco novo não
-- subia: o Flyway aplicava V1 sem que "usuario", "produtos", etc. existissem.
```

Um detalhe que mostra maturidade: a `V0` **não** inclui as colunas que as migrations `V1..V10`
adicionaram depois. A tabela `usuario` na `V0` é a versão "de antes" — sem `assinatura_ciclo`
(que vem na `V1`), sem `empresa_id` (que vem na `V9`), sem `bloqueado` (que vem na `V10`). Cada
coluna continua sendo criada pela sua própria migration. Assim o histórico permanece **correto e
sem duplicação**: `V0` monta a base, e `V1..V10` vão somando por cima, na ordem certa.

### O truque final: baseline em produção

Só que restou um perigo. Se a `V0` cria a tabela `usuario`… o que acontece em **produção**, onde
essa tabela **já existe** (criada pelo Hibernate lá atrás)? A `V0` tentaria criar tudo de novo e
poderia dar conflito. Como fazer a **mesma** `V0` funcionar nos dois mundos — ignorada em
produção, executada num banco vazio?

A resposta são duas linhas no `application.properties`:

```properties
# FLYWAY (MIGRACOES)
spring.flyway.baseline-on-migrate=${FLYWAY_BASELINE_ON_MIGRATE:true}
spring.flyway.baseline-version=${FLYWAY_BASELINE_VERSION:0}
```

Traduzindo o que essas duas configurações combinam:

::fields
::field{name="baseline-on-migrate = true"}
"Se eu encontrar um banco que **já tem tabelas** mas **não tem** a `flyway_schema_history`,
não entre em pânico. Assuma que ele já está numa certa versão de partida (o *baseline*) e comece
a contar dali."
::
::field{name="baseline-version = 0"}
"E essa versão de partida é a **0**. Ou seja: considere que a `V0` **já foi aplicada**."
::
::

O resultado é elegante:

- **Em produção** (banco que já tem as tabelas): o Flyway marca o baseline na versão 0, portanto
  trata a `V0` como já aplicada e a **ignora** (versão ≤ baseline). A produção **não é tocada** —
  o Flyway só roda `V1` em diante, como sempre fez.
- **Num banco vazio** (CI, ambiente novo): não há tabelas, então não há baseline automático. O
  Flyway executa `V0, V1, V2, …` normalmente, **criando tudo do zero**. A API sobe.

Como reforço, a `V0` usa `create table if not exists` em tudo — então, mesmo que rodasse por
engano num banco que já tem as tabelas, ela não quebraria. Cinto **e** suspensório.

::callout{type="tip" title="Como isso é validado de verdade"}
O melhor: isso não é "confia em mim". O **CI** (integração contínua, Aula 12) sobe um Postgres
**vazio** a cada push e roda a suíte com o Flyway ligado. Existe um teste `contextLoads` que
exercita a migração completa do zero, seguida do `ddl-auto=validate`. Ou seja, o robô prova, a
cada mudança, que **um banco novo sobe** e que o resultado bate com as entidades. O bloqueador
nº 1 do go-live saiu do "quebra em produção surpresa" para "está verde no CI".
::

## Resumindo

- O **banco também evolui**; aplicar mudanças na mão em cada ambiente é receita para o caos.
- **Migrations** são arquivos SQL versionados (`V1`, `V2`…) que o **Flyway** aplica **em ordem** e
  **uma única vez** por banco. É o **"Git do banco"**.
- O Flyway registra o que já rodou na tabela **`flyway_schema_history`** e usa *checksum* para
  garantir que migrations antigas **não sejam alteradas** — migration aplicada é **imutável**.
- Migrations são **aditivas**: cada versão **soma** (colunas, tabelas), sem refazer nem destruir
  o que já existe — porque em produção há dados reais e um app publicado dependendo do banco.
- **`ddl-auto=validate`**: o Hibernate **não cria nada** — só **confere** que o banco bate com as
  entidades. Quem cria a estrutura é o **Flyway**.
- Caso real: como a `V1` já assumia a tabela `usuario`, um banco vazio **não subia**. A
  **`V0__baseline_schema.sql`** recriou o schema base, e **`baseline-on-migrate=true` +
  `baseline-version=0`** fazem a `V0` ser **ignorada em produção** e **executada em banco vazio**.

::callout{type="tip" title="Fixando"}
Tente explicar em voz alta: *"por que eu nunca edito uma migration que já rodou?"* Se a sua
resposta mencionar **checksum**, **imutabilidade** e a analogia de que *migration é igual a um
commit do Git que já foi para o repositório de todo mundo* — você entendeu o coração do Flyway.
::
