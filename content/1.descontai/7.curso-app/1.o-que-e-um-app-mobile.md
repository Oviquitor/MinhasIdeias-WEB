Antes de escrever qualquer código, vamos entender **o que é um app**, como ele se diferencia de um
site e — o mais importante para nós — **como ele conversa com a API** que você já conhece. Esta
aula é 100% conceito. É a fundação de tudo.

## O que é um app mobile

Um **app mobile** é um programa que roda **dentro do celular** do usuário — instalado a partir da
Play Store (Android) ou da App Store (iOS). Diferente de um site, que você acessa pelo navegador, o
app é **instalado** e tem acesso a recursos do aparelho: câmera, biometria, notificações, GPS,
armazenamento seguro.

::callout{type="info" title="App x site, na prática"}
Um **site** mora num servidor e você o visita pelo navegador — nada fica instalado. Um **app** é
baixado e vive no seu celular. Por isso o app pode: funcionar (em parte) sem internet, usar a
digital para te logar, receber notificação com o app fechado e abrir a câmera para ler um QR Code.
No Descontai, é justamente isso que o app faz que um site teria dificuldade: **ler o QR do cupom** e
**guardar seu login com biometria**.
::

## O ecossistema do Descontai

O app não vive sozinho. Lembra do curso da API? O Descontai tem **vários programas** conversando com
o **mesmo cérebro central** (a API):

```text
   📱 Descontai-APP (Flutter)      🖥️ Portal Empresas (web)     🖥️ Portal Admin (web)
        │                                │                            │
        └────────────────┬───────────────┴────────────────────────────┘
                         │  falam HTTP (pedem e recebem JSON)
                  ┌──────▼───────┐
                  │  API REST     │  ← o back-end que você já estudou
                  │ (Spring Boot) │
                  └──────┬───────┘
                  ┌──────▼───────┐
                  │  PostgreSQL   │
                  └──────────────┘
```

O **app é um cliente** da API — exatamente como o portal web é. Ele **não** tem banco de dados
próprio, **não** aplica as regras de negócio pesadas. Ele **pede** os dados para a API, **recebe**
em JSON e **transforma em telas**. Toda regra séria ("esse usuário pode usar esse cupom?") mora na
API; o app cuida de **mostrar** e de dar uma boa experiência.

::callout{type="tip" title="A regra de ouro do cliente"}
**Nunca confie só no app para segurança.** O app pode esconder um botão, mas quem **garante** que um
usuário sem assinatura não usa cupom é a **API** (você viu isso no curso da API). O app é a vitrine
educada; a API é o segurança na porta. Um app bem-feito melhora a experiência, mas a fonte da
verdade é sempre o servidor.
::

## Relembrando: cliente, servidor e HTTP (agora pelo lado do app)

No curso da API, você viu que **cliente pede** e **servidor responde**, falando **HTTP**. Aqui, o
**cliente é o app**. Quando o usuário abre a Home, o app faz:

```http
GET /produtos?cidadeId=1 HTTP/1.1
Host: descontai-api.ondigitalocean.app
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

E a API responde com a lista de cupons em **JSON**, que o app usa para montar os cards na tela. É o
mesmo par pergunta/resposta de sempre — só que agora a gente está do lado de **quem pergunta**.

::callout{type="info" title="O que muda ao ser o cliente"}
Do lado do app, aparecem preocupações novas que o servidor não tem: **a internet pode cair no meio**
(e a tela não pode travar), **a resposta demora** (precisa mostrar um "carregando"), o usuário pode
estar **no metrô sem sinal**. Boa parte do trabalho de um dev mobile é lidar com essa
**imprevisibilidade da rede** — assunto que vai voltar várias vezes no curso.
::

## Nativo x cross-platform: por que Flutter?

Existem duas grandes formas de fazer um app:

::fields
::field{name="Nativo" type="uma base por plataforma"}
Você escreve o app **duas vezes**: uma em **Kotlin/Java** para Android e outra em **Swift** para
iOS. Máxima integração com cada sistema, mas **dobro do trabalho** — dois times, dois códigos, dois
bugs para corrigir.
::
::field{name="Cross-platform" type="uma base para as duas"}
Você escreve **uma vez** e roda nas duas plataformas. É o caminho do **Flutter** (e de outros, como
React Native). Menos código, entrega mais rápida, um time só. É a escolha do Descontai.
::
::

::callout{type="info" title="Por que o Descontai escolheu Flutter"}
Uma startup precisa **entregar rápido e com time enxuto**. Manter dois apps nativos (Android + iOS)
custaria o dobro. Com **Flutter**, o Descontai tem **um único código Dart** que vira app Android
**e** iOS, com visual consistente e ótima performance. Foi a decisão certa para o tamanho e a
velocidade do produto — a mesma mentalidade de "simplicidade que serve o objetivo" que guiou a API.
::

## O que o Descontai-APP faz

Para você ter o mapa mental do produto que vamos construir, o app permite ao usuário:

- **entrar** (login, cadastro, biometria, "lembrar de mim");
- ver os **cupons da sua cidade** na Home, com destaques, categorias e busca;
- abrir o **detalhe** de um cupom e **usá-lo** no balcão (por código ou lendo o **QR**);
- **assinar** um plano e **pagar** por Pix ou cartão;
- acompanhar o **histórico de economia**, as **indicações** e as **notificações**;
- gerenciar o **perfil** e a conta.

Cada um desses fluxos vai virar uma aula, sempre com o código real do app na frente.

## Resumindo

- Um **app mobile** é um programa **instalado** no celular, com acesso a recursos do aparelho
  (câmera, biometria, push) — diferente de um site, que roda no navegador.
- O **Descontai-APP é um cliente** da API: ele **pede** dados em HTTP, recebe **JSON** e monta as
  telas. A **regra de negócio e a segurança de verdade ficam na API.**
- Do lado do cliente, entram desafios novos: **internet instável, latência, offline** — lidar com
  isso é metade do trabalho mobile.
- **Nativo** é escrever o app duas vezes (Android e iOS); **cross-platform** (Flutter) é escrever
  **uma vez** para os dois. O Descontai usa Flutter por velocidade e time enxuto.

*"o app (cliente) faz um GET autenticado para `/produtos` na API (servidor), recebe os cupons em JSON
e transforma em cards na tela"* — você já entendeu o papel do app no ecossistema.
::
