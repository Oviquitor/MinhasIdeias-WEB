Toda chamada à API **pode falhar**: a internet cai, o servidor devolve erro, o token expira. A
pergunta é: **como levar essa falha até a tela** de um jeito organizado, sem espalhar `try/catch` por
todo canto e sem deixar uma exceção "explodir" na cara do usuário? A resposta do Descontai é elegante e
vale para qualquer linguagem: o tipo **`Result`**. Esta aula fecha a Parte 3.

## O problema com exceções soltas

Em muitos códigos, um erro vira uma **exceção** que "sobe" até alguém pegar. Se ninguém pegar, o app
quebra. E o pior: olhando a assinatura de um método `Future<List<Produto>> buscarProdutos()`, você
**não tem como saber** que ele pode falhar — a falha fica invisível, escondida, até estourar em
produção.

::callout{type="info" title="Exceção é como um alarme sem dono"}
Uma exceção não tratada é um alarme tocando sem ninguém responsável: ela sobe pelo código até alguém
lembrar de pegá-la — e se ninguém lembrar, o app fecha. O problema não é a exceção existir; é ela ser
**fácil de esquecer**. A gente quer um jeito que **obrigue** quem chama a decidir o que fazer no erro.
::

## A solução: Result (sucesso OU falha, explícito)

O `Result` é um tipo que representa **um de dois estados possíveis**: deu certo (`Success`) **ou** deu
errado (`Failure`). Olha o código real, curtinho (`lib/core/result/result.dart`):

```dart
/// Resultado de uma operação que pode ser sucesso ou falha.
sealed class Result<T> {
  const Result();
}

final class Success<T> extends Result<T> {
  const Success(this.data);
  final T data;          // os dados, quando deu certo
}

final class Failure<T> extends Result<T> {
  const Failure(this.message);
  final String message;  // a mensagem amigável, quando deu errado
}
```

Agora um método que retorna `Result<List<Produto>>` **avisa na própria assinatura**: "eu posso te dar
os produtos **ou** uma mensagem de erro — você vai ter que lidar com os dois". A falha deixou de ser
invisível.

::callout{type="tip" title="Por que sealed class muda tudo"}
A palavra **`sealed`** ("selado") diz ao Dart: *"as ÚNICAS variações de `Result` são `Success` e
`Failure` — não existe uma terceira"*. Isso destrava um superpoder: o compilador **te obriga** a tratar
os dois casos. Se você esquecer de tratar o `Failure`, o código **nem compila**. O erro deixa de
depender da sua memória e passa a ser garantido pela linguagem. Isso é segurança de verdade.
::

## Pattern matching: tratando os dois casos

Como você "abre" um `Result`? Com um `switch` que casa cada variação — o **pattern matching** do Dart.
Lembra do `HomeController` da Aula 7? É exatamente isto:

```dart
final result = await _useCase(usuarioId: ..., cidadeId: ...);

switch (result) {
  case Success(:final data):        // deu certo: 'data' são os produtos
    produtos = data.produtos;
    _buildDerivedLists();
  case Failure(:final message):     // deu errado: 'message' é o texto do erro
    errorMessage = message;
}
```

Leia como duas portas: **se for `Success`, pegue os `data`**; **se for `Failure`, pegue a `message`**.
Não tem terceira porta, e o compilador não deixa você esquecer nenhuma. Limpo e à prova de esquecimento.

## Quem converte a exceção em Result?

O `Result` só funciona se **alguém** transformar as exceções (que o Dio ainda lança) em `Success`/
`Failure`. Esse alguém é o **repository** (Aula 8) — é a regra de ouro: *"o repository converte
exceções em Result"*. Assim, a exceção morre na camada de dados e **nunca chega crua na tela**:

```dart
// dentro de um repository (esquema):
Future<Result<List<Produto>>> getProdutos(int cidadeId) async {
  try {
    final lista = await _service.getProdutos(cidadeId); // pode lançar
    return Success(lista);                              // deu certo
  } catch (e) {
    return Failure(AppError.tratar(e));                // vira Failure com msg amigável
  }
}
```

## AppError: de erro técnico para mensagem humana

Quando algo falha, o Dio devolve um erro **técnico** (`DioExceptionType.connectionTimeout`,
`SocketException`...). O usuário não pode ver isso — ele precisa de uma frase que entenda. Converter
uma coisa na outra é o papel do `lib/core/utils/app_error.dart` (e você viu a mesma lógica no
`AuthService._formatDioError`):

```dart
switch (e.type) {
  case DioExceptionType.connectionTimeout:
    return 'Tempo esgotado ao conectar. Tente novamente.';
  case DioExceptionType.receiveTimeout:
    return 'Servidor demorou para responder. Tente novamente.';
  case DioExceptionType.connectionError:
    return 'Sem conexao com a internet. Verifique sua rede.';
  case DioExceptionType.badCertificate:
    return 'Conexao insegura. Verifique o certificado.';
  case DioExceptionType.badResponse:
    return _formatBadResponse(e.response?.statusCode, context); // 401→"senha inválida" etc.
  // ...
}
```

E, quando a **própria API** manda uma mensagem de erro (lembra do `ApiError` do curso da API?), o app
prefere mostrá-la. Repare como ele procura o texto em vários nomes de campo possíveis:

```dart
final msg = data['message'] ?? data['mensagem'] ?? data['error'] ?? data['erro'];
```

::callout{type="tip" title="Traduzir o erro é respeito com o usuário"}
"`SocketException: Failed host lookup`" não diz nada a quem usa o app. "Sem conexão com a internet.
Verifique sua rede." diz tudo. Normalizar erros técnicos em **mensagens humanas** é o que faz o app
parecer cuidadoso em vez de quebrado. E centralizar isso no `AppError` garante que a **mesma** falha
gere a **mesma** mensagem em todo o app — consistência que o usuário sente como qualidade.
::

## O caminho completo de um erro

Juntando tudo, quando você tenta carregar a Home sem internet:

::steps
### O Dio falha
A requisição estoura o timeout ou dá `connectionError`. O Dio lança um `DioException`.

### O repository captura e converte
O `try/catch` do repository pega a exceção e chama o `AppError`, que devolve *"Sem conexão com a
internet."*. Isso vira um `Failure('Sem conexão...')`.

### O controller recebe o Result
O `switch` no `HomeController` cai no caso `Failure(:final message)` e guarda em `errorMessage`, depois
chama `notifyListeners()`.

### A tela mostra a mensagem
A página (que ouve o controller) vê `errorMessage != null` e desenha um aviso amigável com um botão
"tentar novamente" — em vez de uma tela branca ou um crash.
::

Repare: a exceção **nasceu** na rede e **morreu** no repository, virando um dado (`Result`) que fluiu
calmamente até a tela. Ninguém "explodiu".

## Resumindo

- Exceções soltas são **fáceis de esquecer** e escondem que um método pode falhar. O **`Result`** torna
  a falha **explícita** na assinatura.
- `Result` é uma **`sealed class`** com dois casos: **`Success`** (tem `data`) e **`Failure`** (tem
  `message`). Como é selado, o compilador **obriga** a tratar os dois.
- O **pattern matching** (`switch` com `case Success(:final data)` / `case Failure(:final message)`)
  abre o `Result` — e não deixa esquecer nenhum caso.
- O **repository** é quem converte exceção em `Result` (a exceção morre ali, não chega crua na tela).
- O **`AppError`** traduz erros técnicos do Dio em **mensagens humanas** e prefere a mensagem que a
  própria API mandou — centralizado, para consistência.

::callout{type="tip" title="Fixando"}
Explique por que o Descontai devolve `Result` em vez de deixar a exceção subir: *"porque `Result`
obriga quem chama a tratar o erro (o compilador exige), e porque a exceção técnica é convertida em
mensagem amigável no caminho — a tela recebe um dado tratado, nunca um crash"*. Esse padrão vale em
qualquer linguagem; guarde-o.
::
