Um app guarda coisas no celular: preferências, o último e-mail digitado e — o mais delicado — o
**token de acesso**. Mas nem todo dado pode ser guardado do mesmo jeito. Guardar o token errado é
como deixar a chave de casa embaixo do tapete. Nesta aula você entende **onde** e **como** o Descontai
guarda cada coisa, e como funcionam o **"lembrar login"** e a **biometria**.

## Dois cofres, dois níveis de segurança

O app tem dois lugares para guardar dados locais, e a diferença entre eles é **crítica**:

::fields
::field{name="SharedPreferences" type="dados NÃO sensíveis"}
Um armazenamento simples de chave-valor, guardado em **texto puro** no aparelho. Rápido e prático,
mas **qualquer** app com acesso ao dispositivo (ou um celular rooteado) poderia lê-lo. Serve para o
que não é segredo: `userId`, `cidade`, preferências, o nome do usuário.
::
::field{name="FlutterSecureStorage" type="dados SENSÍVEIS"}
Guarda os dados **criptografados**, usando os cofres do sistema operacional (Keystore no Android,
Keychain no iOS). Muito mais seguro. É onde vão o **token** e o **refresh token**.
::
::

::callout{type="danger" title="A regra de ouro: token NUNCA em SharedPreferences"}
O documento de boas práticas do Descontai é explícito: *"Não persistir token em SharedPreferences"*.
Por quê? Porque SharedPreferences é texto puro — um token ali é uma chave de acesso exposta. Se
vazasse, alguém poderia se passar pelo usuário. O token **exige** o cofre criptografado (secure
storage). Confundir os dois é uma das falhas de segurança mais comuns em apps — e o Descontai a evita
de propósito.
::

Olha a declaração real no `SessionCache`:

```dart
// Cofre criptografado, com Keystore no Android
static const FlutterSecureStorage _secureStorage = FlutterSecureStorage(
  aOptions: AndroidOptions(encryptedSharedPreferences: true),
);

static const String _keyToken = 'auth_token';
static const String _keyRefreshToken = 'auth_refresh_token';
```

E o cuidado ao restaurar a sessão: os dados **não sensíveis** vêm do SharedPreferences, mas o
**token** vem do secure storage:

```dart
static Future<void> restore() async {
  final prefs = await SharedPreferences.getInstance();
  userId = prefs.getString('id');          // não sensível → prefs
  cidadeId = prefs.getInt('cidadeId');     // não sensível → prefs
  // ...
  if (remember) {
    token = await _safeRead(_keyToken);    // sensível → secure storage
  }
}
```

## "Lembrar login": conveniência com responsabilidade

Quando o usuário marca **"lembrar de mim"**, ele quer não digitar a senha toda vez. Na prática, o app
guarda o **token** no secure storage e, ao abrir de novo, o **restaura** — o usuário entra direto.

```dart
// dentro de saveAuth, só grava o token se remember estiver ligado:
if (rememberValue) {
  await _safeWrite(_keyToken, tokenValue);   // token no cofre criptografado
  await _persistRefreshToken();
} else {
  await _clearSecure();                       // sem remember: não deixa token no aparelho
}
```

::callout{type="info" title="Sem 'lembrar', nada fica no aparelho"}
Repare: se o usuário **não** marca "lembrar", o `else` chama `_clearSecure()` — nenhum token fica
gravado. Ao fechar o app, a sessão morre e ele precisa logar de novo. É a escolha do usuário entre
**conveniência** (fica logado) e **segurança** (nada persistido, ideal em aparelho compartilhado). O
app respeita as duas.
::

## Biometria: entrar com a digital

A **biometria** (digital ou reconhecimento facial) é uma forma ainda melhor de "lembrar": em vez de
deixar o token liberado, o app o guarda **bloqueado** e só o **destranca** depois que o usuário prova
que é ele, com a digital. Duas ideias-chave no `SessionCache`:

::fields
::field{name="Token bloqueado" type="_lockedBiometricToken"}
Quando a biometria está ativa, o token fica numa variável **separada e bloqueada**
(`_lockedBiometricToken`), **fora** do `token` que o app usa. Ou seja: o app está "meio logado" — tem
o token guardado, mas ele **não vale** até a digital.
::
::field{name="Destravar" type="unlockBiometricSession()"}
Depois que o usuário passa a digital (via `local_auth`), o app chama `unlockBiometricSession()`, que
**move** o token bloqueado para o `token` ativo. Só então as requisições saem autenticadas.
::
::

```dart
static bool unlockBiometricSession() {
  if (!biometricEnabled || !hasBiometricToken) return false;
  token = _lockedBiometricToken;  // destranca: agora o token vale
  _notifyChanged();               // avisa o router → libera a Home
  return true;
}
```

::callout{type="tip" title="Por que 'token bloqueado' é mais seguro que 'lembrar'"}
No "lembrar", o token já está ativo assim que o app abre — quem pegar o celular destravado entra. Na
**biometria**, mesmo com o app aberto, o token fica **inerte** até a digital. Um cofre a mais entre o
ladrão e a sua conta. Por isso a biometria é a opção mais segura de "não digitar senha toda vez".
::

## Remember e biometria são mutuamente exclusivos

Um detalhe de design importante: o usuário usa **um ou o outro**, nunca os dois. Faz sentido — "manter
o token sempre liberado" (remember) e "manter o token bloqueado até a digital" (biometria) são
**objetivos opostos**. O código garante isso em vários pontos:

```dart
// ao ligar remember, desliga biometria:
if (rememberValue) {
  biometricEnabled = false;
}
// ao ligar biometria, desliga remember:
if (enabled) {
  remember = false;
}
```

::callout{type="warning" title="Estados que se contradizem = bug garantido"}
Deixar "remember" e "biometria" ligados ao mesmo tempo criaria um estado **contraditório** (o token
deveria estar liberado e bloqueado ao mesmo tempo?). O Descontai trata os dois como **mutuamente
exclusivos** e força isso em cada método que mexe neles. Regra geral de programação: se dois estados
não podem coexistir, **garanta no código** que nunca coexistam — não confie na tela para isso.
::

## Robustez: leitura segura do cofre

Por fim, repare que toda leitura/escrita no secure storage passa por helpers `_safeRead`/`_safeWrite`
que **tratam a ausência do plugin** (ex.: rodando na web, onde secure storage não existe) sem quebrar:

```dart
static Future<String?> _safeRead(String key) async {
  if (!_supportsSecureStorage) return null;     // plataforma sem suporte? devolve null
  try {
    return await _secureStorage.read(key: key);
  } on MissingPluginException {
    return null;                                 // sem plugin? não quebra
  }
}
```

É a mesma filosofia defensiva dos models (Aula 10): **degradar com elegância** em vez de estourar.

## Resumindo

- Há **dois cofres**: **SharedPreferences** (texto puro, para dados **não sensíveis**: userId, cidade,
  preferências) e **FlutterSecureStorage** (criptografado, para o **token** e o **refresh token**).
- **Regra de ouro:** token **nunca** em SharedPreferences — sempre no secure storage.
- **"Lembrar login"** guarda o token no cofre e o restaura ao abrir; sem "lembrar", nada fica no
  aparelho.
- **Biometria** guarda o token **bloqueado** (`_lockedBiometricToken`) e só o ativa com a digital
  (`unlockBiometricSession()`) — mais seguro que "lembrar".
- **Remember e biometria são mutuamente exclusivos** — o código força isso para evitar estado
  contraditório.
- Leituras do cofre são **defensivas** (`_safeRead`): degradam sem quebrar onde não há suporte.

::callout{type="tip" title="Fixando"}
Responda: *"por que a biometria é mais segura que o 'lembrar login', se os dois evitam digitar a
senha?"*. Resposta: *"no 'lembrar', o token já está ativo ao abrir o app; na biometria, ele fica
bloqueado até a digital destravá-lo"*. Se você entendeu a diferença entre token **ativo** e token
**bloqueado**, entendeu segurança de sessão no aparelho.
::
