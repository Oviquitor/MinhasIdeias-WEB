Notificação push é o que traz o usuário **de volta** ao app: "seu cupom vence amanhã", "nova parceria
na sua cidade". No curso da API, você viu o back-end **enviar** push via Firebase. Agora vamos ao lado
que **recebe**: como o app se registra para receber, e o que acontece quando a notificação chega em
cada situação. É a outra ponta do mesmo mecanismo.

## O que é push, e por que precisa do Firebase

Uma **notificação push** é uma mensagem que chega no celular **mesmo com o app fechado**. Mas o app
fechado não está rodando para "receber" nada — então quem entrega? O sistema operacional (Android/iOS),
através de um serviço central. Para falar com esse serviço de forma unificada, usa-se o **Firebase
Cloud Messaging (FCM)**.

::callout{type="info" title="A analogia dos Correios"}
Pensa no **FCM como os Correios**. Para receber cartas, você precisa de um **endereço** — o *token
FCM*, um código único do seu aparelho. Você dá esse endereço para o remetente (a API do Descontai).
Quando a API quer te avisar algo, ela entrega a carta aos Correios (FCM) **com o seu endereço**, e o
FCM entrega no seu celular — mesmo que você não esteja "em casa" (app fechado). Sem registrar o
endereço, ninguém consegue te entregar nada.
::

## O fluxo completo do push

```text
1. App pede permissão de notificação ao usuário
2. Firebase gera o TOKEN FCM (o "endereço" do aparelho)
3. App envia esse token para a API  → POST /notificacoes/tokens
4. API guarda o token vinculado ao usuário  (você viu isso no curso da API!)
5. Quando há algo a avisar, a API manda o push via FCM
6. FCM entrega no aparelho → o app reage
```

Repare no passo 3/4: é literalmente o app **registrando seu endereço** no mesmo endpoint que você
estudou no back-end. As duas pontas se encaixam.

## Passo 1 e 2: permissão e token

No `main.dart`, a função `_configureFcm()` cuida disso ao iniciar o app:

```dart
Future<void> _configureFcm() async {
  final messaging = FirebaseMessaging.instance;
  await messaging.requestPermission();          // pede permissão ao usuário

  final token = await messaging.getToken();     // pega o "endereço" (token FCM)
  if (token != null && token.isNotEmpty) {
    final authToken = SessionCache.token;
    if (authToken != null && authToken.isNotEmpty) {
      await NotificationService().registerToken(token); // registra na API (se logado)
    }
  }
  // ...
}
```

::callout{type="tip" title="Por que só registra se estiver logado?"}
Repare: o app só chama `registerToken` se houver um `SessionCache.token` (usuário logado). Faz
sentido — a API guarda o token FCM **vinculado a um usuário**, para saber **para quem** enviar. Sem
usuário logado, não há a quem vincular. Por isso o registro também acontece **depois do login** (você
viu `_registerFcmToken()` no `AuthService`, Aula 12): assim que o usuário entra, o aparelho dele passa
a receber os pushes dele.
::

## Passo 3: o NotificationService

O `NotificationService` é fininho — ele só faz a ponte HTTP com a API para registrar/remover o token:

```dart
// registra o token FCM deste aparelho no backend
await _dio.post(AppEndpoints.notificacoesTokens, data: {'token': fcmToken});
```

E no logout, o simétrico: remover o token (`DELETE /notificacoes/tokens`), para o aparelho parar de
receber os pushes daquele usuário. Registrar ao entrar, remover ao sair.

## O token muda: onTokenRefresh

O endereço FCM **pode mudar** (reinstalação, limpeza de dados, atualização). Se o app não perceber,
a API ficaria mandando push para um endereço morto. Por isso o Descontai **escuta** a troca de token:

```dart
FirebaseMessaging.instance.onTokenRefresh.listen((newToken) async {
  final authToken = SessionCache.token;
  if (authToken != null && authToken.isNotEmpty) {
    await NotificationService().registerToken(newToken); // re-registra o novo endereço
  }
});
```

::callout{type="info" title="Endereço que muda precisa ser re-registrado"}
É como se mudar de casa: você avisa os Correios do novo endereço, senão suas cartas vão para o lugar
errado. O `onTokenRefresh` é o app "avisando o novo endereço" automaticamente sempre que o token muda —
sem isso, as notificações parariam de chegar silenciosamente, um bug difícil de perceber.
::

## Os três estados: onde a notificação cai

Aqui está o que confunde todo iniciante em push: o app pode estar em **três situações** quando a
notificação chega, e cada uma é tratada diferente:

::fields
::field{name="Foreground (app aberto na tela)" type="onMessage"}
O usuário está com o app aberto. O sistema **não** mostra a notificação sozinho — o app decide o que
fazer via `FirebaseMessaging.onMessage.listen(...)` (ex.: mostrar um aviso interno).
::
::field{name="Background (app aberto, mas minimizado)" type="onMessageOpenedApp"}
O app está em segundo plano. O sistema mostra a notificação; se o usuário **toca** nela, o app volta
ao topo e `onMessageOpenedApp` dispara — bom para **navegar** até a tela relacionada (deep link).
::
::field{name="Terminado (app fechado)" type="background handler"}
O app não está rodando. Um **handler de background** especial trata a mensagem. Ele precisa ser uma
função de topo, marcada com `@pragma('vm:entry-point')`, e inicializar o Firebase por conta própria.
::
::

O Descontai registra os três no `main.dart`:

```dart
// handler para quando o app está FECHADO (função de topo, fora de qualquer classe)
@pragma('vm:entry-point')
Future<void> _firebaseMessagingBackgroundHandler(RemoteMessage message) async {
  await Firebase.initializeApp();
}

// ...e dentro do main / _configureFcm:
FirebaseMessaging.onBackgroundMessage(_firebaseMessagingBackgroundHandler);
FirebaseMessaging.onMessage.listen((m) { /* app aberto */ });
FirebaseMessaging.onMessageOpenedApp.listen((m) { /* tocou na notificação */ });
```

::callout{type="warning" title="O detalhe do @pragma('vm:entry-point')"}
O handler de background roda num **isolado separado**, quando o app está morto — o Flutter precisa
conseguir "achar" essa função a partir do nada. A anotação `@pragma('vm:entry-point')` garante que ela
não seja removida na otimização de build e possa ser chamada nesse contexto. É um daqueles detalhes
que, se faltar, o push com app fechado simplesmente não funciona — e é difícil descobrir por quê. Por
isso ele está lá de propósito.
::

## O payload e o deep link

A notificação carrega um **payload** (dados extras) — lembra que, no curso da API, o envio de push
incluía um `payload`, normalmente um JSON com um *deep link*? É esse dado que o app lê no
`onMessageOpenedApp` para **abrir a tela certa**: uma notificação de "cupom novo" pode levar direto ao
detalhe daquele cupom, em vez de só abrir a Home. É a ponte entre a mensagem e a navegação (Aula 6).

## Resumindo

- **Push** é uma mensagem que chega mesmo com o app fechado, entregue pelo SO via **Firebase Cloud
  Messaging (FCM)** — os "Correios" das notificações.
- O aparelho tem um **token FCM** (seu "endereço"). O app pede **permissão**, pega o token e o
  **registra na API** (`POST /notificacoes/tokens`), vinculado ao usuário — só quando logado.
- O token **pode mudar**; o app escuta `onTokenRefresh` e **re-registra** automaticamente (avisa o
  novo endereço).
- Três estados de recebimento: **foreground** (`onMessage`, app decide), **background/tocado**
  (`onMessageOpenedApp`, bom para navegar) e **app fechado** (handler de background com
  `@pragma('vm:entry-point')`).
- O **payload** carrega dados (deep link) que levam o usuário à tela certa.

::callout{type="tip" title="Fixando"}
Explique por que a API precisa do token FCM do aparelho: *"porque o push é entregue por endereço; a
API guarda o token (endereço) vinculado ao usuário para saber para qual aparelho mandar cada
notificação"*. E amarre: *"o app registra esse token ao logar e o remove ao sair"*. Se você viu a
mesma história dos dois lados (API envia, app registra e recebe), fechou o ciclo do push.
::
