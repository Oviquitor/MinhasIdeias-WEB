Um app pode ter todas as telas certas e ainda assim parecer **quebrado** — se ele congela ao carregar,
mostra tela branca sem internet, ou fecha do nada. A diferença entre um app "que funciona" e um app em
que as pessoas **confiam** está nos detalhes de **resiliência**: como ele se comporta quando as coisas
dão errado. Esta aula é sobre esses detalhes.

## Toda tela tem três estados

O erro mental mais comum do iniciante é programar só o "caminho feliz": os dados chegaram, mostra os
dados. Mas toda tela que busca algo da API tem, na verdade, **três estados**, e você precisa desenhar
os três:

::fields
::field{name="Carregando" type="isLoading"}
A requisição está a caminho. Mostre um indicador — nunca uma tela vazia que parece travada.
::
::field{name="Erro" type="errorMessage != null"}
Deu ruim (sem internet, erro do servidor). Mostre a mensagem amigável (Aula 11) e um botão "tentar
novamente".
::
::field{name="Sucesso (com ou sem dados)" type="dados prontos"}
Chegou. Mostre os dados — **e** trate o caso de vir **vazio** ("nenhum cupom nesta cidade") de forma
diferente de um erro.
::
::

Lembra do `HomeController` (Aula 7)? Ele guarda exatamente `isLoading` e `errorMessage` **de
propósito** — para a tela poder desenhar cada estado. A página lê esses campos:

```dart
if (home.isLoading) return const _LoadingSkeleton();       // carregando
if (home.errorMessage != null) return _ErroComRetry(home); // erro
if (home.produtos.isEmpty) return const _NenhumCupom();    // vazio
return _ListaDeCupons(home.produtos);                      // sucesso
```

::callout{type="warning" title="O 'caminho feliz' é uma armadilha"}
Se você só programa o sucesso, seu app parece ótimo no seu wi-fi rápido — e péssimo no metrô do
usuário. **A maior parte da qualidade percebida de um app mora nos estados de erro e carregamento.**
Sempre pergunte, para cada tela: *"e se estiver carregando? e se der erro? e se vier vazio?"*.
::

## Shimmer: carregamento que não parece travado

Em vez de um círculo girando no vazio, apps modernos mostram um **esqueleto** da tela: retângulos
cinza no formato dos cards que vão aparecer, com um brilho passando. É o **shimmer** (a dependência
`shimmer` do `pubspec`).

::callout{type="info" title="Por que shimmer é melhor que um spinner"}
Um esqueleto que já tem o **formato** do conteúdo diz ao cérebro "a informação está vindo, vai
aparecer aqui" — a espera parece mais curta e o app parece mais rápido, mesmo levando o mesmo tempo. É
psicologia de percepção: o **shimmer** transforma um tempo morto num tempo que "promete". O Descontai
usa isso nas listas de cupons.
::

## Detectar a conexão: connectivity_plus

O app usa `connectivity_plus` para saber se há internet. Isso permite antecipar o problema: em vez de
deixar a requisição falhar por timeout (15-30s de espera!), o app pode **avisar na hora** "você está
sem conexão" e oferecer "tentar de novo" quando a rede voltar.

::callout{type="tip" title="Detectar conexão não substitui tratar o erro"}
Cuidado com uma armadilha: `connectivity_plus` diz que você está **conectado ao wi-fi**, não que a
**internet realmente funciona** (o wi-fi pode estar sem saída). Por isso ele é um **complemento**, não
um substituto do tratamento de erro da Aula 11. Regra: use a detecção de conexão para avisar cedo e
melhorar a experiência, mas **sempre** trate o erro da requisição de verdade — porque a rede pode
falhar mesmo "conectado".
::

## Avisos amigáveis: o top_notice

Quando algo dá certo ou errado (cupom usado, erro ao pagar), o app precisa **avisar** o usuário sem
abrir uma tela nova. O Descontai tem um widget compartilhado, `shared/widgets/top_notice.dart`, que
mostra um **aviso flutuante** de sucesso/erro/info no topo da tela.

::callout{type="info" title="Feedback é conversa"}
Todo toque do usuário merece uma resposta. Usou o cupom? Um aviso verde "cupom usado!". Falhou o
pagamento? Um aviso vermelho com o motivo. Sem feedback, o usuário fica na dúvida ("será que
funcionou?") e toca de novo — gerando bug. Centralizar isso num `top_notice` garante que **todo** o
app fale a mesma língua visual de sucesso e erro. Consistência é o que faz parecer cuidado.
::

## Quando tudo falha: Crashlytics

Por mais cuidado que você tenha, **bugs acontecem** em produção — num aparelho específico, numa versão
de Android que você não testou. O problema: se o app fecha na casa do usuário, **você nunca fica
sabendo**... a menos que tenha um "gravador de acidentes". Esse é o **Firebase Crashlytics**.

Lembra do `main.dart` (Aula 3)? Ele configura isso logo no início:

```dart
void _configureCrashReporting() {
  // erros de widget do Flutter → Crashlytics
  FlutterError.onError = (details) {
    FlutterError.presentError(details);
    FirebaseCrashlytics.instance.recordFlutterFatalError(details);
  };
  // erros fora do Flutter (assíncronos) → Crashlytics
  PlatformDispatcher.instance.onError = (error, stack) {
    FirebaseCrashlytics.instance.recordError(error, stack, fatal: true);
    return true;
  };
}
```

E o `main()` inteiro roda dentro de um `runZonedGuarded`, que captura até os erros que escapariam de
tudo — mandando cada um para o Crashlytics.

::callout{type="tip" title="Crashlytics é o traceId da API, no mobile"}
No curso da API, você viu o `traceId` que identifica um erro no servidor. O **Crashlytics** é o
equivalente no app: quando algo quebra no celular do usuário, ele registra **o erro, a pilha de
chamadas e o modelo do aparelho** — e você vê tudo num painel. Sem isso, você depende do usuário
"reclamar direito", o que quase nunca acontece. **Observabilidade não é só coisa de back-end** — o app
em produção também precisa ser enxergado.
::

## Resumindo

- Toda tela que busca dados tem **três estados**: **carregando**, **erro** e **sucesso** (com o
  sub-caso **vazio**). Programar só o "caminho feliz" é o erro clássico — a qualidade mora nos outros.
- O `HomeController` guarda `isLoading` e `errorMessage` **de propósito**, para a tela desenhar cada
  estado.
- O **shimmer** (esqueleto com o formato do conteúdo) faz o carregamento parecer mais rápido que um
  spinner no vazio.
- **`connectivity_plus`** detecta a conexão para avisar cedo — mas **não substitui** tratar o erro da
  requisição (wi-fi conectado ≠ internet funcionando).
- O **`top_notice`** centraliza os avisos de sucesso/erro — todo toque merece feedback.
- O **Crashlytics** grava os crashes de produção (erro + pilha + aparelho), configurado no `main()` —
  é a "observabilidade" do app, o equivalente ao `traceId` da API.

::callout{type="tip" title="Fixando"}
Para qualquer tela que você imaginar no Descontai, responda as quatro perguntas: *"e se estiver
carregando? e se der erro? e se vier vazio? e se quebrar?"*. As respostas são: shimmer, aviso com
retry, mensagem de "nada aqui", e Crashlytics no fundo. Quem responde as quatro constrói apps em que
as pessoas confiam.
::
