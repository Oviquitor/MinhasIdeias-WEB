## Do código ao aplicativo instalável

Durante o desenvolvimento, você roda `flutter run` e o app aparece no emulador/celular em modo
**debug** — cheio de ferramentas, mas lento e não publicável. Para as lojas, você gera um **build de
produção (release)**: otimizado, compacto e assinado.

```bash
# Android — App Bundle (formato exigido pela Play Store)
flutter build appbundle --release

# Android — APK (para instalar direto, testes)
flutter build apk --release

# iOS
flutter build ios --release
```

::callout{type="info" title="APK x App Bundle (AAB)"}
O **APK** é o instalável direto do Android — bom para você mandar para alguém testar. O **App Bundle
(.aab)** é o formato que você **envia para a Play Store**: a Google gera, a partir dele, um APK
otimizado para **cada modelo** de aparelho (menor download para o usuário). Regra: **APK para testar,
AAB para publicar.**
::

## A URL da API: dart-define

Lembra da Aula 9, que o `AppEndpoints.baseUrl` pode ser sobrescrito? É no build que isso importa: você
injeta a URL da API de produção com `--dart-define`, sem mexer no código:

```bash
flutter build appbundle --release --dart-define=BASE_URL=https://sua-api.com
```

::callout{type="tip" title="Mesma filosofia da API: config vem de fora"}
No curso da API, os segredos vinham de **variáveis de ambiente** (`${DB_URL}`), nunca escritos no
código. No app é o mesmo princípio com `--dart-define`: o **mesmo código** vira app de
desenvolvimento (apontando para uma API de teste) ou de produção (apontando para a real), só mudando o
parâmetro do build. E lembre da Aula 9: em release, o app **exige HTTPS** — o token do usuário nunca
trafega em conexão insegura.
::

## Assinatura: a identidade do app

Um app publicado precisa ser **assinado** digitalmente — é a prova de que ele veio de você e não foi
adulterado. Sem assinatura, as lojas não aceitam.

::fields
::field{name="Android" type="keystore"}
Você gera um **keystore** (um arquivo de chaves) e assina o app com ele. **Guarde esse keystore como
ouro:** se você perdê-lo, não consegue mais publicar atualizações do mesmo app. Também precisa do
`google-services.json` (Firebase Android) — já presente no Descontai.
::
::field{name="iOS" type="certificado + provisioning"}
A Apple exige um **certificado de desenvolvedor** e um **provisioning profile**, gerados na conta
Apple Developer (paga). E o `GoogleService-Info.plist` (Firebase iOS) — que, conforme o checklist do
projeto, **ainda falta** no Descontai (o Android já está pronto).
::
::

::callout{type="danger" title="Segredos de build NUNCA no Git"}
O `google-services.json`, o `GoogleService-Info.plist` e o keystore **não vão para o repositório**. A
regra do Descontai é clara: *"utilize secrets do CI/CD para injetá-los no build"*. É a mesma disciplina
de segredos da API — um keystore ou uma credencial Firebase vazada é um problema sério. Config sensível
entra no build pelo ambiente/CI, não versionada.
::

## O caminho das lojas

Publicar é o processo **mais demorado** — comece cedo. Em linhas gerais:

::steps
### Google Play (Android)
Conta de desenvolvedor Google Play (taxa única), ficha da loja (descrição, screenshots), **política de
privacidade** obrigatória, e o envio do `.aab`. A revisão costuma ser rápida.

### Apple App Store (iOS)
Conta Apple Developer (anual), configuração no App Store Connect, e a **revisão da Apple** — mais
rigorosa e mais demorada. Comece por aqui se tiver prazo.

### Depois de publicado
Atualizações seguem o mesmo caminho, sempre subindo o **número de versão** (o `version: 1.0.0+1` do
`pubspec` — lembra da Aula 3). O `+1` é o build number, que precisa crescer a cada envio.
::

## Roteiro: construa o SEU app do zero à loja

Chegou a hora de virar autor. Aqui está o caminho, aplicando **tudo** que você aprendeu. Não faça tudo
num dia — cada passo é um marco. Travou? **Volte na aula correspondente**, com o código do Descontai
como referência.

::callout{type="tip" title="Antes de começar"}
Escolha um app **pequeno e que você entenda**. Nada de "clone do iFood". Pense em: uma lista de
tarefas, um catálogo de filmes que você quer ver, um controle de hábitos. Simples de propósito — o
objetivo é **aplicar a arquitetura**, não impressionar.
::

::steps
### Crie o projeto e a estrutura
`flutter create meu_app`. Organize `lib/` em `core/`, `features/` e `shared/` (Aula 3). Rode com
`flutter run` e brinque com o **hot reload** (Aula 2).

### Monte a primeira tela
Um `Scaffold` com `AppBar` e uma `Column` de widgets. Diferencie um `StatelessWidget` (um card) de um
`StatefulWidget` (um formulário) (Aulas 4 e 5). Centralize cores num arquivo de tema.

### Configure a navegação
Duas ou três telas com **GoRouter** e um **guard** que barra telas privadas sem login (Aula 6).

### Adicione estado com Provider
Um `ChangeNotifier` para o estado de uma tela, registrado no `MultiProvider` do `main`. Use `watch`
para exibir e `read` para agir (Aula 7).

### Conecte a uma API
Configure o **Dio** com baseUrl e timeout, e um **interceptor** que injeta o token (Aula 9). Pode usar
uma API pública gratuita para praticar, ou a sua própria (do curso da API!).

### Crie os models com parse seguro
Uma classe com `fromJson` **defensivo** para cada resposta (Aula 10). Trate campo faltando e tipo
inesperado sem quebrar.

### Trate erros com Result
Faça os repositórios devolverem **`Result`** (`Success`/`Failure`) e a tela tratar os dois casos, com
mensagens amigáveis (Aula 11). Desenhe os **três estados** de cada tela (Aula 17).

### Adicione autenticação e segurança
Login que guarda o token no **secure storage** (Aula 13), interceptor que trata 401, e — se quiser o
desafio — **refresh token** (Aula 14) e **biometria**.

### Ligue notificações (opcional)
Configure o Firebase e o **FCM**: peça permissão, pegue o token, registre-o (Aula 15). Sinta o push
chegar no seu celular.

### Cuide da resiliência
Estados de carregamento (shimmer), avisos de sucesso/erro, e **Crashlytics** capturando os crashes
(Aula 17).

### Publique
Gere o build de release com `--dart-define`, assine o app e suba numa loja (comece pela Play Store,
que é mais rápida). Quando alguém baixar o **seu** app da loja, o ciclo fechou. 🚀
::

::callout{type="info" title="Não precisa ser linear e perfeito"}
No mundo real ninguém faz esses passos numa sequência limpa. Você vai voltar, refatorar, quebrar e
consertar. **Isso é o processo, não um erro.** O importante é ter feito cada peça pelo menos uma vez
com as próprias mãos — depois disso, elas viram naturais.
::

## O kit do dev mobile

Guarde esta tabela. É o seu checklist de "meu app está pronto para o mundo?".

| Peça | O que garante | Aula |
|------|---------------|------|
| Widgets Stateless/Stateful + layout | Telas montadas e organizadas | 4, 5 |
| GoRouter + guards | Navegação e telas protegidas | 6 |
| Provider / ChangeNotifier | Estado reativo e compartilhado | 7 |
| Arquitetura em camadas | Código sustentável, não espaguete | 8 |
| Dio + interceptors | Fala com a API, token automático | 9 |
| Models com parse defensivo | Resiliência a dados inesperados | 10 |
| Result + AppError | Erros tratados e amigáveis | 11 |
| Secure storage + biometria | Token seguro no aparelho | 13 |
| Refresh token (single-flight) | Sessão que não expira na cara do usuário | 14 |
| Push (FCM) | Traz o usuário de volta | 15 |
| Estados de tela + Crashlytics | UX resiliente e crashes monitorados | 17 |
| Build assinado + dart-define | App publicável e configurável | 18 |

## Parabéns — você chegou ao fim

Pensa em quem você era na Aula 1: talvez sem saber a diferença entre um app e um site, sem nunca ter
visto um widget, achando que "publicar na loja" era um mistério. Agora você conhece a **interface**, o
**estado**, a **arquitetura**, a **segurança**, o **push** e a **publicação** de um app de produção.
Isso é o repertório de um dev mobile de verdade.

::callout{type="note" title="As duas metades do sistema"}
Se você fez os **dois** cursos — a [API](../6.curso) e o app — você tem algo raro: enxerga o
**sistema inteiro**. Sabe como o token nasce no back-end e é guardado no front; como uma regra vive na
API e é refletida na tela; como o push sai de um lado e chega no outro. Poucos devs têm essa visão das
duas pontas tão cedo. Ela é o que te faz entender **por que** as coisas são como são — e construir
melhor por causa disso.
::

::callout{type="tip" title="O único segredo daqui pra frente"}
Não existe atalho, mas existe um caminho: **construa**. Faça o app do roteiro. Depois faça outro, um
pouco maior. Cada app que você leva do zero à loja te ensina mais do que dez tutoriais assistidos. O
mercado não quer quem *assistiu* — quer quem *entregou*.
::

aqui para você revisitar.** Travou na navegação enquanto constrói o seu app? Volte na Aula 6 com o

agora com um projeto seu do lado.
