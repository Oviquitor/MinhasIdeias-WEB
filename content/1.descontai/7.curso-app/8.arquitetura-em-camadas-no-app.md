Você já viu as pontas: a **página** (Parte 2) e o **controller** (Aula 7). Mas entre o controller e a
API existem mais peças. Nesta aula, a gente monta o **mapa completo das camadas do app** — e você vai
notar que ele é quase um espelho da arquitetura da API. Não é coincidência: **boa arquitetura é boa
arquitetura**, no back-end ou no front-end.

## O fluxo completo

Quando o usuário abre a Home e os cupons são carregados, os dados atravessam esta sequência:

```text
 Page / Widget          "mostra a tela"
   └─> Controller       "guarda o estado, dispara a ação"  (ChangeNotifier)
        └─> UseCase      "uma ação do app: carregar a Home"
             └─> Repository (interface)   "contrato: o que dá pra pedir"
                  └─> Repository (impl)   "converte exceção em Result"
                       └─> Service        "monta e faz a chamada HTTP"
                            └─> DioClient  "o cliente HTTP compartilhado"
                                 └─> API REST 🌐
```

E a resposta volta pelo mesmo caminho, ao contrário, até virar pixels na tela. Parece muita peça para
"só pegar uns cupons" — mas cada camada tem **uma** responsabilidade, e é isso que mantém o app
sustentável quando ele cresce.

::callout{type="tip" title="Compare com a API — é o mesmo espírito"}
No curso da API, o fluxo era **Controller → UseCase → Service → Repository → Banco**. No app é
**Page → Controller → UseCase → Repository → Service → Dio → API**. Os nomes se repetem porque a
**ideia** é a mesma: separar "quem mostra" de "quem decide" de "quem busca os dados". Se você entendeu
lá, você já entende aqui — muda o que está na ponta (banco lá, API aqui).
::

## O papel de cada camada

::fields
::field{name="Page (a tela)" type="features/.../presentation"}
Só **mostra** o estado e captura toques. Não sabe endpoint, não monta HTTP, não tem regra. "Está
carregando? Mostra spinner. Tem produtos? Mostra os cards." É a vitrine.
::
::field{name="Controller (o estado)" type="ChangeNotifier"}
Guarda o estado (produtos, loading, erro) e **dispara** a ação chamando o use case. Avisa a tela com
`notifyListeners()` (Aula 7). É o gerente da tela.
::
::field{name="UseCase (a ação)" type="domain/usecases"}
Representa **uma ação** do app: "carregar a Home", "usar cupom", "sincronizar usuário". Orquestra o
que precisa (às vezes junta várias chamadas). Uma classe = uma ação — igual à API.
::
::field{name="Repository (o contrato + a tradução)" type="domain + data"}
Uma **interface** (o contrato: "dá para buscar produtos, categorias...") e uma **implementação** que
chama o service e **converte exceções em `Result`** (sucesso/falha) — para o erro nunca "explodir" na
UI. É a fronteira entre o domínio e o mundo da rede.
::
::field{name="Service (a chamada HTTP)" type="core/data/services"}
Sabe **o endpoint** e monta a requisição: `_dio.post('/auth/login', data: {...})`. Cada service
cuida de um assunto (auth, home, pagamento). É quem realmente fala HTTP.
::
::field{name="DioClient (o carteiro)" type="core/network"}
O cliente HTTP compartilhado: coloca o token, define timeout, trata 401. Todos os services usam **o
mesmo** DioClient. É o carteiro que leva e traz todas as cartas.
::
::

## A regra de ouro (a mesma da API)

O documento de arquitetura do app é explícito sobre o padrão preferido:

> *"a tela não conhece endpoint; o controller não monta HTTP; o repository converte exceções em
> Result; o service fala com a API; o model faz parse seguro do JSON."*

::callout{type="warning" title="O erro clássico (também do júnior mobile)"}
O erro nº 1 é a **página chamar o Dio diretamente** ou montar a URL na mão. Funciona hoje, mas
espalha endereços de API por dezenas de telas — no dia em que uma rota muda, você caça em todo lugar,
e o erro de rede vira uma exceção que quebra a tela na cara do usuário. A regra do Descontai é clara:
*"Não chamar Dio diretamente em Pages. Não montar URL manualmente em telas."* Cada coisa na sua
camada.
::

## "Mas eu vi telas chamando o service direto..."

Sim — e o próprio doc de arquitetura é honesto sobre isso: *"Nem todas as telas seguem todas as
camadas. Algumas telas chamam Service diretamente quando o fluxo é simples."* Isso é **pragmatismo**,
não bagunça.

::callout{type="info" title="Pragmatismo com critério"}
Camada demais para um fluxo trivial vira burocracia; camada de menos para um fluxo complexo vira
espaguete. O Descontai usa o **caminho completo** (UseCase → Repository) nas features mais
importantes (auth, home, profile), onde a lógica justifica, e **encurta** onde é simples. O importante
é o **princípio** estar claro (tela não fala HTTP), e a evolução seguir para o padrão completo. É a
mesma "Clean Architecture parcial" madura que a API pratica.
::

## Onde cada coisa mora (feature-first)

Voltando à organização de pastas da Aula 3: como é *feature-first*, uma funcionalidade inteira mora
junta. A `home`, por exemplo:

```text
features/home/
├── presentation/
│   ├── home_page.dart                    # a Page
│   └── controllers/home_controller.dart  # o Controller (ChangeNotifier)
├── domain/
│   ├── usecases/carregar_home_usecase.dart   # o UseCase (a ação)
│   └── repositories/i_home_repository.dart   # o contrato (interface)
├── data/
│   └── repositories/home_repository.dart     # a implementação (converte em Result)
└── widgets/                                  # cards, chips, carrossel da Home
```

O service (`home_service.dart`) e o `DioClient` ficam em `core/` porque são **compartilhados** entre
features. Abrir "a home" é abrir **uma** pasta e ver todas as camadas dela lado a lado.

::callout{type="tip" title="Por que separar contrato (interface) da implementação?"}
Repare no `i_home_repository.dart` (a interface) separado do `home_repository.dart` (a implementação)
— o mesmo padrão da API. O controller depende do **contrato**, não da implementação concreta. Isso
deixa o controller **testável**: num teste, você entrega um repositório **falso** que devolve produtos
de mentira, sem tocar na API de verdade. Interface é o que permite trocar peças sem quebrar quem usa.
::

## Resumindo

- O dado atravessa: **Page → Controller → UseCase → Repository → Service → DioClient → API**, e volta
  pelo mesmo caminho até virar tela.
- Cada camada tem **uma** responsabilidade: página mostra, controller guarda estado e dispara, use
  case representa a ação, repository traduz para `Result`, service fala HTTP, Dio é o carteiro.
- **Regra de ouro:** a tela não conhece endpoint, o controller não monta HTTP, o repository converte
  exceção em `Result`, o service fala com a API. (O mesmo espírito da arquitetura da API.)
- É **pragmático**: fluxos simples podem encurtar camadas; os importantes usam o caminho completo.
- Organização **feature-first**: cada funcionalidade tem suas camadas juntas numa pasta; o que é
  compartilhado (Dio, services) fica em `core/`.
- **Interface separada da implementação** deixa o código testável (trocar por um repositório falso).

::callout{type="tip" title="Fixando"}
Pegue "usar um cupom" e trace o caminho: *"a `ProductDetailPage` chama o controller/serviço → um use
case `usar_cupom` → repository → `HomeService.usarCupom()` → DioClient → `POST /cupons/usar` na API"*.
Se você consegue nomear as camadas que um toque atravessa, você tem o mapa do app na cabeça.
::
