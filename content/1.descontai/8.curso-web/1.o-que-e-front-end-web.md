Antes de qualquer código, vamos entender **o que é um front-end web** e como ele se diferencia do
back-end (que você já conhece) e do app mobile (idem). Esta aula é 100% conceito — a base de tudo.

## Front-end e back-end

Todo sistema web tem dois lados:

::fields
::field{name="Back-end" type="o servidor"}
Roda no servidor, longe do usuário. Guarda os dados, aplica as regras, expõe a API. É o Descontai-API
que você estudou — o cérebro.
::
::field{name="Front-end" type="o navegador"}
Roda **no navegador** do usuário (Chrome, Safari). É o que a pessoa **vê e toca**: telas, botões,
formulários. Ele **pede** dados ao back-end e os **mostra** de forma bonita e usável. É o Portal de
Empresas.
::
::

::callout{type="info" title="A analogia do restaurante, de novo"}
Se o back-end é a **cozinha** (prepara o prato seguindo as regras) e o app mobile é o **garçom de
delivery**, o **front-end web é o salão do restaurante**: a mesa posta, o cardápio bonito, o garçom
que anota. O cliente nunca entra na cozinha — ele interage com o **salão**, que se comunica com a
cozinha nos bastidores. O front-end é esse salão, dentro do navegador.
::

## O front-end também é um cliente da API

Aqui está a ideia que conecta tudo: assim como o app mobile, **o front-end web é um cliente da API**.
Quando a empresa abre a tela de cupons, o portal faz:

```http
GET /empresas/me/produtos HTTP/1.1
Host: descontai-api.ondigitalocean.app
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

E a API responde com os cupons em **JSON**, que o portal transforma em cards na tela. É o mesmo par
pergunta/resposta de sempre — HTTP, Bearer token, JSON — só que agora o cliente é o **navegador**.

::callout{type="tip" title="A regra de ouro continua valendo"}
Como no app (Aula 1 do curso do app): **nunca confie só no front-end para segurança**. O portal pode
esconder um botão de "excluir", mas quem **garante** que uma empresa só mexe nos próprios cupons é a
**API**. O front-end é a vitrine educada; a API é o segurança. Qualquer um pode abrir as ferramentas
do navegador e forjar uma requisição — por isso a validação real mora **sempre** no servidor.
::

## SPA x site tradicional

Existem duas formas de um site funcionar, e a diferença é importante:

::fields
::field{name="Site tradicional (MPA)" type="cada clique = nova página"}
A cada clique, o navegador **pede uma página HTML inteira nova** ao servidor e recarrega a tela do
zero. Simples, ótimo para sites de conteúdo (blogs, notícias), mas dá aquela "piscada" a cada
navegação.
::
::field{name="SPA (Single-Page Application)" type="uma página que se reescreve"}
O navegador carrega **uma vez** e, a partir daí, **reescreve só os pedaços** da tela que mudam,
buscando **só os dados** (JSON) da API. Navegação instantânea, sensação de app. É o que o Portal de
Empresas é.
::
::

::callout{type="info" title="Por que um painel é uma SPA"}
O Portal de Empresas é um **painel** (dashboard) atrás de login — o usuário passa minutos ali,
navegando entre cupons, resgates, faturamento. Uma SPA torna isso fluido: trocar de aba não recarrega
tudo, só busca os dados novos. Já um site de marketing (como o Descontai-WEB) pode preferir o modelo
tradicional para aparecer bem no Google. **Cada tipo de front-end escolhe o modelo que serve seu
objetivo** — você vai ver isso na última aula, quando falarmos de SSR x SPA.
::

## Os três portais do Descontai

O ecossistema tem **três** front-ends web, e todos usam a **mesma stack** (Nuxt + Vue):

| Portal | Para quem | O que faz |
|--------|-----------|-----------|
| **Empresas** | Parceiros | Gerenciar cupons, ver resgates e faturamento (o que vamos estudar) |
| **Admin** | Equipe interna | Dashboards, moderação, gestão de usuários e pagamentos |
| **Web** | Público | Site de apresentação/landing do Descontai |

::callout{type="tip" title="Aprenda um, entenda os três"}
Como os três compartilham a stack, este curso te ensina a **base comum** usando o Portal de Empresas.
Quando você abrir o Admin ou o Web, vai reconhecer tudo: as mesmas pastas, os mesmos componentes, o
mesmo jeito de falar com a API. Muda **o que** cada portal faz, não **como** é construído. É por isso
que vale a pena dominar um a fundo.
::

## O que o Portal de Empresas faz

Para você ter o mapa do produto que vamos construir:

- **Login** da empresa (com o mesmo JWT da API);
- **Visão geral** — indicadores (KPIs), gráfico de resgates, cupons em destaque;
- **Cupons** — listar, criar, editar e remover ofertas;
- **Resgates** — quem usou cada cupom, com busca e export CSV;
- **Faturamento** — movimentação estimada por mês e por cupom;
- **Minha empresa** — editar os dados exibidos no app.

Cada um desses vira uma tela, e várias delas viram aulas.

## Resumindo

- **Back-end** roda no servidor (dados e regras); **front-end** roda no **navegador** (o que o usuário
  vê e toca). O Portal de Empresas é front-end.
- O front-end é **um cliente da API**: pede dados em HTTP (com `Bearer` token), recebe **JSON** e monta
  as telas. A **segurança de verdade continua na API.**
- **Site tradicional (MPA)** recarrega a página inteira a cada clique; **SPA** carrega uma vez e
  reescreve só os pedaços, buscando só os dados. Um painel como o de Empresas é uma **SPA**.
- O Descontai tem **três portais** (Empresas, Admin, Web) na **mesma stack** — aprenda um, entenda os
  três.

::callout{type="tip" title="Fixando"}
Descreva: *"quando a empresa abre a lista de cupons no portal, o que acontece?"*. Resposta: *"o
front-end (cliente, no navegador) faz um GET autenticado para a API (servidor), recebe os cupons em
JSON e reescreve só a área da lista — sem recarregar a página inteira, porque é uma SPA"*. Se isso
ficou claro, você entendeu o papel do front-end.
::
