Você já viu `useApi`, `useAuth`, `useToast` sendo chamados nas telas. Todos começam com `use` — não é
coincidência: são **composables**. Nesta aula você entende o que são, como reaproveitam lógica entre
telas, e a peça-chave para **estado compartilhado**: o `useState`. É o equivalente Nuxt do
Provider/ChangeNotifier do app.

## O problema: lógica repetida entre telas

Várias telas precisam da **mesma** capacidade: falar com a API, saber quem está logado, mostrar um
aviso de sucesso. Você **não** vai reescrever isso em cada página. A solução do Vue/Nuxt são os
**composables**: funções reutilizáveis que **encapsulam lógica** (e às vezes estado) e podem ser
chamadas de qualquer componente.

::callout{type="info" title="Composable = uma ferramenta na caixa"}
Um **composable** é uma função que começa com `use` e empacota uma capacidade pronta para reusar:
`useApi()` te dá os métodos de API, `useAuth()` te dá a sessão, `useToast()` te dá os avisos. É como
pegar uma **ferramenta da caixa**: você não fabrica a chave de fenda toda vez — pega a que já existe e
usa. Cada tela "compõe" o que precisa chamando os composables certos. Daí o nome.
::

E, como você viu na Aula 3, tudo em `app/composables/` é **auto-importado**: você escreve `useToast()`
em qualquer lugar, sem `import`. O Nuxt acha e conecta.

## ref x useState: local x compartilhado

Aqui está a distinção que destrava o estado no Nuxt:

::fields
::field{name="ref" type="estado LOCAL"}
Vive **dentro de um componente**. Cada componente tem o seu. O `busca` da tela de cupons (Aula 6) é um
`ref` — só aquela tela usa. Some quando a tela sai.
::
::field{name="useState" type="estado COMPARTILHADO"}
Um estado **global**, identificado por uma chave, **compartilhado** por todos os componentes e seguro
para SSR. Vários lugares leem e escrevem o **mesmo** dado. É o que você usa quando a informação precisa
ser vista por telas diferentes.
::
::

::callout{type="tip" title="Quando usar qual"}
A pergunta é a mesma do app (setState x Provider): *"esse dado é só desta tela, ou várias telas
precisam dele?"*. Texto de um campo de busca → **`ref`** (local). A sessão do usuário, a lista de
toasts → **`useState`** (compartilhado). Regra: comece com `ref`; suba para `useState` só quando o dado
precisar ser visto por mais de um lugar.
::

## useState na prática: o useToast

O `useToast` é o exemplo perfeito de composable com estado **compartilhado**: qualquer tela dispara um
aviso, e um componente único (`<DcToasts>`, montado no `app.vue`) os exibe. Olha o real:

```ts
export function useToast() {
  // useState cria (uma vez) e reusa um estado global com a chave 'dc-toasts'.
  // Diferente de um ref local, ele sobrevive entre componentes.
  const toasts = useState<Toast[]>('dc-toasts', () => [])

  function push(message: string, type: Toast['type'] = 'default') {
    const id = Date.now() + Math.floor(Math.random() * 1000)
    // Novo array (imutável) para o Vue detectar a mudança e re-renderizar.
    toasts.value = [...toasts.value, { id, message, type }]
    // Remove este toast após 3,5s, filtrando pela id.
    setTimeout(() => {
      toasts.value = toasts.value.filter((t) => t.id !== id)
    }, 3500)
  }

  return {
    toasts,
    success: (m: string) => push(m, 'success'),
    error: (m: string) => push(m, 'error'),
    info: (m: string) => push(m, 'default'),
  }
}
```

Como funciona o compartilhamento:

::steps
### Uma chave, um estado
`useState<Toast[]>('dc-toasts', () => [])` cria a lista de toasts **uma vez** (na primeira chamada) e,
nas próximas, **reusa a mesma**. A chave `'dc-toasts'` é a identidade global desse estado.

### Qualquer tela adiciona
A tela de cupons chama `toast.success('Cupom removido.')`. Isso empurra um item na **mesma** lista
compartilhada — de qualquer lugar do portal.

### Um componente exibe
O `<DcToasts>`, montado uma vez no `app.vue`, **lê** essa lista (o mesmo `useState`) e a renderiza. Como
é o mesmo estado, ele vê o toast que a tela de cupons adicionou.

### Some sozinho
Cada toast se auto-remove após 3,5s, filtrando-o da lista pela `id` — e o `<DcToasts>` reage e o tira
da tela.
::

::callout{type="info" title="Repare: cria um array novo, não muda o antigo"}
Note `toasts.value = [...toasts.value, novo]` em vez de `toasts.value.push(novo)`. Isso cria um **array
novo** (com os itens antigos + o novo), o que garante que o Vue **detecte a mudança** e re-renderize.
Trabalhar com dados **imutáveis** (criar novo em vez de mutar) é um hábito que evita bugs sutis de
reatividade — vale em Vue, em Flutter, em qualquer framework reativo.
::

## O padrão do portal: composables como "camadas"

Repare no desenho geral: os composables são as **camadas de serviço** do front-end, exatamente como no
app. Cada um tem um papel:

| Composable | Papel |
|---|---|
| `useApi` | Fala com a API (todas as chamadas HTTP num lugar) — Aula 11 |
| `useAuth` | Guarda a sessão do usuário e faz login/logout — Aula 12 |
| `useToast` | Sistema global de avisos (toasts) — esta aula |

::callout{type="tip" title="É a mesma arquitetura em camadas — de novo"}
No app, a lógica ficava em controllers/services/repositories. No portal, ela fica em **composables**. A
tela (a página) chama o composable, que cuida da lógica; a página só mostra. Isso mantém as páginas
**enxutas** e a lógica **reutilizável e testável** — a mesma "separação de responsabilidades" que
persegue os três cursos. Nomes diferentes, princípio idêntico.
::

## Resumindo

- Um **composable** é uma função `use...()` que **encapsula lógica reutilizável** — a "ferramenta na
  caixa". Tudo em `app/composables/` é **auto-importado**.
- **`ref`** é estado **local** (de um componente); **`useState`** é estado **global compartilhado**
  (por chave), visto por todos os componentes e seguro para SSR.
- Use `ref` por padrão; suba para `useState` quando o dado precisar ser visto por **mais de uma tela**
  (sessão, toasts).
- O **`useToast`** mostra o padrão: `useState('dc-toasts', ...)` guarda a lista uma vez; qualquer tela
  adiciona; o `<DcToasts>` exibe; cada toast some sozinho. Sempre criando **array novo** (imutável) para
  o Vue reagir.
- Os composables são as **camadas de serviço** do front (`useApi`, `useAuth`, `useToast`) — a mesma
  arquitetura em camadas dos outros cursos.

::callout{type="tip" title="Fixando"}
Explique por que um toast disparado na tela de cupons aparece na tela: *"porque `useToast` guarda a
lista em `useState` (estado global, mesma chave); a tela adiciona um item nessa lista compartilhada, e
o `<DcToasts>` — que lê o mesmo `useState` — o renderiza"*. Se você entendeu estado **compartilhado por
chave**, entendeu o coração da reatividade global do Nuxt.
::
