Chegou a parte que dá vida ao portal: buscar dados reais da API. Nesta aula você conhece o **`$fetch`**
(a ferramenta HTTP do Nuxt), o **`runtimeConfig`** (de onde vem o endereço da API), e — o mais
importante — a **camada única `useApi`**, que centraliza toda a comunicação e resolve, num lugar só, o
que apareceria em toda chamada: o token e o tratamento de erro.

## $fetch: a ferramenta HTTP do Nuxt

O Nuxt já vem com o **`$fetch`**, uma função pronta para fazer requisições HTTP — o equivalente ao Dio
do app. Você diz o método, a rota e o corpo, e ele cuida do resto:

```ts
const cupons = await $fetch('/produtos', { baseURL: 'https://api...', method: 'GET' })
```

Simples. Mas usar `$fetch` **solto** em cada tela traria os mesmos problemas que o app teria sem o Dio
central: repetir o endereço, repetir o token, repetir o tratamento de erro. Por isso o portal
**envelopa** o `$fetch` numa camada única.

## runtimeConfig: o endereço vem de fora

Antes: de onde sai o endereço da API? Do **`runtimeConfig`** (que você viu no `nuxt.config.ts`, Aula 4).
Você o lê com `useRuntimeConfig()`:

```ts
const config = useRuntimeConfig()
config.public.apiBase   // 'https://descontai-api-...ondigitalocean.app/api/v1'
config.public.mockMode  // 'true' ou 'false'
```

::callout{type="tip" title="Config vem do ambiente — pela terceira vez"}
No curso da API, o endereço do banco vinha de `${DB_URL}`. No app, a URL vinha de `--dart-define`. Aqui,
vem do `runtimeConfig`, **sobrescrito por variáveis de ambiente** (`NUXT_PUBLIC_API_BASE`). É **sempre**
o mesmo princípio: o **código não sabe** o endereço; ele vem de fora, para o mesmo build rodar apontando
para a API de teste ou a de produção. Grave esse padrão — ele é universal.
::

## A camada única: useApi

O `app/composables/useApi.ts` é o **único** lugar do portal que fala com a API. Toda tela chama
`useApi()` e usa seus métodos (`listCupons`, `createCupom`…) — nenhuma tela monta URL ou usa `$fetch`
direto. No coração dele está um pequeno **wrapper** do `$fetch`:

```ts
export function useApi() {
  const config = useRuntimeConfig()
  const { token } = useAuth()   // pega o token da sessão

  /** Wrapper de $fetch com baseURL + Authorization + erro amigável. */
  async function api<T>(path: string, opts = {}): Promise<T> {
    try {
      return await $fetch<T>(path, {
        baseURL: config.public.apiBase,          // 1) endereço central
        ...opts,
        headers: {
          // 2) injeta o token, se houver
          ...(token.value ? { Authorization: `Bearer ${token.value}` } : {}),
          ...(opts.headers as Record<string, string> | undefined),
        },
      })
    } catch (e: any) {
      // 3) transforma o erro técnico numa mensagem amigável
      const msg = e?.data?.message || e?.data?.detail || e?.message || 'Erro de comunicação com a API.'
      throw new Error(msg)
    }
  }
  // ... métodos que usam api<T>(...) ...
}
```

Três responsabilidades resolvidas **uma vez** para todas as chamadas:

::fields
::field{name="1. baseURL central"}
Toda chamada usa `config.public.apiBase`. Nenhuma tela escreve o endereço — muda num lugar (o
ambiente), vale para todas.
::
::field{name="2. Authorization automático"}
Se há token na sessão, ele é injetado como **`Bearer <token>`** — o mesmo header que a API valida
(curso da API). Nenhuma tela precisa lembrar de colocar o token. É o interceptor do Dio (curso do app),
na forma de wrapper.
::
::field{name="3. Erro normalizado"}
O `try/catch` transforma o erro técnico do `$fetch` numa **mensagem amigável**, preferindo a mensagem
que a **própria API** mandou (`e.data.message`). É o `AppError` do app, aqui.
::
::

## Os métodos: cada rota, um método

Em cima desse wrapper, o `useApi` expõe um método por operação — a tela nunca vê a URL:

```ts
async listCupons(): Promise<ProdutoResponse[]> {
  if (mock()) { await delay(); return clone(MOCK_CUPONS) }  // modo demonstração (Aula 15)
  return api<ProdutoResponse[]>('/empresas/me/produtos')     // API real
},

async getCupom(id: number): Promise<ProdutoResponse> {
  if (mock()) { /* ...devolve do mock... */ }
  return api<ProdutoResponse>(`/produtos/${id}`)
},

async createCupom(body: ProdutoRequest): Promise<ProdutoResponse> {
  if (mock()) { /* ...cria no mock... */ }
  return api<ProdutoResponse>('/produtos', { method: 'POST', body })
},
```

::callout{type="info" title="A tela pede em alto nível"}
Repare como a tela de cupons (Aula 6) só escreve `await api.listCupons()` — sem saber que por baixo é
`GET /empresas/me/produtos`, com token e tratamento de erro. Isso é o mesmo princípio do Repository (API
e app): **esconder o 'como' e oferecer o 'o quê'**. Se a rota mudar, você troca **um** método no
`useApi` e nenhuma tela é afetada. Centralizar a API é o que mantém o portal manutenível.
::

## Tipagem: o TypeScript conhece a resposta

Note os tipos: `api<ProdutoResponse[]>(...)` diz ao TypeScript que a resposta é uma lista de
`ProdutoResponse` (definido em `app/types/api.ts`). A partir daí, o editor **conhece os campos** — se
você escrever `cupom.titulo` em vez de `cupom.produto`, ele avisa na hora. É o parse com garantia de
tipo que o app fazia à mão, aqui dado pelo TypeScript.

::callout{type="tip" title="Os tipos espelham os DTOs da API"}
O `app/types/api.ts` tem `ProdutoResponse`, `ProdutoRequest`, `EmpresaResponse` — os mesmos contratos
dos DTOs que você viu no curso da API. É literalmente o **contrato** da API descrito no front, para o
front e o back "falarem a mesma língua". Quando a API muda um campo, você atualiza o tipo e o TypeScript
te mostra **todos** os lugares que precisam se ajustar. Contrato tipado é rede de segurança.
::

## Resumindo

- O **`$fetch`** é a ferramenta HTTP do Nuxt (o "Dio" do portal). O endereço da API vem do
  **`runtimeConfig`** (sobrescrito por variáveis de ambiente) — config sempre vem de fora.
- O **`useApi`** é a **camada única** de comunicação: nenhuma tela usa `$fetch` direto nem monta URL.
- O wrapper `api<T>()` resolve, uma vez para todas: **baseURL central**, **`Authorization: Bearer`
  automático** (da sessão) e **erro normalizado** em mensagem amigável (preferindo a da API).
- Cada operação é um **método** (`listCupons`, `createCupom`…) que a tela chama em alto nível — o
  Repository do front.
- Os **tipos** (`app/types/api.ts`) espelham os DTOs da API: o TypeScript conhece os campos e avisa
  erros na hora.

::callout{type="tip" title="Fixando"}
Explique o caminho de `api.listCupons()`: *"a tela chama o método do `useApi`; o wrapper `api<T>()` usa
o `apiBase` do runtimeConfig, injeta o `Bearer` token da sessão, faz o `$fetch`, e se falhar devolve uma
mensagem amigável"*. Se você viu a camada única escondendo endereço + token + erro, entendeu como um
front conversa com a API de forma sustentável.
::
