Você tem a base (HTML, CSS, JS). Agora conheça as duas ferramentas que organizam tudo isso de forma
profissional no portal: o **Vue** e o **Nuxt**. Entender a diferença entre os dois — e a ideia do
**componente Single-File** — faz o projeto inteiro fazer sentido.

## Vue: o framework de interface

**Vue** é um **framework de JavaScript para construir interfaces**. Ele resolve dois problemas que o
JS puro deixa difíceis:

::fields
::field{name="Reatividade"}
Você liga um dado à tela. Quando o dado muda, o Vue **atualiza a tela sozinho** — você não mexe em
elementos na mão. É a UI declarativa (descrevo o "o quê", não o "como").
::
::field{name="Componentes"}
Você quebra a tela em **peças reutilizáveis** (um card, um botão, um KPI). Cada peça tem sua estrutura,
estilo e comportamento juntos. Monta a tela encaixando peças.
::
::

::callout{type="info" title="Componente = peça de Lego (de novo)"}
Se você fez o curso do app, a ideia de "tudo é widget" volta aqui como **componente**. Um componente
Vue é uma peça de interface autocontida — um `DcKpi` (o card de indicador), um `DcModal` (a janela de
confirmação), um `CupomForm` (o formulário). Você monta telas grandes **encaixando** componentes
pequenos. A diferença para o Flutter é só a sintaxe; a ideia é idêntica.
::

## O componente Single-File (SFC)

A marca registrada do Vue é o **SFC** (Single-File Component): um arquivo `.vue` que junta as **três
camadas** da Aula 2 num lugar só, cada uma na sua seção:

```vue
<script setup lang="ts">
// COMPORTAMENTO (JavaScript/TypeScript): dados e lógica
const titulo = 'Meus cupons'
</script>

<template>
  <!-- ESTRUTURA (HTML): o que aparece -->
  <h1>{{ titulo }}</h1>
</template>

<style scoped>
/* APARÊNCIA (CSS): como aparece — o "scoped" limita ao componente */
h1 { color: var(--brand); }
</style>
```

::callout{type="tip" title="Por que juntar as três camadas é genial"}
Tudo que diz respeito àquela peça — a estrutura, o estilo, a lógica — mora **num arquivo só**. Quer
entender o card de KPI? Abre o `DcKpi.vue` e vê tudo. Não precisa caçar o CSS num arquivo, o JS em
outro. E o `scoped` no `<style>` garante que aquele CSS **só** afeta aquele componente — nunca vaza e
quebra outra tela. É organização que escala: cada peça é uma ilha autossuficiente.
::

Repare o `{{ titulo }}` no template: isso é **interpolação** — o Vue insere o valor da variável ali. Se
`titulo` mudar, o `<h1>` muda sozinho. Essa é a reatividade em ação, e você vai vê-la a fundo na Aula 6.

## Nuxt: o meta-framework

Se o Vue é o framework de interface, o **Nuxt** é um **meta-framework** — um framework **em cima** do
Vue que resolve tudo que um app **completo** precisa e que o Vue sozinho não dá pronto: roteamento,
estrutura de projeto, renderização no servidor, build otimizado, deploy.

::callout{type="info" title="Vue x Nuxt, na analogia do carro"}
O **Vue** é o **motor** — a peça que faz o carro andar (a interface reativa). O **Nuxt** é o **carro
completo** montado em volta desse motor: rodas, câmbio, painel, ar-condicionado. Você **poderia**
montar um carro só com o motor Vue, escolhendo e parafusando cada peça na mão (roteador, build...). O
Nuxt te entrega o **carro pronto**, com tudo integrado e testado. Você foca em dirigir (construir as
telas), não em montar o carro.
::

## A magia das convenções

O grande superpoder do Nuxt é **convenção em vez de configuração**: em vez de você **configurar** cada
coisa, o Nuxt **assume** pela **estrutura de pastas**. Alguns exemplos que você vai usar:

::fields
::field{name="pages/ = rotas"}
Cada arquivo em `app/pages/` **vira automaticamente uma URL**. `pages/cupons/index.vue` vira `/cupons`.
Você não configura roteador nenhum — só cria o arquivo (Aula 8).
::
::field{name="components/ = auto-import"}
Qualquer componente em `app/components/` pode ser usado em qualquer template **sem importar**. Escreveu
`<DcKpi />`? O Nuxt acha e importa sozinho.
::
::field{name="composables/ = auto-import"}
Funções em `app/composables/` (como `useApi`, `useAuth`) ficam **disponíveis globalmente** sem import.
Só chamar (Aula 10).
::
::field{name="middleware/ = guards"}
Arquivos em `app/middleware/` rodam **antes das navegações** — o `auth.global.ts` protege as rotas
(Aula 9).
::
::

::callout{type="tip" title="Menos configuração, menos erro"}
Repare que você quase não vê "arquivos de configuração de roteador/importação" no portal — porque o
Nuxt deduz tudo pela estrutura. Isso é ótimo para o iniciante: você aprende **um conjunto de
convenções** e o framework cuida da fiação. O preço é decorar as convenções — mas elas são poucas e se
repetem. É o mesmo espírito do Spring Boot no curso da API: "faça do jeito convencionado e funciona".
::

## Resumindo

- **Vue** é o framework de **interface**: traz **reatividade** (a tela se atualiza quando o dado muda)
  e **componentes** (peças reutilizáveis).
- O **SFC** (`.vue`) junta as três camadas num arquivo: `<script>` (lógica), `<template>` (HTML) e
  `<style scoped>` (CSS que não vaza). `{{ }}` insere valores (interpolação).
- **Nuxt** é um **meta-framework** em cima do Vue: entrega roteamento, estrutura, renderização e deploy
  prontos — o "carro completo" em volta do "motor" Vue.
- O superpoder do Nuxt é **convenção sobre configuração**: `pages/` viram rotas, `components/` e
  `composables/` têm auto-import, `middleware/` roda antes das navegações.

::callout{type="tip" title="Fixando"}
Explique a dupla: *"o Vue cuida de uma tela reativa feita de componentes; o Nuxt monta o app completo
em volta — rotas a partir de pastas, auto-import, build e deploy. Um arquivo `.vue` guarda estrutura,
estilo e lógica juntos."*. Se você separou "Vue = interface" de "Nuxt = app completo", entendeu a
fundação.
::
