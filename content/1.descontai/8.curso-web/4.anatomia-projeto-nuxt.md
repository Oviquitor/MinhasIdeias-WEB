Hora de abrir o projeto. Assim como fizemos com a API e com o app, vamos entender **como um projeto
Nuxt é organizado**, o papel de cada pasta e arquivo, e como as **convenções** da Aula 3 aparecem na
prática. Ter esse mapa na cabeça faz você achar qualquer coisa no portal em segundos.

## O mapa do projeto

Na raiz do `Descontai-Empresas`, os itens que importam:

```text
Descontai-Empresas/
├── app/                  # >>> TODO o código do front-end mora aqui <<<
│   ├── app.vue           # o componente raiz (envolve tudo)
│   ├── assets/css/       # o CSS global e as variáveis do design system
│   ├── components/       # peças reutilizáveis (auto-import): DcKpi, DcModal…
│   ├── composables/      # lógica reutilizável (auto-import): useApi, useAuth…
│   ├── layouts/          # molduras de página (sidebar, tela de login)
│   ├── middleware/       # guards que rodam antes das navegações
│   ├── pages/            # cada arquivo = uma rota (URL)
│   ├── plugins/          # código que roda no início do app
│   ├── types/            # tipos TypeScript (o formato das respostas da API)
│   └── utils/            # funções auxiliares (formatação, dados de mock)
├── public/               # arquivos servidos como estão (favicon, imagens)
├── nuxt.config.ts        # a configuração central do Nuxt
├── package.json          # dependências e scripts
└── tsconfig.json         # config do TypeScript
```

::callout{type="info" title="Repare: cada pasta é uma convenção da Aula 3"}
`pages/` vira rotas, `components/` e `composables/` têm auto-import, `middleware/` roda antes das
navegações, `layouts/` são as molduras. Você não configurou nada disso — o Nuxt **assume** pela pasta.
Decorar esse punhado de pastas é decorar 80% de como o projeto funciona. Cada uma vira uma aula à
frente.
::

## O package.json: enxuto de propósito

Olha o quanto o portal depende de pouca coisa:

```json
{
  "name": "descontai-empresas",
  "scripts": {
    "dev": "nuxt dev",         // roda em desenvolvimento (http://localhost:3000)
    "build": "nuxt build",     // gera a versão de produção
    "preview": "nuxt preview"  // testa o build localmente
  },
  "dependencies": {
    "nuxt": "^4.4.8",
    "vue": "^3.5.35",
    "vue-router": "^5.1.0"
  }
}
```

::callout{type="tip" title="Três dependências para um portal inteiro"}
Repare: **nenhuma biblioteca de UI** (sem Bootstrap, sem Material, sem Tailwind). O README diz isso de
propósito: *"sem dependências de UI externas — design system próprio"*. Por quê? Menos dependências =
menos peso, menos atualizações quebrando, controle total do visual e da identidade da marca. É uma
escolha madura: em vez de puxar uma biblioteca gigante para usar 10%, o time construiu os poucos
componentes de que precisa (`DcKpi`, `DcModal`, `DcBtn`…). Você vê o custo/benefício de cada
dependência — a mesma disciplina de "só adicione o que resolve um problema real" dos outros cursos.
::

O `nuxt dev` é seu melhor amigo: ele sobe o portal em `localhost:3000` com **hot reload** — você salva
um arquivo e o navegador atualiza na hora, mantendo o estado. É o ciclo "muda → vê" instantâneo.

## O nuxt.config.ts: o painel de controle

Este é o arquivo onde você **configura** o que o Nuxt não deduz por convenção. Olha os pontos-chave do
portal:

```ts
export default defineNuxtConfig({
  ssr: false,                          // 1) SPA: renderiza 100% no navegador
  nitro: { preset: 'cloudflare-pages' }, // 2) deploy: Cloudflare Pages
  css: ['~/assets/css/main.css'],      // 3) o CSS global do design system

  runtimeConfig: {                     // 4) configuração exposta em runtime
    public: {
      apiBase: 'https://descontai-api-...ondigitalocean.app/api/v1',
      mockMode: 'true',
    },
  },

  app: { head: { title: 'Descontai · Portal de Empresas', /* meta, fontes */ } }, // 5)
})
```

Vamos aos pontos numerados:

::fields
::field{name="1. ssr: false (SPA)"}
Desliga a renderização no servidor — o portal é uma **SPA** (Aula 1). Faz sentido: é um painel atrás de
login, sem necessidade de SEO. O navegador baixa o app uma vez e cuida de tudo.
::
::field{name="2. nitro: cloudflare-pages"}
Diz ao Nuxt para gerar a saída no formato que a **Cloudflare Pages** (hospedagem gratuita) entende. É
o preparo para o deploy (Aula 16).
::
::field{name="3. css global"}
Carrega o `main.css` — onde moram as variáveis do design system (`--brand`, `--text`…) usadas por todo
o portal (Aula 7).
::
::field{name="4. runtimeConfig.public"}
Configurações que o app lê em tempo de execução: `apiBase` (endereço da API) e `mockMode` (usar dados
de exemplo ou a API real). Podem ser **sobrescritas por variáveis de ambiente** — o mesmo "config vem
de fora" dos outros cursos (Aula 11 e 16).
::
::field{name="5. app.head"}
O `<title>` da aba, as `<meta>` tags e o carregamento das fontes (Sora/Inter). É o "cabeçalho" do
documento HTML.
::
::

::callout{type="tip" title="runtimeConfig é a ponte para o ambiente"}
Aquele `apiBase` e `mockMode` são o coração da flexibilidade do portal. Em desenvolvimento, `mockMode:
'true'` faz o portal rodar com dados de exemplo, **sem API**. Em produção, uma variável de ambiente
(`NUXT_PUBLIC_MOCK_MODE=false`, `NUXT_PUBLIC_API_BASE=...`) aponta para a API real — **sem mudar
código**. Guarde esse par: ele reaparece nas aulas de API (11) e mock mode (15).
::

## O app.vue: a raiz de tudo

O `app/app.vue` é o **componente raiz** — envolve o portal inteiro. Ele normalmente é curtinho: coloca
o sistema de layouts e os elementos globais (como o container de toasts que a gente vê na Aula 14) em
volta da página atual. Toda tela do portal é renderizada **dentro** dele.

## Resumindo

- Todo o front-end mora em **`app/`**, com pastas que são **convenções**: `pages/` (rotas),
  `components/`/`composables/` (auto-import), `layouts/` (molduras), `middleware/` (guards),
  `types/`/`utils/` (tipos e auxiliares).
- O **`package.json`** é enxuto: só **nuxt, vue e vue-router** — **nenhuma biblioteca de UI** (design
  system próprio, por escolha). `npm run dev` sobe com **hot reload**.
- O **`nuxt.config.ts`** configura o que não é convenção: **`ssr: false`** (SPA), deploy na Cloudflare,
  o CSS global, o **`runtimeConfig`** (apiBase + mockMode, sobrescritíveis por ambiente) e o `head`.
- O **`app.vue`** é o componente raiz que envolve todas as telas.

::callout{type="tip" title="Fixando"}
Abra o `nuxt.config.ts` do portal e explique cada linha em voz alta: *"`ssr: false` é SPA, `nitro`
prepara o deploy na Cloudflare, `runtimeConfig.public` guarda o endereço da API e o modo mock que vêm
do ambiente"*. E lembre: quase tudo o mais o Nuxt deduz pela **estrutura de pastas**. Quem entende as
convenções, navega o projeto sem mapa.
::
