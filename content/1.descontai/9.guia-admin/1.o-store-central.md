A maior diferença do Admin para o Empresas está em **como o estado é organizado**. No Empresas, cada
composable cuidava do seu pedaço (`useAuth` da sessão, `useToast` dos avisos). No Admin, como há
**muitos** tipos de dado que várias telas compartilham (empresas, usuários, cupons, pagamentos…),
existe um **store central**: o `useStore`.

## O que é um store central

Um **store** é um lugar único que **guarda todo o estado do domínio** e **expõe as ações** que mexem
nesse estado. Em vez de espalhar dados por vários composables, você concentra tudo num só — e todas as
telas leem e escrevem ali.

::callout{type="info" title="Store = o depósito central da operação"}
Pense num **depósito central** de uma rede de lojas: em vez de cada loja ter seu estoquezinho isolado,
há **um** depósito que todas consultam e abastecem. O `useStore` é esse depósito: a lista de usuários,
de pagamentos, de cupons — tudo mora nele, e qualquer tela (a de usuários, o dashboard) puxa do mesmo
lugar. Uma fonte única da verdade para o portal inteiro.
::

## Estado singleton: refs no escopo do módulo

O truque do `useStore` está em **onde** os `ref` são declarados: **fora** da função, no escopo do
módulo. Isso os torna **singleton** — criados uma vez, compartilhados por todas as telas:

```ts
// app/composables/useStore.ts
// Estes refs vivem no MÓDULO (fora da função) → são únicos e compartilhados.
const empresas = ref<Empresa[]>(clone(mock.empresas))
const usuarios = ref<Usuario[]>(clone(mock.usuarios))
const cupons = ref<Cupom[]>(clone(mock.cupons))
const pagamentos = ref<Pagamento[]>(clone(mock.pagamentos))
// ...+ planos, categorias, cidades, notificacoes, sugestoes, chamados, featureFlags, appConfig

export function useStore() {
  // ... ações que mexem nesses refs ...
  return { empresas, usuarios, cupons, pagamentos, /* ... */ setStatusEmpresa, bloquearUsuario, /* ... */ }
}
```

::callout{type="tip" title="A diferença sutil de onde declarar o ref"}
No curso web (Aula 10), o estado compartilhado usava **`useState`** (a forma do Nuxt). Aqui, o Admin
usa **`ref` no escopo do módulo** — outra forma de conseguir estado singleton compartilhado. Ambas
funcionam; a diferença é técnica. O importante é entender o **conceito**: um `ref` **dentro** da função
seria recriado a cada chamada (local); um `ref` **fora** (no módulo) é criado **uma vez** e reusado por
todos — global. Esse "onde declarar" é o que separa estado local de compartilhado.
::

## As ações moram junto com o estado

O que torna o `useStore` um "store" de verdade (e não só um monte de dados) é que ele **também expõe as
ações** que mudam o estado. A tela não muda os dados na mão — ela **pede** uma ação ao store:

```ts
// dentro do useStore
async function bloquearUsuario(id: number, bloquear: boolean) {
  const u = usuarios.value.find((x) => x.id === id)
  if (!u) return
  if (!isMock) await api.patch(bloquear ? ROUTES.usuarios.bloquear(id) : ROUTES.usuarios.desbloquear(id))
  u.status = bloquear ? 'Bloqueado' : 'Ativo'   // muda o estado central
  registrarLog(...)                              // registra na auditoria (Aula 3)
  toast.success(`Usuário "${u.nome}" ${bloquear ? 'bloqueado' : 'desbloqueado'}.`)
}
```

E a tela de usuários só chama:

```ts
const store = useStore()
store.bloquearUsuario(usuario.id, true)   // a tela não sabe COMO; só pede
```

::callout{type="info" title="Por que juntar estado + ações num store"}
Concentrar "os dados **e** as operações sobre eles" num lugar traz três ganhos: (1) qualquer tela vê o
mesmo dado atualizado (bloqueou o usuário na tela de usuários → o dashboard já reflete); (2) a lógica
de cada ação (mudar status, logar, avisar) fica num só lugar, não duplicada em cada tela; (3) a tela
fica **burra e enxuta**, só disparando ações. É a mesma "separação de responsabilidades" dos outros
cursos, agora num store. Este padrão é o antecessor de bibliotecas como **Pinia** (o store oficial do
Vue) — entendê-lo aqui te prepara para elas.
::

## Quando vale a pena um store central

Nem todo projeto precisa disso. O Empresas **não** tem um store central — e está certo, porque seu
estado é mais simples e local a cada tela. O Admin tem, porque governa **muitos** tipos de dado
interligados que **muitas** telas compartilham.

::callout{type="tip" title="A régua"}
Estado local a uma tela → `ref` na própria tela. Poucos dados compartilhados → um composable com
`useState` (como o Empresas). **Muitos** dados de domínio interligados, usados por muitas telas, com
ações que precisam ser consistentes em todo lugar → um **store central** (como o Admin). Escale a
solução ao tamanho do problema — não comece com um store gigante se um `ref` resolve.
::

## Resumindo

- O Admin usa um **store central** (`useStore`): um lugar único que guarda **todo o estado do domínio**
  (empresas, usuários, cupons, pagamentos…) e **expõe as ações**.
- O estado é **singleton** porque os `ref` são declarados **no escopo do módulo** (fora da função) —
  criados uma vez, compartilhados por todas as telas. (No web usamos `useState`; aqui, `ref` de módulo
  — mesmo conceito, forma diferente.)
- As **ações** (`bloquearUsuario`, `resolverPagamento`…) moram no store, junto do estado. A tela só
  **pede** a ação; não mexe nos dados na mão.
- Store central vale para **muitos** dados de domínio compartilhados (o caso do Admin) — é o padrão que
  antecede o **Pinia**. Projetos menores (como o Empresas) não precisam.
