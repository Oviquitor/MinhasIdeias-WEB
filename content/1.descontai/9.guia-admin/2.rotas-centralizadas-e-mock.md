No curso web, cada método do `useApi` decidia entre mock e API real (Aula 15). O Admin leva essa ideia
adiante com dois refinamentos que valem conhecer: um **catálogo central de rotas** e o padrão de ter,
em cada ação, o caminho mock **e** a chamada real lado a lado.

## O catálogo de rotas: services/routes.ts

Em vez de escrever URLs soltas pelo código, o Admin concentra **todas** num só arquivo,
`app/services/routes.ts`. Cada rota é uma entrada nomeada, muitas geradas por função (para IDs):

```ts
// app/services/routes.ts (esquema)
export const ROUTES = {
  usuarios: {
    bloquear:    (id: number) => `/admin/usuarios/${id}/bloquear`,
    desbloquear: (id: number) => `/admin/usuarios/${id}/desbloquear`,
  },
  pagamentos: {
    aprovar: (id: number) => `/admin/pagamentos/${id}/aprovar`,
    recusar: (id: number) => `/admin/pagamentos/${id}/recusar`,
  },
  cupons: { alterarStatus: (id: number) => `/admin/cupons/${id}/status`, /* ... */ },
  sistema: { config: '/admin/config', featureFlag: (chave: string) => `/admin/flags/${chave}` },
  // ...
}
```

::callout{type="info" title="Um só lugar para todas as URLs"}
Isso é a mesma ideia do `AppEndpoints` do app e do `useApi` do Empresas, levada ao extremo: **nenhuma
tela e nenhuma ação escreve uma URL na mão**. Elas usam `ROUTES.usuarios.bloquear(id)`. Se um endpoint
mudar de caminho, você corrige **uma linha** no catálogo. E o README marca cada rota com **✅ existe**
ou **🆕 falta** — o mesmo `[PROPOSTO]` do Empresas, num catálogo (as que faltam estão listadas em
`MISSING_API_ROUTES.md`). É contract-first (Aula 15 do curso web) organizado num índice.
::

## O padrão mock/real lado a lado

O refinamento mais elegante do Admin: cada ação do store carrega **o caminho mock e o real juntos**. A
linha `if (!isMock) await api...` é a única diferença entre os dois modos:

```ts
async function resolverPagamento(id: number, status: 'approved' | 'rejected') {
  const p = pagamentos.value.find((x) => x.id === id)
  if (!p) return

  // Se NÃO for mock, chama a API real (usando o catálogo de rotas):
  if (!isMock) await api.patch(status === 'approved' ? ROUTES.pagamentos.aprovar(id) : ROUTES.pagamentos.recusar(id))

  // Em AMBOS os modos, atualiza o estado local (o design responde na hora):
  p.status = status
  registrarLog(...)
  toast.success(`Pagamento de ${p.name} ${status === 'approved' ? 'aprovado' : 'recusado'}.`)
}
```

Leia a estrutura: **primeiro** (só no modo real) fala com a API; **depois** (nos dois modos) atualiza o
estado central, registra auditoria e avisa. No modo mock, a linha da API é pulada e só o estado local
muda — o portal "funciona" para demonstração sem back-end.

::callout{type="tip" title="Por que essa ordem importa"}
A API é chamada **antes** de mudar o estado local. Assim, se a chamada real **falhar** (lançar erro), a
execução para ali — e o estado local **não** é alterado indevidamente. Você não mostra "pagamento
aprovado" se a API recusou a operação. Ordem correta: **confirma no servidor, depois reflete na tela.**
É a mesma disciplina do CRUD do curso web (recarregar/refletir a verdade do servidor).
::

## Ligar a API real: uma flag

Como no Empresas, virar do mock para a API real é **só configuração** — sem tocar no código das telas
nem das ações:

```bash
# .env
NUXT_PUBLIC_USE_MOCK=false
NUXT_PUBLIC_API_BASE=https://api.descontai.com.br/api/v1
```

O `useApi` do Admin expõe um `useMock` (lido dessa flag); o `useStore` lê `api.useMock.value` e decide.
Uma flag, e todas as ações passam a falar com a API real, usando os caminhos do `ROUTES`.

::callout{type="warning" title="O mesmo alerta do mock"}
Como no curso web: o mock devolve dados perfeitos e **sempre** deixa a ação "dar certo". A API real vai
recusar operações, exigir permissão de ADMIN, devolver erros. **Antes de produção, teste com
`USE_MOCK=false`** — é aí que aparecem os casos que o mock esconde (um pagamento que não pode ser
aprovado, um usuário que não pode ser bloqueado). O mock acelera; ele não valida.
::

## Resumindo

- O Admin mantém **todas as URLs** num **catálogo central** (`app/services/routes.ts`), muitas geradas
  por função para IDs — nenhuma tela escreve URL na mão. Rotas que ainda não existem estão marcadas
  (🆕 / `MISSING_API_ROUTES.md`) — contract-first organizado.
- Cada ação do store carrega o caminho **mock e real juntos**: a linha **`if (!isMock) await api...`** é
  a única diferença entre os modos.
- A API é chamada **antes** de mudar o estado local — se ela falhar, o estado não é alterado à toa
  (confirma no servidor, depois reflete).
- Ligar a API real é **só uma flag** (`NUXT_PUBLIC_USE_MOCK=false`) — mas teste com ela antes de
  produção.
