O Portal Admin pode **bloquear um usuário, aprovar um pagamento e colocar o app inteiro em
manutenção**. Esse poder muda a engenharia: operações sensíveis exigem **rastro, cuidado e controle**.
Esta última página do guia cobre os três padrões que vêm com essa responsabilidade — e que o Empresas
não precisava.

## Auditoria: rastro de cada ação

Toda ação administrativa deixa um **registro de auditoria**. Repare que **todas** as ações do
`useStore` chamam `registrarLog(...)`:

```ts
function registrarLog(acao: string, entidade: string, detalhe: string) {
  logsAuditoria.value.unshift({
    id: Date.now(),
    ator: 'Admin Descontai',
    acao,                    // "Bloqueou usuário"
    entidade,                // "Usuário #42"
    detalhe,                 // "João Silva"
    ip: '177.x.x.10',
    em: new Date().toLocaleString('pt-BR', ...),
  })
}
```

::callout{type="info" title="Por que auditar tudo"}
Quando alguém tem poder de bloquear contas e mexer em dinheiro, você **precisa saber quem fez o quê e
quando**. Se um usuário foi bloqueado por engano, ou um pagamento aprovado indevidamente, a auditoria
responde "quem, quando, o quê". É prestação de contas — e, em muitos casos, exigência legal. Lembra que
a **própria API** tem uma tabela `audit_log` (curso da API)? O Admin é a ponta que **gera** esses
eventos. Poder sem rastro é risco; a auditoria é a rede de segurança da operação.
::

No Admin (modo mock), o log vive no store; no modo real, o ideal é a **API** registrar a auditoria no
servidor (fonte confiável), já que um registro só no cliente poderia ser burlado. O padrão aqui mostra
a **intenção**: nenhuma ação sensível passa sem deixar rastro.

## Moderação: aprovar, recusar, bloquear

Boa parte do Admin é **moderação** — decidir o que entra e o que sai da plataforma:

- **Usuários/empresas**: mudar status (ativar/pausar/**bloquear**).
- **Pagamentos**: **aprovar ou recusar** assinaturas manualmente.
- **Cupons**: pausar, **ocultar no app**, marcar destaque.
- **Sugestões e reclamações**: aprovar, responder, encerrar.

Repare um detalhe de regra de negócio no store: ao pausar/expirar um cupom, ele também **fica
invisível** automaticamente:

```ts
async function setStatusCupom(id: number, status: Cupom['status']) {
  // ...
  c.status = status
  if (status === 'Pausado' || status === 'Expirado') c.visivel = false  // regra acoplada
  // ...
}
```

::callout{type="tip" title="Ações destrutivas pedem confirmação"}
Bloquear um usuário ou recusar um pagamento **afeta uma pessoa real**. Como no CRUD do curso web (o
modal de remover cupom, Aula 13), toda ação de moderação com impacto deve pedir **confirmação** antes
de executar — e dar **feedback** claro depois (o `toast.success`). Poder de moderação sem uma etapa de
confirmação é acidente esperando para acontecer.
::

## Controle do app: manutenção e feature flags

O padrão mais impactante do Admin: ele controla **o próprio aplicativo** dos usuários, sem precisar de
um novo deploy. Duas alavancas:

::fields
::field{name="Modo de manutenção"}
O admin pode **ligar/desligar o app**: colocar em manutenção, definir a mensagem exibida, a versão
mínima exigida. O app mobile consulta essa config (lembra do endpoint público `/app/config` no curso da
API) e se comporta conforme ela.
::
::field{name="Feature flags"}
"Interruptores" que **ligam/desligam funcionalidades** sem publicar uma nova versão. Uma feature nova
pode ser ativada para todos com um clique — ou desativada na hora se der problema.
::
::

```ts
async function toggleFlag(chave: string) {
  const f = featureFlags.value.find((x) => x.chave === chave)
  if (!f) return
  f.ativo = !f.ativo
  if (!isMock) await api.patch(ROUTES.sistema.featureFlag(chave), { ativo: f.ativo })
  registrarLog('Alterou feature flag', chave, f.ativo ? 'ativada' : 'desativada')
  toast.success(`"${f.nome}" ${f.ativo ? 'ativada' : 'desativada'}.`)
}
```

::callout{type="info" title="Por que feature flags são tão poderosas"}
Sem flags, mudar qualquer comportamento do app exige um novo build e a espera da revisão da loja (dias).
Com flags, o admin **muda na hora**, remotamente. Lançou uma feature e deu ruim? Desliga a flag, sem
publicar nada. É a mesma ideia do "backend selecionável por config" do curso da API — controlar
comportamento por **configuração remota**, não por código. Para uma operação que precisa reagir rápido,
isso é ouro. E, claro: cada flip de flag vai para a **auditoria**.
::

## Resumindo

- O Admin tem **poder** (bloquear, aprovar pagamento, pôr o app em manutenção) — e isso traz padrões
  que o Empresas não precisava.
- **Auditoria**: toda ação chama `registrarLog(...)` — quem, quando, o quê. Poder sensível **sempre**
  deixa rastro (idealmente registrado pela API no servidor, não só no cliente).
- **Moderação**: aprovar/recusar/bloquear/ocultar, às vezes com regras acopladas (cupom pausado fica
  invisível). Ações destrutivas pedem **confirmação** e **feedback**.
- **Controle do app**: **modo de manutenção** e **feature flags** mudam o comportamento do app dos
  usuários **por configuração remota**, sem novo deploy — reação rápida, com auditoria.
