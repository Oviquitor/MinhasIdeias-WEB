## 23. Financeiro leve para lojistas

O Vitrine pode evoluir para um financeiro simples focado no catálogo.

### 23.1 Objetivo

Ajudar o lojista a entender custo, preço, margem e resultado básico.

### 23.2 Recursos futuros

- Custo por produto.
- Margem por produto.
- Relatório de produtos mais rentáveis.
- Controle de despesas simples.
- Taxas de pagamento.
- Relatório de vendas confirmadas.
- Exportação CSV/Excel.

### 23.3 Integração com FinanceOS

Como você também tem a ideia do FinanceOS, futuramente o Vitrine pode se integrar a ele.

Exemplo:

```text
Pedido confirmado no Vitrine
        ↓
Lançamento automático no FinanceOS
        ↓
Receita categorizada
        ↓
Relatório financeiro do negócio
```

Isso criaria um ecossistema forte.

## 24. Pagamentos futuros

### 24.1 Fase correta

Pagamento não deve entrar no MVP inicial. Ele aumenta complexidade de forma significativa.

Entrar quando houver:

- Catálogo validado.
- Lojistas ativos.
- Demanda real por checkout.
- Estrutura de suporte.
- Regras de cancelamento.
- Termos de uso claros.

### 24.2 Formas de pagamento futuras

- Pix.
- Cartão.
- Boleto.
- Link de pagamento.
- Assinatura do lojista.
- Checkout transparente.
- Split de pagamento, se fizer sentido.

### 24.3 Gateways possíveis

- Asaas.
- Stripe.
- Mercado Pago.
- Pagar.me.
- Iugu.

### 24.4 Estratégia técnica

Criar interface:

```text
PaymentGateway
  createPayment()
  getPaymentStatus()
  cancelPayment()
  refundPayment()
  handleWebhook()
```

Implementações:

```text
AsaasPaymentGateway
StripePaymentGateway
FakePaymentGateway
```

### 24.5 Webhooks

Todo gateway precisa de webhooks para atualizar status.

Eventos:

```text
PAYMENT_CREATED
PAYMENT_CONFIRMED
PAYMENT_FAILED
PAYMENT_REFUNDED
SUBSCRIPTION_PAID
SUBSCRIPTION_OVERDUE
```

### 24.6 Regras importantes

- Nunca confiar apenas no front-end para confirmar pagamento.
- Verificar assinatura do webhook.
- Idempotência obrigatória.
- Registrar payload bruto em log seguro.
- Não armazenar dados sensíveis de cartão.

## 25. Nota fiscal futura

### 25.1 Visão

Emissão fiscal é um módulo futuro e deve ser tratado com seriedade.

Existem diferenças entre:

- NFS-e para serviços.
- NF-e para produtos.
- NFC-e para venda ao consumidor.
- Regras municipais, estaduais e nacionais.

### 25.2 Estratégia recomendada

Não implementar emissão fiscal diretamente no começo.

Criar um módulo abstrato:

```text
FiscalGateway
  issueServiceInvoice()
  issueProductInvoice()
  cancelInvoice()
  getInvoiceStatus()
```

Usar integrações com provedores especializados ou API oficial quando viável.

### 25.3 NFS-e

Para prestadores de serviço, a NFS-e é relevante.

O sistema deve permitir no futuro:

- Cadastro de dados fiscais.
- Serviço prestado.
- Código de serviço.
- Tomador.
- Valor.
- Descrição.
- Emissão.
- Cancelamento.
- PDF/XML.

### 25.4 Cuidado

O Vitrine deve evitar parecer consultoria contábil. O sistema pode automatizar processo, mas precisa recomendar que o lojista valide obrigações com contador.

## 26. Assinatura do SaaS

### 26.1 Planos sugeridos

#### Plano Starter

Para quem quer começar.

- Até 30 produtos.
- 1 loja.
- Subdomínio Vitrine.
- Personalização básica.
- Pedido via WhatsApp.
- Métricas básicas.

#### Plano Essencial

Para quem usa com frequência.

- Até 200 produtos.
- Categorias ilimitadas.
- Promoções.
- Cupons.
- Mais templates.
- Analytics.
- IA limitada por mês.

#### Plano Pro

Para lojas mais maduras.

- Produtos ilimitados.
- Campanhas.
- Domínio próprio.
- IA avançada.
- Relatórios.
- Controle de custos.
- Integrações futuras.

#### Plano Business futuro

- Multiusuário.
- Múltiplas unidades.
- Checkout.
- Pagamentos.
- Integrações fiscais.
- Suporte prioritário.

### 26.2 Critérios de bloqueio

Quando plano vencer:

- Não excluir dados.
- Bloquear novas publicações.
- Manter painel acessível por período.
- Vitrine pública pode entrar em modo suspenso após tolerância.

### 26.3 Trial

Trial pode ser:

- 7 dias.
- 14 dias.
- Grátis até publicar X produtos.

Recomendação: trial guiado com foco em publicar a primeira vitrine.

## 28. Multi-tenant e domínios

### 28.1 Modelo inicial

Usar isolamento por `tenant_id` no banco.

Cada registro importante possui:

```text
tenant_id
store_id
```

### 28.2 Subdomínios

Formato:

```text
https://{slug}.vitrine.com.br
```

Exemplos:

```text
casabella.vitrine.com.br
barbeariadosjoao.vitrine.com.br
ateliedacamila.vitrine.com.br
```

### 28.3 Domínio próprio futuro

Permitir:

```text
https://www.minhaloja.com.br
```

Fluxo:

1. Lojista cadastra domínio.
2. Sistema mostra instruções DNS.
3. Lojista cria CNAME.
4. Sistema valida apontamento.
5. Sistema ativa domínio.
6. Certificado SSL é gerado.

### 28.4 Regras de tenant

- Um lojista não acessa dados de outro.
- Toda query privada deve filtrar por tenant.
- Serviços internos devem receber contexto do tenant.
- Testes devem validar isolamento.

### 28.5 Estratégias futuras

1. Banco compartilhado com tenant_id.
2. Schema por tenant.
3. Banco por tenant.

Para o MVP, usar banco compartilhado com tenant_id.
