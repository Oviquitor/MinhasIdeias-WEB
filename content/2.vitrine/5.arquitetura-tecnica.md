## 32. Arquitetura sugerida

### 32.1 Opção recomendada para você

Como você já trabalha com Java, Spring Boot, SQL e tem interesse em arquitetura limpa, a arquitetura recomendada é:

```text
Frontend: Next.js + TypeScript + Tailwind CSS
Backend: Java 21 + Spring Boot
Banco: PostgreSQL
Cache: Redis
Storage: S3 ou Cloudflare R2
CDN: Cloudflare
Fila: RabbitMQ no início, Kafka futuramente se necessário
Infra inicial: VPS/Docker ou Render/Railway/Fly.io
Infra madura: AWS/GCP/Azure
```

### 32.2 Por que Next.js

- Excelente para SEO.
- Server-side rendering.
- Boa performance.
- Ótimo ecossistema.
- Fácil deploy.
- Serve landing, painel e storefront.

### 32.3 Por que Spring Boot

- Forte para regras de negócio.
- Excelente para APIs.
- Segurança madura.
- Bom para arquitetura modular.
- Você já tem familiaridade.

### 32.4 Por que PostgreSQL

- Robusto.
- Ótimo para SaaS.
- Suporta JSONB para customizações.
- Boa consistência.
- Excelente para relatórios.

### 32.5 Monólito modular primeiro

Não começar com microserviços.

Começar com módulos:

```text
auth
identity
stores
catalog
inventory
orders
campaigns
analytics
billing
payments
fiscal
notifications
admin
shared
```

Cada módulo deve ter fronteira clara.

### 32.6 Evolução futura

Somente extrair serviços quando houver necessidade real:

- Serviço de imagens.
- Serviço de analytics.
- Serviço de billing.
- Serviço de pagamentos.
- Serviço fiscal.

## 36. Design system

### 36.1 Estilo visual

O Vitrine deve parecer:

- Moderno.
- Limpo.
- Premium.
- Simples.
- Confiável.
- Acessível.

Referências de sensação:

- Stripe pela clareza.
- Notion pela simplicidade.
- Linear pela elegância.
- Shopify pela organização comercial.

### 36.2 Componentes principais

- Button.
- Input.
- Select.
- Textarea.
- Modal.
- Drawer.
- Card.
- ProductCard.
- ProductGallery.
- Badge.
- Tabs.
- Toast.
- EmptyState.
- DataTable.
- StatCard.
- ColorPicker limitado.
- UploadImage.
- Stepper.
- Sidebar.
- MobileNavbar.

### 36.3 Empty states

Exemplo para produtos:

```text
Sua vitrine ainda não tem produtos.
Cadastre seu primeiro produto para começar a mostrar sua loja para o mundo.
[Adicionar produto]
```

### 36.4 Microcopy

O sistema deve usar linguagem simples:

- “Adicionar produto” em vez de “Cadastrar entidade”.
- “Publicar vitrine” em vez de “Ativar tenant”.
- “Enviar para WhatsApp” em vez de “Finalizar fluxo externo”.
- “Produtos mais vistos” em vez de “Ranking de eventos por entidade”.

### 36.5 Feedback visual

Toda ação deve dar retorno:

- Produto salvo.
- Loja publicada.
- Link copiado.
- Imagem enviada.
- Erro de validação.
- Promoção expirada.

## 38. Segurança, LGPD e privacidade

### 38.1 Princípios

- Coletar apenas dados necessários.
- Informar finalidade.
- Proteger credenciais.
- Criptografar dados sensíveis quando necessário.
- Separar dados por tenant.
- Registrar auditoria.
- Permitir exclusão ou anonimização quando aplicável.

### 38.2 Dados pessoais tratados

Possíveis dados:

- Nome do lojista.
- E-mail.
- Telefone.
- Endereço.
- Dados da loja.
- Nome/telefone de clientes interessados.
- Logs de acesso.

### 38.3 Medidas técnicas

- HTTPS obrigatório.
- Senhas com BCrypt/Argon2.
- JWT curto + refresh token.
- Rate limiting.
- Proteção contra brute force.
- CORS restrito.
- Validação de entrada.
- Sanitização.
- Headers de segurança.
- WAF futuro.
- Backups criptografados.

### 38.4 Isolamento multi-tenant

Testes obrigatórios:

- Token de uma loja não acessa outra.
- Endpoint privado exige storeId autorizado.
- Query sem tenantId deve ser proibida em services críticos.

### 38.5 Políticas necessárias

Antes de produção:

- Termos de uso.
- Política de privacidade.
- Política de cookies.
- Contrato de assinatura.
- Política de cancelamento.
- Política de conteúdo proibido.

## 39. Observabilidade

### 39.1 Logs

Logs estruturados com:

- request_id.
- user_id.
- store_id.
- endpoint.
- status.
- tempo.
- erro.

### 39.2 Métricas

- Latência por endpoint.
- Taxa de erro.
- Uso de CPU/memória.
- Queries lentas.
- Cache hit ratio.
- Uploads.
- Eventos de analytics.
- Webhooks recebidos.

### 39.3 Alertas

- API fora.
- Banco indisponível.
- Erro 500 elevado.
- Fila acumulando.
- Disco cheio.
- Certificado SSL próximo de expirar.
- Falha em backup.

### 39.4 Ferramentas

Início:

- Logs do servidor.
- Sentry.
- UptimeRobot.
- Grafana Cloud free ou similar.

Maduro:

- Prometheus.
- Grafana.
- Loki.
- OpenTelemetry.
- Jaeger/Tempo.

## 40. Testes

### 40.1 Testes unitários

Testar:

- Cálculo de preço.
- Cálculo de desconto.
- Validação de cupom.
- Limite de plano.
- Geração de mensagem WhatsApp.
- Slug.
- Regras de publicação.

### 40.2 Testes de integração

Testar:

- Repositories com PostgreSQL real via Testcontainers.
- Controllers.
- Autenticação.
- Upload.
- Criação de loja.
- Criação de produto.
- Pedido.

### 40.3 Testes E2E

Fluxos:

```text
Cadastrar lojista -> criar loja -> cadastrar produto -> publicar -> acessar vitrine -> adicionar item -> enviar WhatsApp
```

### 40.4 Testes de segurança

- Tenant isolation.
- Acesso sem token.
- Token expirado.
- SQL injection básica.
- XSS em descrição.
- Upload inválido.

### 40.5 Cobertura

Meta inicial:

- Domínio: 80%.
- Aplicação: 70%.
- Controllers críticos: integração.

## 41. CI/CD

### 41.1 Pipeline mínimo

```text
Pull request
  ↓
Lint
  ↓
Testes
  ↓
Build
  ↓
Docker build
  ↓
Deploy staging
```

### 41.2 Pipeline produção

```text
Merge main
  ↓
Rodar testes
  ↓
Criar imagem Docker
  ↓
Publicar registry
  ↓
Rodar migrations
  ↓
Deploy rolling
  ↓
Health check
  ↓
Notificar sucesso/erro
```

### 41.3 Regras

- Não fazer deploy se teste falhar.
- Não fazer deploy se migration falhar.
- Ter rollback.
- Separar staging e produção.
- Secrets fora do repositório.

## 42. Infraestrutura e deploy

### 42.1 Ambiente local

Docker Compose:

```text
web
api
postgres
redis
minio
mailhog
```

### 42.2 Ambiente staging

- Subdomínio staging.
- Banco separado.
- Storage separado.
- Dados fictícios.

### 42.3 Produção inicial de baixo custo

Opção simples:

```text
Vercel para Next.js
Render/Railway/Fly.io para API
PostgreSQL gerenciado
Cloudflare R2 para imagens
Cloudflare DNS/CDN
Sentry para erros
```

Opção VPS:

```text
VPS Ubuntu
Docker Compose
Nginx/Traefik
PostgreSQL gerenciado ou container com backup
Redis
Cloudflare
```

### 42.4 Produção madura

```text
Cloudflare DNS/CDN/WAF
Frontend em Vercel ou CloudFront/S3
API em ECS/Fargate ou Kubernetes
RDS PostgreSQL
ElastiCache Redis
S3/R2 para imagens
SQS/RabbitMQ gerenciado
Observabilidade completa
Backups automáticos
```

### 42.5 Backups

- Backup diário.
- Retenção mínima de 7 a 30 dias.
- Teste de restauração.
- Backup antes de migrations grandes.
