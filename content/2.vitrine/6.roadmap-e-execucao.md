## 44. MVP recomendado

O MVP ideal deve ser simples, mas encantador.

### 44.1 Escopo do MVP

- Landing page do Vitrine.
- Cadastro/login.
- Onboarding guiado.
- Criar loja.
- Escolher template.
- Personalizar logo/cor.
- Cadastrar categorias.
- Cadastrar produtos.
- Upload de imagens.
- Publicar vitrine.
- Vitrine pública por slug.
- Busca e filtro.
- Carrinho/lista de interesse.
- Gerar mensagem WhatsApp.
- Dashboard básico.
- Métricas de clique.

### 44.2 Fora do MVP

- Pagamento online para cliente final.
- Nota fiscal.
- Controle financeiro completo.
- Estoque avançado.
- Multiusuário.
- Domínio próprio.
- Marketplace.
- App mobile nativo.
- Integração com Instagram.
- Automação de mensagens.

### 44.3 Critério de sucesso do MVP

- 10 lojistas usando.
- 5 lojistas com vitrine publicada.
- 3 lojistas compartilhando link com clientes reais.
- Pelo menos 1 lojista disposto a pagar mensalidade.
- Clientes finais conseguindo enviar pedido sem explicação.

## 48. Riscos e decisões

### 48.1 Risco — virar ERP complexo

Mitigação:

- Manter MVP focado.
- Separar roadmap futuro.
- Não adicionar financeiro pesado cedo.

### 48.2 Risco — suporte alto

Mitigação:

- Onboarding claro.
- Templates controlados.
- IA para ajudar cadastro.
- Central de ajuda.

### 48.3 Risco — lojista não atualizar catálogo

Mitigação:

- Lembretes.
- Produtos com atualização rápida.
- Importação por planilha futura.
- Duplicar produto.
- IA para cadastro rápido.

### 48.4 Risco — custo de imagem

Mitigação:

- Compressão.
- Limites por plano.
- CDN.
- Storage barato.

### 48.5 Risco — baixa percepção de valor

Mitigação:

- Templates bonitos.
- Link profissional.
- Analytics.
- IA.
- Controle de custo/margem.

### 48.6 Risco — prometer venda

Mitigação:

- Comunicação honesta.
- Vender organização e presença digital.

## 52. Conclusão

O Vitrine deve nascer como uma ferramenta simples, honesta e extremamente útil para pequenos negócios.

A grande oportunidade não está em criar “mais uma loja virtual”. A oportunidade está em criar uma ponte entre o jeito que o pequeno empresário já vende hoje e uma presença digital mais organizada.

O produto deve respeitar a realidade do lojista:

- Ele usa WhatsApp.
- Ele usa Instagram.
- Ele tem pouco tempo.
- Ele não quer painel complicado.
- Ele não quer pagar caro.
- Ele precisa mostrar o que vende.
- Ele precisa parecer mais profissional.

Por isso, o Vitrine deve ser:

```text
Mais simples que um e-commerce.
Mais organizado que o Instagram.
Mais profissional que um PDF.
Mais acessível que um site sob medida.
Mais útil que apenas um link na bio.
```

A versão perfeita do Vitrine é aquela que ajuda o pequeno negócio a dizer:

> “Agora meus clientes conseguem ver o que eu vendo sem eu precisar explicar tudo toda hora.”

Essa é a essência.

Se essa essência for preservada, o Vitrine pode evoluir com pagamentos, notas fiscais, controle de custos, IA, analytics e gestão, sem perder o foco principal: facilitar a vida do lojista e melhorar a experiência do cliente.

# Anexo A — Histórias de usuário

## A.1 Lojista cria conta

Como lojista, quero criar uma conta para acessar o painel da minha loja.

Critérios de aceite:

- Deve validar e-mail.
- Deve validar senha mínima.
- Deve impedir e-mail duplicado.
- Deve redirecionar para onboarding.

## A.2 Lojista cria loja

Como lojista, quero criar uma loja para publicar minha vitrine.

Critérios de aceite:

- Deve exigir nome.
- Deve sugerir slug automaticamente.
- Deve validar slug único.
- Deve permitir informar WhatsApp.

## A.3 Lojista cadastra produto

Como lojista, quero cadastrar produto com foto, preço e descrição para aparecer na vitrine.

Critérios de aceite:

- Deve permitir salvar rascunho.
- Deve permitir publicar.
- Deve validar preço positivo, se informado.
- Deve permitir produto sob consulta, se loja permitir.

## A.4 Cliente busca produto

Como cliente, quero buscar um produto para encontrar rapidamente o que desejo.

Critérios de aceite:

- Deve buscar por nome.
- Deve ignorar acentos quando possível.
- Deve mostrar vazio amigável quando nada for encontrado.

## A.5 Cliente envia pedido para WhatsApp

Como cliente, quero enviar os itens escolhidos para a loja pelo WhatsApp.

Critérios de aceite:

- Deve gerar mensagem com itens.
- Deve incluir total estimado.
- Deve abrir o WhatsApp.
- Deve registrar evento de clique.

## A.6 Lojista vê métricas

Como lojista, quero ver produtos mais acessados para entender interesse dos clientes.

Critérios de aceite:

- Deve mostrar ranking.
- Deve permitir período.
- Deve diferenciar visualização de clique no WhatsApp.

## A.7 Lojista usa IA para descrição

Como lojista, quero gerar descrição com IA para cadastrar produtos mais rápido.

Critérios de aceite:

- Deve gerar sugestão.
- Deve permitir editar antes de salvar.
- Deve descontar crédito de IA, se houver limite.
- Não deve publicar automaticamente.

# Anexo B — Prompts internos de IA

## B.1 Gerar descrição de produto

Entrada:

```text
Produto: {nome}
Categoria: {categoria}
Características: {caracteristicas}
Tom da marca: {tom}
Público: {publico}
```

Saída esperada:

```json
{
  "shortDescription": "...",
  "description": "...",
  "tags": ["..."],
  "seoTitle": "...",
  "seoDescription": "..."
}
```

Regras:

- Não inventar características não informadas.
- Não prometer resultado inexistente.
- Usar linguagem clara.
- Manter texto comercial, mas honesto.

Entrada:

```text
Custo base: {cost}
Preço de venda: {price}
Desconto: {discount}

```

Saída:

- Custo total estimado.
- Lucro estimado.
- Margem estimada.
- Alerta de risco.
- Sugestão de revisão.

## B.3 Gerar campanha

Entrada:

```text
Objetivo: {objective}
Produtos: {products}
```

Saída:

- Nome da campanha.

# Anexo D — Decisões arquiteturais sugeridas

## ADR-001 — Usar monólito modular

Contexto:

O projeto precisa evoluir rápido e com baixo custo.

Decisão:

Usar monólito modular no backend.

Consequências positivas:

- Menos infraestrutura.
- Mais simples de desenvolver.
- Mais fácil testar.
- Menor custo inicial.

Consequências negativas:

- Exige disciplina modular.
- Pode crescer demais se não houver fronteiras.

## ADR-002 — Usar PostgreSQL

Contexto:

O sistema precisa de consistência, relatórios e multi-tenant.

Decisão:

Usar PostgreSQL.

Consequências positivas:

- Banco robusto.
- Suporte a JSONB.
- Índices fortes.
- Boa maturidade.

## ADR-003 — Usar WhatsApp como fechamento no MVP

Contexto:

Checkout completo aumenta escopo.

Decisão:

Usar WhatsApp para fechamento inicial.

Consequências positivas:

- MVP mais simples.
- Menos risco regulatório.
- Mais aderente ao comportamento dos lojistas.

Consequências negativas:

- Menos automação.
- Conversão final não controlada.
- Pedido não é pagamento confirmado.

## ADR-004 — Personalização limitada

Contexto:

Personalização livre aumenta suporte e prejudica qualidade visual.

Decisão:

Limitar personalização a logo, cores, banners, fontes e templates.

Consequências positivas:

- Produto mais bonito.
- Menos bugs.
- Menos suporte.
- Melhor performance.

# Anexo H — Glossário

**Lojista:** usuário dono da loja.  
**Loja:** tenant dentro da plataforma.  
**Vitrine:** site público da loja.  
**Catálogo:** conjunto de produtos e serviços.  
**Produto:** item físico vendido.  
**Serviço:** atividade vendida ou orçada.  
**Pedido:** intenção enviada pelo cliente.  
**Checkout:** pagamento online futuro.  
**Campanha:** ação promocional com produtos e textos.  
**Cupom:** código de desconto.  
**Template:** modelo visual da vitrine.  
**Slug:** identificador amigável na URL.  
**Tenant:** unidade isolada do SaaS, normalmente uma loja.

# Anexo I — Frase final de direção

Sempre que surgir dúvida sobre uma funcionalidade, pergunte:

> Isso ajuda o lojista a mostrar melhor o que vende, com menos esforço e mais clareza para o cliente?

Se a resposta for sim, a funcionalidade combina com o Vitrine.

Se a resposta for não, provavelmente é distração.
